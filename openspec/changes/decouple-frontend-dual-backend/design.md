## Context

当前仓库已经有一个统一前端壳，工作台页位于 `/`，平台认证与用户中心页位于 `/auth/*`、`/me/*`。其中平台接口已经通过前端 API 层显式指向 `web-backend`，但工作台 HTTP 与 WebSocket 仍大量依赖同源假设。结果是：

- `workstation-backend` 托管前端时，单前端双后端基本可成立
- 前端独立静态托管时，工作台请求与 WebSocket 会误打到前端来源
- `web-backend` 已部分为独立前端场景准备了跨域来源，`workstation-backend` 则仍默认只允许工作台/开发来源

这次变更是跨前端访问层、工作台长连接、后端跨域配置与部署模板的架构调整，适合先完成规划再进入实现。

## Goals / Non-Goals

**Goals:**

- 让单个前端应用可以在独立静态托管形态下，同时访问 `workstation-backend` 与 `web-backend`
- 让工作台 HTTP 和 WebSocket 都通过显式运行时配置解析目标地址
- 让部署模板可以注入前端运行时配置，而不是继续依赖同源或端口猜测
- 让用户侧继续以“一个包、一个启动入口、一个前端地址”的方式使用产品
- 保持现有单前端路由结构，不拆第二个前端应用
- 保持第一版安全边界清晰：`workstation-backend` 仅要求本机或 LAN 可达

**Non-Goals:**

- 不在第一版内支持“公网前端无附加代理地直连任意用户本机 workstation”
- 不重写认证体系为 token 模式
- 不拆分为两个独立前端项目
- 不顺带改造工作台业务流程或平台任务模型
- 不在本次变更内引入 Electron / Tauri 等桌面壳

## Decisions

### 1. 前端运行时引入显式的双后端地址配置

前端需要在运行时区分三类目标：

- `same-origin`：仅保留给兼容态或工作站托管前端
- `workstation`：工作台 HTTP / WS 入口
- `web`：认证、用户中心、平台任务入口

推荐通过前端启动时注入全局配置对象完成解析，例如：

- `__AGENT_THE_SPIRE_API_BASES__.workstation`
- `__AGENT_THE_SPIRE_API_BASES__.web`
- `__AGENT_THE_SPIRE_WS_BASES__.workstation`

选择这个方案而不是只靠端口推断，是因为独立前端部署下，真实地址可能是不同域名、不同协议、反向代理路径或容器服务名映射。端口推断只适合本地联调，不适合作为正式部署契约。

### 2. 工作台 WebSocket 从“浏览器来源派生”改为“从运行时配置派生”

现有工作台长流程全部建立在 `ws://${location.host}` 的假设之上。独立前端站点一旦脱离 `workstation-backend` 托管，这个假设立刻失效。设计上应新增统一的 WS 地址解析层，并让单资产、批量、日志分析、Mod 分析、构建部署等全部复用。

替代方案是继续在前端 Nginx 上统一反代 `/api` 和 `/ws`。我不建议把它作为第一原则，因为这样会把工作台与 Web 的路由分流逻辑硬塞回部署层，增加环境耦合，也不利于本地开发和桌面/局域网场景复用。

### 3. 后端跨域策略改为“显式允许独立前端来源”

`web-backend` 已经部分允许 `http://localhost:8080`，`workstation-backend` 默认没有。设计上应把两者都改为支持配置化的前端来源清单：

- `web-backend`：用于认证、用户中心、平台接口
- `workstation-backend`：用于工作台 HTTP 与 WebSocket 握手

这项调整的目标不是无限开放跨域，而是让独立前端来源变成一项明确配置，而不是硬编码在默认端口集合里。

### 4. 认证会话第一版继续基于 Cookie，但只支持同站或受控跨域部署

当前 `web-backend` 采用 Cookie 会话。第一版不改 token 体系，但要求部署边界满足以下前提之一：

- 前端与 `web-backend` 处于同站或受控子域
- 浏览器对当前 Cookie 策略可以稳定接受并回传

如果未来要支持更复杂的跨站前端部署，再单独立项评估 Cookie 策略升级或 token 化。这样可以避免本次变更把问题扩大成认证体系重构。

### 5. 独立前端部署模板采用“静态资源 + 运行时配置注入”，而非内建后端代理

独立前端模板只负责：

- 托管静态资源
- 注入运行时 endpoint 配置

不要求它内建工作台或 Web 的强耦合代理规则。这样更符合“前端独立部署”的目标，也方便同一前端镜像在不同环境下注入不同后端地址。

### 6. 用户发行形态采用“单安装包 + 两个本地进程”

第一版用户分发不拆成多个下载物，而是在同一个发行包内包含：

- 前端静态资源
- 一个本地静态前端服务
- `workstation-backend`
- 一个负责启动、健康检查、打开浏览器和清理进程的启动器

这样可以同时满足两类目标：

- 从架构上保持前端与 `workstation-backend` 解耦
- 从用户体验上维持“拿一个包、点一个入口即可使用”

替代方案是继续由 `workstation-backend` 托管前端。这个方案短期改动更小，但会削弱本次独立前端改造的价值，并在后续再次遇到前端独立托管与分流问题，因此不作为本 change 的推荐方向。

### 7. 实施顺序与提交节奏写入计划约束

本 change 的实现顺序固定为：

1. 前端 endpoint / WS 解析基础设施
2. 工作台 HTTP / WS 调用点切换
3. 前端运行时配置注入
4. 本地启动器与发行包结构
5. 后端跨域与会话边界收口
6. 文档与验收补齐

执行时采用“小任务完成即提交”的节奏。目标不是机械拆碎，而是让每个任务组形成可回溯、可验证、可独立回滚的提交边界。

## Risks / Trade-offs

- [工作台 WebSocket 仍存在遗漏调用点] → 通过统一 WS 地址解析层收口，并在任务清单里显式覆盖所有工作台 WS 封装
- [独立前端来源配置与实际部署地址不一致] → 前端在 endpoint 缺失时应显式报配置错误，而不是悄悄回退到错误来源
- [Cookie 会话在复杂跨站环境不稳定] → 第一版限制部署边界，只支持同站或受控子域；更复杂场景单独立项
- [部署文档与真实运行形态再次漂移] → 将部署模板、运行时配置约束和验收步骤一起写入 change artifacts
- [兼容态 `same-origin` 被新模型破坏] → 保留兼容态回退，仅在独立前端模式下强制显式目标地址
- [启动器编排过于复杂导致用户侧不稳定] → 第一版保持启动器职责极简，只负责端口检查、生成运行时配置、拉起两个本地进程和打开浏览器
- [“单包双进程”被误解成需要多个安装步骤] → 文档与发行目录明确只暴露一个主启动入口

## Migration Plan

1. 先补前端运行时 endpoint 解析能力与工作台 WS 解析能力。
2. 再把工作台 HTTP/WS 调用点从同源假设切到显式 `workstation` 目标。
3. 补前端静态部署模板与 `runtime-config.js` 一类的运行时配置注入方式。
4. 增加本地启动器，按“单安装包 + 两个本地进程”组织用户侧运行形态。
5. 调整 `workstation` / `web` 的跨域配置与会话边界文档。
6. 以“独立前端 + 本机/LAN workstation + web-backend”作为第一版验收形态。

回滚策略：

- 保留兼容态 `workstation-backend` 托管前端方式
- 运行时未配置独立 endpoint 时，仍允许兼容态同源运行

## Open Questions

- 前端运行时配置注入最终采用独立 `config.js`、模板渲染，还是容器启动脚本写入？
- 用户侧本地静态前端服务最终采用哪种最小依赖实现？
- `workstation-backend` 的第一版 LAN 可达边界是否需要额外限制允许网段或来源白名单？
- 是否要在 UI 中增加“工作台连接配置异常”提示页，以便独立前端部署时快速定位错误？
