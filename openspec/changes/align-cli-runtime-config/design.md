## Context

当前应用通过 `/api/config` 读取和保存仓库内或 release 内的 `config.json`，前端设置页展示的也是这份配置；但在调用 Claude CLI / Codex CLI 时，后端并未完全隔离用户主目录下的 CLI 全局配置。排查结果表明：

- Codex CLI 会继续读取 `~/.codex/config.toml`，其中的 `model_provider`、`base_url` 等配置会影响实际运行。
- Claude CLI 也存在用户级设置来源，且当前后端没有显式把应用中的 `llm.model` 传给 `claude` 命令，导致模型设置天然可能分叉。
- 当前设置接口返回的只是“保存值”，没有表达“生效值”和“外部覆盖来源”，因此用户只能从运行现象反推配置问题。

这次变更跨越后端配置、CLI 运行时封装、设置 API 与前端设置 UI，属于典型跨模块收口问题，适合先用 OpenSpec 固定边界，再进入实现。

## Goals / Non-Goals

**Goals:**
- 让工作站对 Claude CLI / Codex CLI 的运行配置以应用设置为主，不再被用户全局 CLI 配置静默覆盖。
- 让应用中的 `llm.model`、`base_url`、认证信息和关键运行参数在两个 CLI 上都具备明确、可追踪的传递规则。
- 让设置接口返回“保存配置 + 运行时生效摘要 + 外部覆盖来源”，前端能直接提示用户当前真实生效状态。
- 控制首轮改造范围：优先覆盖工作站设置与代码代理主链路，不扩展到所有文本 AI / 图像 AI 调用入口。

**Non-Goals:**
- 不在本轮重做整套 LLM 配置模型或引入多 profile 配置系统。
- 不要求本轮把所有第三方 CLI 的高级配置项都映射到应用设置。
- 不把用户主目录中的 CLI 全局配置完整镜像进应用设置页，只展示必要的覆盖摘要和风险提示。
- 不默认执行全量 build 或重型集成验证，只做定向最小验证。

## Decisions

### 1. 应用必须显式接管 Claude / Codex 的关键运行时配置

工作站在调用 `claude` 与 `codex` 时，必须显式传入或构造应用侧关心的关键运行配置：

- `model`
- `base_url`
- 认证信息
- 与执行模式强相关的关键参数

原因：
- 仅靠环境变量或默认 CLI 行为，无法保证应用设置一定胜过用户级全局配置。
- 当前 Claude CLI 的 `model` 根本没有从应用传递，属于确定性分叉。

替代方案是继续保留“应用设置只提供部分值，剩余让 CLI 自己补”，但这会让用户无法预测真实生效配置，因此不采纳。

### 2. 两条 CLI 链路都要最小化读取用户全局配置

对于 Codex CLI 与 Claude CLI，都应优先采用“受控设置来源”策略，避免无意读取用户主目录配置：

- Codex CLI：需要通过受控 home / config override / 最小配置注入等手段，阻断 `~/.codex/config.toml` 对应用运行的静默影响。
- Claude CLI：需要通过 `--model`、`--settings`、`--setting-sources`、`--bare` 等受控入口评估并固定设置来源。

原因：
- 这次问题本质是“配置所有权不清”，不是单个字段遗漏。
- 只修 `base_url` 或只修 `model` 会留下同类隐患。

### 3. 设置接口要区分“保存值”和“运行时生效值”

后端设置接口不应只返回 `config.json` 的脱敏内容，还应补充运行时摘要，例如：

- `saved_config`: 当前持久化配置
- `runtime_effective.llm`: 当前代码代理预计使用的生效值
- `runtime_effective.overrides`: 检测到的外部 CLI 设置来源或已被屏蔽的覆盖项
- `runtime_effective.warnings`: 需要在前端提示的风险信息

原因：
- 如果后端只做运行时隔离而前端不解释，用户仍然不知道过去为什么出现分叉。
- “保存值”和“生效值”拆开后，后续再扩展其他运行时覆盖也有稳定承载结构。

### 4. 前端设置页优先展示摘要，不直接暴露复杂内部结构

前端设置页应在现有输入项基础上增加运行时摘要区，而不是把所有底层覆盖信息展开成新的可编辑表单：

- 显示当前“保存值”
- 显示当前“代码代理生效值”
- 若检测到用户全局 CLI 配置与应用值不一致，明确提示“已隔离”或“仍存在外部覆盖风险”

原因：
- 用户真正关心的是“我现在改的值有没有真的生效”。
- 直接展示底层配置对象会增加理解负担，也容易把设置页做成调试面板。

## Risks / Trade-offs

- [兼容风险] 完全隔离 CLI 全局配置后，用户原本依赖全局配置提供的某些默认项可能丢失 -> 通过 design 中的运行时摘要和必要的兼容回退，确保关键字段可见且可排查。
- [范围膨胀] 若把所有 LLM 文本调用、图像调用和其他 Provider 一并纳入，会把问题扩大成配置系统重构 -> 首轮只覆盖代码代理主链路和设置页可观测性。
- [实现差异] Claude CLI 与 Codex CLI 支持的配置注入方式不同 -> 分别设计受控注入策略，但统一对外表现为“应用配置为准”。
- [发布差异] 仓库开发模式、workstation release、hybrid release 的配置路径不同 -> 统一从当前运行实例的配置来源生成摘要，避免前端再猜配置位置。

## Migration Plan

1. 在 OpenSpec 中先定义 CLI 运行时配置对齐与设置可观测性的 spec。
2. 后端实现 Claude / Codex 两条链路的受控配置注入与全局配置隔离策略。
3. 后端扩展设置接口，返回保存值与运行时生效摘要。
4. 前端设置页新增运行时摘要与告警提示，帮助用户辨别“保存值”和“实际生效值”。
5. 通过定向测试覆盖：
   - Codex CLI 不再吃用户全局配置
   - Claude CLI 的 `model`、`base_url` 等按应用值生效
   - 设置接口与设置页能正确展示摘要与提示
6. 更新相关文档，记录工作站设置保存位置、debug 打包沿用配置和 CLI 全局配置隔离策略。

回滚策略：
- 若某条 CLI 的受控配置注入导致不可用，可保留运行时摘要与提示，暂时回退到原有调用方式，但必须保留“存在全局配置干扰风险”的显式提示。

## Open Questions

- Claude CLI 在当前版本下，最稳妥的受控设置来源组合应选 `--bare + --settings`，还是 `--setting-sources` 加显式参数覆盖？
- Codex CLI 是否需要为工作站会话生成隔离的临时 `CODEX_HOME`，还是通过 `-c key=value` 已足够覆盖关键项？
- 设置接口的运行时摘要是否只需要服务于前端设置页，还是也要预留给后续诊断接口/日志使用？
