## Context

当前工作区页面已经具备统一入口，但还没有真正的“统一工作台”：

- `App.tsx` 只在内容区顶部提供 tab 按钮
- 四个功能页分别控制自己的最大宽度、外层卡片和视觉节奏
- 设置入口停留在顶栏按钮层，未纳入后台式导航模型
- 颜色体系仍以默认 `amber/slate` 为主，和用户提供的控制台参考图及配色不一致

这次变更属于前端跨模块样式与结构调整，适合先以 OpenSpec 锁定边界、顺序和验收标准，再进入实现。

## Goals / Non-Goals

**Goals:**

- 让工作区 `/` 下四个功能页共享同一套控制台壳层
- 把用户给定的深色侧栏 + 浅色卡片 + 紫蓝强调色方向落实为可复用的主题 token
- 在不改变业务流程的前提下，统一导航、页面标题、内容容器和主要交互按钮的视觉语言
- 保持当前 `tab` 查询参数模型，避免引入额外路由迁移风险
- 确保桌面端接近用户参考图，同时移动端仍然可用

**Non-Goals:**

- 不改 `/auth/*`、`/me/*`、`/me/jobs/:id` 的页面风格
- 不改后端接口、WebSocket、审批流、运行时配置或部署逻辑
- 不顺带重构业务组件状态管理
- 不在本轮引入新的 UI 库、设计系统框架或图标集
- 不追求像素级复刻用户参考图

## Decisions

### 1. 仅统一工作区壳层，不扩散到平台页面

本次只覆盖 `location.pathname === "/"` 的工作区页面，包含：

- `single`
- `batch`
- `edit`
- `log`

`/auth/*` 和 `/me/*` 继续保留现有样式。这样可以把风险集中在单一入口，避免因为全站换肤造成认证、用户中心和任务详情页回归面失控。

### 2. 以语义 token 而不是散落 utility class 实现新配色

新风格应优先落为工作区级主题 token，例如：

- `--workspace-bg`
- `--workspace-surface`
- `--workspace-surface-strong`
- `--workspace-nav-bg`
- `--workspace-accent`
- `--workspace-text-muted`

颜色来源固定为用户提供的 5 个色值，再允许派生少量透明度和浅色背景。这样能避免后续在各页面继续复制 `text-[#724A91]` 这类离散 class，降低换肤和维护成本。

### 3. 在 `App.tsx` 上方建立共享 `WorkspaceShell`

推荐把工作区结构提升为共享壳层，而不是逐页各自模仿后台布局。壳层至少包含：

- 左侧导航轨：用当前四个工作区 tab 作为一级导航
- 顶部栏：页面标题、说明、用户入口、设置入口
- 通知 / Banner 区：承接环境提示、连接提示或后续公告
- 主内容区：统一卡片间距、面板边框、背景和滚动规则

工作区各功能页只渲染自己的内容部分，不再各自决定整页布局。

### 4. 现有功能页改成“内容模块”，优先移除重复外壳

四个功能页都已有自己的外层卡片和留白。实施时应优先处理这些重复壳层：

- `single-asset`：保留左右双栏的工作流结构，但嵌入共享内容容器
- `batch-generation`：保留“左列表 + 右详情”的执行态结构，但输入态和计划态要与共享面板风格一致
- `mod-editor`：把现在的多段独立白卡收敛到统一 section 节奏
- `log-analysis`：去掉独立居中窄栏的工具页感觉，改为与工作台一致的页面层级

原则是只改表现层，不改业务状态机与流程判断。

### 5. 桌面优先贴近参考图，移动端采用“收起侧栏 + 顶部切换”退化策略

用户给的参考图明显是桌面后台控制台，因此桌面端作为第一视觉目标。移动端不强行保持同款侧栏，而采用更稳的退化：

- 大屏：固定左侧图标导航轨 + 顶栏 + 主内容
- 中屏：左侧导航轨缩窄，标题与操作压缩
- 小屏：导航轨切换为顶部横向 tab 或抽屉入口，内容面板改为单列堆叠

这样能保证实现成本可控，也避免在小屏上硬塞一个占宽过大的侧边栏。

### 6. 设置入口继续保留，但归入工作区壳层动作区

`SettingsPanel` 不在本轮重写结构，但触发入口要从“普通页头按钮”提升到工作区控制台动作区的一部分。这样可以统一“环境设置 / 用户入口 / 执行模式”等全局动作的视觉归属。

### 7. 实施顺序按“主题层 -> 共享壳 -> 四个功能页适配 -> 文档与验收”推进

推荐顺序：

1. 建立工作区 token、背景、surface 和基础交互态
2. 抽取 `WorkspaceShell`，在 `App.tsx` 上接管 `/` 路由外壳
3. 逐个适配 `single / batch / edit / log` 四个内容页
4. 补充文档和定向验收记录

这样可以把视觉基线先固定，再处理各页的兼容适配，降低返工。

## Risks / Trade-offs

- [布局回归] 各功能页当前都自带外层卡片，抽壳时容易出现双重边框、双重留白或滚动区域冲突
  - 缓解：把“页面壳层”和“功能内容卡片”拆开处理，逐页消除重复 surface
- [范围膨胀] 视觉统一很容易被继续扩展到 `/auth`、`/me`
  - 缓解：在 proposal、spec 和任务清单中显式锁定只改工作区 `/`
- [样式漂移] 新配色若只靠 utility class 临时拼装，后续很快失控
  - 缓解：先建 token，再做页面适配
- [交互可用性下降] 深色侧栏和新按钮体系可能影响现有按钮层级和可读性
  - 缓解：保留现有交互文案和信息层级，优先改色彩、容器和布局，不改行为路径
- [移动端退化不足] 参考图偏桌面后台，如果直接照搬，小屏会变差
  - 缓解：明确小屏采用顶部 tab / 抽屉退化，而不是强行保留完整侧栏

## Acceptance Criteria

- 工作区 `/` 的 `single / batch / edit / log` 四个 tab 在统一控制台壳层下展示
- 工作区主色不再以 `amber-*` 为主，而是以指定紫蓝色组为核心语义色
- 四个工作区页面切换后，导航、标题区、内容外边距和主要 surface 风格保持一致
- `/auth/*`、`/me/*` 页面视觉与结构不因本次变更被改坏
- 设置入口仍可从工作区访问，且位置与新壳层一致
- 小屏下页面仍可访问四个工作区功能，不出现主导航不可达问题

## Suggested Verification

轻量验证建议如下，默认不自动执行：

- 手工检查四个工作区 tab 的桌面端布局切换
- 手工检查窄屏宽度下导航是否可达
- 定向检查设置面板可打开、可关闭、保存态样式不被新壳层遮挡
- 如需自动化，优先补工作区壳层的最小渲染测试，而不是直接跑全量前端测试
