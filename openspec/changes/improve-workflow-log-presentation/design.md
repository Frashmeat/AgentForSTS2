## Context

当前前端各工作流对日志的处理都是“拿到一段字符串就追加到 `string[]` 再直接渲染”。例如单资产工作流使用 `genLog` / `agentLog`，批量规划使用 `agentLog`，构建部署与日志分析也各自有独立的 `log` 或事件流。这样虽然足以看见文本，但存在几个明显问题：

- 不同来源的日志没有统一的结构，前端无法稳定判断“这是一条原始输出”“这是一条阶段提示”“这是一条 stderr”。
- 当前实际调用的模型名没有统一承载位置，导致前端无法固定显示“现在到底在用哪个模型”。
- 一旦要支持“原始输出 / 优化后输出”双视图，只靠前端对纯字符串做猜测会非常脆弱。

这次改动横跨后端流式输出协议、前端状态组织和共享日志组件，是典型的跨模块体验收口问题，适合先用 OpenSpec 固定协议与交互，再进入实现。

## Goals / Non-Goals

**Goals:**
- 让整个工作流日志统一支持单面板双标签：默认显示“优化后输出”，可切换到“原始输出”。
- 让日志面板固定显示当前实际调用的模型名，并在模型切换时实时更新。
- 让前端保留完整原始输出，同时基于结构化元信息生成更易读的优化视图。
- 首轮覆盖单资产、批量规划、Mod 修改、构建部署、日志分析等工作流主链路，避免各页面继续各自维护日志展示规则。

**Non-Goals:**
- 不在本轮把用户中心历史任务详情页重构成相同的双视图日志系统。
- 不在本轮引入复杂的日志搜索、下载、过滤器或多列调试面板。
- 不用 AI 二次改写日志内容，优化视图只做基于规则的结构化整理与降噪。
- 不默认重做整个工作流页面布局，只聚焦日志面板和相关状态流。

## Decisions

### 1. 日志展示采用“原始缓冲 + 结构化条目 + 优化视图派生”模式

前端不再只保存 `string[]`，而是逐步转向统一的日志条目模型，例如：

- `raw_text`: 原始文本
- `channel`: `raw` / `stderr` / `system` / `stage`
- `source`: `workflow` / `agent` / `build` / `deploy` / `analysis`
- `model`: 可选模型名
- `timestamp` 或顺序索引

然后：
- `原始输出` 直接按条目顺序回放原始文本
- `优化后输出` 由前端基于这些条目做轻量整理与合并

这样做的原因：
- 可以最大程度保留原始信息
- 优化视图不需要修改底层真实日志
- 后续扩展“只看错误”“只看 agent”时也有稳定基础

### 2. 模型名由后端在流式链路中显式提供，不依赖前端猜测

Claude / Codex 相关输出在进入 workflow stream callback 时，需要带上当前实际模型名。非模型阶段不强制携带模型，但日志面板头部应保留最近一次有效模型名作为“当前模型”展示依据。

原因：
- 当前模型可能来自 CLI 默认值、设置项或运行时解析，前端无法仅凭文本稳定猜出。
- 把模型显示做成协议字段，比从日志正文里解析可靠得多。

替代方案是前端只显示设置中的 `llm.model`，但这会把“配置值”和“实际运行值”再次混淆，因此不采纳。

### 3. 优化视图只做轻量整理，不改写事实

优化视图的规则限定为：
- 合并连续重复阶段提示
- 对 stderr、失败、完成、关键动作使用更醒目的前缀或样式
- 对原始大段输出做按行切分与噪声折叠
- 保留原文，不做语义重写

原因：
- 用户仍然需要用优化视图定位真实问题，不能把事实改写成另一套文本。
- 规则越轻，越容易跨工作流复用。

### 4. 单面板双标签作为统一交互，不做多面板并排

日志展示统一为一个面板：
- 顶部固定显示当前模型名
- 顶部提供 `优化后输出` / `原始输出` 两个标签
- 默认落在 `优化后输出`

原因：
- 这是当前用户明确选择的交互方式
- 相比上下双栏或侧边抽屉，对现有页面侵入最小

### 5. 单资产 Code Agent 首轮把优化视图收口为“执行播报”

在不改变双标签结构的前提下，单资产 `Code Agent` 的 `优化后输出` 首轮进一步收口为 `执行播报` 布局：

- 顶部状态头显示 `当前模型`、模式切换和 `X / 6 已完成`
- 主区显示单段 `当前状态`
- 中区显示固定 6 步代码生成进度：
  - `理解任务`
  - `扫描项目`
  - `定位改动点`
  - `生成修改方案`
  - `写入代码`
  - `自检收尾`
- 下方显示 `执行摘要`
- 技术细节进入折叠区，默认弱化

这样做的原因：

- 用户当前更强的诉求是“不要一眼 log 感”，而不是继续增加原始流文本的整理规则
- 代码生成阶段比生图阶段更适合先引入固定进度与摘要说明
- 先在单资产 `Code Agent` 区域收口，可以避免把范围一次性扩展到所有工作流页面

## Risks / Trade-offs

- [兼容风险] 现有页面普遍依赖 `string[]` 日志，直接替换状态结构会波及多处 reducer 和恢复逻辑 -> 首轮可通过兼容适配层逐步迁移，避免一次性推翻全部状态。
- [协议分叉] 如果不同工作流各自定义自己的日志元信息，最终还是会回到分散状态 -> 必须先定义共享日志字段集合，再逐个接入。
- [信息丢失风险] 过早在后端做“优化输出”可能丢失原始信息 -> 原始输出必须始终完整保留，优化视图只从前端派生。
- [显示误导] 非模型阶段若仍显示上一个模型，用户可能误以为当前步骤正在调用该模型 -> UI 需明确语义为“当前工作流最近一次实际调用模型”，必要时在无模型阶段显示“当前阶段未调用模型”。

## Migration Plan

1. 先在 OpenSpec 中定义双视图日志和模型显示两个 capability。
2. 后端为 Claude / Codex 与主要工作流流式事件补统一元信息，至少能表达来源、通道、模型。
3. 前端新增共享日志条目模型与双视图派生逻辑。
4. 优先接入单资产、批量规划、Mod 修改、构建部署、日志分析五类工作流。
5. 为共享日志视图和关键 reducer / 适配层补定向测试。
6. 更新相关文档，记录双视图日志与模型显示的行为边界。

回滚策略：
- 若统一日志条目模型引发兼容问题，可先保留原始 `string[]` 渲染作为兜底，同时让新双视图只在一条工作流上灰度接入；但不得放弃元信息协议定义。

## Open Questions

- 共享日志条目模型应直接取代现有 `string[]`，还是先通过适配函数从旧结构派生？
- 模型名在非 agent 阶段显示“最近一次调用模型”还是显示“未调用模型”，哪个更符合用户预期？
