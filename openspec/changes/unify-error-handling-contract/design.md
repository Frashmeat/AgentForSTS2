## Context

当前后端错误输出分散在路由层、业务层和 WebSocket 事件里，HTTP 主要依赖 `HTTPException(detail=...)`，前端 HTTP 入口只尝试读取 `detail/message`，UI 层又大量直接使用 `Error.message` 或字符串化结果。这样虽然局部能工作，但一旦后端返回结构不一致、异常冒泡或 WebSocket 直接下发自定义错误对象，前端就会把原始 JSON 或底层文本暴露给用户。

这次变更是典型的跨层收口问题：同时涉及后端全局异常出口、HTTP 错误结构、WebSocket 错误事件格式、前端解析层和各页面展示策略。为了避免一次性重构过大，需要先把“错误契约”固定下来，再按入口逐步接入。

## Goals / Non-Goals

**Goals:**
- 定义一套前后端共享的统一错误字段，至少包含稳定的 `code`、用户可展示 `message`、可选 `detail` 和调试/关联字段。
- 让后端 HTTP 有全局异常出口，避免未捕获异常以不稳定格式直接返回。
- 让前端通过统一解析器消费错误，优先展示可读文案，拿不到结构化字段时仍有中文兜底。
- 让工作站 WebSocket 错误事件与 HTTP 错误字段集合尽量一致，降低页面侧分支判断复杂度。
- 控制首轮改造范围：优先收敛共享层和高频入口，不要求本轮把所有错误提示 UI 全部重做成统一组件。

**Non-Goals:**
- 不在本轮引入完整的国际化错误码体系。
- 不要求本轮把所有历史路由都改成同一业务异常类。
- 不把内部 traceback、堆栈或敏感实现细节直接暴露给终端用户。
- 不把所有 WebSocket 协议事件整体重构为新协议，只要求错误事件字段对齐。

## Decisions

### 1. 统一采用“错误 envelope + 兼容旧 detail”策略

后端新增统一错误 envelope，建议字段包含：
- `error.code`: 稳定机器码
- `error.message`: 面向用户的主展示文案
- `error.detail`: 可选补充信息
- `error.request_id`: 可选请求关联 id
- `error.meta`: 可选调试扩展槽位，仅保留安全内容

同时保留 HTTP 层对旧 `detail` 字符串的兼容，避免首轮改造把前端旧逻辑全部打断。

选择这个方案而不是“只继续用 `detail` 字符串”的原因：
- `detail` 无法承载稳定错误码和扩展字段
- WebSocket 事件也需要复用同一语义集合
- 前端后续要做统一展示与埋点时，需要结构化字段

### 2. HTTP 先通过全局异常处理器统一出口

后端在应用装配层注册：
- `HTTPException` handler
- 通用 `Exception` handler
- 必要时补充工作站运行时常见异常映射

理由：
- 能最快覆盖“接口直接返回原始 JSON/未捕获异常”的问题
- 不要求先修改所有路由实现
- 便于先建立稳定协议，再逐步把路由里的 `detail` 优化为更清晰的 `message`

替代方案是逐个路由手改返回结构，但会造成大量重复劳动，也难保证一致性。

### 3. WebSocket 不强制重写协议，只要求错误字段对齐

本轮不推翻现有 `event: error` 事件模式，而是要求错误 payload 内优先对齐：
- `message`
- `code`
- `detail`
- `request_id`

理由：
- 可以最小改动接回现有前端状态机
- 与 HTTP 共享字段集合，降低前端解析复杂度
- 避免把长流程状态协议也一起扩大范围

### 4. 前端新增统一错误解析器，页面侧只消费“可展示消息”

前端共享层应新增两级能力：
- 底层：把 HTTP/WS/未知异常统一解析成标准错误对象
- 上层：生成用户可展示消息，附带中文兜底

页面、controller、API 层应尽量不再自己判断 `detail/message/stringify`。

选择这个方案而不是“继续在各页面 individually 兜底”的原因：
- 当前分散兜底已经造成行为不一致
- 后续扩展 request id、错误码映射时会更容易

## Risks / Trade-offs

- [兼容风险] 旧页面可能仍假设 `Error.message` 是唯一来源 → 通过共享解析器继续产出兼容的 `Error.message` 或等价消息。
- [范围膨胀] 若把所有页面样式和所有 WebSocket 事件一起重做，变更会失控 → 本轮只统一共享层与高频入口，UI 风格统一留到后续。
- [信息泄露] 通用异常若直接输出原始异常内容，可能暴露内部实现 → 通用异常默认输出安全兜底文案，详细信息只进日志或受控字段。
- [迁移不完全] 首轮可能仍有少量页面未接入共享解析层 → tasks 中明确列出优先改造入口与定向验证范围。

## Migration Plan

1. 在 OpenSpec 中先建立统一错误契约 spec。
2. 后端新增错误 envelope 与全局异常 handler，并保持对旧 `detail` 兼容。
3. 前端更新 `requestJson` 与 `resolveErrorMessage`，让共享层可解析新旧两种格式。
4. 优先把 HTTP 高频入口和现有使用 `resolveErrorMessage` 的 controller 接到新解析层。
5. 让工作站 WebSocket `error` 事件按同一字段集合补齐。
6. 以定向测试验证 HTTP 结构化错误、未知异常、WebSocket 错误 payload 和前端兜底展示。

回滚策略：
- 前端若新解析层出现兼容问题，可先回退到旧的 `detail/message` 提取逻辑。
- 后端若全局 handler 引发意外，可暂时保留旧路由行为并仅保留 envelope 工具函数。

## Open Questions

- `request_id` 是否本轮只预留字段，不强制所有请求都生成？
- WebSocket 错误是否需要统一命名为 `error` 之外的更细粒度事件，还是本轮先保持现状？
- 用户中心、认证页这类平台页面是否需要额外的人性化文案映射，还是先统一共享兜底即可？
