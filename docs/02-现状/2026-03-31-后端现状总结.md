## 3. 当时 rollout 开关

以下为 2026-03-31 阶段记录。当前代码中 `config.migration` 会被标准化为空对象，路由挂载改由 `app_factory.create_app("workstation" | "web")` 的运行角色决定。

当时平台相关开关位于 `migration` 配置下：

- `platform_jobs_api_enabled`
- `platform_service_split_enabled`
- `platform_runner_enabled`
- `platform_events_v1_enabled`
- `platform_step_protocol_enabled`

当前挂载与行为：

- `platform_jobs_api_enabled`
  - 控制 `routers.platform_jobs`
- `platform_service_split_enabled`
  - 控制 `routers.platform_admin`
- `platform_runner_enabled`
  - 控制 `workflow.py` / `batch_workflow.py` 是否优先委托平台 compat service
- `platform_events_v1_enabled`
  - 当前主要是 rollout 占位
- `platform_step_protocol_enabled`
  - 当前主要是 rollout 占位

## 4. 当前最小闭环能力

### 4.1 单资产 workflow

已打通最小平台主链闭环：

- `action=start`
- `asset_type=custom_code`
- compat service 写入平台 `job/job_item/job_event`
- 通过 `ExecutionOrchestratorService + WorkflowRunner` 执行
- WebSocket 返回 `agent_stream` 与 `done`

### 4.2 批量 batch

已打通最小平台主链闭环：

- `action=start_with_plan`
- 所有 item 都是 `custom_code`
- `needs_image=false`
- compat service 写入平台主链并串行执行

### 4.3 仍保留 legacy 回退的范围

- 生图主链
- approval-first 路径
- 更复杂的 batch 依赖编排
- 未覆盖的平台事件 v1 独立消费路径

## 8. 当前已知边界与风险

- 平台 runner 路径目前只覆盖最小闭环，不代表所有旧入口都已完全迁移。
- `platform_events_v1_enabled` / `platform_step_protocol_enabled` 已入配置，但当前更多是 rollout 占位，不应误解为完整独立产品能力。
- `workflow.py` / `batch_workflow.py` 仍保留大量 legacy 逻辑，后续审查重点应放在：
  - flag 切换是否稳定
  - compat service 与 legacy 路径是否存在行为漂移
  - 容器装配是否把新主链依赖真正接齐
