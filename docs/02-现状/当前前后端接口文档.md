# 当前前后端接口文档

更新时间：2026-04-02

本文基于当前仓库实现整理，目标是给前端明确后端已暴露的 API 接口、WebSocket 协议、事件流和启用条件。本文覆盖“前端当前已使用接口”和“后端已经预留但需要开关启用的接口”。

## 1. 总览

### 1.1 基础约定

- 后端框架：FastAPI
- 主入口：`backend/main.py`
- 默认 API 前缀：`/api`
- 当前 CORS 白名单：
  - `http://localhost:5173`
  - `http://localhost:7860`
- 当前没有统一鉴权中间件
- 部分平台化接口通过查询参数 `user_id` 标识用户
- WebSocket 地址由前端按 `ws://{location.host}` 拼接

## 2. 通用约定

### 2.1 HTTP 返回风格

当前项目没有统一 REST 响应包裹层，返回风格分三类：

- 直接返回 JSON 对象
- 直接返回 JSON 数组
- 抛出 `HTTPException`
  - 常见格式：`{"detail": "..."}`，状态码如 `404`、`409`、`500`

少数旧接口在参数不合法时不会返回 4xx，而是返回 200 + 业务错误对象，例如：

```json
{
  "error": "..."
}
```

### 2.2 WebSocket 事件公共格式

所有 WebSocket 消息都是 JSON 文本。大多数消息带有 `event` 字段，例如：

```json
{
  "event": "progress",
  "message": "..."
}
```

公共事件：

| 事件名 | 说明 |
| --- | --- |
| `stage_update` | 阶段状态更新，通常用于页面顶部阶段条 |
| `error` | 全局错误 |
| `done` | 单流程完成 |

`stage_update` 通用结构：

```json
{
  "event": "stage_update",
  "scope": "project | text | image | agent | build",
  "stage": "内部阶段标识",
  "message": "给前端显示的阶段文案",
  "item_id": "可选，批量模式下用于标识具体 item"
}
```

### 6.2 客户端首包

基础首包：

```json
{
  "action": "start",
  "asset_type": "card",
  "asset_name": "DarkBlade",
  "description": "一把暗黑匕首，造成8点伤害",
  "project_root": "E:/STS2mod/MyMod"
}
```

`asset_type` 常见值：

- `card`
- `card_fullscreen`
- `relic`
- `power`
- `character`
- `custom_code`

可选字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `batch_size` | `number` | 覆盖当前配置中的生图批量数 |
| `implementation_notes` | `string` | `custom_code` 模式下传给代码代理 |
| `provided_image_path` | `string` | 使用本地图片，跳过生图 |
| `provided_image_b64` | `string` | 使用上传图片，跳过生图 |
| `provided_image_name` | `string` | 上传图片原始文件名 |

### 6.3 客户端后续消息

#### Prompt 确认

服务端发送 `prompt_preview` 后，客户端回复：

```json
{
  "action": "confirm",
  "prompt": "可选，用户修改后的 prompt",
  "negative_prompt": "可选，用户修改后的负向 prompt"
}
```

#### 选图

```json
{
  "action": "select",
  "index": 0
}
```

#### 再生成一张

```json
{
  "action": "generate_more",
  "prompt": "可选，新的 prompt",
  "negative_prompt": "可选，新的负向 prompt"
}
```

#### 审批模式放行

当 `llm.execution_mode = approval_first` 时，服务端会发 `approval_pending`，客户端需回复：

```json
{
  "action": "approve_all"
}
```

### 6.4 服务端事件

| 事件 | 结构 | 说明 |
| --- | --- | --- |
| `stage_update` | `scope/stage/message` | 阶段状态 |
| `progress` | `message` | 普通文本进度 |
| `prompt_preview` | `prompt/negative_prompt/fallback_warning?` | 提示词预览 |
| `image_ready` | `image/index/prompt` | 单张候选图已生成，`image` 为 base64 PNG |
| `approval_pending` | `summary/requests` | 审批模式下动作预览 |
| `agent_stream` | `chunk` | Code Agent 流式输出 |
| `done` | `success/image_paths?/agent_output?` | 完成 |
| `error` | `message/traceback?` | 失败 |

### 6.5 典型事件流

#### 普通单资产流程

1. 客户端连接并发送 `start`
2. 服务端多次发送 `stage_update`、`progress`
3. 服务端发送 `prompt_preview`
4. 客户端发送 `confirm`
5. 服务端循环发送：
   - `stage_update`
   - `progress`
   - `image_ready`
6. 客户端发送 `select` 或 `generate_more`
7. 选定后，服务端进入后处理和代码生成
8. 服务端发送若干 `agent_stream`
9. 服务端发送 `done`

#### 自定义图片流程

如果首包带 `provided_image_path` 或 `provided_image_b64`：

- 跳过 `prompt_preview`
- 跳过 `image_ready`
- 直接进入后处理和代码生成

#### 纯代码流程

如果 `asset_type = custom_code`：

- 跳过生图
- `image_paths` 为空数组

### 6.6 `done` 事件

典型结构：

```json
{
  "event": "done",
  "success": true,
  "image_paths": [
    "E:/STS2mod/MyMod/Resources/..."
  ],
  "agent_output": "Code Agent 最终输出"
}
```

| `name` | `string` | 英文名 |
| `name_zhs` | `string` | 中文名 |
| `description` | `string` | 功能描述 |
| `implementation_notes` | `string` | 给 Code Agent 的技术说明 |
| `needs_image` | `boolean` | 是否需要图片 |
| `image_description` | `string` | 生图描述 |
| `depends_on` | `string[]` | 依赖的 item id |
| `provided_image_b64` | `string?` | 前端上传图片时携带的 base64 |

### 7.4 服务端事件

| 事件 | 结构 | 说明 |
| --- | --- | --- |
| `planning` | 无额外字段 | 开始规划 |
| `plan_ready` | `plan` | 规划完成，等待前端确认 |
| `stage_update` | `scope/stage/message/item_id?` | 全局或单 item 阶段更新 |
| `batch_progress` | `message` | 批量流程全局进度 |
| `batch_started` | `items` | 已开始执行，附 item 列表 |
| `item_started` | `item_id/name/type` | 某个 item 开始处理 |
| `item_progress` | `item_id/message` | 某 item 进度 |
| `item_image_ready` | `item_id/image/index/prompt` | 某 item 出图完成，等待选图 |
| `item_agent_stream` | `item_id/chunk` | 某 item 的代码代理流式输出 |
| `item_approval_pending` | `item_id/summary/requests` | 某 item 或某组资产进入审批 |
| `item_done` | `item_id/success` | 某 item 完成 |
| `item_error` | `item_id/message/traceback?` | 某 item 失败 |
| `batch_done` | `success_count/error_count` | 全部完成 |
| `error` | `message/traceback?` | 全局失败 |

### 7.5 执行特性

- 图像生成支持有限并发，实际并发受 `image_gen.concurrency` 控制
- 代码生成在旧实现下仍有串行锁
- 按依赖关系分组后，支持组内合并生成
- 审批模式下，代码生成前可能停在 `item_approval_pending`
- 非审批模式下，所有 item 完成后会做一次统一 `build_and_fix`

## 14. 已知注意事项

- 当前很多旧接口直接使用 `dict` 作为请求体，没有严格的 OpenAPI schema，前端联调时应以本文档和具体实现为准。
- WebSocket 协议是项目自定义事件流，不是标准订阅模型；前端要按事件名驱动状态机。
- `/api/platform/*` 和 `/api/admin/*` 虽已实现，但默认不开启，前端不要假定线上必然存在。
- `/api/plan` 在需求为空时返回 200 + `{"error": ...}`，不是标准 4xx。
- 当前前端组件层已经不再直接 `fetch("/api/...")`，HTTP 调用统一经 `frontend/src/shared/api/*` 进入。
- 当前前端消费层已进一步补充共享 helper：`runApprovalAction()`、`useDefaultProjectRoot()`、`createBatchPlanningController()`、`createBuildDeployController()`、`resolveErrorMessage()`，用于减少页面重复逻辑。
- 审批对象后端字段比当前前端 TypeScript 类型更多，前端如果要完整展示，可补充：
  - `source_backend`
  - `source_workflow`
  - `created_at`
  - `result`
  - `error`
