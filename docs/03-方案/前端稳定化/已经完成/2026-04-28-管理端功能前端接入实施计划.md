# 管理端功能前端接入实施计划

> 文档定位：本文是“后端已暴露的新管理端能力全部接入前端”的实施计划，负责明确范围、页面结构、接口接线、测试与文档同步要求。
>
> 事实依据：基于当前代码中的 `backend/routers/platform_admin.py`、`backend/routers/platform_jobs.py`、`frontend/src/shared/api/*`、`frontend/src/App.tsx`、`docs/02-现状/当前前后端接口文档.md` 盘点。
>
> 权威入口：上级入口为 `docs/03-方案/前端稳定化/README.md`；接口现状以 `docs/02-现状/当前前后端接口文档.md` 为准。

> 最后更新：2026-04-28

## 当前状态

已完成。

## 目标

把当前后端已经暴露、但前端尚未完整接入的新功能全部接到前端，重点补齐管理员服务器凭据管理、管理员执行记录、管理员退款记录、管理员执行配置列表和生图配置测试入口。

## 不纳入范围

- 不为旧兼容 `/api/platform/jobs*`、`/api/platform/quota` 的显式 `user_id` 链路新增页面入口。
- 不保留新的“兼容旧平台任务管理 UI”。
- 不重构服务器模式 runner、凭据加密、健康检查后端实现。
- 不在本轮新增复杂权限体系；沿用后端已有管理员鉴权。
- 不在计划落地阶段写入真实服务器凭据或修改生产数据库。

## 当前理解

当前用户侧主流程基本已接：

- 登录、注册、邮箱验证、找回密码、重置密码。
- 用户中心、任务列表、任务详情、任务事件流。
- 服务器模式执行配置读取、用户默认服务器配置保存。
- 服务器模式任务创建前的上传资产和服务器工作区预分配。
- queue worker 诊断已调整为 web runtime 入口。

主要缺口集中在管理端：

- 后端已有服务器凭据管理接口，但前端没有 API 封装和页面。
- 后端已有管理员执行配置列表，但前端没有 API 封装和页面。
- 后端已有管理员执行记录、执行详情、退款记录接口，前端已有 API 封装但没有页面消费。
- 后端已有 `/api/config/test_imggen`，前端没有设置页测试入口。

## 风险

1. 服务器凭据涉及敏感字段，前端不得回显原始 `credential / secret`。
2. 编辑凭据时，空字符串在后端语义中表示“保留原密文”，前端必须明确表达，避免用户误以为已经清空。
3. `auth_type = ak_sk` 当前健康检查会返回 `degraded / unsupported_auth_type`，前端不能把它展示成系统异常。
4. 管理端页面需要区分“未登录”和“非管理员”，错误提示应复用现有友好化文案。
5. 当前项目没有正式前端测试 runner 脚本，局部验证需要使用 `node --test` 与 `npx tsc --noEmit -p frontend/tsconfig.json`。

## 验收标准

- 管理员可以查看服务器执行配置列表。
- 管理员可以新增、编辑、启用、禁用服务器凭据。
- 管理员可以对服务器凭据执行手动健康检查，并看到 `healthy / degraded / auth_failed / rate_limited / quota_exhausted / disabled` 等状态。
- 当某个 profile 下存在 `enabled = true` 且 `health_status = healthy` 的凭据后，普通用户设置页对应服务器配置可选。
- 管理员可以按 `job_id` 查看执行记录，并查看单次执行详情。
- 管理员可以查看退款记录，并按 `user_id` 筛选。
- 设置页图像配置区可以触发 `/api/config/test_imggen`，并展示成功或失败提示。
- 前端新增功能不新增旧 `/api/platform/jobs* + user_id` 页面入口。
- 接口文档和前端稳定化索引同步更新。

## 设计概览

- `/admin/runtime-audit` 页面顶部 actions 增加：
  - `凭据管理`
  - `执行记录`
  - `退款记录`
- `App.tsx` 新增路由：
  - `/admin/server-credentials`
  - `/admin/executions`
  - `/admin/refunds`

后续如果管理员页面继续增加，再抽统一 `AdminNav`。

`frontend/src/shared/api/config.ts` 新增：

- `testImageGenerationConfig()`

### 服务器凭据页面

页面职责：

- 读取管理员执行配置列表。
- 读取凭据列表，可按 profile 筛选。
- 展示凭据元数据，不展示原始密钥。
- 提供新增/编辑表单。
- 提供启用、禁用、健康检查按钮。

表单字段：

- `execution_profile_id`
- `provider`: `openai | anthropic`
- `auth_type`: `api_key | ak_sk`
- `credential`
- `secret`
- `base_url`
- `label`
- `priority`
- `enabled`

编辑语义：

- `credential` 留空表示保留原值。
- `secret` 留空表示保留原值或忽略，具体遵循后端现有契约。
- 成功后刷新列表。

### 执行记录页面

页面职责：

- 输入 `job_id`。
- 调用 `listAdminJobExecutions(jobId)`。
- 列表展示 `id / job_id / job_item_id / status / provider / model`。
- 点击执行记录后调用 `getAdminExecution(executionId)` 并展示详情。

### 退款记录页面

页面职责：

- 可选输入 `user_id`。
- 调用 `listAdminQuotaRefunds(userId?)`。
- 展示 `ai_execution_id / charge_status / refund_reason`。

### 设置页生图测试入口

页面职责：

- 在图像配置区增加“测试生图配置”按钮。
- 点击后调用 `testImageGenerationConfig()`。
- 成功展示 `ok / size`。
- 失败走 `resolveErrorMessage()` 和现有 `StatusNotice`。

## 建议执行顺序

推荐先做 Task 1、Task 3。

原因：

- 服务器凭据管理是当前服务器 profile 不可选的直接前端缺口。
- 接完后可以立刻通过管理员前端录入凭据，再观察普通用户设置页 profile 可用性变化。
- 执行记录和退款记录虽然重要，但不会阻断服务器模式可用性。

## 多 Agent 说明

该任务可以拆成多个独立子任务并行处理，例如凭据页、执行记录页、退款页、设置页测试按钮。但按照本项目规则，多 Agent 默认关闭；除非用户明确同意，否则执行时使用单 Agent 顺序推进。
