# 阶段 1：工作流会话层实施计划

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 收敛前端 WebSocket 会话管理，优先把直接 `new WebSocket(...)` 的链路迁回统一 session/facade 体系，降低连接生命周期和错误处理分散带来的不稳定性。

**Architecture:** 保留现有 `WorkflowClient -> WorkflowSocketFacade` 分层，在此基础上补一个可复用的“端点会话封装层”。不同业务链路只声明自己的路径和事件类型，页面组件只订阅事件和发送命令，不直接处理原始 `WebSocket` 生命周期。

**Tech Stack:** React 18、TypeScript、Node `node:test`、现有 `frontend/src/shared/ws/*` 体系

## Chunk 2：统一 session/facade 抽象

### 目标

让不同 WebSocket 页面都通过统一 facade 接入，而不是各自手写 `new WebSocket`、`onopen`、`onmessage`、`onerror`。

### 设计要求

- [x] `WorkflowClient` 继续作为最底层 JSON WebSocket 客户端
- [x] `WorkflowSocketFacade` 继续负责 typed event 订阅
- [x] 新的业务 socket 封装只负责：
  - 指定路径
  - 限定事件类型
  - 暴露 `on / send / waitOpen / close`
- [x] 需要持续错误处理的链路，统一通过 facade 暴露，而不是页面自己绑 `onerror`

### 验收标准

- [x] `BuildDeploy.tsx` 不再直接 `new WebSocket`
- [x] `log-analysis/view.tsx` 不再直接 `new WebSocket`
- [x] `mod-editor/view.tsx` 的分析链路不再直接 `new WebSocket`

## 风险

- 页面迁移时容易顺手改动状态逻辑，导致范围失控
- 如果把不同业务事件强行并成一个联合类型，会降低可读性
- `mod-editor` 同时包含“分析链路”和“修改链路”，本轮只动分析链路，不能误伤修改链路

## 完成标准

- 三处直接原生 WebSocket 使用被替换

- 未引入额外状态管理重构
