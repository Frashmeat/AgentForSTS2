# 设置面板异步退出安全与任务恢复实施计划

## 1. 背景

设置界面存在多类异步操作：配置加载、自动保存、路径自动检测、知识库检查与刷新、服务器偏好保存、队列 worker 诊断刷新等。当前部分异步链路已在 `useEffect` 内做了局部 `cancelled` 保护，但事件处理函数和部分 Promise 回调仍可能在设置面板关闭或卸载后继续回写组件状态。

用户已经确认本次目标不是只做“关闭后不报错”的最小止血，而是同时覆盖：

1. 退出设置界面时的卸载安全止血。
2. 关闭设置界面前尽量保存当前修改。
3. 长任务关闭后可恢复/可见。

## 2. 当前理解

### 2.1 主要风险点

`frontend/src/components/SettingsPanel.tsx` 中需要重点检查以下链路：

- 初始加载：`loadAppConfig`、`loadKnowledgeStatus`。
- 工作区配置自动保存：`save`、自动保存定时器。
- 路径相关：`detectPaths`、路径检测轮询、`cancelDetectPaths`、`choosePath`。
- 知识库相关：`handleCheckKnowledge`、`handleRefreshKnowledge`、知识库刷新轮询。
- 服务器配置：服务器资料加载、`handleSaveServerPreference`、服务器偏好自动保存定时器。
- 诊断刷新：`refreshQueueWorkerStatus`。
- 提示清理：保存提示、路径提示、知识库提示等延迟清理定时器。

### 2.2 行为边界

- 关闭设置面板不等于取消后端长任务。
- 后端长任务可以继续执行，但已卸载的设置面板不能再接收状态回写。
- 用户正在编辑的配置应在关闭前尽量保存，避免“刚改完就关掉导致未保存”。
- 路径检测、知识库刷新等长任务在重新打开设置面板后应尽量恢复进度展示。

## 3. 目标

### 3.1 必须实现

- 组件卸载后，任何异步回调不得继续调用该组件的 `setState`。
- 设置面板关闭入口统一收口，关闭前尝试保存脏配置。
- 路径检测和知识库刷新任务在关闭设置面板后不被误取消。
- 重新打开设置面板时，能够恢复仍在运行的长任务进度，或者明确显示最近任务结果/失败信息。
- 更新相关实施文档和基线说明。

### 3.2 暂不扩大

- 不把设置面板所有状态整体迁入全局状态管理。
- 不为所有 API 一次性改造 `AbortController`，除非调查发现现有 API 封装已经天然支持。
- 不改变后端任务状态机语义。
- 不改变路径检测、知识库刷新、服务器偏好保存的业务接口返回结构，除非长任务恢复必须补齐查询入口。

## 4. 方案分层

### 4.1 第一层：卸载安全止血

在 `SettingsPanel` 内建立统一的挂载状态保护：

- 组件挂载时标记 `mountedRef.current = true`。
- 组件卸载时标记 `mountedRef.current = false`。
- 所有 `await` 后、`.then/.catch/.finally` 回调中，在状态回写前检查 `mountedRef.current`。
- 轮询逻辑继续保留局部 `cancelled` 标记，同时叠加组件级 mounted 判断。
- 定时器在 cleanup 中清理；定时器回调执行前也不依赖已卸载组件。

验收重点：关闭设置面板后，不出现卸载组件状态更新、悬挂轮询继续更新 UI、父组件回调被过期任务触发等问题。

### 4.2 第二层：关闭前保存

将设置面板关闭入口统一收口到 `handleCloseSettings`：

- 抽象出“关闭前 flush”流程。
- 如果工作区配置存在 `configDirty`，先调用当前保存逻辑。
- 如果服务器偏好存在 `serverSelectionDirty`，先调用服务器偏好保存逻辑。
- 保存过程中避免重复触发已有自动保存定时器。
- 保存成功后关闭设置面板。
- 保存失败时优先保留设置面板并展示错误，避免用户误以为已经保存。

需要注意：如果用户点击遮罩关闭和点击右上角关闭，两者必须走同一入口。

### 4.3 第三层：长任务恢复/可见

长任务分为两类处理：

#### 路径检测任务

优先调查后端是否已有“当前/最近路径检测任务”查询能力：

- 如果已有：设置面板挂载时查询活跃任务并恢复 `detectionTaskId` 与轮询。
- 如果没有：第一版将 `detectionTaskId` 提升到设置面板外层或模块级轻量状态，保证关闭再打开仍能恢复同一前端会话中的任务。

#### 知识库刷新任务

优先调查后端是否已有“当前/最近知识库刷新任务”查询能力：

- 如果已有：设置面板挂载时查询活跃任务并恢复 `knowledgeTaskId` 与轮询。
- 如果没有：第一版将 `knowledgeTaskId` 提升到设置面板外层或模块级轻量状态，保证同一前端会话内恢复。

#### 可见性原则

- 设置面板关闭期间不强行弹全局浮层，避免扩大 UI 范围。
- 重新打开设置面板后恢复浮动状态提示和任务进度。
- 如果任务已经结束，展示最近结果或刷新后的最终状态。

## 5. 推荐实施顺序

1. 调查现有 API 与后端任务存储能力，确认是否支持按“当前活跃任务”恢复。
2. 先落地组件卸载安全止血，覆盖所有异步回写点。
3. 收口关闭入口，实现关闭前保存。
4. 实现长任务恢复：优先复用后端查询能力；没有后端能力时先做同一前端会话内恢复。
5. 更新相关文档与任务状态。
6. 执行最小验证：`npx tsc --noEmit`。
7. 如用户同意，再执行完整前端 build 或 release 打包验证。

## 7. 风险与约束

- 如果关闭前保存失败，建议阻止关闭并展示错误；否则用户可能误判为已保存。
- 如果采用模块级任务 ID 缓存，只能覆盖同一前端会话，页面刷新后仍会丢失任务恢复能力。
- 如果要支持页面刷新后的任务恢复，必须依赖后端活跃任务查询或持久化任务索引。
- 不应把“关闭设置面板”自动解释为“取消检测/取消刷新”。取消任务仍应由用户显式点击取消入口触发。
- 避免在本轮引入大规模状态管理重构。

## 8. 验收标准

- 设置面板关闭后，未完成异步请求返回不会再更新已卸载组件状态。
- 设置面板关闭前，工作区配置和服务器默认配置的脏数据会被保存或明确报错阻止关闭。
- 路径检测进行中关闭设置面板，再打开后能继续看到检测进度或最终结果。
- 知识库刷新进行中关闭设置面板，再打开后能继续看到刷新进度或最终状态。
- 面板打开期间原有交互行为保持不变。
- `npx tsc --noEmit` 通过。
- 文档已同步更新。

## 9. 待确认问题

- 保存失败时是否一定阻止关闭？当前建议：阻止关闭并展示错误。
- 长任务恢复是否要求跨页面刷新？当前建议：先支持同一前端会话；如后端已有活跃任务查询能力，则顺手支持刷新后恢复。
- 是否需要设置面板关闭期间显示全局任务提示？当前建议：暂不做，避免扩大 UI 范围。
