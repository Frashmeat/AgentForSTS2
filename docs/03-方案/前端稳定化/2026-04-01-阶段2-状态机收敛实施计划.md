# 阶段 2：状态机收敛实施计划

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把单资产和批量流程中的分散 `setState` 收敛为可测试的 reducer / 状态变更函数，让页面状态流可预测，并降低协议事件变更时的改动面。

**Architecture:** 优先抽取“纯状态层”，不先动 UI 结构。第一批先做单资产 reducer，把 `App.tsx` 中与 workflow 直接相关的状态抽离成一份纯函数 reducer 和 action 集；第二批再对批量模式的 item/global 状态做同样处理。

**Tech Stack:** React 18、TypeScript、Node `node:test`

## 风险

- 如果把表单输入和 workflow 运行态混进一个 reducer，会让 reducer 过大
- 如果同时动单资产和批量模式，回归面会太大
- 如果 reducer 设计过细，action 会膨胀；过粗，又会失去状态机意义

## 完成标准

- 单资产 workflow 有独立 reducer 和测试
- `App.tsx` workflow 状态切换主要通过 dispatch
- 为批量模式下一轮 reducer 收敛做好边界准备
