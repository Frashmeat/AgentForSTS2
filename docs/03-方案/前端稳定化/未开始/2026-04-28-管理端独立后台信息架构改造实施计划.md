# 管理端独立后台信息架构改造实施计划

> 文档定位：本文是“管理员相关功能从设置页与零散入口中收敛为独立管理台”的实施计划，负责明确页面结构、迁移范围、展示语义、测试与验收标准。

## 当前状态

未开始。

## 当前理解

当前管理员功能已经能用，但信息架构比较松散：

- 管理端入口散落在设置页服务器模式区、运行审计页 actions、独立管理页之间。
- `/admin/runtime-audit` 承担了“审计页”和“管理入口导航”两类职责。
- 管理页面之间没有统一布局、统一导航、统一详情区模式。
- 若直接展示后端原生字段，例如 `job_id`、`credential_ref`、`runtime.queue_worker.*`、`auth_failed`，管理人员阅读成本偏高。
- 设置页应该只保留普通用户视角的配置，不承载管理员维护动作。

因此本轮目标不是继续堆新功能，而是把已有管理员能力整理成高信息密度、低解释文案、业务语言友好的独立后台。

## 目标

- 新增独立管理端壳层，把管理员功能统一收敛到 `/admin` 下。
- 设置页不再承载管理员功能，只保留必要的“服务器模式可用性”用户视角信息。
- 管理端页面采用高信息密度布局：指标、筛选、表格、详情、操作优先，减少描述性文案。
- 展示层使用中文业务字段，不直接暴露后端原生字段名。
- 保留当前已接的管理能力：运行状态、执行记录、审计事件、执行配置、服务器凭据、健康检查、退款记录。

## 不纳入范围

- 不新增后端接口。
- 不修改凭据加密、健康检查、退款、queue worker 的后端业务逻辑。
- 不在本轮实现完整用户管理或额度调整后端能力；`用户与额度` 可先作为管理端扩展位或只展示已有可用数据。
- 不重新设计普通用户工作站主流程。
- 不新增旧 `/api/platform/* + user_id` 管理入口。

## 目标路由结构

```text
/admin
/admin/runtime
/admin/executions
/admin/audit
/admin/execution-profiles
/admin/server-credentials
/admin/credential-health
/admin/refunds
/admin/users
```

说明：

- `/admin`：管理台首页，高密度概览，不放长说明。
- `/admin/runtime`：运行状态、queue worker、队列积压。
- `/admin/executions`：任务执行记录与执行详情。
- `/admin/audit`：审计事件流。
- `/admin/execution-profiles`：服务器执行配置。
- `/admin/server-credentials`：服务器凭据管理。
- `/admin/credential-health`：凭据健康检查汇总。
- `/admin/refunds`：退款记录。
- `/admin/users`：用户与额度扩展位；第一阶段可静态占位或只接已有用户资料能力，具体实现前再确认。

## 展示语义规则

管理端展示层不得直接把后端原生字段名当作主要 UI 文案。

| 后端/内部字段 | UI 展示 |
| --- | --- |
| `job_id` | 任务编号 |
| `execution_id` | 执行编号 |
| `credential_ref` | 使用凭据 |
| `idempotency_key` | 请求标识 |
| `provider` | 服务商 |
| `auth_type` | 认证方式 |
| `api_key` | 密钥 |
| `ak_sk` | 访问密钥 |
| `healthy` | 健康 |
| `degraded` | 需复检 |
| `disabled` | 已停用 |
| `auth_failed` | 认证失败 |
| `rate_limited` | 调用限流 |
| `quota_exhausted` | 额度耗尽 |
| `succeeded` | 已成功 |
| `failed` | 已失败 |
| `retrying` | 重试中 |
| `runtime.queue_worker.leader_renewed` | 调度权续租 |
| `runtime.queue_worker.job_claimed` | 领取任务 |
| `runtime.queue_worker.workspace_locked` | 工作区占用 |
| `quota.refunded` | 额度返还 |

必要排障字段可以放在详情区的“技术信息”折叠区域，不能作为列表主列或页面主标题。

## 设计原则

- 管理界面优先展示事实数据：指标、表格、筛选、状态、操作。
- 删除解释型卡片和大段说明文案。
- 页面首屏应尽量呈现可操作信息，而不是介绍功能。
- 复用工作站视觉风格：深紫侧栏、淡蓝灰背景、半透明白面板、紫色渐变主按钮。
- 管理端卡片允许比工作站更紧凑；不要使用营销式大 hero。
- 列表页统一“左列表 / 右详情”或“上筛选 / 下表格 / 右详情”的信息结构。
- 敏感凭据只允许写入，不允许回显。

## 风险

1. 路由迁移可能影响已有 `/admin/runtime-audit` 链接，需要决定是否保留跳转兼容。
2. 设置页移除管理员入口后，管理员需要明确从 `/admin` 进入管理台，不能丢失可发现性。
3. 展示语义转换如果散落在页面中，后续容易重新出现原生字段名，建议集中到 `adminDisplay.ts`。
4. `用户与额度` 当前后端能力可能不足，第一阶段不要承诺完整用户管理。
5. 管理页视觉更高密度后，需要注意移动端可用性，至少保证表格横向滚动和操作不重叠。

## 验收标准

- 管理员可以从 `/admin` 进入统一管理台。
- 设置页不再展示管理员维护型入口；普通用户设置页只显示自身服务器模式配置和可用性。
- 管理端侧边导航覆盖运行状态、执行记录、审计事件、执行配置、服务器凭据、健康检查、退款记录、用户与额度。
- 管理端首页不出现长段描述性文案，首屏包含指标、待处理事项、服务器能力矩阵和最近执行/退款。
- 管理端列表主列不直接展示 `job_id`、`credential_ref`、`runtime.queue_worker.*`、`auth_failed` 等后端原生字段名。
- 凭据页不回显原始密钥。
- 已有管理端 API 调用不退回旧 `/api/platform/* + user_id` 入口。
- 局部测试和 TypeScript 验证通过。

## 建议讨论点

1. `/admin/runtime-audit` 是否保留为重定向到 `/admin/audit`，还是直接删除旧路径？
2. `用户与额度` 第一阶段是否只做占位页，还是需要同步补后端用户列表接口？
3. 设置页是否完全移除管理员入口，还是在用户菜单/顶部导航保留一个“管理台”入口？
4. 管理端首页指标是否需要完全来自真实接口，还是允许第一阶段先用已有接口 + 局部占位？
5. 技术字段折叠区是否需要默认显示给超级管理员，还是默认收起？
