# 前端稳定化四阶段计划

更新时间：2026-04-02

## 1. 目标

当前前端已经具备可用功能，但“接口调用层、工作流状态层、页面展示层”之间仍有较强耦合。目标不是立刻重做界面，而是在不大改产品行为的前提下，把现有前端做稳，降低以下问题的发生概率：

- WebSocket 断线或异常后页面状态错乱
- 单资产和批量流程状态分散、难以维护
- 后端协议调整后，需要在多个页面重复改动
- 审批、重试、刷新恢复等中断场景不稳定

## 2. 总体顺序

建议按下面顺序推进，不要倒序做：

1. 收敛工作流会话层
2. 收敛页面状态机
3. 补齐异常恢复和可恢复性
4. 统一接口层并解耦页面

原因：

- 第 1 步先收拢 WebSocket 生命周期和命令发送，才能减少协议分散
- 第 2 步再收敛状态管理，才能让页面行为可预测
- 第 3 步基于前两步补恢复能力，避免“修一个点，裂三个点”
- 第 4 步做长期结构治理，让后续接新接口时不继续发散

## 5. 阶段计划清单

### 阶段 1：收敛工作流会话层

#### 目标

把 WebSocket 连接、事件监听、错误处理、发送命令、关闭连接等能力，从页面组件中抽离成统一的 workflow session 层。

#### 当前问题

- 单资产流程和批量流程各自管理 socket，行为重复
- 页面组件直接理解底层协议细节
- `onmessage`、`send`、错误处理分散在页面逻辑中
- 很难统一处理断线、超时、主动关闭、重连策略

#### 要做的事

- 定义统一的 session 抽象，至少覆盖：
  - 建连
  - 发命令
  - 订阅事件
  - 主动关闭
  - 区分用户主动关闭与异常关闭
- 将单资产和批量模式都接入该抽象
- 把“协议事件名”与“页面响应行为”分离
- 统一 WebSocket 错误提示策略

#### 产出

- 单资产 session 层
- 批量 session 层
- 统一的事件订阅接口
- 统一的关闭与错误处理约定

#### 风险

- 如果抽象过大，容易把单资产和批量模式强行揉成一套，失去可读性
- 如果抽象过小，只是换了文件名，没有真正降耦合

#### 验收标准

- 页面组件不再直接 new 原始 `WebSocket`
- 页面不再直接处理底层 `onmessage`
- 单资产和批量模式的连接生命周期逻辑可独立测试和复用

---

### 阶段 2：收敛页面状态机

#### 目标

把当前 scattered `setState` 推进的流程，改成基于事件驱动的 reducer 或显式状态机。

#### 当前问题

- `App.tsx` 和 `BatchMode.tsx` 中状态切换路径较多
- 同一个事件可能同时影响日志、阶段、图片列表、审批状态、错误状态
- 用户操作和后端事件共同驱动状态，容易出现遗漏或不一致

#### 要做的事

- 为单资产流程定义一份明确状态图
- 为批量流程定义一份明确状态图
- 建立“事件 -> 状态变更”唯一入口
- 区分：
  - 服务端事件
  - 用户动作
  - 派生 UI 状态
- 把日志、阶段条、审批区、图片选择区的状态来源收敛

#### 推荐划分

- 单资产状态：
  - 初始输入
  - prompt 确认
  - 等待出图
  - 等待选图
  - 审批中
  - 代码生成中
  - 完成
  - 错误
- 批量状态：
  - 输入需求
  - 规划中
  - 审阅计划
  - 执行中
  - 审批阻塞
  - 部分失败
  - 完成
  - 错误

#### 风险

- 如果在这个阶段同时改 UI 结构，容易把“状态治理”变成“大重构”
- 如果不先约束事件入口，后续恢复机制依旧会乱

#### 验收标准

- 任何一个页面状态都能回答“它由哪个事件或用户动作进入”
- 新增一个协议事件时，只需要改 session/reducer 层，不需要到处找 `setState`
- 单资产和批量模式的状态流图能独立写出来并自洽

---

### 阶段 3：补齐异常恢复和可恢复性

#### 目标

让前端能正确处理“中途中断”的场景，而不是只在理想流程下可用。

#### 重点场景

- WebSocket 意外断开
- 用户刷新页面
- 审批中页面切走再回来
- 批量执行中部分 item 失败
- 重试某个 item 后状态残留
- 图片已生成但未选中时页面切换
- 上传图片后页面刷新导致数据丢失

#### 要做的事

- 明确哪些状态需要持久化，哪些不需要
- 为批量流程保留最小恢复数据
- 对审批态建立刷新后恢复策略
- 对错误态建立“清理旧状态后再重试”的规则
- 统一“页面初始化时如何恢复旧 session / 旧 UI”

#### 恢复优先级建议

1. 先恢复计划和批量 item 状态
2. 再恢复审批待办
3. 最后评估是否需要恢复单资产中间态

#### 风险

- 过度持久化会把临时状态和真实状态混在一起
- 如果后端没有 session resume 能力，前端只能恢复 UI 快照，不能恢复底层连接

#### 验收标准

- 页面刷新后不会把用户带到完全错误的阶段
- 批量模式可恢复上次计划和关键执行上下文
- 重试逻辑能清理旧错误、旧图片、旧审批残留

---

### 阶段 4：统一接口层并解耦页面

#### 目标

把 HTTP API、DTO、错误转换、类型收敛到统一接口层，避免组件里到处直接 `fetch`。

#### 当前问题

- 配置接口、审批接口、工作流接口分布在多个位置
- 有的组件直接 `fetch`
- 有的走 `lib/*.ts`
- 有的走 `shared/api/*`
- 同类接口风格不统一

#### 当前进度（2026-04-02）

- 已统一 `shared/api/http.ts` 入口，并完成 `config`、`approvals`、`workflow`、`platform`、`admin` 五类 API 模块归位。
- `BuildDeploy` 已改为消费 `buildDeployController.ts`，组件本身不再同时维护 WebSocket 和 HTTP 两套动作编排。
- `App.tsx` 与 `BatchMode.tsx` 的审批异步骨架已收口到 `shared/approvalAction.ts`。
- `App.tsx`、`BatchMode.tsx`、`ModEditorFeatureView` 的默认项目路径加载已收口到 `shared/useDefaultProjectRoot.ts`。
- `BatchMode` 的 WebSocket 规划入口和 HTTP fallback 规划入口已收口到 `features/batch-generation/planningController.ts`。
- 错误消息归一已补 `shared/error.ts`，减少 controller / hook 内重复的 `Error` 解析分支。
- 页面消费层的状态命名已进一步收口，当前更适合继续处理视图展示拼装，而不是零散别名替换。

#### 要做的事

- 统一 HTTP client 入口
- 按领域划分 API 模块：
  - config api
  - approvals api
  - workflow api
  - platform api
  - admin api
- 为后端返回值做前端 DTO/类型收敛
- 统一错误对象映射
- 组件只使用 domain-facing API，不直接拼路径

#### 推荐原则

- 页面拿到的是“业务语义函数”，不是 URL
- 协议兼容和字段兜底逻辑放在接口层
- 接口层负责把后端不规则响应转成前端稳定结构

#### 风险

- 如果接口层和 session 层职责不清，会再造一层混乱
- 如果一口气改完全部模块，回归成本较高

#### 验收标准

- 组件层不再直接 `fetch("/api/...")`
- 后端字段调整时，优先只修改接口层
- 未来新增 `platform/admin` 页面时，可复用现有 API 模块

## 6. 分阶段执行建议

建议按以下粒度推进：

1. 先做阶段 1 设计和边界梳理
2. 确认后落单资产 session
3. 再把批量模式迁移到同一套 session 思路
4. 然后做阶段 2 的 reducer / 状态机收敛
5. 最后再补阶段 3、4

不要一开始就同时改：

- 页面结构
- 状态管理
- API 接入方式
- UI 视觉表现

这样风险太高，不利于验证。

## 7. 本计划的验收标准

如果整个“四阶段稳定化计划”完成，应至少达到以下结果：

- 单资产和批量模式的工作流不会因为局部异常而整体状态错乱
- 前端对 WebSocket 协议的处理有统一入口
- 页面组件主要负责展示和交互，不直接承载底层协议细节
- 批量模式的恢复与重试行为可预测
- HTTP 接口调用方式统一，后续接新接口不会继续发散

## 8. 下一步建议

阶段 4 的接口层主线已经推进到 Chunk 7，下一步更适合继续做“消费层收口”，而不是再横向扩更多 API 模块。优先级建议如下：

1. 继续把单资产、批量、Mod 编辑里仍然分散的 WebSocket 生命周期和事件适配往 controller / session 层下沉。
2. 统一前端错误字段和 busy 状态命名，减少 `errorMsg / globalError / analysisError / modifyError`、`projectCreateBusy / createProjectBusy` 并存。
3. 在不引入重型验证的前提下，继续补消费层 adoption test，锁定共享 helper 与页面的真实接入关系。
