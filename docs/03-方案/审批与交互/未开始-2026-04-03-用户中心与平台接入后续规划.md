# 2026-04-03 用户中心与平台接入后续规划

> 本文档只记录本次功能实现后仍然存在、但不建议在本轮一并解决的残留风险与后续方向。

## 目的

本轮实现聚焦于：

- 登录 / 注册 / 邮箱验证最小闭环
- 当前用户视角平台查询
- 用户中心
- 生成前本机 / 服务器分流
- 平台历史、返还与净消耗展示

为避免范围膨胀，以下事项不在本轮强行落地，而是进入后续规划。

## 残留风险与后续方向

### 1. 真实邮件发送能力

当前计划采用“伪邮件 / 控制台验证码”跑通注册、验证、找回密码流程。后续需要补：

- SMTP / 邮件服务商接入
- 邮件模板
- 重发频控
- 投递失败监控

建议后续单独做“认证基础设施深化”专题，不和本轮用户中心耦在一起。

### 2. 账户安全能力不足

本轮只做最小认证闭环，后续还需要考虑：

- 登录失败频控
- 密码重试限制
- 邮箱验证码过期和重放防护
- 异常登录告警
- 会话管理与多端登录策略

建议后续单列“账号安全加固”任务，不阻塞本轮主链。

### 3. 本机执行历史的可视化缺位

按 2026-03-27 草案，本机执行历史不能进入平台用户中心。这会带来一个自然的后续需求：

- 用户也许仍希望查看本机执行历史
- 但这部分必须留在工作站本地视图，而不是平台用户中心

建议后续单独规划“工作站本地历史页”，数据源基于本地快照/本地数据库，而不是平台 `job`。

### 4. 独立 Web 前端站点仍未解决

本轮已提升 `web-backend` 独立部署优先级，并将其纳入先行实施范围；但“独立 Web 前端站点”仍未纳入本轮。如果平台控制台能力持续扩大，后续仍可能需要：

- 独立 Web 前端构建与托管入口
- 面向平台的独立导航与部署形态
- 工作站端与 Web 端更明确的前端边界

建议等本轮 `web-backend` 独立部署、用户中心和平台查询稳定后，再评估是否需要继续拆分前端站点。

### 5. 商业化与计费规则后台配置

本轮只做“统一次数池 + 模块不同扣减值”的产品表达，不深入商业化配置能力。后续可能需要：

- 后台可配置扣减规则
- 用户权益等级
- 套餐、订阅、赠送次数
- 更细的周期额度规则

这部分不应与本轮认证/用户中心主链捆绑上线。

### 6. 返还判定矩阵仍可细化

当前按草案，只强调：

- 系统异常自动返还
- BYOK 不进平台计费
- 取消后已发出的调用可能仍计费

后续仍需细化：

- 超时是否返还
- 第三方限流是否返还
- 用户取消到哪一阶段开始不返还
- 部分成功批量任务如何逐项解释返还

建议在平台计费专题中继续深化。

## 后续规划建议顺序

1. 认证基础设施深化：真实邮件发送、验证码管理、频控
2. 账号安全加固：锁定、审计、会话管理
3. 工作站本地历史页：独立于平台用户中心
4. 平台计费规则深化：返还矩阵、规则配置、套餐扩展
5. 评估独立 Web 前端站点：视平台控制台能力增长决定

## 结论

本轮应优先把“当前用户视角 + 用户中心 + 平台历史 + 分流逻辑”做完整，剩余风险不应在本轮继续扩张范围。后续按专题逐步推进，比一次性把所有账户、平台、工作站边界问题都塞进本轮更稳。
