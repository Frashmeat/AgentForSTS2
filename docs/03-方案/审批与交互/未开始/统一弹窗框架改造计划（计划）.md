# 统一弹窗框架改造计划

> 文档定位：本文用于规划前端弹窗、确认框与抽屉的统一框架改造，不直接规定后端接口或业务流程。
>
> 事实依据：基于 2026-04-29 对 `frontend/src` 中 `ConfirmDialog`、`ExecutionModeDialog`、`KnowledgeGuideDialog`、`SettingsPanel` 与确认类按钮的静态调查。
>
> 权威入口：上级入口为 `docs/03-方案/审批与交互/README.md`。
>
> 最后更新：2026-04-29

## 当前状态

未开始，等待方案讨论确认。

## 背景

前端已经移除服务器任务开始链路中的浏览器原生 `window.confirm`，并新增 `ConfirmDialog` 承接“确认开始服务器任务”。但当前弹窗体系仍不统一：

- `ConfirmDialog` 是简单确认框。
- `ExecutionModeDialog` 是业务复杂弹窗，自带遮罩与容器。
- `KnowledgeGuideDialog` 是说明类弹窗，自带遮罩与容器。
- `SettingsPanel` 是右侧设置抽屉，自带遮罩与容器。
- 部分高风险操作仍是按钮直接执行，例如“清空默认配置”“停止生成”“重新开始”。

因此需要抽象一套统一框架，避免后续业务组件继续手写 `fixed inset-0`、z-index、遮罩、圆角、阴影和可访问性属性。

## 目标

- 所有弹窗和抽屉共享同一套遮罩、层级、容器、标题区、内容区、底部操作区与可访问性约定。
- 简单二次确认统一走 `ConfirmDialog`。
- 复杂业务弹窗复用底层 `AppDialog`，不强行塞进 `ConfirmDialog`。
- 右侧抽屉使用 `AppDrawer`，与 `AppDialog` 共享底层视觉规则，但保持职责清晰。
- 禁止新增浏览器原生 `window.confirm / alert / prompt`。

## 不纳入范围

- 不引入 Radix、Headless UI、Ant Design、MUI 等外部弹窗库。
- 不重做页面视觉风格，只统一现有弹窗壳层。
- 不改变服务器任务、知识库、设置保存、生成流程的业务语义。
- 不把 Toast 类 `StatusNoticeStack` 改成弹窗。

## 建议组件边界

### `AppDialog`

底层居中弹窗框架，只负责结构与视觉壳层：

- `open: boolean`
- `title: string`
- `kicker?: string`
- `description?: string`
- `size?: "sm" | "md" | "lg" | "xl"`
- `onClose?: () => void`
- `closeOnBackdrop?: boolean`
- `children`
- `footer?`

职责：

- 渲染统一遮罩。
- 渲染统一容器、标题区、正文区、底部区。
- 提供 `role="dialog"`、`aria-modal="true"` 等基础可访问性属性。
- 统一 z-index、圆角、阴影、边框、背景色和间距。

### `AppDrawer`

右侧抽屉框架，用于设置页等侧边面板：

- `open: boolean`
- `title?: string`
- `onClose?: () => void`
- `closeOnBackdrop?: boolean`
- `children`

职责：

- 复用统一遮罩规则。
- 使用右侧滑出容器。
- 不承载确认语义。

### `ConfirmDialog`

基于 `AppDialog` 的简单二次确认组件：

- `open: boolean`
- `title: string`
- `message: string`
- `confirmLabel?: string`
- `cancelLabel?: string`
- `tone?: "default" | "warning" | "danger"`
- `onConfirm: () => void`
- `onCancel: () => void`

职责：

- 承接“是否继续”“是否停止”“是否清空”等简单确认。
- 不承载复杂业务表单或列表。

## 迁移范围

第一批应迁移：

- `frontend/src/components/ConfirmDialog.tsx`
- `frontend/src/components/KnowledgeGuideDialog.tsx`
- `frontend/src/components/ExecutionModeDialog.tsx`
- `frontend/src/components/SettingsPanel.tsx`

第二批应评估接入 `ConfirmDialog` 的操作：

- 设置页“清空默认配置”
- 单资产“停止生成”
- 单资产“重新开始”
- 批量生成“停止生成”

## 实施步骤

1. 新增 `AppDialog` 与 `AppDrawer`。
2. 重构 `ConfirmDialog` 复用 `AppDialog`，保持现有调用 API 不变。
3. 迁移 `KnowledgeGuideDialog` 到 `AppDialog`。
4. 迁移 `ExecutionModeDialog` 到 `AppDialog`。
5. 迁移 `SettingsPanel` 抽屉模式到 `AppDrawer`，页面模式保持现有页面布局。
6. 为高风险按钮增加 `ConfirmDialog` 二次确认。
7. 增加回归测试：
   - 禁止 `window.confirm / alert / prompt`。
   - 除 `AppDialog` / `AppDrawer` / `StatusNoticeStack` 外，业务组件不直接写 `fixed inset-0` 弹窗壳。
8. 同步更新前端行为文档。

## 风险

- 弹窗层级变化可能影响多个弹窗同时出现时的覆盖顺序。
- `SettingsPanel` 同时支持 `drawer` 和 `page` 两种模式，迁移时要避免影响独立设置页。
- `ExecutionModeDialog` 内部有较多服务器配置状态，迁移时应避免误改 disabled 状态和按钮行为。
- 高风险按钮接入确认框后会增加一次点击，可能影响用户操作节奏，需要确认哪些动作确实需要二次确认。

## 验收标准

- 搜索 `frontend/src`：
  - 无 `window.confirm / window.alert / window.prompt`。
  - 除统一框架和 Toast 外，无业务组件直接声明弹窗遮罩 `fixed inset-0`。
- `ConfirmDialog`、`KnowledgeGuideDialog`、`ExecutionModeDialog`、`SettingsPanel` 视觉壳层统一。
- 设置页页面模式不受抽屉迁移影响。
- 定点测试与 TypeScript 检查通过。
- 相关文档同步更新。
