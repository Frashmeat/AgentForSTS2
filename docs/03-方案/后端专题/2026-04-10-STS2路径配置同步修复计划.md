# 2026-04-10 STS2 路径配置同步修复计划

## 1. 背景

当前工作站设置页已经允许用户配置：

- `sts2_path`
- `godot_exe_path`
- `default_project_root`

但实际 STS2 mod 项目的 `dotnet publish` 仍然依赖项目目录下的 `local.props` 与 `.csproj` 内的 MSBuild 属性解析。现场排查表明，设置值与项目实际构建输入之间存在断链，导致：

- 设置页里 `sts2_path` 已正确配置
- 项目内 `local.props` 仍保留旧默认值
- `dotnet publish` 找不到 `sts2.dll` / `0Harmony.dll`

## 2. 已确认事实

### 2.1 设置与项目构建输入脱节

- `backend/project_utils.py::ensure_local_props()` 只在 `local.props` 不存在时才生成文件。
- 现有项目若已存在 `local.props`，设置页后续修改不会同步进项目。
- `backend/routers/build_deploy.py` 在构建前只读取 `cfg["sts2_path"]` 用于部署目录检查，不会把设置回写到项目的 `local.props`。

### 2.2 自动生成模板字段名与项目模板实际消费字段不一致

- `backend/app/shared/resources/prompts/runtime_system.md` 当前 `project_utils_local_props_template` 使用：
  - `STS2GamePath`
  - `GodotExePath`
- 但 STS2 模板 `.csproj` 实际消费的是：
  - `SteamLibraryPath`
  - `GodotPath`

这意味着即便自动生成 `local.props`，也可能无法被 `.csproj` 正确读取。

### 2.3 同类风险不只影响 `sts2_path`

- `godot_exe_path` 也走同一条 `local.props` 链路，存在相同断链风险。
- Code Agent prompt 与知识文档里存在“`local.props` 已正确、不要重建”的口径，如果底层同步机制失效，会持续放大旧配置残留问题。

## 3. 根因判断

主要矛盾不是用户没配置路径，而是：

1. 运行时设置真源与项目构建输入源分裂
2. `local.props` 只做“缺失时生成”，不做“已有项目同步”
3. 自动生成模板字段与 `.csproj` 实际读取字段不一致

## 4. 修复目标

### 4.1 主目标

- 设置页里的 `sts2_path` / `godot_exe_path` 能稳定进入项目构建输入
- 已有项目不需要手工改 `local.props`
- 新项目初始化时不再生成错误字段

### 4.2 非目标

- 不改 STS2 模板的 DLL 引用结构
- 不引入新的构建系统
- 不做全量 build / 全量测试

## 5. 修复方案

### 5.1 修复 `local.props` 生成模板

- 把运行时模板字段改为 `.csproj` 实际可消费的字段
- 优先使用：
  - `SteamLibraryPath`
  - `GodotPath`

### 5.2 增加受控同步逻辑

- 新增/重构 `project_utils` 中的 `local.props` 同步能力
- 不再使用“文件存在就直接跳过”的策略
- 同步逻辑只管理本工具负责的字段，避免无脑覆盖用户可能手工保留的其他字段

### 5.3 构建前补兜底同步

- 在 Build & Deploy 或其 facade 进入 `dotnet publish` 前，先用当前设置同步项目 `local.props`
- 覆盖“已有项目 + 设置已改 + 项目未重建”的场景

### 5.4 更新测试与文档口径

- 补 `project_utils` 定向测试
- 补构建链路测试
- 更新相关说明文档，明确 `local.props` 来自当前设置同步，而不是永远“已经正确”

## 6. 影响范围

- `backend/project_utils.py`
- `backend/routers/build_deploy.py`
- `backend/app/modules/platform/application/services/build_deploy_facade_service.py`
- `backend/app/shared/resources/prompts/runtime_system.md`
- 运行时 artifact 对应文件
- `backend/tests/test_project_utils.py`
- `backend/tests/test_build_deploy_ws.py`
- `backend/tests/platform/services/test_build_deploy_facade_service.py`
- 相关实施基线 / 审查记录 / 说明文档

## 7. 风险

### 7.1 覆盖用户手工配置

如果直接整文件重写 `local.props`，可能覆盖用户自定义字段。

应对：

- 仅更新工具托管字段
- 其他未知字段原样保留

### 7.2 运行时副本与源码漂移

当前存在 `backend/` 与 `tools/latest/artifacts/.../backend/` 双份实现。

应对：

- 同步修改两侧关键文件
- 最终用 diff / 测试确认

### 7.3 路径派生错误

`.csproj` 需要 `SteamLibraryPath`，而设置里存的是游戏根目录。

应对：

- 明确从 `sts2_path` 推导 `SteamLibraryPath`
- 若路径不满足预期结构，直接回退为错误并阻止静默写错

## 8. 验收标准

- 设置页配置 `sts2_path=E:/SteamLibrary/steamapps/common/Slay the Spire 2` 后，项目 `local.props` 中会同步得到可用于 `.csproj` 的路径字段
- 已有项目即使历史上已有旧 `local.props`，构建前也会同步到最新设置
- `godot_exe_path` 走同样同步链路
- 定向测试覆盖：
  - `local.props` 初次生成
  - `local.props` 已存在但字段过期
  - Build & Deploy 构建前同步
- 文档口径不再继续误导“`local.props` 一定已经正确”

## 9. 实施顺序

1. 先修计划文档与实施口径
2. 再修 `project_utils` 的模板与同步逻辑
3. 再修构建前兜底同步
4. 再补测试
5. 最后更新文档收口
