# 2026-04-10 知识库资产化改造计划（后续阶段）

## 0. 2026-04-14 当前状态说明

- 本文不再表示“从零开始的新计划”，而是知识库真源主线之后的后续阶段计划。
- 已被现有代码吸收的前置能力：
  - `backend/app/modules/knowledge/application/knowledge_facade.py` 已存在，统一了知识来源决策入口。
  - `backend/app/modules/knowledge/infra/knowledge_runtime.py` 已把 `runtime/knowledge/` 收口为运行时唯一真源，并提供 seed 初始化。
  - `backend/app/modules/codegen/api.py` 已通过 facade 注入 API lookup 上下文。
  - `frontend/src/components/KnowledgeGuideDialog.tsx`、`README.md` 已能说明运行时知识目录位置与用户可编辑语义。
- 因此本文当前真正剩余的工作，不是“再做一遍真源收口”，而是继续推进：
  - 更明确的知识资产边界与清单管理
  - 刷新链路、说明链路、状态表达的长期收口
  - 把知识库更新彻底从工作流链路中剥离成更独立的资产运维能力

## 1. 目标

把当前 STS2 / BaseLib 知识来源从“运行时临时拼装”改造成“独立资产目录 + 统一消费 facade + 工作流只读不构建”的结构。

## 2. 当前问题

- 仍有部分历史文档、设计稿与接口命名沿用旧的 `decompiled_src_path / game_decompiled / baselib_decompiled` 口径，容易让维护者误解当前真源边界。
- 运行时真源虽然已经收口，但知识库作为“独立资产”的生命周期管理仍偏分散，manifest、刷新、缓存、说明和版本表达还没有完全抽成长期稳定约束。
- `sts2_api_reference.md` 是仓库静态参考，但过去缺少明确版本标注，容易被误用成当前游戏版本真源。
- 知识库更新依赖 `ilspycmd`，但工具存在方式和查找顺序不清晰。

## 3. 冻结后的结构

### 3.1 运行时知识资产目录

- `runtime/knowledge/knowledge-manifest.json`
- `runtime/knowledge/game/`
- `runtime/knowledge/baselib/`
- `runtime/knowledge/resources/sts2/`
- `runtime/knowledge/cache/`

### 3.2 仓库静态 seed source

- `backend/agents/sts2_api_reference.md`
- `backend/agents/baselib_src/BaseLib.decompiled.cs`
- `backend/app/modules/knowledge/resources/sts2/*.md`

### 3.3 统一消费原则

- 工作流执行时不再触发知识库构建。
- Planning / Codegen / Analyzer 只读取运行时知识状态与运行时知识目录。
- `runtime/knowledge/` 是唯一运行时真源；仓库静态文件只用于开发期与打包期生成初始化种子。

## 6. 第一阶段实施范围

### 6.0 当前判断

- 第一阶段的核心实现已被 `知识库运行时唯一真源实施计划` 吸收并落地。
- 本节保留，主要用于说明后续若继续扩展时的既有落点与设计意图。

### 6.1 新增统一 facade

新增：

- `backend/app/modules/knowledge/application/knowledge_facade.py`

负责：

- 统一决策当前游戏知识来源
- 统一决策当前 Baselib 知识来源
- 输出 codegen prompt 所需的 API lookup context

### 6.2 收口 codegen 入口

修改：

- `backend/app/modules/codegen/api.py`

要求：

- 不再直接组合 `get_active_game_decompiled_src_path()`、`get_decompiled_src_path()`、`get_active_baselib_src_path()`
- 统一改为调用 knowledge facade

### 6.3 补测试

新增：

- `backend/tests/test_knowledge_facade.py`

覆盖：

- runtime game 目录优先
- runtime baselib 优先
- missing 场景
- `build_api_lookup_context()` 输出符合 codegen 预期

### 6.4 文档与工具前提

- `sts2_api_reference.md` 明确标注当前游戏 build 版本
- `install.ps1` 确保 `ilspycmd`
- `knowledge_runtime.py` 查找 `ilspycmd` 时先查项目目录，再查 PATH

## 7. 第二阶段实施范围

- 统一设置页、知识库说明和工作流 banner 的来源展示
- 让前端明确看到当前运行时知识目录位置，并说明用户修改后直接生效

## 8. 第三阶段实施范围

- 提供独立知识库刷新脚本
- 把知识库更新彻底从工作流执行链中剥离

## 9. 验收标准

- `codegen.api` 只从 knowledge facade 读取知识来源
- 工作流执行不再临时构建知识库
- 发行包直接包含 `runtime/knowledge/` 初始内容
- 用户修改 `runtime/knowledge/` 后，后续知识读取直接变化
- Baselib / 游戏源码当前实际来源可被前后端统一表达为运行时知识目录
- 第一阶段定向测试通过
