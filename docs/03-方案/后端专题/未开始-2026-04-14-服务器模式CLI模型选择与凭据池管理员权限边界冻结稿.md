# 2026-04-14 服务器模式 CLI 模型选择与凭据池管理员权限边界冻结稿

## 1. 冻结目标

本文只解决 4 件事：

1. 明确哪些接口属于普通用户接口，哪些属于管理员接口
2. 冻结管理员能力模型的最小表达
3. 冻结管理员写接口的鉴权前置条件
4. 明确在管理员权限边界未补齐前，哪些能力禁止进入实现

本文不解决：

- 具体用户表 / 权限表 migration
- 具体 JWT / session claims 结构
- 具体依赖注入和 router 装饰器代码
- 管理员前端页面实现

## 2. 当前事实

基于现有仓库与文档，当前事实已经明确：

- `/api/admin/*` 已存在查询型接口
- 当前管理员链路更偏“查询入口”，而不是“高风险写操作入口”
- 当前未见独立管理员权限模型的正式冻结基线
- 因此，不能把现有 `/api/admin/*` 的存在误解为“已经具备服务器凭据写接口的安全前提”

## 3. 第一版边界冻结

## 3.1 普通用户接口

普通用户接口继续冻结为：

- `GET /api/platform/execution-profiles`
- `GET /api/me/server-preferences`
- `PUT /api/me/server-preferences`

要求：

- 必须以普通登录用户身份访问
- 不要求管理员权限
- 只允许读取或修改“当前用户自己的默认服务器配置”
- 不暴露任何服务器凭据明文或单把凭据细节

## 3.2 管理员接口

管理员接口第一版冻结为：

- `GET /api/admin/platform/server-credentials`
- `POST /api/admin/platform/server-credentials`
- `POST /api/admin/platform/server-credentials/{id}/health-check`

要求：

- 必须以管理员身份访问
- 不能依赖“普通登录即可访问”的现有会话模型
- 写操作与检测操作都必须经过强制管理员鉴权

## 4. 管理员能力模型冻结

当前先冻结“能力”而不是“具体实现字段名”。

第一版管理员最小能力集合：

- `platform_admin_read`
  - 可查看平台管理员侧查询接口
- `platform_admin_credentials_write`
  - 可新增 / 修改 / 启用 / 禁用服务器凭据
- `platform_admin_credentials_check`
  - 可执行手动健康检测

当前建议：

- 第一版实际实现时，允许先把以上三种能力收口成一个总能力，例如 `platform_admin`
- 但在冻结稿中先把读 / 写 / 检测三个能力拆开写清楚，避免后续接口继续无边界扩张

## 5. 鉴权规则冻结

## 5.1 普通用户接口鉴权

- 必须登录
- `GET /api/me/server-preferences`
  - 只能读取当前会话用户自己的偏好
- `PUT /api/me/server-preferences`
  - 只能修改当前会话用户自己的偏好

## 5.2 管理员接口鉴权

第一版冻结以下规则：

- `GET /api/admin/platform/server-credentials`
  - 需要管理员能力
- `POST /api/admin/platform/server-credentials`
  - 需要管理员写能力
- `POST /api/admin/platform/server-credentials/{id}/health-check`
  - 需要管理员检测能力

额外要求：

- 不能只靠“路径前缀是 `/api/admin/*`”视为天然安全
- 不能把当前管理员查询接口默认扩展为写接口
- 任何写接口都必须有可审计的操作者身份

## 5.3 禁止的做法

第一版明确禁止：

- 未补齐管理员权限模型前直接开放凭据写接口
- 让普通用户会话直接命中凭据管理接口
- 把“是否登录”混同于“是否管理员”
- 在没有操作者身份记录的情况下执行健康检测或修改凭据

## 6. 管理员接口归属冻结

当前继续冻结管理员接口归属为：

- 路由前缀：`/api/admin/platform/*`

原因：

- 本专题属于平台服务器能力，不属于工作站本机配置
- 应沿现有平台管理员查询链路增量扩展
- 避免把服务器凭据管理散落到普通用户接口或工作站接口里

## 7. 审计要求冻结

管理员写操作和检测操作必须至少能记录：

- `operator_user_id`
- `action`
- `target_type`
- `target_id`
- `created_at`

第一版建议最少覆盖的动作：

- `server_credential_created`
- `server_credential_updated`
- `server_credential_enabled`
- `server_credential_disabled`
- `server_credential_health_checked`

## 8. 在本稿完成前禁止进入的实现项

以下能力在管理员权限边界未冻结前，禁止直接进入实现：

- `POST /api/admin/platform/server-credentials`
- 任何服务器凭据编辑接口
- 任何服务器凭据启用 / 禁用接口
- 任何服务器凭据检测接口

允许继续推进的仅包括：

- 普通用户接口
- 表结构冻结
- 普通用户前端交互设计

## 9. 本稿验收标准

当以下条件满足时，可认为“管理员权限边界冻结稿”完成：

- 普通用户接口与管理员接口边界明确
- 管理员能力模型最小集合明确
- 管理员接口的鉴权前置条件明确
- 高风险接口在权限未补齐前被明确禁止实现
- 后续实现人员不需要再回到草案对管理员边界做二次裁决

## 10. 当前建议

本稿完成后，下一步才适合进入：

1. 评估当前认证 / 权限模型如何承载 `platform_admin*` 能力
2. 再决定是否开始 migration 和管理员接口实现

在此之前，不建议直接开始服务器凭据管理写接口开发。
