# 知识库运行时唯一真源实施计划 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让发放给用户的 `runtime/knowledge/` 成为知识库唯一运行时真源，保证用户修改发行包内知识文件后，运行时读取结果直接生效。

**Architecture:** 保持“仓库内知识文件作为打包种子来源、运行时知识目录作为唯一消费真源”的双阶段结构。运行时不再把 `backend/agents/*` 与 `backend/app/modules/knowledge/resources/sts2/*` 当作并列知识来源；这些文件只在打包或首次初始化时复制到 `runtime/knowledge/`，之后所有读取统一走运行时目录，避免用户修改不生效。

**Tech Stack:** Python, FastAPI, pathlib, PowerShell, pytest, React, TypeScript

---

## 调查结论

- 当前运行时知识缓存目录已经存在于 `release/runtime/knowledge/`，入口常量定义在 `backend/app/modules/knowledge/infra/knowledge_runtime.py`。
- 当前运行时消费仍然依赖仓库内 fallback：
  - `backend/agents/sts2_api_reference.md`
  - `backend/agents/baselib_src/BaseLib.decompiled.cs`
  - `backend/app/modules/knowledge/resources/sts2/*.md`
- `backend/app/modules/knowledge/application/knowledge_facade.py` 明确存在 `runtime_decompiled -> repo_reference/repo_fallback -> missing` 的回退逻辑。
- `backend/agents/sts2_docs.py` 仍直接从仓库路径读取静态知识资源和 API 参考文件，这意味着即使用户修改发行包内知识文件，也不一定能影响运行时。
- 因此本次主要矛盾不是“知识文件在哪”，而是“运行时是否仍把 repo 文件当并列真源”。只有把运行时真源单收口到 `runtime/knowledge/`，用户修改才能稳定生效。

## 范围冻结

- 本次范围只处理“知识库运行时真源、打包种子复制、消费链路切换、展示说明与测试收口”。
- 本次不引入 zip 资产解包、prerelease 支持、知识编辑 UI。
- 本次不改知识内容本身，只改知识文件位置、初始化方式和读取路径。

## 目标目录结构

发行包与运行时统一使用以下结构：

- `runtime/knowledge/knowledge-manifest.json`
- `runtime/knowledge/game/`
- `runtime/knowledge/baselib/`
- `runtime/knowledge/resources/sts2/`
- `runtime/knowledge/cache/`

说明：

- `game/`：用户可查看和替换的游戏反编译知识目录。
- `baselib/`：用户可查看和替换的 BaseLib 反编译知识目录。
- `resources/sts2/`：用户可查看和编辑的静态规则文档目录。
- `cache/`：刷新知识库时的下载缓存目录。
- 仓库内现有 `backend/agents/*` 与 `backend/app/modules/knowledge/resources/sts2/*` 只作为种子来源，不再作为运行时并列真源。

## 风险清单

- `backend/agents/sts2_docs.py` 当前是模块导入时加载资源，若直接切到运行时目录，需要避免在目录尚未初始化时就导入失败。
- 打包脚本需要兼容 `workstation / hybrid / web / full` 多条链路，不能只改一条发行目标。
- 现有测试大量写死 repo fallback 语义，实施时要整体改口径，避免一半新语义一半旧语义。
- 当前工作区已有其他未提交改动，实施时必须避开：
  - `backend/llm/text_runner.py`
  - `backend/tests/test_deploy_docker_script.py`
  - `backend/tests/test_text_runner_selection.py`

## 验收标准

- 发行包内包含完整 `runtime/knowledge/` 种子内容。

- 用户修改 `runtime/knowledge/` 下的知识文件后，后续知识读取结果直接变化。
- 知识状态、前端说明、README、实施基线文档口径一致。
