# 剩余兼容层彻底删除实施计划 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在项目未上线前提下，删除仓库内剩余仍会影响当前运行边界、配置真源、知识源、旧入口与旧数据恢复的兼容层，最终收敛到单一路径实现。

**Architecture:** 本轮不再保留“迁移期并存”策略，而是按真实主链逐块收口。先删除已经被 `me.ts`、`runtime/*.config.json` 等新路径取代的兼容 API 和脚本回退，再处理后端配置真源、知识库运行时 fallback、旧导入 shim、本地恢复旧字段和顶层脚本 wrapper，最后做一次文档与搜索口径清扫，确保仓库不再保留活动兼容层。

**Tech Stack:** Python/FastAPI、TypeScript/React、PowerShell、Pytest、Node test

## 风险与边界

- `settings.py` 删除根 `config.json` 回退后，旧本地环境若未生成 `runtime/*.config.json` 会直接暴露配置缺口；这是本轮有意为之，不再兼容。
- 删除知识库 `repo_fallback` 后，未完成运行时种子的环境会从“stale”变为“missing”；这也是预期行为。
- 删除 tools 顶层 wrapper 会打断旧命令习惯，但不会影响 `tools/tools.ps1` 与 `tools/start/*`、`tools/install/*` 真入口。
- 本计划中的“兼容层”只指项目自身为了旧路径、旧配置、旧数据、旧入口保留的桥接；普通容错型 fallback（如错误文案、CPU fallback）不在本轮删除范围。

## 验收标准

- 前端不再保留显式 `user_id` 平台兼容 API
- 后端运行时不再回退到根目录 `config.json`
- 发布脚本不再迁移 `services/workstation/config.json`
- 知识库运行时不再返回 `repo_reference / repo_fallback`
- 仓库主代码不再使用 `agents.code_agent`
- 前端恢复模块不再兼容旧快照字段名
- tools 顶层旧 wrapper 全部删除
- 当前有效 README / tools 文档不再把兼容入口或兼容来源写成可用主链
