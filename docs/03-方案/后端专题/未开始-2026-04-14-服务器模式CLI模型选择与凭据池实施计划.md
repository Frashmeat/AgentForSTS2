## 1. 定位与目标

本计划用于落地“服务器模式下用户只选择 `CLI + 模型`，平台后台维护同组合多凭据池并允许同组合内自动切换一次”的能力。

本专题要解决的核心问题是：

- 普通用户如何在服务器模式下稳定地选择平台提供的执行能力
- 平台如何把“用户可见组合”和“后台真实凭据”分层管理
- 任务如何同时记录“用户选择了什么”和“实际命中了什么”
- 管理员如何安全地维护服务器 key、查看状态并进行检测
- 高并发下如何避免服务器模式把平台主链拖垮

## 3. 已冻结的设计结论

- 用户进入服务器模式后，只选择 `CLI + 模型`，不选择 provider、base URL 或具体 key。
- 即使用户本地已有自己的 key，仍可主动使用服务器模式。
- 同一个 `CLI + 模型` 组合下允许挂多组服务器凭据。
- 首次执行失败时，只允许在同组合内自动切换一次。
- 自动切换不得跨 CLI、不得跨模型、不得跨组合降级。
- 每个用户全局只记录一个默认服务器配置。
- 默认服务器配置以服务端字段为真源，设置页与运行时弹窗共用同一份默认值。
- 设置页“服务器模式”与运行时选择流程复用同一套服务器配置选择弹窗。
- 管理员可以配置服务器凭据，并对单个凭据或某个组合进行状态检测。

## 5. 代码现状带来的关键约束

基于当前仓库代码，后续实现必须遵守以下约束：

- `platform_admin` 当前是查询型路由，不应被误认为已经具备管理员写操作的鉴权基线。
- 当前未见独立管理员权限模型，因此“管理员凭据写接口”不能直接挂到现有 `/api/admin/*` 上裸跑。
- 当前 `ai_execution` 主体仍与 `job_item` 绑定，若要补服务器模式快照，必须明确“用户选择快照”和“实际命中事实”的字段落点。
- 现有平台主链已具备 `job / job_item / ai_execution / audit` 基础结构，新能力应沿既有平台主链增量扩展，而不是旁路另起一套记录系统。

## 8. 推荐执行顺序

若进入正式落地，建议严格按以下顺序推进：

1. 先冻结 API 草案和字段口径
2. 再补齐管理员权限边界
3. 再进入平台后端数据层与服务层实现
4. 再落地前端选择器与设置页 Tab 化
5. 最后补抗压保护、文档回写与基线收口

该顺序当前不建议调整，原因是：

- 如果接口和表结构不先冻结，前后端会反复返工
- 如果管理员权限边界不先补齐，凭据管理接口风险过高
- 如果运行保护太晚才考虑，服务器模式一旦放量会直接放大平台压力

## 9. 风险

### 9.1 管理员权限边界缺失

- 若直接在现有 `/api/admin/*` 上扩写接口，会把“查询入口”误当成“已具备高权限基线”的能力，安全风险最高。

### 9.2 用户选择与实际执行脱节

- 若只记录用户选择、不记录实际命中凭据，后续排障、审计和返还解释都会失真。

### 9.3 数据落点摇摆

- 若 `execution_profile` 快照在 `job`、`job_item`、`ai_execution` 间反复摇摆，前后端语义会难以统一。

### 9.4 设置页与运行时分裂

- 若设置页和运行时各自维护一套选择器与默认值逻辑，后续一定会出现列表、状态和说明文案不一致。

### 9.5 高并发下缺少保护

- 若服务器模式在没有入队、限流、熔断前就放量，高并发下会拖慢 `web` 主链并放大坏凭据重试风暴。

## 10. 验收标准

当以下条件满足时，可认为本专题第一批闭环完成：

- 普通用户在服务器模式下只能看到 `CLI + 模型` 能力列表，不暴露底层凭据细节。
- 每个用户只有一份服务端真源的默认服务器配置，设置页与运行时共用。
- 平台能同时记录：
  - 用户选择的服务器配置组合
  - 单次执行实际命中的凭据与是否发生切换
- 管理员可以安全地维护凭据、启停凭据并进行手动检测。
- 默认配置失效时，运行时会要求重新选择，而不是静默失败。
- 服务器模式具备最小入队、限流和熔断保护。
- 实现完成后，相关实施计划、接口文档和基线文档已同步回写。
