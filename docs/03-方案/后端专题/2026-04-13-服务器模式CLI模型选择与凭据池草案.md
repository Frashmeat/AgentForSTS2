# 2026-04-13 服务器模式 CLI / 模型选择与凭据池方案草案

> 适用范围：本文用于沉淀“服务器模式下用户直接选择 `CLI + 模型`，后台通过同组合多凭据池自动执行”的设计草案。本文当前属于讨论稿，不等同于实施基线；待用户选择配置、平台数据模型和接口字段进一步确认后，再决定是否升格到实施计划或基线文档。

## 1. 目标

当前产品已经存在“本机执行 / 服务器模式”的分流，但服务器模式下仍缺少以下能力的明确设计：

- 用户如何选择服务器侧文本 / 代码执行配置
- 服务器如何管理多个服务商凭据
- 同一 `CLI + 模型` 下如何做可控的自动切换
- 平台任务如何记录“用户选择了什么”与“实际执行用了什么”

本文目标是在不改变既有后端双角色边界的前提下，为服务器模式补齐上述设计。

## 2. 已收口结论

截至 2026-04-13，当前讨论已确认以下口径：

1. 用户即使本机已有自己的 key，仍然可以主动选择服务器模式。
2. 用户进入服务器模式后，直接选择 `CLI + 模型`，而不是选择 provider、key 或 base URL。
3. 同一个 `CLI + 模型` 组合后面允许挂多组服务器凭据。
4. 运行时优先使用同组合内的一组可用凭据执行。
5. 若首次执行失败，且失败类型属于允许切换的错误，则在同组合内自动切换到下一组可用凭据重试一次。
6. 自动切换只允许发生一次，且不得跨 CLI、不得跨模型、不得跨组合降级。
7. 服务器模式仍然只通过 `web` 对外；服务器侧 `workstation` 仅作为内部执行器。

## 3. 上位约束

本草案默认遵循以下既有文档口径：

- `web` 与 `workstation` 仍然是长期唯一后端角色。
- 前端公开目标仍然只有 `workstation` 与 `web`。
- 服务器模式的任务入口、状态、历史、事件、计费、审计仍由 `web` 统一对外。
- 服务器侧 `workstation` 只接收 `web` 下发的内部执行请求，不对前端直接暴露。

因此，本草案讨论的是“服务器模式内部如何选择和解析执行配置”，而不是重新定义前端目标或后端角色。

## 4. 设计原则

### 4.1 用户只看能力，不看凭据

普通用户在服务器模式下只应看到可选能力项，例如：

- `Codex CLI / gpt-5.4`
- `Claude CLI / claude-sonnet-4-6`

用户不应直接看到：

- provider 名称
- 原始 API key
- base URL
- 单把服务器凭据的状态细节

### 4.2 平台内部保留可替换的映射层

尽管用户界面只展示 `CLI + 模型`，后端内部仍应保留一层独立的执行配置映射，以便：

- 后续轮换 key
- 临时下线某组凭据
- 为同一组合挂多组凭据
- 保持历史任务可追溯

### 4.3 任务必须记录“用户选择”与“实际执行”

平台链路必须同时能够回答两个问题：

- 这个任务创建时，用户选择的是哪一个服务器侧组合
- 这次具体执行时，系统实际命中了哪一组服务器凭据

没有这两个层次，后续审计、排障和计费解释都会变得模糊。

## 5. 核心模型

## 5.1 服务器执行配置（用户可见）

建议新增“服务器执行配置”这一逻辑对象，作为用户在服务器模式下实际可选的能力单元。

建议最小字段：

- `id`
- `code`
- `display_name`
- `agent_backend`
- `model`
- `enabled`
- `recommended`
- `description`
- `sort_order`

其中：

- `display_name` 用于前端直接展示，例如 `Codex CLI / gpt-5.4`
- `agent_backend` 用于表达 CLI 类型，例如 `codex`、`claude`
- `model` 用于表达该组合对应的模型名
- `recommended` 用于前端默认推荐展示

### 5.2 服务器凭据池项（后台私有）

建议新增“服务器凭据池项”，用于承载真正的服务商凭据。

建议最小字段：

- `id`
- `execution_profile_id`
- `provider`
- `auth_type`
- `credential_ciphertext`
- `base_url`
- `priority`
- `enabled`
- `health_status`
- `last_error_code`
- `last_error_at`

说明：

- 同一个 `execution_profile_id` 下面允许存在多条凭据池项
- 前端与普通用户接口不暴露 `credential_ciphertext`
- `priority` 用于同组合内的优先顺序
- `health_status` 用于避免坏凭据被反复命中

### 5.3 任务执行快照（平台事实）

任务创建或启动时，应把用户当时选择的服务器侧组合固化为任务事实的一部分。

建议至少记录：

- `selected_execution_profile_id`
- `selected_agent_backend`
- `selected_model`

单次 `ai_execution` 再记录实际命中的执行事实：

- `provider`
- `model`
- `credential_ref`
- `retry_attempt`
- `switched_credential`

这样可以区分：

- 用户选的是 `Codex CLI / gpt-5.4`
- 实际第一次用了凭据 A，失败后切到凭据 B

## 6. 执行流程

## 6.1 创建任务

用户在前端选择“服务器模式”后：

1. 前端请求服务器侧可用执行配置列表
2. 用户选择一个 `CLI + 模型` 组合
3. 前端创建平台任务时，提交该组合对应的 `execution_profile_id`
4. `web` 在任务主链中记录用户选择快照

### 6.2 启动与执行

任务进入执行阶段后：

1. `web` 根据 `execution_profile_id` 读取该组合下可用凭据池项
2. 按优先级选择第一组健康凭据
3. 向服务器侧 `workstation` 下发内部执行请求
4. `workstation` 使用解析后的最终参数执行
5. `web` 记录本次 `ai_execution` 的实际命中信息

## 7. 自动切换与重试规则

### 7.1 允许自动切换的前提

第一版仅在同组合内允许自动切换一次，且只对以下错误类型生效：

- `401/403`
- 额度耗尽
- 限流
- 上游临时不可用
- 连接失败

### 7.2 不允许的行为

第一版明确禁止：

- 从 `Codex CLI` 自动切到 `Claude CLI`
- 从 `gpt-5.4` 自动切到其他模型
- 在未记录审计信息的情况下隐式更换凭据
- 对任意未知异常都盲目反复重试

### 7.3 审计要求

每次自动切换至少要记录：

- 首次命中的 `credential_ref`
- 是否发生自动切换
- 切换后的 `credential_ref`
- 触发切换的错误分类
- 最终执行结果

## 8. 对现有接口与平台主链的影响

本草案初步判断，后续实现将至少影响以下几个层面：

- 前端服务器模式选择弹窗与创建任务流程
- 平台任务创建 contract
- 平台后台执行配置列表接口
- 平台 `job / job_item / ai_execution` 记录结构
- `web -> server workstation` 的内部执行请求结构
- 管理员审计与执行明细视图

当前仍不建议直接把现有工作站全局 `llm` 配置硬扩成平台服务器凭据池，因为二者职责不同：

- 工作站 `llm` 配置更接近单机运行时真源
- 服务器凭据池属于平台能力配置和内部调度范畴

## 9. 用户选择配置问题

当前已确认用户在服务器模式下直接选择 `CLI + 模型`。但具体交互仍需进一步讨论，重点包括：

### 9.1 选择入口

当前倾向：

- 用户先选择 `本机执行` 或 `服务器模式`
- 只有在选择 `服务器模式` 后，才展示服务器侧 `CLI + 模型` 列表

进一步确认后的推荐主流程为：

1. 用户先选择 `服务器模式`
2. 系统读取该用户的全局默认服务器配置
3. 若默认配置存在且当前可用，则直接用于本次任务
4. 若默认配置不存在，或已下线 / 不可用，则弹出服务器配置选择窗口
5. 用户本次选择完成后：
   - 当前任务立即使用该 `execution_profile_id`
   - 同步更新该用户的全局默认服务器配置

### 9.1.1 默认配置规则

截至当前讨论，默认配置规则已进一步收口为：

- 每个用户全局只记一个默认服务器配置
- 默认配置保存在服务端用户设置，不以浏览器本地缓存作为真源
- 设置页与运行时弹窗写入同一份后端字段，避免双真源
- 即使存在默认配置，用户在任务发起时仍应始终可以手动更换

建议最小字段：

- `default_execution_profile_id`
- `default_execution_profile_updated_at`

当前建议该默认配置不要直接挂入认证用户主表，而是落在独立的 `user_platform_preferences` 表中。

推荐原因：

- 默认服务器配置属于平台偏好，不属于认证身份事实
- 避免把 `auth users` 扩展成杂项配置表
- 后续若增加更多平台偏好字段，独立表更易演进
- 平台模块可独立维护，不必反向污染认证域边界

### 9.2 选择展示信息

第一版建议用户只看到以下信息：

- 组合名称，如 `Codex CLI / gpt-5.4`
- 一句简短说明
- 是否推荐
- 当前是否可用

第一版暂不默认展示：

- 底层 provider
- 具体凭据数量
- 凭据健康细节
- 自动切换细节

### 9.3 选择结果的提交方式

当前推荐由前端提交：

- `execution_profile_id`

而不是直接提交：

- `provider`
- `api_key`
- `base_url`

这样可以让前端交互保持稳定，同时把后台调整凭据池、轮换 key 的自由度留在服务端。

## 9.4 设置页信息架构调整建议

当前设置页已经包含：

- 项目路径与工具配置
- 知识库状态与维护动作
- 本机 LLM 配置
- 图像生成配置

若继续直接在当前长表单中增加“服务器模式默认配置”，设置界面会进一步变重。因此当前讨论建议同步把设置页改成顶部横向 Tabs。

### 9.4.1 第一版推荐 Tabs

第一版建议收口为以下 5 个 Tab：

- `项目与工具`
- `本机 AI`
- `服务器模式`
- `图像与后处理`
- `知识库`

拆分原则如下：

- `本机 AI` 仅表达本机工作站 / BYOK 配置
- `服务器模式` 仅表达用户在平台侧的默认服务器配置偏好
- `知识库` 作为运行时维护页独立存在，不继续夹在普通配置表单中间
- `图像与后处理` 保持独立，避免与文本 / 代码设置混杂

### 9.4.2 为什么推荐顶部横向 Tabs

当前讨论已确认第一版使用顶部横向 Tabs，而不是侧边导航，原因包括：

- 与现有设置页改造成本更匹配
- 当前分类数量控制在 5 个以内，适合顶部切换
- 对抽屉模式和页面模式都更容易统一

## 9.5 服务器模式 Tab 设计草案

### 9.5.1 目标

`服务器模式` Tab 的职责是：

- 展示当前默认服务器配置
- 提供更换和清除动作
- 说明默认配置与运行时弹窗之间的关系

该 Tab 第一版不承担以下职责：

- 展开全部可选 `CLI + 模型` 列表
- 展示 provider 细节
- 展示服务器凭据池状态

### 9.5.2 固定说明文案

建议在 `服务器模式` Tab 顶部固定显示一段简短说明：

- 本页只管理服务器模式的默认选择
- 发起任务时若默认配置可用，将优先直接使用
- 若默认配置不存在或不可用，运行时会弹窗要求重新选择
- 本页不展示服务器凭据细节

### 9.5.3 三态卡片

当前讨论建议把 `服务器模式` Tab 的主体收口成一张状态卡片，并固定为以下三种状态：

#### 状态 A：未设置默认服务器配置

- 标题：`未设置默认服务器配置`
- 说明：首次使用服务器模式时会弹出配置选择窗口，也可以现在先设置一个默认组合
- 主操作：`选择默认配置`

#### 状态 B：默认服务器配置可用

- 标题：`当前默认服务器配置`
- 主信息：展示当前组合名称，例如 `Codex CLI / gpt-5.4`
- 说明：服务器模式下会默认使用该组合，但用户在任务发起时仍可手动更换
- 主操作：`更换配置`
- 次操作：`清除默认配置`

#### 状态 C：默认服务器配置已不可用

- 标题：`默认服务器配置已不可用`
- 主信息：若可取得历史记录，则展示上次保存的组合名称
- 说明：该组合当前已下线或暂不可用，需要重新选择
- 主操作：`重新选择配置`
- 次操作：`清除默认配置`

### 9.5.4 与运行时弹窗的关系

当前建议的边界如下：

- 设置页负责“查看和管理默认值”
- 运行时弹窗负责“当前任务具体要用哪个服务器配置”

在默认配置存在且可用时：

- 运行时可以直接复用默认配置
- 但应始终保留“更换配置”入口

在默认配置不存在或不可用时：

- 运行时必须弹出配置选择窗口
- 用户选择成功后，除用于当前任务外，还要同步更新默认值

### 9.5.5 选择器复用原则

当前讨论已进一步确认：

- 设置页中的“更换配置”直接复用运行时同一套服务器配置选择弹窗
- 不再为设置页单独实现另一套内嵌选择器

这样收口后的边界为：

- 服务器配置选择弹窗是唯一的 `CLI + 模型` 选择器
- 运行时入口与设置页入口共用：
  - 同一份可选列表
  - 同一份推荐标记
  - 同一份可用状态
  - 同一份说明文案

两种调起来源的差异只体现在提交后的动作：

- `运行时选择`
  - 当前任务使用所选 `execution_profile_id`
  - 同步更新默认服务器配置
- `设置页更换默认配置`
  - 仅更新默认服务器配置
  - 不直接创建任务

该原则的价值在于：

- 避免设置页与运行时出现两套不一致的配置列表
- 避免后续维护两套展示逻辑
- 降低前端实现与联调成本

## 10. 当前开放问题

在进入正式实现前，仍需继续确认以下问题：

1. 用户选择服务器配置时，列表是否需要按“推荐优先”排序
2. 用户是否需要看到更强的说明文案，例如“适合复杂代码修改”“更稳定但更慢”
3. 若某个组合临时不可用，前端是灰显、隐藏，还是展示不可用原因
4. `execution_profile_id` 应落在 `job`、`job_item` 还是两者都落

## 11. 实现前最小确认清单

若后续要把本草案推进到正式实现，建议至少先冻结以下事项：

1. 服务器配置列表的排序规则是否采用“推荐优先”
2. 列表项是否展示额外说明文案
3. 不可用组合在前端的表现方式
4. 默认服务器配置字段的存放位置与接口归属
5. `execution_profile_id` 与执行快照的最终落库位置
6. 自动切换失败后的用户可见错误文案
7. 服务器配置选择弹窗的最终字段与展示结构

## 11.1 用户侧 API 草案一：获取服务器配置列表

当前建议优先冻结的第一个用户侧接口为：

- `GET /api/platform/execution-profiles`

### 11.1.1 用途

该接口用于以下场景共用：

- 运行时服务器配置选择弹窗
- 设置页 `服务器模式` Tab 中的“更换配置”
- 读取当前普通用户可选择的 `CLI + 模型` 组合列表

### 11.1.2 请求

第一版建议：

- 无请求体
- 暂不增加筛选参数
- 默认返回当前用户可见的全部可选组合

### 11.1.3 响应草案

```json
{
  "items": [
    {
      "id": 1,
      "display_name": "Codex CLI / gpt-5.4",
      "agent_backend": "codex",
      "model": "gpt-5.4",
      "description": "适合复杂代码修改，默认推荐。",
      "recommended": true,
      "available": true
    },
    {
      "id": 2,
      "display_name": "Claude CLI / claude-sonnet-4-6",
      "agent_backend": "claude",
      "model": "claude-sonnet-4-6",
      "description": "适合通用代码生成与分析。",
      "recommended": false,
      "available": true
    }
  ]
}
```

### 11.1.4 字段说明

- `id`
  - 前端提交任务和保存默认服务器配置时使用的主键
- `display_name`
  - 前端主展示名称，不要求前端自行拼接文案
- `agent_backend`
  - 轻量表达 CLI 类型；当前第一版建议冻结为 `codex | claude`
- `model`
  - 当前组合对应的模型名
- `description`
  - 给用户看的简短说明
- `recommended`
  - 是否推荐，用于前端排序和标记
- `available`
  - 当前是否允许用户选择；若为 `false`，前端可灰显或提示不可用

### 11.1.5 第一版不返回的字段

当前建议以下字段不进入普通用户接口响应：

- `provider`
- `credential_count`
- `health_status`
- `base_url`
- 任何 key 相关字段

原因是这些都属于后台实现细节，不应暴露给普通用户选择层。

### 11.1.6 返回规则建议

当前建议第一版按以下规则返回：

- 只返回 `enabled = true` 的 `execution_profile`
- `available` 由后端根据该组合下是否存在健康可用凭据综合判断
- 默认排序规则建议为：
  1. `recommended desc`
  2. `sort_order asc`
  3. `id asc`

### 11.1.7 后续计划

围绕该接口，当前建议的后续推进顺序如下：

1. 先冻结响应字段，不继续扩散到底层 provider 或凭据细节
2. 再冻结 `available` 的后端判定规则
3. 再让设置页与运行时弹窗统一改为消费该接口
4. 最后根据交互反馈决定是否增加筛选参数或更丰富的说明文案

### 11.1.8 风险

该接口的主要风险包括：

- 若 `available` 的判定依赖实时探测，上线后容易变慢并放大服务器压力
- 若响应中暴露过多后台字段，会破坏用户选择层与管理员配置层的边界
- 若设置页和运行时最终没有共用这同一个接口，后续容易出现配置列表不一致
- 若 `description` 文案没有约束，前端不同入口可能对同一组合解释不一致

## 11.2 用户侧 API 草案二：读取当前用户默认服务器配置

当前建议优先冻结的第二个用户侧接口为：

- `GET /api/me/server-preferences`

### 11.2.1 用途

该接口用于以下场景共用：

- 设置页 `服务器模式` Tab 读取当前默认服务器配置
- 运行时选择服务器模式时，判断是否可直接复用默认配置
- 判断当前默认配置是否仍然可用，决定是否需要弹出选择窗口

### 11.2.2 请求

第一版建议：

- 无请求体
- 必须要求当前用户已登录
- 只返回当前会话用户自己的默认服务器配置

### 11.2.3 响应草案

```json
{
  "default_execution_profile_id": 1,
  "display_name": "Codex CLI / gpt-5.4",
  "agent_backend": "codex",
  "model": "gpt-5.4",
  "available": true,
  "updated_at": "2026-04-13T16:20:00Z"
}
```

当用户尚未设置默认服务器配置时，第一版建议返回：

```json
{
  "default_execution_profile_id": null,
  "display_name": "",
  "agent_backend": "",
  "model": "",
  "available": false,
  "updated_at": null
}
```

### 11.2.4 字段说明

- `default_execution_profile_id`
  - 当前用户默认服务器配置的主键；为空表示尚未设置
- `display_name`
  - 当前默认配置的前端展示名称
- `agent_backend`
  - 当前默认配置对应的 CLI 类型
- `model`
  - 当前默认配置对应的模型
- `available`
  - 当前默认配置是否仍可直接使用
- `updated_at`
  - 默认配置最后一次更新的时间

### 11.2.5 第一版不返回的字段

当前建议以下字段不进入该用户接口响应：

- provider
- base_url
- credential_ref
- 凭据数量
- 检测失败详情
- 任何 key 相关字段

原因是该接口只负责“读取当前用户默认服务器配置”，不负责暴露后台凭据细节。

### 11.2.6 返回规则建议

当前建议第一版按以下规则处理：

- 若用户未登录，返回认证失败
- 若用户已登录但从未设置默认服务器配置，返回空对象语义，而不是 404
- `available` 由后端根据该默认配置当前是否仍存在且可用来判定
- 若 `default_execution_profile_id` 已存在，但组合已下线或暂不可用：
  - 保留 `default_execution_profile_id`
  - 尽量返回历史 `display_name`
  - `available = false`

### 11.2.7 后续计划

围绕该接口，当前建议的后续推进顺序如下：

1. 先冻结空值语义，避免前后端对“未设置”采用不同表达
2. 再冻结 `available` 的判定规则和失效配置的返回口径
3. 再让设置页 `服务器模式` Tab 和运行时入口统一消费该接口
4. 最后再决定是否需要补充更丰富的失效说明字段

### 11.2.8 风险

该接口的主要风险包括：

- 若“未设置默认配置”与“默认配置失效”没有明确区分，前端会难以正确展示三态卡片
- 若失效配置直接返回 404，前端会丢失历史配置名称，用户体验会很差
- 若该接口混入后台凭据细节，会破坏用户偏好层与管理员配置层边界
- 若设置页和运行时不共用该接口，默认值判断很容易出现分叉

## 11.3 用户侧 API 草案三：更新当前用户默认服务器配置

当前建议优先冻结的第三个用户侧接口为：

### 11.3.1 用途

该接口用于以下场景共用：

- 设置页 `服务器模式` Tab 中更新默认服务器配置
- 运行时弹窗完成选择后，同步写回当前用户默认值
- 清除当前用户默认服务器配置

### 11.3.2 请求草案

第一版建议请求体如下：

```json
{
  "default_execution_profile_id": 1
}
```

清除默认服务器配置时，建议请求体如下：

```json
{
  "default_execution_profile_id": null
}
```

### 11.3.3 响应草案

更新成功后，建议直接返回最新默认服务器配置快照：

```json
{
  "default_execution_profile_id": 1,
  "display_name": "Codex CLI / gpt-5.4",
  "agent_backend": "codex",
  "model": "gpt-5.4",
  "available": true,
  "updated_at": "2026-04-13T16:35:00Z"
}
```

清除成功后，建议返回：

```json
{
  "default_execution_profile_id": null,
  "display_name": "",
  "agent_backend": "",
  "model": "",
  "available": false,
  "updated_at": "2026-04-13T16:40:00Z"
}
```

### 11.3.4 校验规则建议

当前建议第一版至少做以下校验：

- 用户必须已登录
- `default_execution_profile_id` 若不为空，必须指向一个存在的 `execution_profile`
- 目标 `execution_profile` 必须对普通用户可见
- 目标 `execution_profile` 若当前不可用，可选择拒绝写入，或允许保存但返回 `available = false`

当前更推荐第一版采用：

- **允许保存已存在但当前不可用的 profile，仅返回 `available = false`**

原因是这样更符合“用户记住默认值，但运行时发现失效后再要求重选”的产品语义，也更便于前端展示“默认配置已失效”的状态。

### 11.3.5 第一版不承担的职责

该接口第一版不负责：

- 创建任务
- 触发服务器执行
- 返回 provider 级别的底层细节
- 返回凭据健康详情
- 返回管理员侧检测结果

它只负责“更新当前用户默认服务器配置”。

### 11.3.6 后续计划

围绕该接口，当前建议的后续推进顺序如下：

1. 先冻结“设置默认值”和“清除默认值”的请求语义
2. 再冻结对不可用 profile 的写入策略
3. 再让设置页和运行时选择器统一调用该接口
4. 最后根据真实交互反馈决定是否补充更明确的结果提示字段

### 11.3.7 风险

该接口的主要风险包括：

- 若设置页和运行时分别调用不同接口写默认值，后续很容易出现双真源
- 若对不可用 profile 的写入策略不统一，前后端会对“可保存但不可直接使用”产生理解偏差
- 若清除默认值没有标准语义，前端会出现一部分入口用空串，一部分入口用 `null` 的混乱情况
- 若更新成功后不直接返回最新快照，前端还要额外补一次读取请求，容易增加请求量和状态抖动

## 11.4 管理员侧 API 草案一：查看服务器凭据列表

当前建议优先冻结的第一个管理员侧接口为：

- `GET /api/admin/server-credentials`

### 11.4.1 用途

该接口用于管理员后台查看当前服务器凭据列表及其基本状态，支撑以下场景：

- 查看所有服务器凭据
- 查看某个 `execution_profile` 下挂载了哪些凭据
- 查看凭据当前是否启用、是否健康、最近一次检测情况

### 11.4.2 请求建议

- `execution_profile_id`：可选，仅查看某个组合下的凭据
- `enabled`：可选，筛选启用 / 禁用状态
- `health_status`：可选，筛选健康状态

### 11.4.3 响应草案

```json
{
  "items": [
    {
      "id": 101,
      "execution_profile_id": 1,
      "execution_profile_display_name": "Codex CLI / gpt-5.4",
      "provider": "openai",

      "priority": 10,
      "enabled": true,
      "health_status": "healthy",
      "last_checked_at": "2026-04-13T16:45:00Z",
      "last_error_code": "",
      "last_error_message": ""
    }
  ]
}
```

### 11.4.4 字段说明

- `id`
  - 凭据记录主键
- `execution_profile_id`
  - 该凭据挂载到哪个用户可见组合
- `execution_profile_display_name`
  - 方便管理员直接识别组合名称
- `provider`
  - 底层服务商标识，仅管理员侧可见
- `label`
  - 管理员自定义备注名称
- `masked_credential`
  - 掩码后的 key 标识，用于人工识别，不回显原始值
- `base_url`
  - 当前凭据对应的服务地址
- `priority`
  - 同组合内优先级
- `enabled`
  - 是否启用
- `health_status`
  - 当前健康状态
- `last_checked_at`
  - 最近一次主动 / 被动检测时间
- `last_error_code`
  - 最近一次错误分类
- `last_error_message`
  - 最近一次错误摘要

### 11.4.5 第一版不返回的字段

当前建议以下字段不在管理员列表接口中返回：

- 原始 `credential`
- 原始 `secret`
- 密文字段
- 完整检测历史明细

原因是列表接口应保持“可巡检、可判断、不可泄密”的边界。

### 11.4.6 返回规则建议

当前建议第一版按以下规则返回：

- 仅管理员可访问
- 列表接口默认按：
  1. `execution_profile_id asc`
  2. `priority desc`
  3. `id asc`
  排序
- `masked_credential` 必须由后端统一生成，前端不得自行处理
- `last_error_message` 建议为可读摘要，不直接暴露原始 traceback

### 11.4.7 后续计划

围绕该接口，当前建议的后续推进顺序如下：

1. 先冻结管理员侧最小列表字段
2. 再冻结 `health_status` 的枚举口径
3. 再补单个凭据详情页或检测历史接口
4. 最后视后台使用情况再考虑增加更复杂筛选与分页

### 11.4.8 风险

该接口的主要风险包括：

- 若原始 key 或密文字段意外进入响应，将直接造成严重安全问题
- 若 `health_status` 与检测记录口径不稳定，管理员会误判可用性
- 若列表接口返回过重字段或做实时探测，会增加后台巡检成本和服务器压力
- 若 `masked_credential` 规则不统一，管理员难以准确识别是哪一把 key

## 11.5 管理员侧 API 草案二：新增服务器凭据

该接口用于管理员新增一条服务器凭据，并将其挂载到某个 `execution_profile`。

典型场景包括：

- 为已有 `CLI + 模型` 组合补一把新 key
- 新增同组合下的冗余凭据，用于同组合自动切换
- 新增不同 provider 但服务于同一组合的后台能力

### 11.5.2 请求草案

```json
{

  "auth_type": "api_key",

  "label": "主 OpenAI Key",
  "priority": 10,
  "enabled": true

```

字段说明建议如下：

- `execution_profile_id`
  - 该凭据挂载到哪个 `CLI + 模型` 组合
- `provider`
  - 底层服务商标识
- `auth_type`
  - 鉴权方式，例如 `api_key`、`ak_sk`
- `credential`
  - 主凭据；创建时明文提交，后端必须加密保存
- `secret`
  - 辅助密钥；如不需要则为空
- `base_url`
  - 服务地址
- `label`
  - 管理员备注
- `priority`
  - 同组合内优先级
- `enabled`
  - 创建后是否立即启用

创建成功后，建议返回脱敏后的凭据摘要：

```json
{

  "execution_profile_id": 1,
  "execution_profile_display_name": "Codex CLI / gpt-5.4",
  "provider": "openai",
  "label": "主 OpenAI Key",
  "masked_credential": "****abcd",
  "base_url": "https://api.openai.com/v1",
  "priority": 10,
  "enabled": true,
  "health_status": "unknown",
  "last_checked_at": null,
  "last_error_code": "",
  "last_error_message": ""
}
```

### 11.5.4 校验规则建议

当前建议第一版至少做以下校验：

- 仅管理员可访问
- `execution_profile_id` 必须存在
- `provider` 与 `auth_type` 必须在允许范围内
- `credential` 不能为空
- 若 `auth_type = ak_sk`，则 `secret` 不能为空
- `base_url` 如存在，必须为合法 URL
- `priority` 必须为整数并落在允许范围内

### 11.5.5 安全要求

该接口第一版建议明确以下安全要求：

- 原始 `credential` 与 `secret` 只允许出现在请求体中
- 后端必须在写库前完成加密
- 响应体不得回显原始 `credential` 或 `secret`
- 日志中不得打印原始凭据

### 11.5.6 第一版不承担的职责

该接口第一版不负责：

- 自动创建新的 `execution_profile`
- 自动触发任务执行
- 自动暴露给普通用户
- 返回完整检测历史

创建完成后若需要检测，可由管理员显式调用检测接口。

### 11.5.7 后续计划

围绕该接口，当前建议的后续推进顺序如下：

1. 先冻结请求字段和响应脱敏规则
2. 再冻结 provider / auth_type 枚举
3. 再补后端加密存储实现
4. 最后根据使用情况决定是否增加“创建后立即检测”的可选能力

### 11.5.8 风险

该接口的主要风险包括：

- 若日志或响应误回显原始凭据，会造成高风险泄露
- 若 `execution_profile_id` 校验不严，可能把凭据挂到错误组合
- 若创建后默认直接启用但未做任何检测，坏 key 可能立即进入可调度集合
- 若 provider / auth_type 不先收口，后续实现与前端管理页很容易分叉

## 11.6 管理员侧 API 草案三：手动检测服务器凭据

当前建议优先冻结的第三个管理员侧接口为：

- `POST /api/admin/server-credentials/{id}/check`

### 11.6.1 用途

该接口用于管理员主动触发某把服务器凭据的一次健康检测。

典型场景包括：

- 新增凭据后立即检测是否可用
- 某把凭据近期失败较多，管理员手动复核
- 某个组合显示不可用时，定位是哪把 key 出问题

### 11.6.2 请求建议

第一版建议：

- 使用路径参数 `id` 指定待检测凭据
- 无请求体
- 仅管理员可访问

若后续需要补充更细的检测模式，再考虑增加请求体字段，例如“快速检测 / 完整检测”。

### 11.6.3 响应草案

第一版建议返回本次检测结果摘要：

```json
{
  "credential_id": 102,
  "status": "healthy",
  "error_code": "",
  "error_message": "",
  "started_at": "2026-04-13T16:55:00Z",
  "finished_at": "2026-04-13T16:55:03Z"
}
```

若检测失败，建议返回：

```json
{
  "credential_id": 102,
  "status": "auth_failed",
  "error_code": "http_401",
  "error_message": "鉴权失败，请检查该凭据是否失效。",
  "started_at": "2026-04-13T16:56:00Z",
  "finished_at": "2026-04-13T16:56:02Z"
}
```

### 11.6.4 检测结果状态建议

当前第一版建议至少支持以下结果：

- `healthy`
- `degraded`
- `rate_limited`
- `auth_failed`
- `quota_exhausted`
- `unreachable`

这些状态既可用于管理员手动检测，也可与运行时被动健康更新复用。

### 11.6.5 行为规则建议

当前建议第一版按以下规则执行：

- 检测完成后必须写入 `credential_health_checks`
- 同步更新对应 `server_credentials.health_status`
- 若检测结果为明确不可用状态，应影响后续调度
- `error_message` 应为可读摘要，不直接回传原始 traceback

### 11.6.6 第一版不承担的职责

该接口第一版不负责：

- 自动修复凭据
- 自动切换其他凭据
- 自动触发全组合批量检测
- 直接返回完整诊断日志

它只负责“检测当前这一把凭据，并更新健康状态和检测记录”。

### 11.6.7 后续计划

围绕该接口，当前建议的后续推进顺序如下：

1. 先冻结检测状态枚举
2. 再冻结最小探针策略和超时规则
3. 再补 `credential_health_checks` 写入与 `health_status` 更新
4. 最后根据后台使用情况决定是否增加批量检测或异步检测能力

### 11.6.8 风险

该接口的主要风险包括：

- 若检测本身过重或过慢，会增加管理员后台操作压力
- 若检测结果直接依赖一次偶发网络状态，可能误判凭据健康度
- 若检测成功 / 失败后不及时更新 `health_status`，调度层和管理层会看到不一致状态
- 若错误摘要过于技术化或直接暴露底层异常，管理员体验和安全边界都会变差

## 12. 服务器模式抗压第一版必做清单

若后续开始正式实现，以下事项建议作为服务器模式抗压第一版的最低防线，优先级高于多节点扩容、复杂调度和全量流式体验优化。

### 12.1 任务必须先入队再执行

- `web` 负责创建任务、开始任务、查询状态、记录事件
- 重执行不应由 HTTP 请求直接承担完整生命周期
- 平台状态至少明确区分 `draft`、`queued`、`running`

### 12.2 后台执行器必须做资源池限流

第一版至少建议拆分以下资源池：

- `codegen_pool`
- `image_pool`
- `build_pool`
- `deploy_pool`

原则是不同类型的重任务不能共用一个粗粒度全局并发数，避免某一类任务爆发时拖垮其他能力。

### 12.3 单用户并发上限必须存在

- 限制同一用户同时运行中的服务器任务数
- 超限时不应继续直接占用执行资源
- 可根据实现阶段选择“拒绝创建”“允许创建但保持排队”“拒绝开始”等策略，但必须有明确上限

### 12.4 创建任务与开始任务接口必须限流

至少建议覆盖：

- 创建任务
- 开始任务
- 更换服务器配置相关接口

目的在于防止重复点击、脚本刷接口与短时间洪峰直接冲击后端。

### 12.5 凭据池必须具备健康状态与快速熔断

- 同组合内允许自动切换一次
- 坏凭据不能被反复命中
- 对 `401/403`、额度耗尽、限流、连接失败、上游临时不可用等错误，必须能及时更新凭据健康状态

### 12.6 任务状态查询必须保持轻量

- 列表与详情接口只读取平台主链已聚合结果
- 查询接口不应顺手做重计算、实时探测或执行判断
- 前端第一版优先采用低频轮询，而不是高频实时流式拉取

### 12.7 系统繁忙时必须提供明确降级反馈

至少要覆盖以下场景：

- 资源池已满
- 排队时间过长
- 默认服务器配置失效
- 当前组合暂不可用

要求是向用户返回明确的 `queued` / `繁忙中` / `需重新选择配置` 等反馈，而不是长时间无响应或模糊超时。

### 12.8 执行审计信息必须最小可追溯

第一版至少应记录：

- 用户选择的 `execution_profile_id`
- 实际命中的 `credential_ref`
- 是否发生自动切换
- 切换后的 `credential_ref`
- 最终失败分类

否则在并发压力和重试场景下，平台将无法有效排查问题来源。

### 12.9 设置页与运行时必须共用同一套服务器配置来源

- 设置页和运行时弹窗共用同一套服务器配置选择器
- 共用同一份可选列表、推荐标记、可用状态和说明文案

该项不仅是交互一致性要求，也属于抗压要求的一部分，因为它能避免两套列表和两套判定逻辑造成额外请求、额外缓存分歧和额外排障成本。

### 12.10 文档与运维口径必须同步

正式实现前，至少应同步明确：

- 哪些接口属于轻接口
- 哪些操作会进入排队
- 哪些状态表示排队、繁忙、失败或需要重新选择配置

否则在并发场景下，前端、后端与运维之间会快速出现口径漂移。

### 12.11 可后置的增强项

以下能力有价值，但不属于第一版抗压最低防线，可在流量上来后再逐步补齐：

- 多节点 worker / 多台服务器执行器
- 更复杂的调度策略
- SSE / WebSocket 全面流式化
- 自动扩缩容
- 套餐分层优先级
- 更复杂的监控与看板

## 13. 管理员配置服务器凭据与状态检测

除普通用户侧的“选择 `CLI + 模型`”之外，服务器模式还需要一组明确的管理员侧能力，用于维护平台真实可用的服务器凭据。

## 13.1 目标

管理员侧至少需要支持以下事情：

- 添加服务器使用的 key / 凭据
- 把凭据挂载到某个 `CLI + 模型` 组合
- 启用 / 禁用某把凭据
- 检测某把凭据当前是否可用
- 查看每把凭据的最近状态和错误原因

该能力应属于平台后台配置能力，而不是普通用户设置能力。

## 13.2 当前前提与风险

结合当前代码现状，管理员凭据管理在正式实现前必须先注意以下前提：

- 当前用户模型尚未形成完整的管理员角色字段
- 当前 `/admin` 路由已经存在，但当前实现更偏查询入口，不应直接默认视为已具备完整管理员鉴权

因此，后续若进入正式实现，建议把“管理员权限边界补齐”视为本能力的前置条件，而不是在现有管理员接口上直接裸扩。

## 13.3 建议拆分的管理员能力

### 13.3.1 服务器凭据管理

管理员应能：

- 添加凭据
- 编辑凭据备注、provider、base_url、优先级
- 启用 / 禁用凭据
- 绑定到某个 `execution_profile`

安全要求：

- 原始 key 只允许在创建 / 更新时提交
- 创建后不回显原始 key
- 列表与详情只显示掩码和元数据

### 13.3.2 凭据状态检测

管理员应能：

- 手动检测单个凭据
- 批量检测某个 `CLI + 模型` 组合下的全部凭据
- 查看最近一次检测状态、错误分类和检测时间

检测结果可至少分为：

- `healthy`
- `degraded`
- `rate_limited`
- `auth_failed`
- `quota_exhausted`
- `disabled`

### 13.3.3 组合绑定管理

管理员应能：

- 查看某个 `execution_profile` 当前挂载了哪些凭据
- 调整同组合内凭据优先级
- 判断该组合当前是否还有健康可用凭据

这项能力直接关系到“同组合内自动切换一次”是否能正常工作。

## 13.4 建议的数据模型

若后续进入实现，建议最少拆成以下四类数据：

### 13.4.1 `execution_profiles`

职责：

- 表达用户可见的 `CLI + 模型` 能力组合

### 13.4.2 `server_credentials`

职责：

- 表达管理员维护的真实服务器凭据
- 存储 provider、密文凭据、base_url、优先级、启用状态、健康状态等

### 13.4.3 `credential_health_checks`

职责：

- 记录每次主动或被动检测结果
- 为凭据熔断、恢复和排障提供依据

### 13.4.4 `user_platform_preferences`

职责：

- 保存用户在平台服务器模式下的默认偏好

当前第一版建议最少字段：

- `user_id`
- `default_execution_profile_id`
- `updated_at`

当前不建议把该信息直接塞入 `auth users`，原因已在 `9.1.1 默认配置规则` 中说明。

## 13.5 检测方式建议

第一版建议采用“被动检测 + 主动检测”两层结合：

### 13.5.1 被动检测

来源：

- 实际执行时的运行结果

行为：

- 若命中凭据执行失败，按错误类型更新凭据健康状态
- 为自动切换和熔断提供第一手信号

### 13.5.2 主动检测

来源：

- 管理员手动点击检测
- 后续可扩展为定时巡检

行为：

- 通过最小探针请求验证凭据是否有效
- 检查 base_url 是否连通
- 尝试识别是否处于限流、额度耗尽或鉴权失败状态

第一版当前建议先做：

- 手动检测
- 运行时失败后的被动健康更新

定时巡检可以作为后续增强项。

## 13.6 第一版必须同批具备的能力

若要上线管理员凭据管理，当前建议以下事项属于同批最低要求：

- 管理员权限边界补齐
- 管理员权限模型
- 管理员接口鉴权
- 凭据密文存储
- 原始 key 不回显
- 启用 / 禁用开关
- 健康状态字段
- 手动检测接口
- 检测日志记录

## 13.7 可后置的增强项

以下能力有价值，但可以在流量和管理需求进一步上升后补齐：

- 定时自动巡检
- 更复杂的健康评分
- provider 维度统计看板
- 更复杂的自动摘除 / 自动恢复策略
- 成本与用量分析面板

## 13.8 管理界面建议

后续若补管理员页面，建议与普通用户配置完全隔离，至少拆成两块：

- `服务器配置`
  - 管理 `CLI + 模型` 组合
- `服务器凭据`
  - 管理具体 key、状态、检测与优先级

这样可以避免把“用户选择的组合”和“管理员维护的真实 key”混在同一套界面里。

## 13.9 与用户侧交互的边界

管理员维护的服务器凭据属于后台私有能力，普通用户侧不应直接感知：

- 原始 key
- 单把凭据状态细节
- 凭据数量
- provider 级别的路由细节

普通用户仅需要感知：

- 某个 `CLI + 模型` 当前可用或不可用
- 当前默认服务器配置是否需要重新选择

## 13.10 当前建议的推进顺序

若后续要推进管理员凭据能力，建议顺序如下：

1. 先补管理员权限边界
2. 再冻结 `execution_profile / server_credentials / credential_health_checks` 数据模型
3. 再补管理员接口
4. 最后再实现管理员页面

这里的“补管理员权限边界”应明确包括：

- 在用户模型或权限模型中补足管理员能力表达
- 为管理员接口补齐强制鉴权，而不是默认沿用当前普通用户会话能力
- 在文档中明确哪些接口属于管理员专用接口

## 13.11 任务快照字段落点建议

关于 `execution_profile_id` 在平台主链中的落点，当前讨论建议第一版优先落在 `job`，而不是 `job_item`。

建议理由如下：

- 当前用户交互语义是“发起一次服务器任务时，选择一个 `CLI + 模型`”
- 当前并未要求同一任务内的不同 `job_item` 分别选择不同配置
- 若第一版直接落到 `job_item`，会提前引入不必要的调度、审计和前端展示复杂度

因此，当前推荐：

- `job` 记录用户选择快照：
  - `selected_execution_profile_id`
  - `selected_agent_backend`
  - `selected_model`
- `ai_execution` 记录实际命中事实：
  - `provider`
  - `model`
  - `credential_ref`
  - `retry_attempt`
  - `switched_credential`

第一版暂不建议在 `job_item` 上额外落 `execution_profile_id`。

## 13.12 已确认设计项的后续计划与风险

为避免前文只给出结论而缺少推进顺序与风险边界，当前对已确认的关键设计项补充如下。

### 13.12.1 用户默认服务器配置

对应结论：

- 每个用户全局只记一个默认服务器配置
- 默认配置落在 `user_platform_preferences`
- 设置页与运行时共用同一份默认值真源

后续计划：

1. 先冻结 `user_platform_preferences` 的最小字段
2. 再冻结读取 / 更新默认服务器配置接口
3. 再让设置页与运行时都改为消费同一份默认值
4. 最后补“默认配置失效”时的兜底交互

风险：

- 若默认配置继续散落在多个入口，本地缓存和服务端状态容易不一致
- 若把该字段直接塞进认证用户主表，后续平台偏好扩展会污染认证域
- 若不处理默认配置失效场景，运行时会出现静默失败或错误自动兜底

### 13.12.2 设置页 Tab 化与统一选择器

对应结论：

- 设置页改成顶部横向 Tabs
- 新增 `服务器模式` Tab
- 设置页“更换配置”与运行时复用同一套服务器配置选择弹窗

后续计划：

1. 先冻结 Tab 分类与 `服务器模式` Tab 的三态结构
2. 再抽出统一的服务器配置选择器组件
3. 让设置页入口与运行时入口共用同一组件和同一接口
4. 最后统一状态文案、不可用提示和推荐标记

风险：

- 若设置页与运行时继续维护两套选择器，列表和状态很容易分裂
- 若 Tab 分类不先冻结，后续新增服务器模式内容时设置页会再次变重
- 若三态文案不统一，用户会误解默认配置与当前任务选择的关系

### 13.12.3 管理员凭据管理与状态检测

对应结论：

- 管理员可以配置服务器用的 key
- 管理员可以对单个 key 或某个组合进行状态检测
- 原始 key 创建后不回显

后续计划：

1. 先补管理员权限边界
2. 再冻结 `server_credentials` 与 `credential_health_checks` 字段
3. 再实现管理员接口和手动检测链路
4. 最后补管理员页面与被动健康更新

风险：

- 若管理员权限边界未补齐就开放接口，风险最高
- 若密钥只做明文保存或可回显，会直接引入严重安全问题
- 若检测结果不落库，后续自动切换与熔断将缺少依据

### 13.12.4 任务快照字段落点

对应结论：

- `execution_profile_id` 第一版优先落在 `job`
- `ai_execution` 记录实际命中事实
- `job_item` 第一版不额外落 `execution_profile_id`

后续计划：

1. 先在 contracts 和表结构草案里冻结快照字段
2. 再补用户中心、管理员明细与审计接口的读取口径
3. 再让执行器把实际命中信息写回 `ai_execution`
4. 最后补自动切换后的审计展示

风险：

- 若快照字段落点摇摆不定，前后端会很难统一数据语义
- 若只记录用户选择、不记录实际命中，重试与切换后无法排障
- 若过早落到 `job_item`，会提前引入不必要的复杂度

### 13.12.5 同组合自动切换一次

对应结论：

- 同组合内失败自动切换一次
- 不跨 CLI、不跨模型、不跨组合
- 只对约定错误类型生效

后续计划：

1. 先冻结允许触发切换的错误分类
2. 再冻结凭据健康状态更新规则
3. 再补审计记录和用户可见错误文案
4. 最后根据真实运行数据决定是否需要更复杂策略

风险：

- 若错误分类不清，会出现误重试或漏重试
- 若缺少健康状态更新，坏凭据会被重复命中
- 若自动切换范围扩散，会让用户感知和审计边界失控

## 13.13 表结构草案一：`execution_profiles`

当前建议优先冻结的第一张核心表为：

- `execution_profiles`

该表表达用户可见的服务器配置组合，也就是用户在服务器模式下实际选择的 `CLI + 模型` 能力单元。

### 13.13.1 第一版推荐字段

- `id`
- `code`
- `display_name`
- `agent_backend`
- `model`
- `description`
- `enabled`
- `recommended`
- `sort_order`
- `created_at`
- `updated_at`

### 13.13.2 字段含义建议

- `id`
  - 主键
- `code`
  - 稳定内部标识，便于日志、配置与迁移
- `display_name`
  - 给前端直接展示的名称，例如 `Codex CLI / gpt-5.4`
- `agent_backend`
  - 表达 CLI 类型，例如 `codex`、`claude`
- `model`
  - 该组合对用户暴露的模型名
- `description`
  - 简短说明文案
- `enabled`
  - 是否在普通用户可见列表中展示
- `recommended`
  - 是否推荐
- `sort_order`
  - 前端排序辅助字段
- `created_at`
  - 创建时间
- `updated_at`
  - 最后更新时间

### 13.13.3 第一版边界建议

当前建议该表第一版只表达“用户可见的组合”，不直接承担以下职责：

- 保存真实 key
- 保存凭据健康状态细节
- 保存 provider 级别路由策略
- 保存用户默认值

这些职责分别由：

- `server_credentials`
- `credential_health_checks`
- `user_platform_preferences`

承担。

### 13.13.4 约束建议

当前建议第一版至少具备以下约束：

- `code` 唯一
- `display_name` 应保持稳定，避免频繁变更造成用户认知漂移
- `agent_backend` 取值范围先冻结
- `sort_order` 默认为整数

### 13.13.5 后续计划

围绕该表，当前建议的后续推进顺序如下：

1. 先冻结字段集合和唯一约束
2. 再冻结 `agent_backend` 枚举
3. 再由用户侧接口和设置页统一消费这张表
4. 最后再决定是否需要补更丰富的展示字段

### 13.13.6 风险

该表的主要风险包括：

- 若把 provider、健康状态等后台细节也塞进来，会破坏“用户可见组合”和“后台凭据池”边界
- 若 `display_name` 直接依赖前端拼接，后续不同入口文案可能不一致
- 若 `code` 不稳定，后续日志、接口和迁移很容易漂移
- 若 `enabled` 与 `available` 概念混淆，前后端会难以统一“下线”和“暂不可用”的语义

## 13.14 表结构草案二：`server_credentials`

当前建议优先冻结的第二张核心表为：

- `server_credentials`

该表表达管理员维护的真实服务器凭据，是服务器模式后台能力的核心事实之一。

### 13.14.1 第一版推荐字段

- `id`
- `execution_profile_id`
- `provider`
- `auth_type`
- `credential_ciphertext`
- `secret_ciphertext`
- `base_url`
- `label`
- `priority`
- `enabled`
- `health_status`
- `last_checked_at`
- `last_error_code`
- `last_error_message`
- `created_at`
- `updated_at`

### 13.14.2 字段含义建议

- `id`
  - 主键
- `execution_profile_id`
  - 该凭据挂载到哪个用户可见组合
- `provider`
  - 底层服务商标识
- `auth_type`
  - 鉴权方式，例如 `api_key`、`ak_sk`
- `credential_ciphertext`
  - 加密后的主凭据
- `secret_ciphertext`
  - 加密后的辅助密钥；如不需要可为空
- `base_url`
  - 服务地址
- `label`
  - 管理员备注名称
- `priority`
  - 同组合内优先级
- `enabled`
  - 是否启用
- `health_status`
  - 当前健康状态
- `last_checked_at`
  - 最近一次检测时间
- `last_error_code`
  - 最近一次错误分类
- `last_error_message`
  - 最近一次错误摘要
- `created_at`
  - 创建时间
- `updated_at`
  - 最后更新时间

### 13.14.3 第一版边界建议

当前建议该表第一版表达的是“后台真实凭据与当前健康摘要”，不直接承担以下职责：

- 保存完整检测历史
- 保存用户默认值
- 保存任务执行快照

这些职责分别由：

- `credential_health_checks`
- `user_platform_preferences`
- `job / ai_execution`

承担。

### 13.14.4 约束建议

当前建议第一版至少具备以下约束：

- `execution_profile_id` 外键关联 `execution_profiles`
- `priority` 为整数
- `provider` 与 `auth_type` 枚举值先冻结
- `credential_ciphertext` 必须非空
- `enabled = false` 的凭据不得进入可调度集合

### 13.14.5 安全边界建议

当前建议第一版明确以下边界：

- 表中只保存密文，不保存明文 key
- 原始 key 只允许在创建 / 更新请求中短暂出现
- `last_error_message` 只保存可读摘要，不保存敏感原文
- 任何普通用户接口都不直接读取这张表

### 13.14.6 后续计划

围绕该表，当前建议的后续推进顺序如下：

1. 先冻结字段集合与安全边界
2. 再冻结 `provider / auth_type / health_status` 枚举
3. 再补后端加密存储与解密使用链路
4. 最后再根据调度需要考虑是否增加更细的路由辅助字段

### 13.14.7 风险

该表的主要风险包括：

- 若明文 key 进入数据库或日志，会造成严重安全问题
- 若 `health_status` 更新不及时，调度层会频繁命中坏凭据
- 若 `last_error_message` 直接保存底层异常细节，可能暴露敏感信息
- 若该表和 `execution_profiles` 边界不清，后续会混淆“用户可见组合”和“后台真实凭据”

## 15. 风险

### 15.1 管理员权限边界不完整

若在未补齐管理员权限模型和鉴权边界前直接开放凭据管理接口，风险最高：

- 非管理员误用管理员接口
- 服务器 key 暴露或被误改
- 后续难以审计“是谁修改了哪把 key”

### 15.2 认证域与平台域混杂

若把平台偏好、服务器配置和管理员能力直接硬塞进认证域：

- 用户表会变成杂项配置表
- 认证模块职责被持续污染
- 后续迁移和演进成本升高

### 15.3 用户选择与实际执行不一致

若不明确区分“用户选择的组合”和“实际命中的凭据”：

- 用户中心难以解释任务为何失败
- 管理员难以判断问题来自配置、凭据还是执行器
- 审计与排障会非常困难

### 15.4 同组合自动切换规则失控

若自动切换缺少明确边界：

- 可能发生跨模型、跨 CLI 的隐式降级
- 用户感知会混乱
- 同时会放大排障复杂度

### 15.5 运行时压力控制不足

若未同步落地入队、限流和资源池：

- 高并发时 `web` 请求会被长任务拖住
- 某类重任务会拖垮其他能力
- 坏凭据和重试风暴可能导致雪崩

### 15.6 设置页与运行时口径分裂

若设置页和运行时分别实现两套服务器配置来源：

- 用户会看到不一致的列表和状态
- 代码维护成本显著增加
- 额外增加请求量、缓存分歧和联调成本

### 15.7 检测结果不可靠

若凭据检测只做极简探针，或完全依赖单次请求结果：

- 可能把暂时性失败误判成永久失效
- 可能把有问题的凭据错误标成健康
- 进而影响调度和自动切换的正确性

### 15.8 文档与实现脱节

该草案涉及用户选择、后台管理、任务主链、管理员权限和运行保护多个层面。

若实现推进时不及时回写实施计划、实施基线和接口文档：

- 讨论结论会与代码事实脱节
- 后续参与者难以知道当前真正规则
- 测试、前端、后端和运维会快速出现口径漂移

## 16. 当前建议

当前建议的推进顺序如下：

1. 先冻结 API 草案和字段口径
2. 再补齐管理员权限边界
3. 再进入后端数据层与服务层实现
4. 再落前端选择器和设置页 Tab 化
5. 最后补抗压保护、文档回写与验证
