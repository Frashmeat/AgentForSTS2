# 硬编码 Prompt 与知识文本模板化改造计划

## 目标

本计划旨在将当前仓库内适合外置的硬编码 Prompt 与静态知识文本，统一迁移为“资源文件 + 代码装配”的结构，以降低多行字符串散落在业务代码中的维护成本，提升 Prompt 复用性、可读性与后续演进效率。

本次改造优先覆盖以下范围：

- `codegen`
- `planning`
- `approval`
- `image`
- `analyzer`
- `STS2` 知识库

本计划仅聚焦 Prompt 与知识文本模板化改造，不包含 `backend/llm/text_runner.py` 中 LiteLLM 依赖问题的修复工作。相关依赖治理应作为独立事项单独评估与推进。

## 目标结构

建议将改造后的资源组织为如下结构：

```text
backend/app/modules/codegen/prompts
backend/app/modules/planning/prompts
backend/approval/prompts
backend/image/prompts
backend/routers/prompts
backend/agents/knowledge
backend/app/shared/prompting
```

其中：

- Prompt 模板目录用于承载大段系统提示词、行为约束文本、说明性段落等内容。
- 知识资源目录 `backend/agents/knowledge` 用于承载大段静态知识文档，优先使用 Markdown 资源文件。
- 通用加载层 `backend/app/shared/prompting` 负责统一加载、格式化、异常处理与最小 fallback 机制。

本次改造遵循以下原则：

- 大段 Prompt 文本进入模板文件维护。
- 大段静态知识文档进入 Markdown 资源文件维护。
- 变量注入、条件分支、运行时 selector、路径判断、fallback 判断保留在 Python 代码中。
- `skill` 体系不作为运行时真源，运行时文本应以仓库内可控资源文件为准。

## 技术选择

第一版实现坚持轻量与可控原则，仅采用 Python 标准库完成模板装配：

- 使用 `pathlib.Path.read_text` 读取模板与知识资源文件。
- 使用 `str.format` 或 `string.Template` 完成基础变量注入。
- 测试体系继续沿用 `pytest`。

若后续模板复杂度显著上升，例如出现大量条件渲染、循环片段拼接或多层模板继承，再单独评估是否引入 `Jinja2`。在第一版中不预设额外模板依赖，以避免增加实现与运维复杂度。

## 推荐执行顺序

建议按以下顺序推进，以先固化基础设施，再逐步迁移调用方，降低返工风险：

1. `prompt_loader`
2. `prompt_assembler.py`
3. `code_agent.py`
4. `planning/services.py`
5. `approval/action_prompt.py`
6. `image/prompt_adapter.py`
7. `log_analyzer.py / mod_analyzer.py`
8. `sts2_docs.py`

该顺序的核心考虑是先建立统一的加载与装配能力，再迁移依赖面较广的 Prompt 构建点，最后处理迁移量更大、知识文本更重的 STS2 资源。

## 风险与注意事项

本次改造的主要风险如下：

- 模板变量名不统一：不同模块若对同一语义使用不同变量名，容易造成模板渲染失败或维护成本上升，需尽早建立命名约定。
- 控制流字符串误外置：若将本应保留在代码中的条件判断、selector 或 fallback 描述错误迁移到资源文件，可能导致业务行为难以追踪与调试。
- `sts2_docs.py` 迁移量大：该模块可能包含体量较大的知识文本与复杂引用关系，迁移时应优先做资源切分与引用映射设计。
- analyzer/image 缺少快照测试导致 Prompt 漂移：若缺乏稳定的输出断言，后续微调模板时容易出现行为变化而未被及时发现。

实施过程中需特别注意以下事项：

- 模板只承载稳定文本，不承载业务判定。
- 所有资源路径应由统一加载层集中处理，避免业务模块自行拼路径。
- 迁移时优先保留兼容层，避免一次性替换造成调用链中断。
- 测试不仅验证“能读取”，还应验证“关键语义未丢失”。

## 完成定义

当以下条件全部满足时，可认为本次模板化改造完成：

- 主要大段 Prompt 不再直接以内嵌 Python 多行字符串形式维护。
- STS2 大块知识文本不再以内嵌常量形式维护。
- 业务 selector、路径判断、fallback 分支仍保留在代码中。
- 关键测试通过，至少覆盖模板加载、变量装配、关键 Prompt 输出与主要调用链回归。
- 已新增一份架构说明文档，明确后续新增 Prompt 与知识资源的接入规范。
