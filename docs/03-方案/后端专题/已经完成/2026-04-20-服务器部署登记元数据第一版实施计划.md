# 2026-04-20 服务器部署登记元数据第一版实施计划

## 0. 文档定位

- 本文是“服务器部署目录占用保护第一版”之后的下一阶段实施计划。
- 本文只规划“如何为 `Mods/{project_name}` 写入最近一次成功部署事实”，不直接等同于多实例调度方案。

  - [2026-04-20-服务器部署目录隔离与占用策略冻结稿.md](./2026-04-20-服务器部署目录隔离与占用策略冻结稿.md)

## 1. 当前理解

当前已经有：

- `project_name` 级别部署目录锁
- 同名部署冲突回收到 `queued + server_deploy_target_busy`
- 单 `web` 实例后台排队消费

但当前仍无法回答：

- `Mods/{project_name}` 最近一次是谁写入的
- 最近一次部署来自哪个 `server_project_ref`
- 最近一次部署对应哪个工作区

原因是：

- 当前部署锁文件只表达“谁正在占用”
- 锁释放后文件即被删除
- 现有部署复制逻辑不会写任何持久登记元数据

## 2. 目标

第一版目标是：在不改数据库和不改锁语义的前提下，为 `Mods/{project_name}` 写入一份“最近一次成功部署事实”。

本阶段希望满足：

1. 平台 `build.project` 成功部署后，写入部署登记元数据
2. 旧 `/api/ws/build-deploy` 成功部署后，也写入同一格式的部署登记元数据
3. 元数据至少可回答：
   - `project_name`
   - `server_project_ref`
   - `user_id`
   - `deployed_at`
   - `source_workspace_root`

## 3. 非目标

本阶段不做：

- 数据库部署真源
- 多实例 leader / 租约 / 去重
- 完整部署历史表
- 回滚与版本管理
- 基于元数据反向驱动新的调度逻辑

## 4. 代码现状调查结论

### 4.1 当前真实部署写入点

当前部署复制只发生在两处：

- `backend/app/modules/platform/runner/build_project_handler.py`
- `backend/routers/build_deploy.py`

它们都会在复制到 `Mods/{project_name}` 前获取部署锁。

### 4.2 当前锁文件的边界

`backend/app/modules/platform/application/services/server_deploy_target_lock_service.py` 当前会写：

- `*.lock.json`

但它只表示“当前占用者”，释放锁后即删除，因此不能作为“最近一次成功部署”的持久真源。

### 4.3 当前最小可复用切口

`backend/app/modules/platform/infra/build_output_files.py` 当前已统一承担：

- 查找最新构建产物
- 复制构建产物到 `Mods/{project_name}`

因此，部署登记最合适的写入时机是：

- 复制成功之后
- 锁仍在持有期间

## 5. 第一版推荐方案

当前推荐方案冻结为：

- **独立部署登记服务 + 目标目录内单文件登记**

### 5.1 基本思路

1. 新增独立部署登记服务
2. 在 `Mods/{project_name}` 下写入一份元数据文件
3. 平台 `build.project` 与旧 `/api/ws/build-deploy` 都复用这份服务
4. 每次成功部署只覆盖“最近一次部署事实”

### 5.2 推荐字段

第一版建议至少记录：

- `project_name`
- `server_project_ref`
- `user_id`
- `deployed_at`
- `source_workspace_root`

同时建议顺手补齐：

- `job_id`
- `job_item_id`
- `entrypoint`
- `deployed_to`
- `file_names`

原因：

- 不额外增加太多复杂度
- 但可明显提升后续排障与多入口追溯能力

### 5.3 推荐落点

当前建议直接放到：

- `Mods/{project_name}/.server-deploy.json`

原因：

- 跟真实部署目录同生命周期
- 不需要先建数据库或全局注册中心
- 旧 ws 构建部署链也能直接复用

## 6. 风险与边界

### 6.1 只记录最近一次事实

第一版只覆盖最新部署，不保留完整历史。

### 6.2 不等同于调度真源

即使有部署登记元数据，也还不等于已经解决：

- 多实例重复消费
- 跨进程协调
- 崩溃恢复后的锁真源

### 6.3 写入时机必须晚于复制成功

若在复制前或复制中途写元数据，容易把未完成部署误记成成功部署。

## 7. 影响范围

预计会涉及：

- `backend/app/modules/platform/application/services/`
- `backend/app/modules/platform/runner/build_project_handler.py`
- `backend/routers/build_deploy.py`
- 定向测试
- 接口文档、方案文档、阶段总结

## 8. 验收标准

当以下条件满足时，可视为第一版部署登记元数据落地：

1. `Mods/{project_name}` 下存在稳定的部署登记文件
2. 平台 `build.project` 成功部署后会写入登记信息
3. 旧 `/api/ws/build-deploy` 成功部署后也会写入同一格式的登记信息
4. 定向测试能覆盖平台部署与旧 ws 部署两条入口
5. 文档已明确：
   - 这是“最近一次部署事实”
   - 不是多实例调度真源

## 9. 2026-04-20 实现回写

截至 `2026-04-29`，本文对应的第一版最小实现已落地，当前状态为已经完成：

- 已新增独立部署登记服务，统一负责写入与读取：
  - `Mods/{project_name}/.server-deploy.json`
- 当前写入时机固定为：
  - 本次构建产物复制成功之后
  - 部署锁仍在持有期间
- 平台 `build.project` 当前会写入：
  - `entrypoint = "platform.build.project"`
- 旧 `/api/ws/build-deploy` 当前会写入：
  - `entrypoint = "legacy.ws.build_deploy"`
- 当前登记字段包括：
  - `schema_version`
  - `project_name`
  - `job_id`
  - `job_item_id`
  - `user_id`
  - `server_project_ref`
  - `source_workspace_root`
  - `deployed_at`
  - `deployed_to`
  - `entrypoint`
  - `file_names`

当前仍未完成的部分不再属于本文第一版范围：

- 跨进程 / 多实例锁真源
- 部署历史累积而不只是最近一次覆盖
- 基于部署登记反向驱动调度与冲突治理

## 10. 后续步骤

建议按以下顺序继续推进：

1. 先把“部署锁文件”和“.server-deploy.json”收口成跨进程可复用的真源模型
   - 明确“当前占用”和“最近一次成功部署”各自负责什么
   - 明确进程异常退出后的过期、清理与恢复口径

2. 再把后台 queue worker 升级为多实例可协调消费
   - 明确谁能消费
   - 明确重复消费、抢占和去重策略

3. 最后再评估是否需要补部署历史累积
   - 若只是排障，最近一次事实可能已经足够
   - 若要支持回滚或版本管理，再进入历史记录设计
