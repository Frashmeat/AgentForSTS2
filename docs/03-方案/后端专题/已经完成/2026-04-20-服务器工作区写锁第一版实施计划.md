## 1. 当前理解

当前服务器模式已经把工作区隔离到 `runtime/platform-workspaces/{user_id}/{token}/{project_name}`，但写型链路仍可能并发命中同一 `server_project_ref`。

这会带来三个直接问题：

1. 源码互相覆盖
2. 构建中间产物互相污染
3. 结果归属和失败定位混乱

因此，当前最先要补的不是排队体验，而是“同一工作区不能被并发写”这条正确性底线。

## 2. 本次范围

### 2.1 本次要做

- 新增 `server_project_ref` 级别工作区写锁
- 在平台执行编排层识别写型 workflow
- 命中写型链路时先尝试获取锁
- 锁冲突时保持 `job=queued`、`item=ready`
- 查询接口返回 `queued_reason_code / queued_reason_message`
- 前序同工作区任务完成后，继续尝试拉起下一个 `queued` 任务
- 补最小定向测试
- 同步回写实施计划与阶段总结

### 2.2 本次不做

- 不接真实后台排队消费器
- 不做 `project_name` 级别部署锁
- 不做数据库租约锁
- 不做多实例场景治理

## 3. 设计口径

### 3.1 资源粒度

- 锁键固定为 `server_project_ref`

### 3.2 写型链路

第一版至少覆盖：

- `code.generate`
- `asset.generate`
- `build.project`

### 3.3 挂点

- 锁应挂在执行编排层
- 不应散落到单个 handler 内部

### 3.4 冲突语义

- 开始任务接口返回：`200 {"status":"queued"}`
- 查询摘要返回：
  - `queued_reason_code = server_workspace_busy`
  - `queued_reason_message = server workspace is busy`
- 当前已补同进程内自动续跑，但还不包含独立后台 worker / 跨进程消费器

## 5. 风险

### 5.1 锁粒度挂错

若只锁 `build.project`，前置代码生成仍可能互相覆盖。

### 5.2 锁服务放错层

若把锁散落到 handler，会导致后续排队、日志与部署占用保护很难统一。

### 5.3 同进程续跑与未来真实排队仍有差距

本次已先把工作区占用收口为显式 `queued + reason_code`，并补了“前序任务完成后顺手续跑下一个 queued 任务”的同进程能力；但还没有独立后台消费器。后续若接真实排队，需要保留当前 `server_workspace_busy` 口径，避免前后状态断裂。

## 6. 验收标准

- 同一 `server_project_ref` 的第二条写型任务不能并发执行
- 当前先返回 `queued`，并带 `queued_reason_code = server_workspace_busy`
- 非写型链路不被误伤
- 至少一条定向测试覆盖该行为
