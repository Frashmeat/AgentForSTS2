# 2026-04-18 服务器模式平台任务 Payload 收口方案

## 1. 结论

2026-04-29 进度复核：当前状态为已经完成。平台 runner 当前统一读取 `item_name`，平台创建任务已拒绝 `name / asset_name / project_root / has_uploaded_image / provided_image_*` 等旧字段，本文保留为 payload 收口口径参考。

当前服务器模式专题已经把“低成本可接通的纯文本最小闭环”扩展到 13 条，下一阶段不应继续机械补 workflow 注册，而应转入 **平台任务 payload 收口**。

基于当前项目 **尚未正式上线**、**不需要保留迁移兼容包袱** 的约束，本轮建议直接采用“单一主线 contract”：

- 平台服务器任务只接受“服务器可执行 payload”
- 工作站本地工作流继续使用“本地执行 payload”
- 不再为 `name / asset_name`、`project_root`、`has_uploaded_image` 等历史口径保留并存兼容

一句话结论：

**下一阶段主攻方向应从“继续补点”切换为“前后端平台 payload 一次性收口到单一语义”，否则后续越往前推进，`deferred`、runner 入参和前端发包口径会越来越散。**

### 1.1 2026-04-18 后端第一阶段落地补充

截至 `2026-04-18`，后端第一阶段收口已经完成：

- 平台 runner 当前已统一只认 `item_name`
- `single_asset_plan_handler` 不再兼容读取 `asset_name / name`
- `batch_custom_code_handler` 不再兼容读取 `name / asset_name`
- 平台 workflow 注册当前会在 step payload 内部注入规范化 `asset_type`
- 对当前已接通的资产类 workflow，platform caller 不再必须显式传 `asset_type`
- `JobApplicationService.create_job()` 当前会直接拒绝以下旧字段：
  - `name`
  - `asset_name`
  - `project_root`
  - `has_uploaded_image`
  - `provided_image_b64`
  - `provided_image_name`
- `/api/platform/jobs` 与 `/api/me/jobs` 当前会把这类输入稳定返回 `400`

截至 `2026-04-18`，前端第一阶段收口也已完成：

- 单资产平台任务当前统一写入 `item_name`
- 批量平台任务当前统一写入 `item_name`
- 前端平台任务当前不再写入：
  - `project_root`
  - `has_uploaded_image`
  - `asset_name`
  - `name`
- 若用户在服务器模式下选择“上传图片”，前端当前会先走上传接口，再把 `uploaded_asset_ref` 注入平台任务 payload

因此，平台 payload 收口的一期目标已完成，前后端当前已经重新回到同一条主线 contract。

## 3. 新的设计原则

基于“项目未上线，不保留兼容包袱”，本轮收口采用以下原则：

1. **平台任务 payload 只保留一套字段，不保留双字段并存**
2. **工作站本地语义与平台服务器语义分离，不再共用同一 payload 结构**
3. **前端不再向平台任务写入明知会触发 deferred 的字段**
4. **后端不再继续为旧字段做长期兼容读法**
5. **所有阻塞语义都必须前后端一致，不能靠临时分支解释**

## 4. 目标 Payload 设计

### 4.1 工作站本地工作流 payload

用途：

- 单资产工作流 WebSocket
- 批量工作流 WebSocket
- 本地图片生成 / 本地 Code Agent / 本地构建

允许字段：

- `asset_type`
- `item_name`
- `description`
- `project_root`
- `image_mode`
- `provided_image_b64`
- `provided_image_name`
- `auto_mode`

说明：

- 这是一条纯本地执行链，保留 `project_root` 和本地图片内容是合理的
- 该 payload 不应直接复用为平台任务 payload

### 4.2 平台服务器任务 payload

用途：

- `/api/me/jobs`
- `/api/platform/jobs`
- 所有服务器模式 runner

统一字段：

- `item_name`
- `description`
- `image_mode`

由后端 workflow 内部注入：

- `asset_type`

按需补充字段：

- `goal`
- `detailed_description`
- `implementation_notes`
- `depends_on`
- `scope_boundary`
- `dependency_reason`
- `acceptance_notes`
- `affected_targets`
- `coupling_kind`

### 4.3 平台任务明确禁止的字段

平台服务器任务 payload 不再允许出现：

- `project_root`
- `provided_image_b64`
- `provided_image_name`
- `has_uploaded_image`
- `name`
- `asset_name`

## 5. 具体收口策略

### 5.1 字段名统一

平台链路统一使用：

- `item_name`

不再保留：

- `name`
- `asset_name`

这意味着：

- `single_asset_plan_handler` 不再做 `asset_name -> name` 双读
- `batch_custom_code_handler` 不再做 `name -> asset_name` 双读
- 平台任务创建接口与 runner 统一只认 `item_name`

### 5.2 `project_root` 完全退出平台任务

`project_root` 是用户本机目录语义，只属于工作站链路。

因此：

- 前端创建平台任务时不再发送 `project_root`
- 后端平台执行链也不再把 `project_root` 视为可执行输入
- `local_project_root_required` 仅作为防御性残余判定，目标是后续不再从主链触发

### 5.3 上传图片语义收口

当前 `has_uploaded_image` 只能表达“用户上传过图片”，不能表达“服务器能取到图片”。

因此本轮建议：

- 在服务器任务链路里直接禁止上传图片型任务进入 runner
- 前端服务器模式创建任务时，不再写入 `has_uploaded_image`
- 若后续要支持上传图片，必须引入真正的服务器侧 `uploaded_asset_ref`

### 5.4 `deferred` 角色收缩

本轮目标不是删除 `deferred`，而是让它回到更真实的职责：

- 用于尚未注册的 workflow
- 用于仍未完成服务器资产持久化或服务器工作目录治理的阻塞点

而不是继续替前端错误发包兜底。

## 7. 验收标准

满足以下条件时，可认为 payload 收口完成：

- 平台任务 payload 中不再出现 `name / asset_name / project_root / has_uploaded_image`
- 平台任务统一使用 `item_name`
- 单资产与批量同类资产的服务器 payload 口径一致
- 上传图片型服务器任务不会再用模糊字段进入主链
- 后端 runner 不再依赖兼容字段 fallback
- 文档、测试和代码行为一致

## 8. 后端兼容层扫描结论

以下结论只针对 `backend/` 活代码，不包含归档文档、测试样板和第三方资源。

### 8.1 必须纳入下一阶段改造的兼容口径

1. `backend/app/modules/platform/runner/single_asset_plan_handler.py`
   - 当前 `_resolve_item_name()` 仍采用：
     - `asset_name`
     - `name`
   - 这是典型的双字段兼容层
   - 在“未上线、不保留兼容包袱”的约束下，后续应直接收口到 `item_name`

2. `backend/app/modules/platform/runner/batch_custom_code_handler.py`
   - 当前 `_resolve_item_name()` 仍采用：
     - `name`
     - `asset_name`
   - 同样属于平台 payload 的兼容读法
   - 建议和 `single_asset_plan_handler` 同步收口

### 8.2 明显带有历史兼容痕迹、但需谨慎处理的点

1. `backend/routers/workflow.py`
   - `_maybe_await_approval()` 的返回约定此前曾带有：
     - `(False, output): 保留给兼容分支；当前单资产流程不会使用`
   - 该项已于 `2026-04-19` 清理完成
   - 它属于工作站本地链路，不再构成当前平台 payload 主线阻塞

2. `backend/routers/workflow.py`
   - `_ws_run_with_provided_image()` 中此前曾保留：
     - “优先用 base64，fallback 到本地路径”
   - 该项已于 `2026-04-19` 收口为只认 `provided_image_b64`
   - 当前不再保留本地路径 fallback

3. `backend/app/shared/infra/config/settings.py`
   - `DEFAULT_LLM_CONFIG["execution_mode"] = "legacy_direct"`
   - 这里的 `legacy` 更像历史命名残留，而不是运行时双路径兼容实现
   - 可列入后续命名收口，但不属于当前 payload 主矛盾

### 8.3 不建议误删的 fallback

以下 fallback 当前更偏“运行时容错”，不建议混入“兼容层删除”处理：

- `backend/image/prompt_adapter.py` 中的 prompt fallback
- `backend/image/postprocess.py` 中的 CPU fallback
- `backend/app/shared/prompting/prompt_loader.py` 中的模板 fallback

这些 fallback 不是旧口径并存，而是正常的容错策略。
