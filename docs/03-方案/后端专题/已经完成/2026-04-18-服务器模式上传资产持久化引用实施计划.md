# 2026-04-18 服务器模式上传资产持久化引用实施计划

## 1. 目标

在服务器模式下，为“用户上传图片”建立最小可用的持久化引用链路，让平台任务不再只能因为“上传图片无法被服务器读取”而被前端拦截或后端 deferred。

本轮目标不是一次性做完完整对象存储系统，而是先做一条 **最小、受控、可审计** 的服务器引用链：

- 前端上传图片
- 后端返回稳定 `uploaded_asset_ref`
- 平台任务 payload 使用 `uploaded_asset_ref`
- 后端 runner 能通过该引用读取图片

### 1.1 2026-04-18 当前进度补充

截至 `2026-04-29`，本计划的第一阶段最小链路已经落地，并已从 `未开始/` 移入 `已经完成/`：

- 后端已新增最小上传资产服务
- 已新增 `POST /api/me/upload-assets`
- 前端 `createAndStartPlatformFlow()` 当前会先上传图片，再把 `uploaded_asset_ref` 注入平台任务 payload
- 单资产与批量平台任务当前都已具备“预上传 -> 创建平台任务”的最小链路
- `ExecutionOrchestratorService` 当前会通过 `UploadedAssetService.get_asset()` 与 `get_asset_content_path()` 解析上传资产
- `asset_generate_handler` 当前已能读取 `uploaded_asset_ref` 指向的图片内容并进入服务器侧图片后处理链路

当前仍未完成的部分不再属于本计划第一版范围：

- 完整对象存储、清理策略、配额策略和跨实例文件存储治理

## 2. 当前事实

### 2.1 当前上传图片只存在本地语义

- 单资产工作流当前通过 `provided_image_b64 / provided_image_name` 直接走工作站 WebSocket
- 批量工作流当前通过 `provided_image_b64` 直接走工作站 WebSocket
- 平台任务链路当前已经拒绝：
  - `has_uploaded_image`
  - `provided_image_b64`
  - `provided_image_name`

### 2.2 当前服务器模式的用户体验

- 前端已经在服务器模式下直接阻止“上传图片型任务”
- 这保证了平台 payload 主线收口，但也意味着：
  - 服务器模式暂时无法覆盖“上传图片驱动”的任务

## 3. 本轮范围

### 3.1 在范围内

- 最小上传资产存储目录
- 最小上传资产引用生成规则
- 最小上传接口
- 单资产 / 批量平台任务改为可选携带 `uploaded_asset_ref`
- 后端平台 runner 能读取 `uploaded_asset_ref`

### 3.2 不在范围内

- 不接外部对象存储
- 不做图片 CDN
- 不做复杂权限系统
- 不做长期归档与大规模清理策略
- 不处理 `project_root` 服务器工作目录语义

## 4. 最小设计

### 4.1 资产落点

建议落在受控运行时目录，例如：

- `runtime/platform-upload-assets/`

每个上传对象最少包含：

- 文件实体
- 元数据：
  - `user_id`
  - `created_at`
  - `content_type`
  - `original_name`

### 4.2 引用格式

建议第一版使用稳定内部引用：

- `uploaded-asset:{id}`

或

- `platform-upload:{id}`

要求：

- 不直接暴露物理路径
- 不把原始文件名当主键
- 后续可迁移到数据库 id 或对象存储 key

### 4.3 平台任务 payload

新增字段：

- `uploaded_asset_ref`

说明：

- 有图任务在服务器模式下，若想继续进入平台链路，必须先获得该字段
- 不再继续使用：
  - `has_uploaded_image`
  - `provided_image_b64`
  - `provided_image_name`

第一阶段当前现状：

- 前端已在创建平台任务前预先调用上传接口
- 后端平台链路当前接受并校验 `uploaded_asset_ref`

## 5. 推荐实施顺序

1. 先补上传资产最小后端存储
2. 再补最小上传接口
3. 再改单资产平台任务链路
4. 再改批量平台任务链路
5. 最后补 runner 读取与回归测试

当前执行状态：

- 1 / 2 / 3 / 4 已完成第一阶段最小落地
- 5 已完成最小回归
- 下一步应决定是否继续让 runner 读取引用元数据，或者进入真正的图片内容消费

## 6. 验收标准

- 服务器模式下，前端可先上传图片并获得 `uploaded_asset_ref`
- 平台任务 payload 使用 `uploaded_asset_ref`
- 后端不再因为“上传图片只存在本地语义”阻塞这类任务
- 文档、接口和测试同步更新
