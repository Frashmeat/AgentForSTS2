## 1. 冻结目标

本文只解决 4 件事：

1. 明确服务器部署目录冲突的真实资源粒度
2. 冻结“最终部署目录名能否脱离 `project_name`”这一关键判断
3. 选定第一版推荐的隔离与占用策略
4. 明确当前已有实现里哪些行为必须被禁止继续沿用

本文不解决：

- 完整的发布中心或版本管理系统
- 多版本并存部署
- 回滚与版本切换 UI
- 最终部署清理策略

## 2. 当前代码事实

基于当前仓库代码，以下事实已经明确：

- 服务器工作区构建仍在各自 `server_workspace_root` 下进行，因此“构建阶段”已经具备物理目录隔离。
- `build.project` 若命中部署，会把产物落到共享 `Mods/{project_root.name}` 目录。
- 当前知识与 prompt 口径仍普遍把 `project_root.name / mod_name / Godot 资源目录` 视为同一语义主轴。
- 因此，“最终部署目录名”当前不能被简单替换为随机 token，而不影响现有代码生成与资源路径约束。

## 3. 冲突资源粒度冻结

第一版服务器部署冲突的主资源粒度冻结为：

- `project_name`

更准确地说，是共享部署目录：

- `Mods/{project_name}`

原因：

- 当前最终部署路径就是按该目录组织。
- 构建产物一旦进入部署阶段，真正冲突的不是 `server_project_ref`，而是共享的最终 Mod 目录。
- 不同用户、不同服务器工作区、不同 `server_project_ref`，只要 `project_name` 相同，就会指向同一部署资源。

## 4. 最终目录命名冻结

第一版冻结以下判断：

- **最终部署目录继续保持 `Mods/{project_name}`**

当前不建议第一版直接改成：

- `Mods/{server_project_ref}`
- `Mods/{user_id}-{token}`
- `Mods/{project_name}__{token}`

原因：

- 现有知识与 prompt 仍要求 `ModId / assembly / folder name` 保持一致。
- 若直接把最终目录改成 token 风格，会把当前模板、资源路径和运行时认知全部一起打散。
- 该改动远大于“先把多人并行冲突收口”的目标。

## 5. 第一版推荐策略冻结

当前推荐策略冻结为：

- **构建隔离，部署独占**

含义：

- 每个 `server_project_ref` 继续在自己的服务器工作区内独立构建
- 只有最终写入 `Mods/{project_name}` 这一步，才视为共享资源占用
- 同一 `project_name` 的部署必须串行

### 5.1 推荐主语义

- 同一时刻只允许一条部署链写入 `Mods/{project_name}`
- 若另一条任务命中相同 `project_name`：
  - 若排队已落地，进入 `queued`
  - 若排队尚未落地，返回稳定 `busy/conflict` 语义

### 5.2 允许的第一版边界

- 允许“同名项目只能有一个活动部署者”
- 暂不要求同名项目来自不同用户时自动并存

这意味着第一版优先保证“不互相覆盖”，而不是一开始就支持“同名项目多版本共存”。

## 6. 当前禁止继续沿用的行为

当前实现里最需要被禁止继续扩大的是：

- 目标目录存在旧 `.dll/.pck` 时，直接把旧文件当作当前成功部署结果返回

该行为的问题是：

- 它会把“已有历史产物”误判成“本次部署成功”
- 无法区分旧产物属于哪个 `server_project_ref`
- 在多人并行或同名项目场景下会直接制造错误归属

因此，第一版冻结以下规则：

- 部署成功必须来自“当前工作区本次构建产物”的明确复制结果
- 不能继续把目标目录现存文件直接当作成功捷径

## 7. 部署占用元数据冻结

为了让部署冲突可解释，第一版建议至少记录：

- `project_name`
- `server_project_ref`
- `user_id`
- `deployed_at`
- `source_workspace_root`

第一版推荐落点：

- `Mods/{project_name}` 下的部署元数据文件
- 或独立的服务器部署登记文件

当前不强制第一版立刻上数据库真源，但必须保证后续可以回答：

- 这个部署目录最近是由哪个服务器工作区写入的
- 当前冲突是“同一工作区重部署”还是“不同工作区同名冲突”

## 8. 状态流转与冲突返回口径冻结

第一版部署占用保护应尽量复用现有平台主状态，而不是新发明“deploying_busy”一类专用主状态。

### 8.1 推荐状态流转

对命中 `build.project -> deploy` 的任务，当前建议冻结为：

1. `draft`
2. `queued`
3. `running`
4. 终态

其中：

- 因 `project_name` 同名部署占用而等待的任务，状态仍为 `queued`
- 不应因为部署目录被占用而进入 `deferred`

### 8.2 与工作区忙的边界

当前需要明确区分两类共享资源：

- `server_project_ref`
  - 代表服务器工作区写入占用
- `project_name`
  - 代表最终部署目录占用

冻结要求：

- 同工作区冲突，按“服务器工作区忙”解释
- 同名部署冲突，按“部署目录占用”解释
- 不能把两类冲突都混写成笼统的 `busy`

### 8.3 事件复用建议

第一版优先继续复用 `job.queued`，并对 payload 增补：

- `queue_reason_code`
- `queue_reason_message`
- `resource_type`
- `resource_key`

部署占用等待场景建议固定为：

- `queue_reason_code = server_deploy_target_busy`
- `queue_reason_message = 当前服务器部署目录正在被另一条同名项目部署链占用，请等待其完成后再继续执行。`
- `resource_type = server_deploy_target`
- `resource_key = project_name`

### 8.4 无排队前的同步冲突口径

若部署占用保护先于统一排队主链落地，第一版允许临时返回同步冲突。

推荐返回约束：

- HTTP：`409`
- detail：`server deploy target is busy`

同时建议在错误 payload 或日志中补充：

- `reason_code = server_deploy_target_busy`
- `project_name`
- `server_project_ref`

### 8.5 `deferred` 边界冻结

第一版明确禁止把以下情况写成 `deferred`：

- 当前部署目录已被同名项目占用
- 当前只是在等待另一条部署链释放 `Mods/{project_name}`
- 当前构建已完成，但最终部署阶段需要排队

`deferred` 仍只用于“当前运行时能力不能消费”的场景，而不是共享资源暂时繁忙。

## 9. 当前部署成功判定冻结

为了避免“旧产物即成功”的误判，第一版冻结以下成功判定原则：

- 只有当本次执行明确从当前工作区复制出构建产物到目标目录后，才能判定为部署成功
- 若只是检测到目标目录已存在旧 `.dll/.pck`，不能直接返回成功
- 若目标目录被占用但当前未复制任何新产物，应返回等待或冲突，而不是“已部署”

推荐最小成功依据：

- 当前工作区存在本次构建产生的输出文件
- 当前执行完成了复制动作
- 当前部署元数据已更新到本次 `server_project_ref`

## 10. 备选方案评估冻结

## 10.1 方案 A：直接改成 token 目录

结论：

- 当前不采纳

原因：

- 与现有 `project_name / mod_name / folder name` 口径冲突过大

## 10.2 方案 B：强制全局唯一项目名

结论：

- 当前不采纳

原因：

- 会把部署冲突问题粗暴转嫁为用户命名约束，体验过重
- 也不能替代部署阶段的占用保护

## 10.3 方案 C：构建隔离，部署独占

结论：

- 第一版采纳

原因：

- 能最小代价收口当前共享目录冲突
- 不要求同时重写模板、prompt 与资源路径语义

## 11. 代码落点建议

后续若进入实现，优先关注：

- `backend/app/modules/platform/infra/build_output_files.py`
- `backend/app/modules/platform/runner/build_project_handler.py`
- `backend/app/modules/platform/application/services/build_deploy_facade_service.py`
- 新增的部署占用锁或部署登记抽象

冻结要求：

- “部署占用保护”不能只散落在 `build.project` 一个函数里临时判断
- 需要有明确的部署资源抽象，至少能围绕 `project_name` 做统一占用判断

## 12. 验收标准

当以下条件满足时，可认为“服务器部署目录隔离与占用策略冻结稿”完成：

- 明确了最终冲突资源是 `Mods/{project_name}`
- 明确了第一版不改最终目录命名语义
- 明确了第一版推荐策略是“构建隔离，部署独占”
- 明确禁止继续把目标目录旧产物视为当前成功部署结果
- 明确了部署占用等待应复用 `queued`，并区分 `server_workspace_busy` 与 `server_deploy_target_busy`
- 明确了后续至少需要最小部署元数据来解释冲突来源

## 13. 当前建议

本稿完成后，下一步才适合进入：

1. 先补 `project_name` 级别部署占用保护
2. 再清理“旧产物即成功”的现有捷径逻辑
3. 最后补部署登记与对应测试

在此之前，不建议继续扩大“自动部署”覆盖面。

## 14. 2026-04-20 实现回写

截至 `2026-04-29`，本冻结稿对应的第一版最小实现已落地，当前状态为已经完成：

- 已新增 `project_name` 级别部署目录锁服务，资源键固定为 `Mods/{project_name}`
- `build.project` 当前只在最终部署复制阶段获取该锁，保持“构建隔离，部署独占”
- 同名部署目录被占用时，平台任务当前会回收到：
  - `job.status = queued`
  - `job_item.status = ready`
  - `queued_reason_code = "server_deploy_target_busy"`
  - `queued_reason_message = "server deploy target is busy"`
- 当前已补同进程内最小续跑：前序同名部署完成后，会继续尝试拉起下一个同名 `project_name` 的 `queued` 任务
- 旧 `/api/ws/build-deploy` 当前也已接入同一类部署目录锁；若部署目录正忙，会直接返回 `server_deploy_target_busy`
- `build.project` 与旧 `/api/ws/build-deploy` 当前都不再把目标目录里已有旧 `.dll/.pck` 直接当作本次成功结果
- 当前已补第一版部署登记元数据：
  - 成功部署后会在 `Mods/{project_name}/.server-deploy.json` 写入最近一次成功部署事实
  - 平台 `build.project` 与旧 `/api/ws/build-deploy` 当前共用同一格式
  - 当前登记字段已覆盖 `project_name / server_project_ref / user_id / deployed_at / source_workspace_root`

当前仍未完成的部分不再属于本冻结稿第一版范围：

- 跨进程 / 多实例真源排队消费
- 多版本并存与回滚语义
