## 0.1 2026-04-30 补充：Codex 文本步骤执行路径

服务器模式执行配置选择 `Codex CLI / gpt-5.4` 时，文本规划步骤必须与代码生成步骤一样尊重 `agent_backend=codex`，走 `agent_cli -> codex_cli` 路径，而不是走 `litellm -> OpenAI completion` 路径。Web / Workstation 容器镜像必须内置 Codex CLI 与 Claude Code CLI，缺 CLI 是部署依赖错误，不应回退到第三方中转站。

本次排障事实：

- 管理站点截图显示 `gpt-5.4` 在 `Codex` 类型下正常消费，只能证明 Codex 路径可用。
- 失败日志显示 `LiteLLM completion() model=gpt-5.4; provider=openai`，说明 Web 平台文本步骤实际走的是 OpenAI-compatible completion 请求。
- 两者请求形态不同，不能用 Codex 路径成功来证明 LiteLLM completion 路径可用。
- 旧 Web / Workstation Docker 模板只安装 Python 依赖，不默认安装 Node/Codex CLI；因此托管容器里选择 Codex profile 时，文本步骤强制走 CLI 会在本地 `PATH` 检查阶段失败，表现为 `未找到 Codex CLI`。
- 服务器模式目标是 Web 上部署 Codex / Claude，不是通过文本步骤回退中转站绕过 CLI；因此修复点应在镜像依赖，而不是运行时降级。

已收口约束：

- `text_generate_handler.build_text_llm_config()` 对 `agent_backend=codex` 返回 `mode=agent_cli`。
- `llm.text_runner` 的 `codex_cli` 文本后端必须把执行期 `api_key/base_url` 注入为 `OPENAI_API_KEY/OPENAI_BASE_URL`。
- Web / Workstation Docker 模板必须安装 `nodejs npm`、`@openai/codex` 与 `@anthropic-ai/claude-code`，保证服务器容器具备 Codex / Claude CLI。
- `llm.text_runner` 与 `llm.agent_backends.codex_cli` 均保持缺 CLI 即失败，避免文本和代码路径在部署依赖不完整时伪成功。
- `agent_backend=claude` 的既有 API/LiteLLM 文本路径暂不扩大修改，避免影响无关链路。

## 0.2 2026-04-30 补充：OpenAI-compatible LiteLLM 模型名规范化

服务器凭据选择 `provider=openai` 且配置了自定义 `base_url` 时，系统语义是“OpenAI-compatible 网关”，不等于“只能调用 OpenAI 官方模型”。这类配置下若模型名是 `deepseek-v3.2` 这种裸模型名，LiteLLM 本地会先尝试从模型名前缀推断 provider；由于裸 `deepseek-v3.2` 不能被 LiteLLM 识别，调用会在本地解析阶段失败，表现为：

```text
LLM Provider NOT provided. You passed model=deepseek-v3.2
```

本次排障事实：

- 该错误没有到达上游网关，也不是账号、内容安全或区域策略拒绝。
- 根因是执行绑定中的 `provider` 没有进入文本 runner 配置，且 `normalize_llm_config()` 会丢弃 `provider`。
- 对 `provider=openai`、`base_url` 非空、模型名未带 provider 前缀的 LiteLLM 文本路径，运行时应传 `openai/<model>`，例如 `openai/deepseek-v3.2`。
- 已显式配置为 `openai/deepseek-v3.2` 的模型名不重复加前缀。
- LiteLLM 文本 completion 与 stream 请求附加 `User-Agent: AgentTheSpire/0.1.0`，用于让第三方中转站识别这是本应用的后端服务请求；不伪装成 Chrome/Safari 等浏览器客户端。

这解释了为什么用户工作站通常不会遇到该问题：用户工作站的 Codex / Claude CLI 路径直接把模型名交给对应 CLI，并通过环境变量注入 key/base_url；它不经过 LiteLLM 的 provider 推断逻辑。只有 Web 服务器模式下走 `claude_api -> litellm` 的文本步骤，才会暴露这个裸模型名解析问题。

## 0. 当前状态

- 当前状态：进行中
- 触发背景：
  - 服务器模式任务已经能路由到 Web 托管 Workstation，并进入上游模型调用。
  - 失败日志出现 `reason_code=upstream_request_blocked`，但管理端只能看到任务摘要，不能直接查看 Web 工作站 stdout / stderr。
  - 当前上游错误分类过粗，`Your request was blocked` 会被固定解释为“可能触发内容安全策略”，无法区分内容安全、网关、凭据、权限、区域、限流或其他上游失败。
- 本文目标：先把管理员诊断闭环打通，再把上游错误分型做细，避免后续继续靠猜测定位失败。

## 1. 当前代码事实

### 1.1 Web 托管 Workstation 运行态

当前 `WorkstationRuntimeManager` 已具备：

- 自动启动 Web 托管 Workstation。
- 将子进程 stdout / stderr 写入固定日志文件：
  - `runtime/logs/web-workstation.stdout.log`
  - `runtime/logs/web-workstation.stderr.log`
- `get_runtime_status()` 返回：
  - `available`
  - `running`
  - `managed`
  - `capabilities`
  - `last_error`
  - `stdout_log_path`
  - `stderr_log_path`

当前不足：

- 管理端只能看到日志路径，不能在页面直接查看日志内容。
- 排障时仍需要登录服务器读文件，不符合 Web 管理后台职责。
- 如果上游模型失败的 `raw_error` 只写在 Web 工作站 stderr 或后端 warning 日志里，管理员无法在后台快速确认真实失败点。

### 1.2 管理端运行状态页

当前管理端已有：

- `GET /api/admin/platform/workstation-runtime-status`
- `AdminRuntimePage`
- “Web 托管 Workstation”运行状态展示
- capabilities 展示
- stdout / stderr 路径展示

当前不足：

- 没有日志 tail 接口。
- 没有 stdout / stderr 切换面板。
- 没有“刷新日志”“日志为空”“日志文件不存在”“日志被截断”等状态表达。

### 1.3 上游错误分类

当前 `text_generate_handler` 中的分类规则是字符串包含判断：

- `request was blocked`
- `content policy`
- `safety policy`
- `content_filter`
- `content filter`

当前不足：

- `request was blocked` 语义过宽，可能来自内容安全，也可能来自网关、账号策略、区域限制、模型权限或代理策略。
- 用户友好文案固定写“可能触发内容安全策略”，容易误导排障方向。
- `error_payload` 虽包含 `raw_error`，但没有结构化的 `upstream_category`、`retryable`、`http_status`、`provider_error_code` 等诊断字段。

### 1.4 Prompt 安全边界

服务器单资产 prompt 已说明：

- 任务仅涉及虚构电子游戏 Mod 机制与代码实现。
- “伤害”“攻击”“毒”等词均指游戏内数值规则，不涉及现实世界伤害或违法行为。

当前不足：

- 仍可继续强化“只输出虚构电子游戏 Mod 代码实现方案，不提供现实世界伤害、违法行为、医疗、金融或安全规避建议”。
- Prompt 只能降低误判概率，不能替代后端错误分型与管理端日志可见性。

## 2. 目标

### 2.1 核心目标

1. 管理员能在 Web 后台直接查看 Web 托管 Workstation 的 stdout / stderr 日志内容。
2. 上游 AI 失败从单一 `upstream_request_blocked` 拆成可诊断的错误分类。
3. 普通用户看到可行动的友好文案，管理员能看到足够排障信息。
4. 所有日志读取能力只面向管理员，且只允许读取固定日志文件。

### 2.2 用户体验目标

管理员在“运行状态”页面应能完成：

- 查看 Web 托管 Workstation 是否运行。
- 查看 capabilities 是否可用。
- 直接打开 `stderr`，确认最近失败的 raw error 或启动错误。
- 切换到 `stdout`，查看普通运行日志。
- 点击刷新，重新读取日志 tail。
- 根据错误分类判断下一步动作：
  - 改写描述
  - 切换模型
  - 切换凭据
  - 检查 base_url
  - 等待限流恢复

## 3. 非目标

本计划不要求：

- 给普通用户开放 Web 工作站日志。
- 支持读取任意服务器文件。
- 一次性建设完整集中式日志系统。
- 在第一版中实现日志实时 WebSocket 流。
- 保证上游模型永不拒绝请求。
- 把 Prompt 防误判当成唯一修复手段。

## 4. 后端修改计划

### 4.1 Web 工作站日志 tail 能力

建议在 `WorkstationRuntimeManager` 增加固定日志 tail 方法：

```python
read_runtime_log_tail(stream: str, tail_bytes: int = 65536) -> WorkstationRuntimeLogTail
```

`stream` 只允许：

- `stdout`
- `stderr`

返回字段建议：

```json
{
  "stream": "stderr",
  "path": "runtime/logs/web-workstation.stderr.log",
  "exists": true,
  "size_bytes": 12345,
  "tail_bytes": 65536,
  "truncated": false,
  "content": "..."
}
```

安全要求：

- 不允许前端传任意路径。
- `tail_bytes` 设置上限，建议最大 262144。
- 文件不存在时返回 `exists=false`，不抛 500。
- 读取失败时返回明确错误或通过 HTTP 500 交给管理端展示，不吞掉异常。

### 4.2 管理端日志接口

新增接口：

```http
GET /api/admin/platform/workstation-runtime-logs?stream=stderr&tail_bytes=65536
```

约束：

- 必须复用 `require_admin_user`。
- `stream` 非 `stdout` / `stderr` 时返回 HTTP 400。
- manager 未注册时返回 HTTP 503 或结构化不可用结果。
- 日志内容仅管理员可见。

### 4.3 上游错误分类器

建议新增：

```text
backend/app/modules/platform/runner/upstream_error_classifier.py
```

分类结果建议：

```python
class UpstreamErrorClassification:
    reason_code: str
    upstream_category: str
    reason_message: str
    retryable: bool
    http_status: int | None
    provider_error_code: str
    raw_error: str
```

第一版错误码：

| reason_code | 触发条件 | 用户文案方向 | retryable |
| --- | --- | --- | --- |
| `upstream_content_policy_blocked` | 明确包含 `content_filter` / `content policy` / `safety policy` | 调整描述后重试 | false |
| `upstream_gateway_blocked` | 只有泛化 `request was blocked`，无内容安全标识 | 联系管理员检查网关、代理、账号策略或模型权限 | false |
| `upstream_auth_or_region_blocked` | 401 / 403 / permission / unauthorized / region unavailable / model access | 检查凭据、模型权限、base_url 或区域 | false |
| `upstream_rate_limited` | 429 / rate limit / quota exceeded | 稍后重试或切换配置 | true |
| `upstream_bad_request` | 400 / invalid request / unsupported parameter | 检查请求参数、模型配置或 provider 兼容性 | false |
| `upstream_unclassified_error` | 其他上游异常 | 管理员查看日志定位 | true |

关键规则：

- `Your request was blocked` 不再直接等同内容安全。
- 只有出现明确内容安全标识时才归 `upstream_content_policy_blocked`。
- 对外文案避免猜测；管理员诊断依赖 `raw_error` 和日志。

### 4.4 `text_generate_handler` 接入分类

调整方向：

- 移除 `_is_upstream_request_blocked()` 的粗粒度布尔判断。
- 使用分类器返回结构化结果。
- 对可识别上游失败抛出统一异常，例如 `UpstreamTextGenerationError`。
- `to_error_payload()` 输出结构化字段：
  - `reason_code`
  - `reason_message`
  - `upstream_category`
  - `retryable`
  - `http_status`
  - `provider_error_code`
  - `raw_error`

日志要求：

- warning 日志记录 `reason_code`、`upstream_category`、`retryable`、`provider`、`model`、短 `raw_error`。
- 不记录 API key、完整 prompt、完整响应体。

## 9. 风险与约束

### 9.1 日志敏感信息风险

风险：

- stdout / stderr 可能包含上游 SDK 异常细节。
- 极端情况下第三方库可能把敏感配置写入日志。

控制：

- 仅管理员接口。
- 只读固定日志文件。
- 限制读取大小。
- 后端继续禁止主动记录 API key、credential、secret。

### 9.2 错误分型误判风险

风险：

- 上游供应商错误格式不统一。
- 字符串规则仍可能误判。

控制：

- 分类器集中维护。
- `raw_error` 保留给管理员诊断。
- 未明确内容安全时不输出内容安全文案。

### 9.3 前端展示过载风险

风险：

- 日志内容过长导致页面难用。

控制：

- tail 读取。
- 固定高度滚动。
- 默认 64KB，最大 256KB。

## 10. 验收标准

第一版完成后必须满足：

1. 管理员能在 Web 后台运行状态页直接查看 Web 托管 Workstation 的 stdout / stderr 日志内容。
2. 非管理员不能访问日志读取接口。
3. 前端不能指定任意服务器文件路径。
4. 日志文件不存在、为空、过大时均有明确状态。
5. `Your request was blocked` 不再默认提示“内容安全”。
6. 明确 `content_filter` 类错误仍归内容安全。
7. 上游失败日志包含可诊断字段，但不泄露密钥。
8. 相关接口文档和方案文档同步更新。

## 11. 推荐执行顺序

1. 后端补日志 tail 能力与管理端接口。
2. 前端运行状态页接入 stdout / stderr 查看。
3. 后端补上游错误分类器并替换粗粒度判断。
4. Prompt 边界补强。
5. 更新接口文档与相关方案。
6. 执行后端局部测试和前端 TypeScript 检查。

## 12. 当前结论

Web 工作站已经成为服务器模式执行面的关键节点。只显示日志路径不足以支撑管理员排障；上游失败也不能继续统一解释为内容安全。当前应把“日志可见性”和“上游错误分型”作为同一条诊断闭环推进：先让管理员看到事实，再让系统把事实分成可行动的错误类型。
