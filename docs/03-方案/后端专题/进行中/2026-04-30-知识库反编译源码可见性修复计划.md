# 知识库反编译源码可见性修复计划

> 文档定位：记录知识库包中文件数量偏少、游戏反编译源码缺失提示不清、Web 工作站 Prompt lookup 文案误导的修复计划。
>
> 事实依据：基于 2026-04-30 对 `runtime/knowledge`、知识库导入导出逻辑、单资产服务器 Prompt 组装链路和管理端知识库包页面的代码调查。
>
> 权威入口：上级入口为 `docs/03-方案/后端专题/README.md`。
>
> 最后更新：2026-04-30
>
> 当前状态：进行中

## 当前理解

当前知识库包文件数量偏少不是前端截断。管理端知识库包页面直接展示后端 `pack.files`，后端上传包时也会把 zip 中实际解出的文件列表写入 `pack-meta.json`。

本机当前 `runtime/knowledge/game/` 只有 `sts2_api_reference.md`，没有大量游戏反编译 `.cs`。`runtime/knowledge/baselib/BaseLib.decompiled.cs` 存在，`resources/sts2/*.md` 也存在。

Web 工作站单资产文本方案的 Prompt 拼接链路是存在的：`single_asset_plan_handler` 会组装 `guidance`、`facts`、`lookup` 并渲染到 `runtime_agent.platform_single_asset_server_user`。问题在于只有 `sts2_api_reference.md` 时，当前逻辑仍把游戏知识标成 `runtime_decompiled`，lookup 文案会误导模型去读“runtime knowledge directory”。

## 计划

1. 调整游戏知识来源判定：只有存在 `.cs` 文件时才标记为 `runtime_decompiled`；只有 `sts2_api_reference.md` 时标记为 `reference_only`。
2. 调整 lookup 文案：`reference_only` 使用“STS2 API reference summary”这类标题，明确这是摘要参考文件，不是完整反编译源码目录。
3. 为知识库包元数据增加可观测统计：`resource_md_count`、`game_cs_count`、`baselib_cs_count`，让管理端无需肉眼从文件列表判断源码是否齐全。
4. 管理端知识库包页面展示源码统计，并在 `game_cs_count = 0` 时提示缺少游戏反编译源码。
5. 用局部测试覆盖 source mode、lookup 文案、知识库包统计和前端展示。

## 影响范围

- 后端知识库运行时：`backend/app/modules/knowledge/infra/knowledge_runtime.py`
- 后端知识库 facade：`backend/app/modules/knowledge/application/knowledge_facade.py`
- 后端 lookup provider：`backend/app/modules/knowledge/infra/sts2_lookup_provider.py`
- 管理端知识库包页面与类型：`frontend/src/pages/admin/AdminKnowledgePacksPage.tsx`、`frontend/src/shared/api/admin.ts`
- 相关测试与接口文档。

## 风险

- 这次修复不自动生成完整游戏反编译源码；完整 `.cs` 仍依赖工作站配置了 STS2 路径和 `ilspycmd` 后执行“更新知识库”。
- 旧知识库包没有新增统计字段，前端必须兼容字段缺失。
- 如果已有逻辑把 `runtime_decompiled` 当成“有任意游戏参考资料”，本次改为 `reference_only` 后需要同步测试预期。

## 验收标准

- 只有 `sts2_api_reference.md` 时，`get_game_source_info()` 返回 `source_mode = reference_only`。
- lookup 中不再把 md 文件描述成 runtime knowledge directory。
- 上传和导出的知识库包元数据包含 `resource_md_count`、`game_cs_count`、`baselib_cs_count`。
- 管理端知识库包页面显示源码统计，并对游戏 `.cs` 为 0 的包给出明确提示。
- 局部后端与前端测试通过。
