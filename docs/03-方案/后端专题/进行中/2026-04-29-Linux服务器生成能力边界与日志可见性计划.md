- 管理员应该能看到更细的执行诊断信息，但仍不得暴露密钥、完整 prompt 和用户敏感内容。

## 1. 概念边界

核心边界：

- 用户工作站负责用户本机能力。
- Web 工作站负责服务器生成能力。
- Web 控制面负责用户、任务、额度、凭据、知识库版本和结果记录。

三者不要互相假装能访问对方环境。

| 角色 | 运行位置 | 能访问 | 主要产出 | 不应负责 |
| --- | --- | --- | --- | --- |
| 用户工作站 | 用户电脑 | 用户本机 STS2、Godot、Mods、项目目录、本地日志 | 本机构建、本机部署、本机验证结果 | Web 用户认证、额度、服务器凭据、其他用户任务 |
| Web 工作站 | Web 服务器同机 | 服务器工作区、服务器上传资产、管理员激活的知识库包、Web 下发的执行期凭据 | 服务器生成结果、项目包、执行事件 | 访问用户本机、部署到用户 Mods、Linux 第一版 Godot 构建 |
| Web 控制面 | Web 服务器 | 数据库、任务、用户、额度、服务器凭据、artifact 索引 | 任务状态、事件、审计、下载入口 | 拼业务 prompt、直接跑生成、直接构建部署 |

### 1.1 用户本机工作站

用户本机工作站运行在用户自己的电脑上，能访问：

- 用户本机 STS2 安装目录。
- 用户本机 Godot Mono。
- 用户本机 Mods 目录。
- 用户本机项目目录。

因此，本机工作站适合做：

- 本地构建。
- 本地部署到 STS2 Mods。
- 本地游戏验证。
- 读取本机工具链状态。
- 本地知识库制作或导出。
- 导入 Web 生成的项目包后继续构建和部署。

不应该负责：

- Web 平台用户认证。
- Web 额度扣减。
- Web 任务队列。
- 服务器凭据管理。
- 替 Web 保存其他用户的任务结果。

### 1.2 Web 自启动 Workstation

Web 自启动 Workstation 运行在 Web 服务器同一台机器上，默认只能访问：

- 服务器文件系统。
- 服务器 runtime 目录。
- 服务器上传资产目录。
- 服务器工作区目录。
- 服务器配置中声明的工具路径。

它不能访问用户本机：

- 不能写用户电脑磁盘。
- 不能部署到用户本机 STS2 Mods。
- 不能读取用户本机 Godot。
- 不能假设用户本机游戏目录存在于服务器。

适合负责：

- 服务器侧文本生成。
- 服务器侧代码生成。
- 服务器工作区内项目文件生成。
- 生成项目包 / 源码包 artifact。
- 执行进度事件上报。
- 使用管理员激活的知识库版本。
- 返回结构化结果给 Web 控制面。

不应该负责：

- 访问用户电脑上的 STS2。
- 访问用户电脑上的 Godot。
- 部署到用户本机 Mods。
- 假设服务器安装了 STS2。
- Linux 第一版执行 Godot 构建。
- 直接管理 Web 用户、额度、订单、审计。

服务器生成链路应明确为：

```text
用户浏览器
  -> Web 控制面
  -> Web 同机 Workstation
  -> 服务器 runtime/server-workspaces/*
  -> 服务器 artifact / 结果落库
  -> 用户下载或同步到本机工作站
```

### 1.3 Web 控制面

Web 控制面不是 Web 工作站。Web 控制面负责“谁在用、做什么任务、扣多少额度、用哪份凭据、结果如何保存”，不负责具体业务生成。

Web 控制面负责：

- 登录、用户、权限。
- 任务创建、排队、状态、事件。
- 额度、计费、退款。
- 服务器凭据管理。
- 知识库包上传、激活、回滚，并展示包内相对文件清单用于上传后校对。
- 管理员本机 Workstation 当前知识库一键上传；该链路由浏览器桥接，不意味着 Web 服务器可直接访问管理员本机文件系统。
- 派发任务给 Web 工作站。
- 保存结果和 artifact 索引。
- 向用户提供下载入口。

Web 控制面不应该：

- 拼业务 prompt。
- 直接跑代码生成。
- 直接构建项目。
- 直接访问用户本机资源。
- 直接持有 Workstation 内部执行状态作为唯一真源。

### 1.4 典型链路

服务器生成：

```text
用户浏览器
  -> Web 控制面创建任务
  -> Web 工作站生成文本 / 代码 / 项目包
  -> Web 控制面记录结果和 artifact
  -> 用户下载项目包
  -> 用户本机工作站导入
  -> 用户本机构建 / 部署
```

本机生成：

```text
用户本机工作站
  -> 直接读取本机 STS2 / Godot / Mods
  -> 本地生成 / 构建 / 部署
  -> 可选同步结果或日志给 Web
```

一句话边界：

```text
Web 工作站产出可下载的服务器生成结果；
用户工作站产出可在用户电脑上构建、部署、验证的本地结果。
```

## 2. Linux 部署能力边界

### 2.1 支持能力

Linux Web Workstation 第一版支持：

- 文本规划。
- 代码生成。
- 基于内置知识库和 code facts 的 prompt 组装。
- 服务器工作区内项目文件生成。
- 生成结果落库。
- 生成项目包 / 源码包供用户下载。
- 用户可见执行进度。
- 管理端诊断。

### 2.2 不支持能力

Linux Web Workstation 第一版不支持：

- 服务器侧 Godot 构建。
- 服务器侧部署到 STS2 Mods。
- 自动部署到用户本机游戏目录。
- 假设服务器存在 STS2 游戏安装目录。
- 假设服务器存在可用 Godot Mono。

### 2.3 可选增强能力

后续可选增强：

- Linux 服务器安装并诊断可用 Godot Mono 后，单独开启 `build_only` 能力。
- 通过远端执行节点或用户本机工作站回调，实现“服务器生成，本机构建部署”。
- 支持下载项目包后，由本机工作站导入并继续构建部署。

这些增强不进入第一版默认能力。

## 3. 知识库策略

代码生成需要 STS2 知识，但不应强依赖服务器安装 STS2 游戏。

第一版策略：

- 默认使用仓库内置知识资源：
  - `backend/app/modules/knowledge/resources/sts2/*`
  - 已固化的 prompt guidance。
  - 已整理的 code facts / lookup。
- 管理员可上传知识库 zip 包，后端保存到 `runtime/knowledge/packs/{pack_id}`。
- 管理员可激活或回滚知识库包；激活指针保存在 `runtime/knowledge/active-knowledge-pack.json`。
- 管理端前端已提供知识库包页面，支持列表、上传 zip、从本机 Workstation 导出当前知识库后上传、激活、回滚和包内文件清单校对。
- 本机一键上传使用管理员浏览器访问本机 Workstation；若管理端不是 localhost origin，本机 Workstation 需要允许对应 Web origin，否则会被浏览器 CORS 拦截。
- 运行时读取优先级为 active knowledge pack，其次现有 runtime/内置 seed 知识库。
- `sts2_path` 只作为可选增强配置。
- 没有 `sts2_path` 时，服务器生成不应直接失败。
- 需要刷新真实游戏 DLL / 版本信息时，再要求管理员配置 `sts2_path`。

能力口径：

| 能力 | 是否需要 `sts2_path` | Linux 第一版 |
| --- | --- | --- |
| 文本规划 | 否 | 支持 |
| 代码生成 | 否，优先内置知识库 | 支持 |
| 游戏版本检测 | 是 | 可选 |
| 部署到 Mods | 是 | 不支持 |

## 4. 构建与部署策略

### 4.1 服务器生成不等于构建部署

服务器模式第一版应拆成：

```text
生成项目源码包
  -> 用户下载
  -> 用户本机工作站导入
  -> 用户本机构建
  -> 用户本机部署到 STS2 Mods
```

因此，Web 页面文案不应把服务器模式描述为“服务器构建并部署”。

建议文案：

- “服务器生成项目包”
- “下载后在本机工作站构建”
- “服务器暂不支持构建和部署”
- “部署到游戏需要在本机完成”

避免文案：

- “服务器已部署”
- “已安装到游戏”
- “服务器构建成功”作为默认完成态

### 4.2 Artifact 交付

服务器生成完成后，用户应获得：

- 生成摘要。
- 项目包下载入口。
- 生成文件列表。
- 后续本机构建提示。

artifact 类型建议：

| artifact_type | storage_provider | 用途 |
| --- | --- | --- |
| `source_project` | `server_workspace` | 服务器生成的项目源码目录或压缩包 |
| `source_patch` | `server_workspace` | 代码修改补丁或变更摘要 |
| `generation_report` | `server_workspace` | 生成报告 |

当前代码已优先落 `source_project` 压缩包：

- 生成位置：服务器工作区旁的 `_source_artifacts/{project}.source.zip`。
- 排除目录：`bin/`、`obj/`、`.godot/`、`.git/`。
- 下载入口：`GET /api/me/artifacts/{artifact_id}/download`。
- 前端导入入口：用户中心任务详情页可选择本机目录，将项目包导入本机 Workstation。
- 当前残留：`object_key` 仍是服务器 zip 文件路径；后续如果要完全隐藏服务器路径，应改为 Web 可解析的逻辑对象 key。

## 5. 用户可见流程

用户应看到的是产品化进度，不是原始日志。当前代码已把 Workstation step events 同步到 `job_events`，用户中心任务详情页已将 Workstation 事件渲染为阶段化文案，不再直接展示事件 JSON payload。

建议事件阶段：

```text
已创建任务
已分配服务器工作区
正在准备知识上下文
正在生成方案
正在生成代码
正在整理项目文件
正在生成下载包
已生成可下载项目包
```

失败示例：

```text
服务器暂不支持构建，请下载项目包后在本机工作站构建。
```

用户可见事件字段建议：

```json
{
  "event_type": "workstation.step.started",
  "job_item_id": 2103,
  "payload": {
    "phase": "code_generation",
    "message": "正在生成代码",
    "progress": 40
  }
}
```

用户侧不展示：

- API key。
- 执行期凭据。
- `X-ATS-Workstation-Token`。
- 完整 prompt。
- 完整模型请求 / 响应。
- 服务器绝对路径。
- Python traceback。
- 内部异常栈。

## 6. 管理员诊断日志

管理员可以看到更细的执行诊断，但仍必须脱敏。

管理员可见：

- `job_id`
- `job_item_id`
- `execution_id`
- `workstation_execution_id`
- `step_id`
- `step_type`
- `provider`
- `model`
- `credential_ref`
- `reason_code`
- 简短错误摘要。
- 轮询状态。
- Workstation runtime 状态。
- 耗时、重试次数、并发队列状态。

管理员不可见：

- 解密后的 `credential`。
- `secret`。
- token。
- cookie。
- 完整 prompt。
- 完整模型响应。
- 用户上传文件的私密绝对路径。

## 7. 事件回传设计

当前已有用户事件接口：

- `GET /api/me/jobs/{job_id}/events`
- `GET /api/me/jobs/{job_id}/items`

当前已让 Workstation 执行过程中的 step 事件通过轮询结果回传到 Web，并由 Web 写入 `job_event`。

第一版建议采用“轮询结果顺带同步事件”，不先做回调。

```text
Web 派发执行
  -> Workstation 记录 step events
  -> Web 轮询 execution result
  -> Web 同步新增 events
  -> Web 写入 job_event
  -> 用户通过 /api/me/jobs/{job_id}/events 查看
```

Workstation 轮询结果建议扩展：

```json
{
  "workstation_execution_id": "ws-exec-2203",
  "status": "running",
  "step_id": "single.custom_code.codegen",
  "events": [
    {
      "sequence": 1,
      "event_type": "workstation.step.started",
      "occurred_at": "2026-04-29T10:00:00Z",
      "payload": {
        "step_id": "single.custom_code.codegen",
        "step_type": "code.generate",
        "message": "正在生成代码"
      }
    }
  ]
}
```

Web client 当前在单次 `dispatch_and_poll` 内按 `sequence` 去重同步事件，避免同一轮询链路重复写入。

当前落地约束：

- Workstation 事件类型为 `workstation.step.started` 与 `workstation.step.finished`。
- 用户可见 payload 只包含 `phase`、`step_id`、`step_type`、`message`、`workstation_execution_id`、`sequence` 与 `occurred_at`。
- 事件不得包含执行期凭据、控制 token、完整 prompt、完整模型响应或服务器绝对路径。

{

Web 管理端还应通过 `GET /api/admin/platform/workstation-runtime-logs?stream=stderr&tail_bytes=65536` 读取固定 stdout / stderr 日志 tail。该能力仅用于管理员诊断 Web 托管 Workstation 启动、capabilities 和上游 AI raw error，不面向普通用户展示，也不允许读取任意服务器路径。

## 9. 配置策略

Web 配置负责控制面：

- 数据库。
- auth secret。
- 平台凭据管理。
- `platform_execution.workstation_url`。
- `platform_execution.workstation_config_path`。
- 并发、轮询、超时。

Workstation 配置负责执行面：

- 知识库路径或内置知识库开关。
- `sts2_path`，可选。
- `godot_exe_path`，Linux 第一版不要求。
- `mod_template_path`，可选。
- 工作区根目录。
- 本地工具链能力。

当前已补的配置项：

```json
{
  "platform_execution": {
    "workstation_config_path": "runtime/workstation.config.json"
  }
}
```

Web 启动 Workstation 子进程时，当前已显式注入：

```text
SPIREFORGE_CONFIG_PATH=<workstation_config_path>
ATS_WORKSTATION_CONTROL_TOKEN=<generated token>
```

这样可以避免 Linux 发布态中 Workstation 误读 Web 配置。

## 10. 实施计划

### Task 1：冻结 Linux 服务器模式能力口径（已落地，继续清扫展示文案）

目标：

- 将服务器模式文案和后端状态从“构建/部署”收口为“生成项目包”。
- 明确 Linux Web Workstation 不支持构建和部署。

当前代码事实：

- Workstation capabilities 已明确返回服务器构建和服务器部署不支持。
- Workstation workflow registry 当前不包含 `build.project` 步骤。
- 用户侧仍存在 `delivery_state` 的“已构建/已部署”通用标签渲染，后续要确认服务器生成详情页是否需要按 artifact 类型隐藏这类标签。

验收：

- 用户侧不会看到“服务器已部署到游戏”的误导文案。
- 后端不会在 Linux 默认链路中要求 `sts2_path` 或 `godot_exe_path`。

### Task 2：补 Workstation 配置路径注入（注入已落地，诊断展示待补）

目标：

- 增加 `platform_execution.workstation_config_path`。
- Web 自启动 Workstation 时注入 `SPIREFORGE_CONFIG_PATH`。

验收：

- Web 和 Workstation 可以读取不同配置文件。
- Workstation 托管启动时已注入配置路径；runtime status 当前还未直接返回配置文件路径，后续可补 `workstation_config_path` 或 `config_path` 字段辅助排障。

### Task 3：新增 Workstation capability 诊断（已落地）

目标：

- 增加 `GET /api/workstation/capabilities`。
- 管理端 runtime status 聚合 capabilities。

验收：

- Linux 下明确返回 `server_build_supported = false`。
- Linux 下明确返回 `server_deploy_supported = false`。
- 缺少 `sts2_path` 不影响文本 / 代码生成能力。

### Task 4：服务器生成 artifact 打包下载（已落地，逻辑 key 待收口）

目标：

- 生成完成后把服务器工作区整理为 `source_project` artifact。
- 用户可下载项目包。

当前代码事实：

- `WorkstationPlatformExecutor` 会在成功结果包含 `server_workspace_root` 且目录存在时生成 zip。
- `ExecutionOrchestratorService` 会持久化 `output_payload.artifacts`。
- 用户中心任务详情页会对 `source_project + server_workspace` 展示“下载项目包”。
- 用户中心任务详情页可选择本机目录并调用 Workstation 导入项目包。
- 当前 `object_key` 仍是服务器文件路径，后续可改为逻辑对象 key。

验收：

- 用户能在任务详情中看到下载入口。
- 下载包中包含生成后的项目文件。
- 不要求服务器侧 Godot 构建。

### Task 5：Workstation 事件回传（已落地）

目标：

- Workstation 执行过程中记录 step events。（已落地）
- Web 轮询时同步新事件到 `job_event`。（已落地）
- 用户通过现有 jobs events 接口看到进度。（已落地）
- 用户中心已渲染面向用户的阶段文案，不直接展示 payload JSON。（已落地）

验收：

- 用户能看到阶段进度。
- 管理员能看到更细的 step 诊断。
- 事件不暴露凭据、完整 prompt、token 或内部绝对路径。

### Task 6：管理员知识库包上传、激活、回滚（已落地）

目标：

- 管理员上传服务器知识库包。（已落地）
- 管理员可从本机 Workstation 一键导出当前知识库并上传到服务器。（已落地）
- 管理员激活指定知识库包。（已落地）
- 管理员回滚到上一知识库包或内置 fallback。（已落地）
- Workstation 知识读取优先 active pack，缺项回退到现有 runtime/内置知识库。（已落地）
- 管理端前端页面支持查看、上传、从本机 Workstation 上传、激活、回滚和包内相对文件清单校对。（已落地）

验收：

- Linux 服务器不依赖本机 STS2 游戏也能使用管理员上传的知识库。
- 知识库更新不要求重启 Web 服务器。
- 上传、激活、回滚写入运行时审计事件。
- 包内路径不允许目录穿越。

### Task 7：本机工作站接续构建部署（下载与前端导入闭环已落地）

目标：

- 用户下载服务器生成项目包后，可由本机工作站导入。（下载入口、本机导入 API 与用户中心“导入本机工作站”动作已落地）
- 本机工作站负责 Godot 构建和部署到 STS2 Mods。

验收：

- 服务器生成和本机构建职责清晰。
- 本机构建失败不会被归因为服务器生成失败。

## 11. 风险

- 如果继续把服务器模式描述为“构建部署”，用户会误以为 Web 服务器能写入自己的电脑。
- 如果 Workstation 配置不独立，Linux 发布态容易误读 Web 配置，导致知识库、工具链和路径诊断混乱。
- 如果没有事件回传，用户会认为服务器生成卡死。
- 如果管理员日志过细，可能泄露 prompt、凭据或服务器路径。
- 如果过早支持 Linux Godot 构建，会把部署环境复杂度提前引入第一版。

## 12. 当前推荐

第一版推荐目标：

- Linux Web Workstation 只负责生成。
- 服务器生成结果以项目包 / 源码包 artifact 交付。
- 构建和部署交给用户本机工作站。
- 能力诊断、项目包 artifact、下载入口、导入入口、Workstation step events、用户侧阶段化展示和管理端运行状态页能力展示已落地；下一步优先收口 artifact 逻辑 key 和更完整的管理端执行诊断，再讨论 Linux Godot 构建是否作为增强能力。

## 13. 后续简化规划

> 状态：仅作为后续规划，当前阶段不实施。

### 13.1 简化目标

后续产品表达应尽量从“三个技术角色”收敛为“两个用户可理解入口”。

内部技术角色仍然是：

```text
Web 控制面
  -> Web 工作站
  -> 用户工作站
```

但用户侧产品表达建议简化为：

```text
服务器生成
  -> 本机部署
```

一句话心智：

```text
服务器帮你生成项目包；本机工作站帮你构建并安装到游戏。
```

### 13.2 用户侧入口

#### 服务器生成

用户感知：

- 在 Web 上提交需求。
- 服务器生成项目包。
- 用户查看生成进度。
- 用户下载项目包。

用户不需要知道：

- Web 控制面。
- Web 工作站。
- Workstation runtime。
- 内部派发协议。
- 控制 token。

服务器生成不承诺：

- 服务器构建。
- 服务器部署。
- 写入用户本机 Mods。

#### 本机部署

用户感知：

- 打开本机工作站。
- 导入服务器下载的项目包。
- 本机构建。
- 本机部署到 STS2 Mods。
- 本机验证游戏效果。

本机部署负责所有依赖用户电脑环境的能力：

- 读取本机 STS2。
- 读取本机 Godot。
- 写入本机 Mods。
- 分析本机游戏日志。

### 13.3 管理端入口

管理端可以继续保留更细的技术概念：

- Web 工作站 runtime status。
- Workstation capabilities。
- 知识库包上传、激活、回滚。
- 执行记录。
- step 诊断。
- 服务器凭据。

这些内容不进入普通用户主流程。

### 13.4 不建议的简化

不建议把 Web 工作站和用户工作站合并成同一个产品概念。

原因：

- 两者运行位置不同。
- 两者可访问的文件系统不同。
- 两者安全边界不同。
- 两者对 Godot、STS2、Mods 的访问能力不同。
- 合并后容易让用户误以为服务器能部署到自己电脑。

因此，内部仍应保留：

- Web 控制面。
- Web 工作站。
- 用户工作站。

但 UI 和文案层面只暴露：

- 服务器生成。
- 本机部署。

### 13.5 可简化项

后续可以简化：

- 普通用户 UI 不展示“Web 工作站”字样。
- 服务器任务完成态统一叫“项目包已生成”。
- artifact 统一包装为“下载项目包”。
- Linux 第一版隐藏服务器构建部署选项。
- 知识库更新统一为管理员上传知识库包。
- 管理端能力诊断用“服务器生成能力”聚合展示，而不是暴露底层进程细节。

### 13.6 后续验收标准

当进入简化实施阶段，验收标准为：

- 普通用户能清楚理解：服务器负责生成，本机负责部署。
- 普通用户不会看到误导性的“服务器已部署到游戏”。
- 管理员仍能看到 Web 工作站健康状态和能力诊断。
- 已生成项目包能被本机工作站导入。
- 本机工作站能继续构建和部署项目包。
