## 1. 当前理解

当前前端已经存在 `/admin/users` 页面，且当前工作树已经把它从扩展占位推进为真实管理界面：

- 通过 `frontend/src/shared/api/admin.ts` 调用管理端用户、额度、账本和人工调整接口。
- 页面可展示用户列表、选中用户详情、固定额度摘要和近期额度账本。
- 人工调整会提交 `direction / amount / reason`，由后端写入固定余额和账本。

当前后端已有部分额度基础设施：

- `users`：用户基础资料、邮箱验证、管理员标记。
- `quota_accounts`：用户额度账户。
- `quota_balances`：固定总次数余额主线。
- `quota_buckets`：旧日 / 周周期额度桶，当前只作为历史兼容来源。
- `usage_ledgers`：额度使用账本，当前已可通过 `quota_balance_id` 关联固定余额。
- `GET /api/admin/quota/refunds`：管理员查看退款记录。
- `GET /api/me/quota` 和 `GET /api/platform/quota`：当前仍返回 `daily_limit / weekly_limit` 视图。

现有模型与已确认产品口径存在冲突：

- 产品只需要固定总次数。
- 代码和接口仍表达日/周周期额度。
- 管理端如果继续展示日/周额度，会让用户与管理员看到不一致的概念。

因此，本计划建议先收口额度主线，再补管理端完整版。

## 2. 目标

### 2.1 额度模型目标

把当前额度主线从周期桶收口为固定总次数余额：

- `total_limit`：总次数上限。
- `used_amount`：已消耗次数。
- `refunded_amount`：已返还次数。
- `adjusted_amount`：人工调整净额。
- `remaining`：剩余次数。
- `status`：额度状态。

执行任务继续按每次消耗固定次数，第一版默认每次消耗 `1`。

### 2.2 管理端目标

补齐 `/admin/users` 管理闭环：

1. 用户列表
2. 用户详情
3. 额度详情
4. 额度账本
5. 人工增加 / 扣减次数
6. 异常账号识别
7. 审计与追溯

### 2.3 用户侧目标

用户中心额度展示同步改为固定次数：

- 不再展示 `daily / weekly`。
- 展示固定总次数、已用、已返还、人工调整、剩余次数。
- 用户侧和管理端使用同一套额度语义。

## 3. 非目标

本计划不做：

- 订阅、套餐、自动续期、会员等级。
- 支付、订单、账单、发票。
- 多租户复杂权限系统。
- 批量用户导入。
- 批量额度调整。
- 保留新的日/周周期额度产品口径。
- 为旧 `/api/platform/quota?user_id=` 新增管理页面入口。

## 4. 数据模型方案

### 4.1 新增 `quota_balances`

新增固定余额表作为当前主线：

```text
quota_balances
  id
  user_id
  quota_account_id
  total_limit
  used_amount
  refunded_amount
  adjusted_amount
  status
  created_at
  updated_at
```

建议约束：

- `user_id` 唯一，第一版一个用户一条当前余额。
- `quota_account_id` 关联 `quota_accounts.id`。
- `total_limit / used_amount / refunded_amount / adjusted_amount` 使用整数。
- `status` 第一版复用或对齐额度账户状态：`active / suspended / closed`。

### 4.2 `usage_ledgers` 调整

`usage_ledgers` 继续作为事实账本。

建议新增或调整关联字段：

```text
quota_balance_id nullable
quota_bucket_id nullable
```

第一版推荐：

- 新增 `quota_balance_id`。
- 保留 `quota_bucket_id`，但固定额度主线不再写入它。
- 固定额度新账本都写入 `quota_balance_id`。

建议账本类型：

| 类型 | 含义 |
| --- | --- |
| `reserve` | 执行开始预留 / 消耗次数 |
| `capture` | 执行完成确认 |
| `refund` | 执行失败或取消返还 |
| `admin_grant` | 管理员增加次数 |
| `admin_deduct` | 管理员扣减次数 |

### 4.3 旧 `quota_buckets`

旧表不作为当前主线。

第一版建议：

- ORM 可以暂时保留，避免一次性清理造成无关风险。
- 新服务不再通过 `QuotaBucketRecord` 判断可用额度。
- 文档中把旧日/周字段标记为历史口径或迁移前口径。

## 5. 后端接口设计

### 5.1 用户侧接口

#### `GET /api/me/quota`

返回当前登录用户固定额度视图：

```json
{
  "total_limit": 100,
  "used_amount": 12,
  "refunded_amount": 2,
  "adjusted_amount": 10,
  "remaining": 100,
  "status": "active"
}
```

说明：

- `remaining = max(total_limit + adjusted_amount - used_amount + refunded_amount, 0)`。
- 若额度账户或余额不存在，第一版建议返回全 0 的 inactive 视图，或在注册流程初始化余额。实现前需二选一冻结。

### 5.2 管理端接口

#### `GET /api/admin/users`

查询参数建议：

- `query`：匹配 `user_id / username / email`
- `email_verified`
- `is_admin`
- `quota_status`
- `anomaly`
- `limit`
- `offset`

返回：

```json
{
  "items": [
    {
      "user_id": 1,
      "username": "alice",
      "email": "alice@example.com",
      "email_verified": true,
      "is_admin": false,
      "created_at": "2026-04-28T00:00:00+00:00",
      "quota": {
        "total_limit": 100,
        "used_amount": 12,
        "refunded_amount": 2,
        "adjusted_amount": 10,
        "remaining": 100,
        "status": "active"
      },
      "anomaly_flags": []
    }
  ],
  "total": 1,
  "limit": 50,
  "offset": 0
}
```

#### `GET /api/admin/users/{user_id}`

返回用户基础资料、额度摘要、异常标记和最近活动摘要。

#### `GET /api/admin/users/{user_id}/quota`

返回用户固定额度详情。

#### `GET /api/admin/users/{user_id}/quota/ledger`

查询参数建议：

- `ledger_type`
- `after_id`
- `limit`

返回账本记录：

```json
{
  "items": [
    {
      "ledger_id": 10,
      "ledger_type": "admin_grant",
      "amount": 20,
      "balance_after": 120,
      "reason_code": "manual_adjustment",
      "reason": "补偿测试额度",
      "ai_execution_id": null,
      "created_at": "2026-04-28T00:00:00+00:00"
    }
  ]
}
```

#### `POST /api/admin/users/{user_id}/quota/adjust`

请求：

```json
{
  "direction": "grant",
  "amount": 20,
  "reason": "补偿测试额度"
}
```

规则：

- `direction` 取值：`grant / deduct`。
- `amount` 必须为正整数。
- `reason` 必填。
- 扣减后剩余次数不允许低于 `0`。
- 成功后写入 `quota_balances` 和 `usage_ledgers`。
- 成功后写入审计事件。

返回更新后的额度视图。

## 6. 后端服务设计

### 6.1 新增或重命名服务

建议新增固定额度服务：

- `QuotaBalanceService`
- `QuotaBalanceRepository`
- `QuotaBalanceQueryRepository`

职责：

- 查询固定额度视图。
- 执行预留 / 返还。
- 执行管理员人工调整。
- 计算剩余次数。
- 写入账本。

### 6.2 改造 `QuotaBillingService`

当前 `QuotaBillingService` 依赖 active bucket。

改造后：

- `has_available_quota()` 查询 `quota_balances`。
- `reserve()` 增加 `used_amount` 并写 `reserve` 账本。
- `refund()` 增加 `refunded_amount` 并写 `refund` 账本。
- `capture()` 继续写 `capture` 账本，但不再改变余额。

### 6.3 管理端查询服务

`AdminQueryService` 扩展：

- `list_users()`
- `get_user_detail()`
- `get_user_quota()`
- `list_user_quota_ledgers()`

建议新增命令服务：

- `AdminQuotaCommandService.adjust_user_quota()`

避免把有副作用的调整动作塞进 query service。

## 11. 风险

### 11.1 旧日/周口径残留

如果只改管理端而不改用户侧，用户会看到“日/周额度”，管理员看到“固定次数”，语义会冲突。

应同步收口用户侧 quota view。

### 11.2 账本与余额不一致

人工调整如果只改余额、不写账本，会失去追溯能力。

所有余额变化必须写账本。

### 11.3 迁移数据口径

现有测试数据可能已经使用 `quota_buckets`。

实现时需要决定：

- 测试数据直接迁到 `quota_balances`。
- 或 migration 读取当前 active bucket 初始化 balance。

当前项目未正式上线，优先建议测试数据直接改到新主线，不保留复杂历史迁移包袱。

### 11.4 管理员误扣

扣减不允许把剩余次数扣到 0 以下。

如果需要表达惩罚或冻结，应使用额度状态，而不是负余额。

## 12. 验收标准

第一版落地需满足：

1. 用户侧 `GET /api/me/quota` 返回固定次数视图。
2. 用户中心不再展示日/周额度。
3. 执行任务时按固定余额判断并消耗次数。
4. 执行失败或取消时按固定余额返还次数。
5. 管理员可以查看真实用户列表。
6. 管理员可以查看用户额度详情。
7. 管理员可以查看用户额度账本。
8. 管理员可以增加或扣减用户次数。
9. 人工调整必须写入账本，并要求原因。
10. 扣减不允许导致剩余次数低于 0。
11. 普通用户访问管理端用户与额度接口返回 403。
12. 接口文档同步更新。
