# 2026-04-18 后端兼容层进一步收口方案

## 1. 结论

2026-04-29 进度复核：

- 当前状态：进行中
- 平台 runner 的 `item_name` 主线、平台创建任务旧字段拒绝、`provided_image_path` 本地 fallback 清理已经由代码事实确认。
- 本文继续保留为进行中，是因为 `legacy_direct` 命名、工作站本地链路历史注释和 prompt/容错 fallback 边界仍需要单独复核，不能把所有 fallback 都误判为兼容层。

基于当前项目 **尚未正式上线**、**不需要为迁移期或旧口径并存额外保留兼容包袱** 的约束，后端仍残留的兼容层应继续收口，但要区分：

- **必须继续删除的兼容口径**
- **当前不应误删的正常容错 fallback**

一句话结论：

**下一轮后端兼容层收口，应优先处理平台 runner 中的双字段兼容读法，谨慎处理工作站本地链路里的历史输入 fallback，不要把容错机制误删成“兼容层清理”。**

## 2. 背景

仓库已经在此前阶段完成过一轮兼容层主干收口，核心结论包括：

- 项目未上线，不保留长期兼容层
- `workstation + web` 是当前有效运行边界
- 平台主链与本地工作流主链已基本分开

但在近期服务器模式专题推进中，仍出现了少量新的“兼容式读法”或历史命名残留，主要集中在：

- 平台 runner 的 payload 解析
- 工作站路由的历史输入 fallback
- 配置项中的旧命名

## 3. 本轮扫描结论

### 3.1 必须优先收口的兼容口径

#### 1. `backend/app/modules/platform/runner/single_asset_plan_handler.py`

当前 `_resolve_item_name()` 仍采用双字段兼容：

- `asset_name`
- `name`

问题：

- 平台 payload 收口后，这种双读会继续纵容单资产与批量字段不一致
- 会把历史输入形态固化为长期 contract

建议：

- 下一轮直接收口到 `item_name`
- 不再继续保留 `asset_name / name` 的平台链路兼容读取

#### 2. `backend/app/modules/platform/runner/batch_custom_code_handler.py`

当前 `_resolve_item_name()` 仍采用双字段兼容：

- `name`
- `asset_name`

问题：

- 和 `single_asset_plan_handler` 一样，会让平台链路长期依赖双字段
- 与“未上线、不保留兼容包袱”的项目约束冲突

建议：

- 与 `single_asset_plan_handler` 同步收口到 `item_name`

### 3.2 需要谨慎处理的兼容痕迹

#### 3. `backend/routers/workflow.py`

截至 `2026-04-19` 之前，`_maybe_await_approval()` 的返回值注释里曾有：

- `(False, output): 保留给兼容分支；当前单资产流程不会使用`

判断：

- 这是明显的历史兼容注释/口径残留
- 但它属于工作站本地链路，不是当前平台 payload 收口的主矛盾

建议：

- 该项已在 `2026-04-19` 当天完成清理
- 后续无需再把它作为独立阻塞项

#### 4. `backend/routers/workflow.py`

截至 `2026-04-19` 之前，`_ws_run_with_provided_image()` 曾保留：

- 优先使用 `provided_image_b64`
- fallback 到 `provided_image_path`

判断：

- 这更像“工作站本地输入兼容”
- 是否删除，取决于是否还要支持非浏览器/非前端调用方直接给本地路径

建议：

- 该项已在 `2026-04-19` 当天完成收口
- 当前本地自定义图片链已统一只认 `provided_image_b64`

#### 5. `backend/app/shared/infra/config/settings.py`

截至 `2026-04-19` 之前，`DEFAULT_LLM_CONFIG` 中仍有：

- `execution_mode = "legacy_direct"`

判断：

- 这里更偏历史命名残留，而不是运行时双路径兼容实现

建议：

- 该项已在 `2026-04-19` 当天完成收口
- 当前默认命名已统一为 `direct_execute`

### 3.3 不应误判为兼容层的 fallback

以下机制当前更偏“正常容错”，不建议作为兼容层清理目标：

#### 1. `backend/image/prompt_adapter.py`

- prompt fallback
- provider / 透明背景相关 fallback

原因：

- 这是运行期兜底，不是旧字段或旧协议并存

#### 2. `backend/image/postprocess.py`

- CPU fallback

原因：

- 这是运行环境差异兜底，不是迁移兼容层

#### 3. `backend/app/shared/prompting/prompt_loader.py`

- 模板 fallback

原因：

- 这是资源读取容错，不是历史协议桥接
