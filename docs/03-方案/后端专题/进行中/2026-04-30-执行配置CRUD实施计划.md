# 2026-04-30 执行配置 CRUD 实施计划

> 文档定位：本文规划管理端执行配置从只读默认种子项扩展为可新增、编辑、启用、禁用和受保护删除的实施方案。

> 事实依据：基于当前 `execution_profiles` 表、`ServerExecutionRepositorySqlAlchemy.DEFAULT_EXECUTION_PROFILES`、`platform_admin` 管理端路由、`AdminExecutionProfilesPage` 与服务器凭据管理页面代码整理。

## 1. 当前理解

当前管理端执行配置页只能展示两个默认 seed 出来的 profile：

- `Codex CLI / gpt-5.4`
- `Claude CLI / claude-sonnet-4-6`

后端表结构已经有 `code / display_name / agent_backend / model / description / enabled / recommended / sort_order`，但只有列表接口，没有新增、编辑、启用、禁用或删除接口。服务器凭据可以新增和编辑，并通过 `execution_profile_id` 挂到某个执行配置下。

## 2. 目标

1. 管理员可新增执行配置，例如 `Codex CLI / gpt-5.5`。
2. 管理员可编辑执行配置的名称、后端、模型、描述、推荐状态、排序和启用状态。
3. 管理员可启用/禁用执行配置；普通用户和执行路由只使用启用配置。
4. 管理员可删除未被引用的执行配置；已有引用时返回明确冲突，提示改用禁用。
5. 执行配置 CRUD 与服务器凭据管理保持一致的管理端体验。

## 3. 非目标

- 不改变服务器凭据的加密、健康检查和优先级机制。
- 不引入 provider 到执行配置；provider 仍属于凭据。
- 不支持任意 agent backend，第一版仍限制为 `codex / claude`。
- 不迁移历史任务的 `execution_profile_id`。

## 4. 后端方案

### 4.1 Contract

新增或扩展：

- `CreateExecutionProfileCommand`
- `UpdateExecutionProfileCommand`
- 管理端 list item 保持包含 `id / code / display_name / agent_backend / model / enabled / recommended / sort_order`。

校验：

- `code`、`display_name`、`agent_backend`、`model` 必填。
- `agent_backend` 仅允许 `codex` 或 `claude`。
- `code` 唯一。
- `sort_order` 转为整数。

### 4.2 Repository / Service

建议新增 `ExecutionProfileAdminService` 或在现有 admin service 中补命令方法：

- `create_execution_profile(command)`
- `update_execution_profile(profile_id, command)`
- `set_execution_profile_enabled(profile_id, enabled)`
- `delete_execution_profile(profile_id)`

删除保护：

- 若存在 `server_credentials.execution_profile_id` 引用，返回冲突。
- 若存在 `user_platform_preferences.default_execution_profile_id` 引用，返回冲突。
- 若存在历史 job 引用，返回冲突。
- 无引用时才物理删除。

### 4.3 HTTP API

管理端新增：

```http
POST   /api/admin/platform/execution-profiles
PUT    /api/admin/platform/execution-profiles/{profile_id}
POST   /api/admin/platform/execution-profiles/{profile_id}/enable
POST   /api/admin/platform/execution-profiles/{profile_id}/disable
DELETE /api/admin/platform/execution-profiles/{profile_id}
```

错误口径：

- `400`：字段非法或 code 重复。
- `404`：profile 不存在。
- `409`：profile 已被引用，不能物理删除。

## 5. 前端方案

在 `AdminExecutionProfilesPage` 中补齐：

- 新增/编辑表单。
- 启用/禁用操作。
- 删除操作；删除失败时展示后端返回原因。
- 刷新列表。

表单字段：

- 编号 `code`
- 名称 `display_name`
- Agent 后端 `codex / claude`
- 模型 `model`
- 描述 `description`
- 排序 `sort_order`
- 推荐 `recommended`
- 启用 `enabled`

## 7. 测试计划

后端定向测试：

- 管理员可创建执行配置。
- 重复 `code` 返回 `400`。
- 管理员可编辑执行配置。
- 管理员可启用/禁用执行配置。
- 删除未引用配置成功。
- 删除已被凭据或用户偏好引用的配置返回 `409`。

前端定向测试：

- API client 覆盖新增、编辑、启用、禁用、删除调用。
- 页面至少覆盖展示新增入口和调用 API 的基础行为。

## 8. 验收标准

1. 管理端可创建新的执行配置，并在执行配置列表中显示。
2. 新执行配置可被服务器凭据表单选择。
3. 禁用后的执行配置不进入普通用户可选列表。
4. 被引用的执行配置不能物理删除，返回明确错误。
5. 未引用的执行配置可以删除。
6. 定向后端测试通过，前端 API 测试通过。
