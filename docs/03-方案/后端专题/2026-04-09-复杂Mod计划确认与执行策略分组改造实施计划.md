# 复杂 Mod 计划确认与执行策略分组改造 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在复杂 Mod 批量生成链路中引入 `P0 item 详细确认` 与 `P1 执行策略分组确认` 两道正式闸门，并支持用户选择判断严格程度，以降低黑箱感和执行偏差。

**Architecture:** 保持现有 `plan_mod -> review_items/review_bundles -> confirm_plan/start_with_plan -> batch_workflow` 主链不被推翻，在规划结果与真正执行之间插入新的“计划校验 + 执行策略预览”层。后端新增 `plan validator`、`execution bundle` 计算服务和轻量 `plan review` 复核接口，前端把现有 `review_plan` 拆成 `review_items` 与 `review_bundles` 两段确认流，并通过统一协议承载严格程度、item 校验状态、bundle 预览结果。

**Tech Stack:** Python, FastAPI, dataclasses, TypeScript, React, WebSocket, Node test runner, pytest

---

## Scope Freeze

- 本次改造只覆盖复杂 Mod 批量生成链路，不改单资产工作流。
- 继续复用现有 `plan_mod`、`start_with_plan`、`confirm_plan`、`retry_item`、`resume` 等执行入口；允许新增轻量 `plan review` 复核接口为前端确认页服务。
- 第一阶段不实现“任意拖拽 bundle 编排”，只支持系统建议 + 用户确认 + 要求拆开/补充后重算。
- 第一阶段不做 planner 自动重写整份计划，不做基于项目扫描的深层语义推断。
- 第一阶段默认不自动执行 build、全量测试、重型验证，只做定向轻量验证。

## Risks

- planner 一次生成不出足够字段，导致首版 `待补充` item 过多
- 严格程度策略与 validator / bundle 阈值耦合后，调参成本增加
- `review_items -> review_bundles -> executing` 状态切换会触及现有恢复逻辑
- 恢复、审批、重试链路如果沿用旧阶段名，可能出现状态不同步

## Acceptance

- 规划产出后，前端先进入 `review_items`，而不是直接执行
- item 模糊时必须补充说明，不能静默放行
- `dependency group` 与 `execution bundle` 分层展示
- 用户能在执行前看到 bundle 风险状态
- 严格程度切换会同时影响 P0 校验与 P1 分组
- 后端在 `confirm_plan/start_with_plan` 前仍能守住最终放行闸门
