## 1. 当前理解

当前服务器模式已经把工作区隔离到 `runtime/platform-workspaces/{user_id}/{token}/{project_name}`，但写型链路仍可能并发命中同一 `server_project_ref`。

这会带来三个直接问题：

1. 源码互相覆盖
2. 构建中间产物互相污染
3. 结果归属和失败定位混乱

因此，当前最先要补的不是排队体验，而是“同一工作区不能被并发写”这条正确性底线。

## 2. 本次范围

### 2.1 本次要做

- 新增 `server_project_ref` 级别工作区写锁
- 在平台执行编排层识别写型 workflow
- 命中写型链路时先尝试获取锁
- 锁冲突时返回 `409 server workspace is busy`
- 补最小定向测试
- 同步回写实施计划与阶段总结

### 2.2 本次不做

- 不接统一 `queued`
- 不做 `project_name` 级别部署锁
- 不做数据库租约锁
- 不做多实例场景治理

## 3. 设计口径

### 3.1 资源粒度

- 锁键固定为 `server_project_ref`

### 3.2 写型链路

第一版至少覆盖：

- `code.generate`
- `asset.generate`
- `build.project`

### 3.3 挂点

- 锁应挂在执行编排层
- 不应散落到单个 handler 内部

### 3.4 冲突语义

- HTTP：`409`
- detail：`server workspace is busy`

## 5. 风险

### 5.1 锁粒度挂错

若只锁 `build.project`，前置代码生成仍可能互相覆盖。

### 5.2 锁服务放错层

若把锁散落到 handler，会导致后续排队、日志与部署占用保护很难统一。

### 5.3 过渡语义和未来排队冲突

本次先用 `409` 收口，因此后续接 `queued` 时必须保留稳定 `reason_code`，避免前后口径断裂。

## 6. 验收标准

- 同一 `server_project_ref` 的第二条写型任务不能并发执行
- 当前先返回 `409 server workspace is busy`
- 非写型链路不被误伤
- 至少一条定向测试覆盖该行为
- 文档同步更新为“第一版工作区锁已开始实现”
