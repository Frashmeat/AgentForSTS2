# 2026-03-27 产品讨论纪要

本文为当前阶段的讨论纪要，用于沉淀已确认信息、建议方向与开放问题，便于后续产品、前端与后端继续对齐。本文不是最终规格说明（spec），后续若有新结论应继续补充或修订。

## 一、当前讨论结论

1. 当前产品主形态仍以工作站为主，Web 端仅承担少量补充能力。
2. AI 接入需要同时支持两种模式：
   - 平台模式：由平台服务器承接请求并提供 AI 服务，服务器保存任务与结果，并纳入计费、限额、异常返还与审计体系。
   - BYOK 模式：用户自带 key，本地执行、本地存储，不经过平台服务器。
3. Web 功能定位为控制台或管理台，不承担 BYOK 直连执行职责，且只查看平台模式的数据，不查看 BYOK 本地历史。
4. 前端技术方向短期内不做大规模框架更换，继续沿用 React 18 + TypeScript + Vite 5 + Tailwind CSS。
5. 当前计费长期方向为“月付套餐 + 次数付费”并存，但现阶段可先不落月付，优先预留按周期限额的次数能力。
6. 这份文档用途是讨论纪要，应明确区分“已确认事项”“建议方向”“开放问题”。

## 二、产品需求整理

### 1. 已确认事项

- 产品运行形态以工作站为核心，Web 不是主承载端。
- Web 更适合作为控制台、配置台、账号与配额管理入口，以及平台能力的可视化入口。
- AI 运行模式必须兼容平台模式与 BYOK 模式两套路径。
- 前端层需要统一抽象两种模式，避免业务页面直接耦合到底层接入方式。
- 平台模式下，任务与结果保存在服务器侧，并进入平台的计费、限额、异常返还与审计体系。
- BYOK 模式下，任务执行与结果存储都应保留在本地，服务器不保存 BYOK 的任务记录与结果。

### 2. 建议方向

- 产品叙事上应明确区分“执行端”和“控制端”。
- 工作站负责高执行自由度、本地能力与 BYOK 直连体验。
- Web 负责平台能力承接、状态查看、配置管理、配额呈现与部分任务控制。
- 后续产品文案、权限设计、接口命名都应围绕这两个角色边界展开。

### 3. 仍待继续讨论的问题

- 平台模式下，Web 控制台除观察与配置之外，是否允许发起部分平台托管任务。
- 工作站与 Web 之间的数据同步粒度如何定义，例如平台模式下的会话、任务历史、配额消耗、设备状态等。
- BYOK 模式下除 key 之外，是否还存在需要跨端同步的非敏感配置；若有，同步边界应如何定义。

## 三、前端现状与建议

### 1. 当前前端现状

- 技术栈已确认是 React 18 + TypeScript + Vite 5 + Tailwind CSS。
- 主界面目前更像工作台或控制台应用，而不是典型的多页面网站。
- 当前未引入完整路由体系，整体结构更偏单页工作区。

### 2. 建议方向

- 短期继续沿用现有技术栈，避免在产品方向尚未完全收敛前进行大规模迁移。
- 逐步补齐以下基础设施：
  - 路由能力，用于区分控制台页面、配置页面、配额页面等。
  - 请求状态管理，用于统一处理加载、失败、重试、空态与错误提示。
  - 模式适配层，用统一前端接口屏蔽平台模式与 BYOK 模式差异。
- 建议将“平台模式 / BYOK 模式”抽象为同一组前端服务接口，例如统一的会话、任务、连接状态与错误语义，页面层只依赖抽象接口，不直接感知底层实现细节。

### 3. 当前已识别风险

- WebSocket URL 存在硬编码风险，后续在多环境部署与模式切换时容易失控。
- 连接失败缺少统一兜底，用户体验与故障可观测性都不够稳定。
- 缺少 `onclose` 与资源清理机制，可能带来状态残留、重复连接或内存泄漏风险。
- 页面体量偏大，且存在 `any` 回退，说明当前边界划分与类型约束仍需加强。

### 4. 建议优先级

1. 先补连接生命周期与错误兜底。
2. 再补模式适配层与统一接口。
3. 随后补路由与页面拆分。
4. 最后逐步压缩 `any`、完善类型模型与模块边界。

## 四、Web 与工作站边界

### 1. 工作站职责

- 承担主要执行能力。
- 保留 BYOK 直连模式，不经过平台服务器。
- 适合承接本地化、长连接、执行自由度高的交互与任务。

### 2. Web 职责

- 作为平台控制台存在。
- 承接平台模式下的管理、配置、查看、配额与状态呈现能力。
- 不承担 BYOK 直连执行，不直接替代工作站。
- 仅查看平台模式的数据与记录，不承载 BYOK 本地历史查看。

### 3. 边界原则

- 凡是依赖用户本地 key、强调本地执行路径或不应经过服务器的能力，应优先归于工作站。
- 凡是平台托管、账号体系、配额计费、运维状态、统一控制等能力，应优先归于 Web。
- 两端应通过一致的产品术语与接口抽象协同，但不强求功能完全镜像。

## 五、最新后端结论补充

### 1. 模式边界

- 工作站仍是主产品，Web 仍是控制台。
- 平台模式走服务器 AI，服务器保存任务与结果，并纳入计费、限额、异常返还与审计体系。
- BYOK 模式由用户自己提供 key，本地执行、本地存储，不经过服务器；服务器不保存 BYOK 的任务记录与结果。
- Web 控制台只查看平台模式的数据，不查看 BYOK 本地历史。

### 2. 建议的后端模型

- 平台模式建议采用 `job` + `ai_execution` 双层模型。
- `job` 表示用户视角的大任务，用于承载一次操作或一组批量操作的聚合状态。
- `ai_execution` 表示一次实际 AI 调用，是最小计费单元，也是配额消耗与异常返还的核心记录单位。
- 一个 `job` 可以拆分为一个或多个 `ai_execution`；批量任务本质上是同一 `job` 下的多次 AI 调用。

### 3. 状态与结算概念

- 平台模式典型任务状态建议包括：`pending`、`running`、`succeeded`、`partial_quota_exhausted`、`failed_business`、`failed_system`、`cancelled`。
- 平台模式典型结算状态建议包括：`reserved`、`captured`、`refunded`。
- 这里仍是讨论纪要层面的建议收敛，不代表最终数据库字段或接口枚举已经冻结。

### 4. 计费、配额与返还规则

- 只有“实际调用 AI”的操作才算计费单元；与 AI 无关的操作不计费。
- 计费最小单位不是整个 `job`，而是单次 AI 调用，即 `ai_execution`。
- 批量任务按“边执行边检查配额”的方式运行，不预占整批额度。
- 若运行过程中额度不足，则中止后续 AI 子任务，已完成部分保留，并反映为部分完成或额度耗尽状态。
- 系统异常时自动返还次数；是否返还由系统自动判定，管理员只查看记录，不做人肉改判。

## 六、计费与配额预留

### 1. 当前共识

- 长期计费模型需要支持：
  - 月付套餐。
  - 次数付费。
- 当前阶段可以先不落月付，但系统设计不能把月付路径堵死。
- 当前次数扣减应围绕 `ai_execution` 建模，而不是按整个 `job` 一次性扣减。

### 2. 近期优先预留

- 日限额次数。
- 周限额次数。

这意味着无论前端展示还是后端计数模型，都应具备按周期统计与限流的能力，而不是只做一次性的总次数扣减。

### 3. 产品与技术含义

- 前端需要能展示用户当前配额、剩余额度、周期刷新时间与超限提示。
- 后端需要能表达至少两类周期配额：按日、按周，并支持按次 AI 调用实时检查与扣减。
- 后续若增加月付套餐，应尽量复用同一套配额与权益模型，而不是重新起一套计费结构。

## 七、后端待讨论问题

1. 平台模式的 `job` 与 `ai_execution` 分层应如何落到具体表结构、接口返回与事件模型中。
2. 配额模型如何抽象，是否统一为“周期窗口 + 消耗事件 + 权益规则”，并覆盖日限额与周限额。
3. 日限额与周限额的统计口径如何定义，按用户、按设备、按工作区还是按具体功能计数。
4. 平台模式与工作站之间是否需要统一的任务 ID、会话 ID 与状态机，以便控制台可观测。
5. Web 控制台若未来发起平台托管任务，需要什么鉴权、审计与风控能力。
6. 连接失败、断线重连、任务中断等事件是否需要服务端统一事件语义，供前端模式适配层复用。
7. 自动返还的判定标准如何细化，例如网络抖动、模型服务超时、业务校验失败等场景的归类标准。

## 八、下一步建议

1. 先由产品与后端确认“平台模式 / BYOK 模式”的服务边界，避免前端抽象方向反复。
2. 基于当前共识，整理一版更接近 spec 的接口草案，重点覆盖 `job`、`ai_execution`、连接状态与配额模型。
3. 前端按不换框架的前提推进治理，优先修复连接生命周期、错误兜底和模式适配层。
4. 将 Web 明确定位为控制台，避免在早期把 BYOK 直连执行错误塞进 Web。
5. 提前为日限额、周限额和异常自动返还设计数据结构与展示位，月付套餐后续再接入。
