# 兼容层收口与审查缺陷修复实施计划 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 收口当前项目中未上线阶段不再需要的兼容口径、迁移开关与 compat 实现，并一并修复审查中发现的认证、安全和敏感信息处理缺陷。

**Architecture:** 以后端双角色 `workstation + web` 作为唯一有效运行边界，删除 `full` 兼容态、迁移 flags 和 platform compat bridge，让工作流与平台 API 均回到单一路径。安全修复以“先封堵越权与账户接管面，再收紧前端敏感信息暴露面”为顺序，文档口径同步更新，避免代码与说明继续分叉。

**Tech Stack:** Python, FastAPI, SQLAlchemy, pytest, React, TypeScript, Vite

## 2026-04-10 执行状态

## 背景与收口结论

- 当前项目尚未正式上线，不需要继续维持迁移期开关、兼容桥接和多套口径并存。
- 当前主链路已经清晰：
  - `workstation` 承接本地工作流、配置、审批、构建与部署
  - `web` 承接认证、用户中心、平台任务与额度接口
- 当前需一起修复的审查缺陷包括：
  - 默认会话密钥导致可伪造登录态
  - `/auth/forgot-password` 直接返回重置码
  - `/platform/*` 通过 `user_id` 参数直接越权访问
  - 平台模式未严格执行邮箱验证约束
  - 注册/找回密码流程把一次性验证码写入 URL

## 风险与防线

- 风险：删除 compat 相关文件后，测试基线会集中报错。
  - 防线：先改定向测试，再删实现，避免“一次删光后无法判断坏在哪”。

- 风险：README 和历史实施文档仍可能残留过时口径。
  - 防线：把文档收口列为独立 chunk，不作为“顺手修改”。

- 风险：前端验证码不进 URL 后，用户刷新页面会丢临时态。
  - 防线：保留手动输入入口，把路由 state 仅作为便捷路径，不作为唯一入口。

- 风险：平台接口收口时可能误伤现有用户中心流程。
  - 防线：优先保住 `/me/*` 主链路，再处理 `/platform/*`。

## 实施顺序

1. 先删开关和 compat bridge
2. 再收口工作流单路径
3. 再修认证/平台安全缺陷
4. 最后修前端验证码 URL 暴露与文档口径
