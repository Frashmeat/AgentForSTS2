# 2026-04-27 工作站与 Web 端异常处理统一计划

> 文档定位：本文规划工作站本机链路与 Web 平台链路的异常分类、取消语义和资源清理规则统一，不直接作为具体代码实现记录。
>
> 事实依据：基于当前代码中的 `http_errors.py`、`ws_errors.py`、工作站 WebSocket 工作流、Web 平台任务链路以及本次讨论整理。
>
> 权威入口：上级入口为 `docs/03-方案/后端专题/README.md`。
>
> 最后更新：2026-04-27

## 0. 当前状态

- 当前状态：未开始
- 已确认：
  - HTTP 异常外壳已有统一基座。
  - WebSocket 错误事件已有统一工具。
  - 工作站与 Web 端的业务异常语义尚未统一。
  - 用户取消、客户端断开和资源清理还没有成为统一的一等终止原因。
- 本文建议：统一核心语义，不强行统一传输载体。

## 1. 当前理解

当前项目同时存在两类运行链路：

- 工作站本机链路：
  - 主要通过 `/api/ws/create`、`/api/ws/batch`、`/api/ws/build-deploy` 等 WebSocket 推进。
  - 错误更偏当前会话即时提示。
  - 前端关闭连接或重置 UI 不等同于后端取消执行。
- Web 平台链路：
  - 主要通过 HTTP 创建任务，再通过 `jobs / job_items / ai_executions / job_events` 持久化状态。
  - 错误更偏可追溯任务事实。
  - 具备 `cancel_job` 标记入口，但取消与正在执行 runner 的中止尚未完全打通。

因此，推荐目标不是让两端返回完全相同的结构，而是让同类问题拥有同一个错误码、同一套终止语义和同一套资源清理规则。

## 2. 当前代码事实

### 2.1 已有统一点

- HTTP 统一异常 envelope：
  - `backend/app/shared/infra/http_errors.py`
  - 当前输出形态为 `{"error": {"code", "message", "detail", "request_id"}}`
  - `app_factory.create_app()` 已统一安装 `install_http_error_handlers(app)`
- WebSocket 错误事件工具：
  - `backend/app/shared/infra/ws_errors.py`
  - 当前输出形态为 `{ "event": "error", "code", "message", "detail", "request_id", "traceback" }`
- 前端展示层已有轻量消息归一：
  - `frontend/src/shared/error.ts`
  - 当前主要负责从不同错误形态中提取 `message`

### 2.2 当前不统一点

- HTTP 与 WebSocket 没有共享错误码表。
- 工作站 WebSocket 里的错误分类多为局部实现，例如单资产 workflow 自带 `_classify_workflow_error()`。
- 批量、构建部署、日志分析、Mod 分析各自决定错误码和提示。
- Web 平台任务错误会进入 `error_summary / error_payload / job_events`，但与工作站 WebSocket 错误码没有统一对应关系。
- `DomainError` 已存在，但还没有作为业务异常主线广泛使用。
- 用户取消和客户端断开没有形成统一终止原因。

## 3. 目标

### 3.1 核心目标

统一以下四类规则：

1. 错误码分类
2. 取消与断开语义
3. 资源清理要求
4. 前端展示与后续动作

### 3.2 传输载体目标

不同链路允许继续使用不同载体：

- HTTP：继续使用 `{"error": ...}` envelope。
- WebSocket：继续使用 `{event:"error", ...}` 或后续新增 `{event:"cancelled", ...}`。
- Web 平台任务：继续持久化到 `job_events / error_payload / status`。

但三类载体必须能表达同一个 `code` 和同一个终止原因。

## 4. 非目标

本计划不要求：

- 把 WebSocket 错误强行改成 HTTP envelope。
- 把工作站本机任务全部改造成 Web 平台 job。
- 一次性重写所有 router 的异常处理。
- 一次性覆盖所有第三方 provider 的远端取消能力。
- 把用户主动取消记录为系统失败。

## 5. 推荐统一错误码

第一版建议先冻结最小公共码表：

| code | 含义 | 工作站行为 | Web 端行为 |
| --- | --- | --- | --- |
| `user_cancelled` | 用户主动取消 | 停止后续调用，显示“已取消” | 标记取消请求，停止/跳过后续执行 |
| `client_disconnected` | 客户端断开 | 触发本地执行清理 | 若有任务事实，记录断开或转取消策略 |
| `api_key_invalid` | 本机或平台凭据无效 | 提示去设置页更新 key | 记录凭据健康异常，可切换凭据 |
| `credential_invalid` | 平台凭据不可用 | 通常不适用或提示服务器配置异常 | 记录 server credential 异常 |
| `rate_limited` | 限流 | 提示稍后重试或更换 key | 可触发重试、切 key 或退款 |
| `quota_exhausted` | 额度耗尽 | 提示额度不足 | 标记额度阻塞或返还 |
| `network_error` | 网络错误 | 提示检查网络 | 记录可重试系统异常 |
| `timeout` | 超时 | 停止当前等待，提示重试 | 记录可重试系统异常 |
| `resource_busy` | 资源占用 | 提示当前工作区或部署目录被占用 | 转 `queued` 或保留排队原因 |
| `workflow_not_supported` | 当前模式不支持该任务 | 提示换模式或等待接入 | 记录 deferred 原因 |
| `validation_error` | 输入校验失败 | 高亮用户输入问题 | HTTP 400 或任务创建失败 |
| `internal_error` | 未分类系统异常 | 兜底失败提示 | 500 或系统失败 |

后续可以在 `meta` 中保留细分字段，例如：

- `provider`
- `resource_key`
- `job_id`
- `job_item_id`
- `ai_execution_id`
- `retryable`
- `cleanup_required`

## 6. 取消语义冻结

### 6.1 用户主动取消

`user_cancelled` 应被视为“终止”，不是“失败”。

要求：

- 不显示为红色系统异常。
- 不触发普通失败重试。
- 应触发资源清理。
- 若已有计费或占用，应按链路决定退款、释放锁或标记取消。

### 6.2 客户端断开

`client_disconnected` 应被视为独立终止原因。

第一版建议：

- 工作站本机链路：默认按取消处理，立即停止后续 AI 调用和轮询。
- Web 平台链路：如果任务已持久化，不应仅因页面断开就自动取消；应继续以平台 job 状态为准。

### 6.3 取消与失败的区别

取消类终止不应写成：

- `internal_error`
- `workflow_runtime_error`
- `failed_system`

除非取消过程本身发生清理失败，并且清理失败需要单独审计。

## 7. 资源清理规则

第一版至少要求以下资源具备统一清理口径：

- 工作站 WebSocket 当前执行上下文
- 图像生成轮询任务
- CLI agent 子进程
- 批量生成中的 asyncio 子任务
- 工作区写锁
- 部署目录锁
- Web 平台的 `ai_execution` 运行状态
- 已预留但未捕获的额度

清理规则建议：

1. 用户取消或客户端断开时，停止后续新请求。
2. 已提交到第三方的远端任务如果 provider 不支持取消，本地也必须停止轮询、下载和重试。
3. CLI runner 必须支持取消时终止子进程。
4. 资源锁释放必须放在 `finally` 或统一执行上下文清理阶段。
5. Web 平台任务的取消必须与 `ai_execution` 状态、额度返还和事件记录保持一致。

## 8. 实施步骤

### Step 1：冻结共享错误码与终止原因

- 新增或扩展共享错误定义模块。
- 明确 `code / message / detail / meta` 基本字段。
- 将 `user_cancelled / client_disconnected` 纳入第一版。

### Step 2：统一 WebSocket 错误与取消事件构造

- 保留 `send_ws_error()`。
- 增加取消类事件或在 error payload 中明确 `code=user_cancelled`。
- 避免把取消显示成普通失败。

### Step 3：工作站本机链路接入取消上下文

- `/api/ws/create` 接收 `{"action":"cancel"}`。
- `/api/ws/batch` 接收全局取消消息。
- WebSocket 断开时触发统一取消上下文。
- 图像生成、prompt 适配、代码 agent 和批量子任务检查取消信号。

### Step 4：CLI runner 支持可取消执行

- `run_streaming()` 支持取消信号。
- 取消时终止子进程。
- Windows 下优先保证子进程树能被清理。

### Step 5：Web 平台任务取消语义补齐

- `cancel_job` 不只标记 `cancel_requested_at`。
- 正在执行的 `ai_execution` 需要能进入取消路径。
- 额度、锁、事件记录要与取消状态一致。

### Step 6：前端按错误码分流展示

- `resolveErrorMessage()` 保留兜底。
- 新增基于 `code` 的行为映射：
  - 去设置页
  - 稍后重试
  - 显示已取消
  - 显示排队/占用
  - 展开 traceback

### Step 7：文档与规范回写

- 更新 `docs/02-现状/当前前后端接口文档.md`。
- 回填 `.trellis/spec/backend/error-handling.md` 的实际项目约定。
- 如涉及前端统一展示，也同步回填前端相关规范。

## 10. 风险

### 10.1 过度统一风险

如果强行统一 HTTP、WebSocket、数据库任务事件的完整结构，会增加迁移成本，并破坏当前各链路的自然形态。

应只统一核心字段和语义，不统一所有载体细节。

### 10.2 远端任务不可取消

部分图像 provider 已提交的远端任务可能无法真正撤销。

第一版验收重点应是：

- 本地停止轮询
- 不继续下载
- 不继续重试
- 不继续发起后续 AI 调用

而不是承诺所有 provider 都能释放远端已提交任务。

### 10.3 取消误判为失败

若取消仍走普通异常分支，前端会误报失败，Web 端也可能误触发退款、重试、降级凭据等错误动作。

必须先把 `user_cancelled` 与 `client_disconnected` 从普通异常中分离。

## 11. 验收标准

第一版满足以下条件即可视为计划落地：

1. 工作站和 Web 端同类异常使用同一 `code`。
2. HTTP、WebSocket、任务事件都能表达 `code / message / detail / meta`。
3. 用户主动取消不会被记录为系统失败。
4. 工作站 WebSocket 断开会停止本地后续 AI 调用和轮询。
5. CLI agent 在取消时能终止子进程。
6. 批量生成取消时能取消未完成子任务。
7. Web 平台取消能与 `job / job_item / ai_execution / quota` 状态一致。
8. 前端能基于错误码做最小分流展示。
9. 相关接口文档和 error-handling 规范已同步更新。

## 12. 推荐推进顺序

推荐先推进最能解决当前痛点的部分：

1. 冻结 `user_cancelled / client_disconnected` 语义。
2. 工作站 WebSocket 链路接入取消上下文。
3. CLI runner 与图像生成轮询接入取消。
4. 前端加停止按钮和取消展示。
5. 再把 Web 平台任务取消语义补齐。
6. 最后统一更多业务错误码。

## 13. 当前判断

工作站与 Web 端异常处理不需要完全同形，但必须同义。

当前最优先的突破口是取消语义，而不是完整错误体系重构。只要先把 `user_cancelled` 和 `client_disconnected` 做成统一终止原因，就能直接解决“工作站无法主动打断、强退后 key 仍被使用”的主要风险，并为后续统一错误码表打基础。
