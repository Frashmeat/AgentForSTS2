# 2026-04-20 服务器后台排队消费第一版实施计划

## 1. 当前理解

当前 `queued` 任务已经有两类稳定原因码：

- `server_workspace_busy`
- `server_deploy_target_busy`

但自动恢复仍有明显边界：

- 只有在某次 `start_job()` 调用链结束时，才会顺手尝试 `drain`
- 如果占用资源释放后，没有新的 `start_job()` 触发，排队任务可能一直停在 `queued`
- 当前没有后台 worker、没有 app 启动后的轮询任务、也没有跨进程真源消费器

因此，当前已经完成的是：

- `queued` 语义收口
- 原因码收口
- 同进程内最小自动续跑

当前还没完成的是：

- 服务器级持续排队消费
- 跨进程 / 多实例一致消费
- 部署登记真源

## 2. 目标

本阶段目标不是一步做到“分布式任务系统”，而是先把当前 `queued` 恢复机制从“顺手续跑”升级为“web 后端自带后台轮询消费”。

第一版目标：

1. `web` 进程启动后，后台可周期性扫描可尝试恢复的 `queued` 任务
2. 后台恢复继续复用现有 `JobApplicationService.start_job()` 主链，而不是另起一套执行逻辑
3. 对命中 `server_workspace_busy / server_deploy_target_busy` 的任务，继续保持原语义，不把资源忙误写成失败
4. 明确第一版只承诺“单 web 实例下的后台排队消费”，不声称已解决多实例一致性

## 3. 非目标

本阶段不做：

- 多实例分布式 leader election
- 数据库租约锁 / Redis 队列 / 外部消息队列
- 完整优先级调度系统
- 部署登记元数据真源
- 新的主状态枚举

## 4. 代码现状调查结论

### 4.1 当前恢复逻辑在哪里

当前恢复逻辑只在：

- `JobApplicationService.start_job()`
- `JobApplicationService._drain_queued_jobs_for_resources()`

特点：

- 恢复入口不是后台线程，而是用户显式触发的 `start_job()` 调用链
- 资源释放后，只会顺手拉起“同工作区”或“同部署目录”的下一条 `queued` 任务

### 4.2 当前已有的真源可复用点

当前可直接复用：

- `jobs.status = queued`
- `job_items.status = ready`
- `job_event.event_type = job.queued`
- `queued_reason_code / queued_reason_message`
- `find_next_queued_job_for_server_workspace()`
- `find_next_queued_job_for_server_deploy_target()`

### 4.3 当前缺口

当前还没有：

- “找一条全局最早可尝试恢复的 queued 任务”的仓储接口
- app 生命周期里的后台轮询挂点
- 后台消费的重入保护
- 后台消费者的单实例边界说明

## 5. 第一版推荐方案

当前推荐方案冻结为：

- **单 web 实例、数据库轮询、复用 `start_job()` 主链**

### 5.1 基本思路

1. `web` app 启动时注册一个轻量后台轮询器
2. 固定间隔扫描“候选 queued 任务”
3. 每次只挑一条最早候选任务尝试恢复
4. 调用现有 `JobApplicationService.start_job(triggered_by="system_queue_worker")`
5. 若恢复后仍命中资源忙：
   - 保持 `queued`
   - 不视为错误
6. 若恢复成功并释放资源：
   - 继续利用现有 `_drain_queued_jobs_for_resources()` 顺带拉后续任务

### 5.2 候选任务筛选建议

第一版建议新增统一仓储查询，例如：

- `find_next_runnable_queued_job(exclude_job_ids=...)`

筛选条件建议固定为：

- `jobs.selected_execution_profile_id is not null`
- `jobs.status = queued`
- 至少存在一条 `job_item.status = ready`
- 按：
  - `jobs.started_at asc`
  - `jobs.id asc`

第一版不在查询层直接判断资源是否空闲。

原因：

- 当前资源可用性判断已经封装在现有 `start_job()` 主链里
- 后台 worker 只负责“挑任务并尝试”，不再复制一套资源判定逻辑

### 5.3 轮询节奏建议

第一版建议：

- 固定 2~5 秒轮询一次
- 单次最多恢复 1 条任务

原因：

- 避免在尚无更细粒度租约前引入过度并发
- 先把语义跑通，再考虑批量恢复和吞吐

## 6. app 挂点建议

当前代码事实表明：

- `app_factory.create_app()` 已在 `web` 模式做启动期初始化
- 目前只有 `_bootstrap_web_execution_profiles(app)`

因此第一版建议新增：

- `web` app 启动后注册后台 queue worker
- `web` app 关闭时取消该 worker

第一版不建议：

- 把后台轮询藏进 router
- 把后台轮询塞进某个请求链中惰性启动

## 7. 风险与边界

### 7.1 单实例边界

第一版必须明确：

- 只承诺“单 web 实例下有效”
- 若未来出现多个 `web` 进程同时启动后台轮询器，当前不能声称已完全避免重复消费竞争

### 7.2 重入风险

后台 worker 若频繁命中同一条持续忙的任务，可能造成：

- 重复写入 `job.queued`
- 空转扫描

因此第一版建议至少增加最小退避之一：

- 记录最近一次后台尝试时间
- 或基于最近 `job.queued` 事件时间做最小冷却窗口

### 7.3 不要复制业务逻辑

后台 worker 不应复制：

- 工作区锁判断
- 部署锁判断
- quota 判断
- workflow 注册判断

这些都应继续由：

- `JobApplicationService.start_job()`
- `ExecutionOrchestratorService`

统一负责。

## 8. 影响范围

预计会涉及：

- `app_factory.py`
- `backend/app/modules/platform/application/services/job_application_service.py`
- `backend/app/modules/platform/domain/repositories/job_repository.py`
- `backend/app/modules/platform/infra/persistence/repositories/job_repository_sqlalchemy.py`
- 新增后台 queue worker service
- 定向测试
- 接口/阶段总结文档回写

## 9. 验收标准

当以下条件满足时，可视为第一版后台排队消费落地：

1. 在没有新的用户 `start_job()` 请求进入时，`queued` 任务仍能被后台继续尝试恢复
2. 命中 `server_workspace_busy` 或 `server_deploy_target_busy` 时，不会被误写成失败
3. 单 web 实例下，资源释放后排队任务能在合理时间内自动推进
4. 至少补：
   - 后台 worker 轮询恢复测试
   - 工作区忙重试不误失败测试
   - 部署目录忙重试不误失败测试
5. 文档已明确第一版仍是单实例语义，不伪装成分布式真源调度

## 10. 当前建议的实施顺序

建议按以下顺序推进：

1. 先补统一候选任务查询接口
2. 再补后台 queue worker service
3. 再把 worker 挂到 `web` app 生命周期
4. 最后补定向测试和文档回写

## 11. 当前判断

当前最值得做的不是继续扩新 workflow，而是先把 `queued` 从“有状态但无后台消费”推进到“单实例可持续恢复”。

这是把当前服务器模式从“可演示”推进到“更接近真实多人并行使用”的最短路径。

## 12. 2026-04-20 实现回写

截至 `2026-04-20`，本文对应的第一版最小实现已开始落地，当前状态为：

- `web` app 启动后当前会注册单实例后台 queue worker
- worker 会周期性扫描可恢复的服务器 `queued` 任务
- 当前候选任务查询已新增统一仓储入口：
  - `find_next_runnable_queued_job(now, cooldown_seconds=...)`
- 候选条件当前固定为：
  - `jobs.status = queued`
  - `selected_execution_profile_id is not null`
  - 至少存在一条 `job_item.status = ready`
  - 按 `started_at / id` 顺序挑最早候选项
- worker 当前直接复用：
  - `JobApplicationService.start_job(triggered_by="system_queue_worker")`
- 最小退避当前基于最近一次 `job.queued` 事件时间做冷却窗口
- 工作区忙与部署目录忙的任务，当前都会继续保持 `queued`

当前仍未完成：

- 多实例 / 多进程协同消费
- 部署登记元数据真源
- leader / 租约 / 去重消费策略
