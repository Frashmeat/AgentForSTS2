# 2026-04-18 服务器模式上传资产与项目目录语义收口方案

## 1. 结论

当前平台任务 payload 的前后端第一阶段收口已经完成，下一阶段主矛盾不再是字段名不一致，而是：

1. 上传图片目前只有本地内存或本地路径语义，服务器无法消费
2. `project_root` 仍然是“用户本机目录”语义，服务器无法消费

因此，下一阶段不应继续补小型 workflow，而应在以下两条线中择一作为主攻方向：

- **上传资产持久化引用**
- **`project_root` 服务器工作目录语义**

一句话结论：

**若目标是尽快扩大服务器可执行任务边界，优先级应先做“上传资产持久化引用”；`project_root` 的服务器工作目录语义应放在其后。**

## 2. 当前事实

### 2.1 上传图片当前只存在本地链路语义

单资产当前行为：

- `frontend/src/features/single-asset/controller.ts`
  - 本地工作流在 `imageMode === "upload"` 时，会发送：
    - `provided_image_b64`
    - `provided_image_name`
- 这些字段只在工作站 WebSocket 链路中被消费

批量当前行为：

- `frontend/src/features/batch-generation/view.tsx`
  - `PlanItem` 上仍保留 `provided_image_b64`
  - 批量工作流 WebSocket 本地链路会直接消费该 base64

服务器模式当前行为：

- 平台 payload 现在已经不再接受：
  - `has_uploaded_image`
  - `provided_image_b64`
  - `provided_image_name`
- 前端当前会在服务器模式下直接阻止上传图片任务进入平台链路

结论：

- 上传图片目前已经被彻底限制在本地工作流
- 服务器链路没有任何可持续、可复用、可审计的图片引用机制

### 2.2 `project_root` 当前仍是本机目录语义

单资产和批量本地链路当前都仍会使用：

- `project_root`

主要消费位置：

- `backend/routers/workflow.py`
- `backend/routers/batch_workflow.py`
- `backend/routers/build_deploy.py`
- `backend/app/modules/platform/application/services/build_deploy_facade_service.py`

这些位置都会把 `project_root` 直接当作：

- 本地项目目录
- 本地构建目录
- 本地输出目录
- 本地图像写入目录

服务器模式当前行为：

- 后端平台链路已拒绝 `project_root`
- `local_project_root_required` 当前只保留为阻塞语义说明

结论：

- `project_root` 目前是纯本地概念
- 服务器链路没有“服务器工作目录 / 项目快照 / 远端仓库引用”的替代语义

### 2.3 服务器工作区最小引用链已落地

截至 `2026-04-18`，与 `project_root` 服务器语义相关的第一步最小能力已经补上：

- 后端已新增最小服务器工作区服务
- 已新增 `POST /api/me/server-workspaces`
- 平台链路当前已经可以接受并校验 `server_project_ref`

当前还没有完成的部分是：

- 前端当前已接工作区预分配链路，会先创建 `server_project_ref` 再创建平台任务
- runner 当前已能把 `server_project_ref` 的工作区元数据注入执行输入，`batch_generate/custom_code` 与 `single_asset_plan` 已开始读取服务器工作区摘要来组织文本实现方案；`batch_generate/custom_code` 与 `single_generate/custom_code` 还已进一步接到 `code.generate + build.project`，会把代码写入并在服务器工作区内完成项目构建，构建产物也已开始落入平台 `artifacts`
- 若服务器配置了可用的 `sts2_path/Mods`，`build.project` 当前还会顺带完成最小部署；否则会显式停在“已构建未部署”状态
- build / deploy 仍未真正消费服务器工作区目录语义
- 也就是说，当前已经从“仅可引用”推进到“部分文本任务真实消费 + custom_code 真实写代码 + 服务器构建”，并开始具备最小部署能力，但还没有进入完整部署执行语义

## 3. 两条路线的复杂度对比

### 3.1 上传资产持久化引用

目标：

- 把“用户上传了一张图片”从本地内存语义，转换成平台服务器可读的稳定引用

需要补的能力：

1. 上传图片保存位置
   - 本地文件系统临时目录
   - 数据库存 metadata
   - 或专用对象存储（如果未来需要）

2. 平台任务 payload 新字段
   - 例如 `uploaded_asset_ref`

3. 后端运行时消费
   - runner 能根据 `uploaded_asset_ref` 读取图片内容

4. 生命周期治理
   - 何时清理
   - 谁拥有引用
   - 如何审计

收益：

- 一旦打通，最直接能扩大“服务器模式 + 上传图片”任务范围
- 对单资产和批量都有效

代价：

- 需要新增存储层或持久化落点
- 需要补资产权限和清理策略

### 3.2 `project_root` 服务器工作目录语义

目标：

- 让服务器链路拥有自己的“项目目录”语义，而不是继续依赖用户本机路径

候选做法：

1. 服务器侧工作目录模板
2. 平台项目快照目录
3. 远端仓库/上传包恢复出的工作目录

需要补的能力：

- 服务器侧项目创建
- 服务器侧文件写入
- 服务器侧 build / package / deploy 语义
- 项目生命周期与清理

收益：

- 一旦打通，服务器模式就能真正进入“写代码 / 构建 / 产物输出”链

代价：

- 远高于上传资产引用
- 会牵动 codegen、build、deploy、artifact、工作目录清理等多条链

## 4. 推荐优先级

推荐顺序：

1. **先做上传资产持久化引用**
2. **再做 `project_root` 服务器工作目录语义**

原因：

- 上传图片当前已经有明确的输入来源，只差“变成服务器可读引用”
- `project_root` 则需要重新定义服务器侧项目生命周期，改动面更大
- 上传资产引用一旦打通，可以立即释放一批“文本 + 图片输入”类服务器任务

## 5. 最小实施方案建议

### 5.1 若先做上传资产持久化引用

建议最小方案：

1. 新增平台上传资产存储
   - 落地到 `runtime/` 下受控目录，先不引入外部对象存储

2. 新增平台任务字段
   - `uploaded_asset_ref`

3. 前端在服务器模式下改成：
   - 先上传图片
   - 拿到 `uploaded_asset_ref`
   - 再创建平台任务

4. 后端 runner 先只支持“读取引用”
   - 暂不扩展完整图片工作流

### 5.2 若先做 `project_root` 服务器语义

建议最小方案：

1. 只先支持服务器侧临时工作目录
2. 只给最小文本/代码型任务使用
3. 暂不接构建部署

但即便如此，整体复杂度仍明显高于上传资产引用。

## 6. 验收标准

### 上传资产持久化引用路线

- 前端服务器模式不再因为“上传图片”被直接拦截
- 平台任务 payload 使用 `uploaded_asset_ref`
- 后端 runner 能读取该引用
- 旧的 `uploaded_asset_not_persisted` defer 口径已不再作为当前主路径

### `project_root` 服务器语义路线

- 平台任务不再依赖用户本机 `project_root`
- 服务器侧拥有可审计的工作目录
- 后端 build / codegen / artifact 路径都能消费服务器侧目录语义

## 7. 当前建议

当前建议不要直接并行推进这两条线，而是：

1. 先选定“上传资产持久化引用”为下一阶段主攻方向
2. 先冻结最小方案
3. 再进入代码实现
