# 2026-03-27 数据库表设计草案

> 适用范围：本文面向平台模式后端第一版数据库设计，用于指导后续 PostgreSQL DDL、ORM 模型、索引、状态机与接口实现。本文是“数据库表设计草案”，不是最终 DDL；字段命名、类型细节、约束颗粒度可在实现阶段继续收敛。当前仅覆盖用户维度，不覆盖团队 / 组织维度。

## 一、设计前提

### 1.1 已确认边界

- 第一版后端建议采用 PostgreSQL。
- 当前只做用户维度，不做团队 / 组织维度。
- 平台模式进入后端数据库体系；BYOK 模式本地执行、本地存储，不进入服务器任务、结果、计费与配额体系。
- 第一版统一核心任务模型为 `job -> job_item -> ai_execution`。
- 首批平台任务类型为：
  - `single_generate`
  - `batch_generate`
  - `log_analysis`
  - `mod_edit`
- 计费最小单位是 `ai_execution`，非 AI 操作不进入计费主链。
- 用户侧只看 `job` 与 `job_item` 聚合结果；`ai_execution` 仅管理员与系统内部可见。
- 结果存储策略为“数据库存摘要，对象存储存文件 / 大结果”。

### 1.2 设计原则

- 第一版优先保证最小闭环可落地，不追求一次性穷尽所有商业化与多租户能力。
- 枚举字段优先采用字符串枚举，不急着拆独立字典表。
- `jobs` 与 `job_items` 上直接保留聚合计数，避免列表页和详情页反复实时聚合底层执行数据。
- `input_summary`、`result_summary`、`error_summary` 等展示摘要字段，第一版可直接采用 `text`。
- `event_payload` 建议采用 `jsonb`，兼容轮询与后续 WebSocket 共用事件模型。
- 大结果、导出文件、日志附件等不直接塞入主表，主表只保留摘要与对象存储引用。
- 本文中标注为“建议冗余”的字段，应优先考虑直接落表；标注为“后续可选”的字段，可在第一版完成后补齐。

## 二、核心表总览

| 表名 | 用途 | 是否第一版核心 | 说明 |
| --- | --- | --- | --- |
| `users` | 用户主体 | 是 | 若仓库后续确认已有统一用户表，建议直接复用，不重复造表 |
| `jobs` | 用户视角的大任务 | 是 | 面向列表与详情展示的聚合主表 |
| `job_items` | 任务拆分后的可观测子项 | 是 | 面向普通用户可见的最小工作单元 |
| `ai_executions` | 单次实际 AI 调用 | 是 | 最小计费单元，仅管理员可见 |
| `execution_charges` | 单次执行的结算记录 | 是 | 与 `ai_execution` 一一对应 |
| `quota_accounts` | 用户额度账户 | 是 | 承载额度主体与当前账户状态 |
| `quota_buckets` | 周期额度桶 | 是 | 承载日 / 周等周期额度窗口 |
| `usage_ledgers` | 消耗与返还流水 | 是 | 不可篡改导向的额度流水 |
| `artifacts` | 对象存储产物元数据 | 是 | 存文件 / 大结果引用 |
| `job_events` | 统一事件流 | 是 | 供轮询与后续推送模型共用 |

## 三、表字段建议

以下为建议字段集合，重点强调第一版应优先落的核心字段、冗余字段与后续可选字段。

### 3.1 `users`

用途：承载用户主体，作为任务、配额、流水的归属对象。

建议字段：

- `id`
- `created_at`
- `updated_at`

说明：

- 若系统已有统一用户表，建议直接复用，不在本草案内重复定义完整用户资料字段。
- 后续可选：
  - `status`
  - `display_name`
  - `email`

### 3.2 `jobs`

用途：用户视角的大任务，承载任务聚合状态、聚合统计、展示摘要与生命周期时间。

建议核心字段：

- `id`
- `user_id`
- `job_type`
- `status`
- `input_summary`
- `result_summary`
- `error_summary`
- `total_item_count`
- `pending_item_count`
- `running_item_count`
- `succeeded_item_count`
- `failed_business_item_count`
- `failed_system_item_count`
- `skipped_quota_exhausted_item_count`
- `skipped_cancelled_item_count`
- `created_at`
- `updated_at`
- `confirmed_at`
- `started_at`
- `finished_at`
- `cancel_requested_at`

建议冗余字段：

- `total_item_count`
- `pending_item_count`
- `running_item_count`
- `succeeded_item_count`
- `failed_business_item_count`
- `failed_system_item_count`
- `skipped_quota_exhausted_item_count`
- `skipped_cancelled_item_count`

这些字段建议直接冗余存储在 `jobs`，用于列表查询、详情统计与快速判断整体进度。

后续可选字段：

- `source`
- `created_from`
- `client_request_id`
- `last_error_code`
- `last_error_at`

### 3.3 `job_items`

用途：`job` 下的可执行子项，是普通用户可见的细粒度观察单位。

建议核心字段：

- `id`
- `job_id`
- `user_id`
- `item_index`
- `item_type`
- `status`
- `input_summary`
- `result_summary`
- `error_summary`
- `attempt_count`
- `last_ai_execution_id`
- `created_at`
- `updated_at`
- `started_at`
- `finished_at`

建议冗余字段：

- `attempt_count`
- `last_ai_execution_id`
- `result_summary`
- `error_summary`

这些字段建议直接冗余保留，避免前端查询子项结果时还要下钻 `ai_executions`。

后续可选字段：

- `business_key`
- `target_ref`
- `last_error_code`
- `last_error_at`

### 3.4 `ai_executions`

用途：单次实际 AI 调用记录，是最小计费单元，也是审计、排障、返还的核心主表。

建议核心字段：

- `id`
- `job_id`
- `job_item_id`
- `user_id`
- `status`
- `provider`
- `model`
- `input_summary`
- `result_summary`
- `error_summary`
- `request_idempotency_key`
- `started_at`
- `finished_at`
- `created_at`
- `updated_at`

说明：

- `provider`、`model` 用于保留平台模式下的调用上下文。
- `request_idempotency_key` 建议预留，方便后续处理重试与重复扣费保护。
- `input_summary` / `result_summary` 第一版只保留摘要，不强求保存完整请求响应体。

后续可选字段：

- `prompt_template_version`
- `provider_request_id`
- `token_input`
- `token_output`
- `latency_ms`
- `operator_type`

### 3.5 `execution_charges`

用途：记录单次 `ai_execution` 的结算状态与计费结果，建议与 `ai_executions` 保持一对一关系。

建议核心字段：

- `id`
- `ai_execution_id`
- `user_id`
- `charge_status`
- `charge_unit`
- `charge_amount`
- `reserved_at`
- `captured_at`
- `refunded_at`
- `refund_reason`
- `created_at`
- `updated_at`

说明：

- 第一版建议通过唯一索引保证 `ai_execution_id` 在本表中一对一。
- 若第一版只做“次数扣减”，`charge_amount` 也建议保留，避免后续扩展时再迁移主链。

后续可选字段：

- `pricing_version`
- `billing_note`
- `external_bill_id`

### 3.6 `quota_accounts`

用途：表达用户额度账户，是周期额度桶与流水的归属主体。

建议核心字段：

- `id`
- `user_id`
- `status`
- `created_at`
- `updated_at`

说明：

- 第一版只做用户维度，因此 `quota_accounts` 可一对一归属 `user_id`。
- 若后续进入团队 / 组织维度，可再扩展账户归属模型。

后续可选字段：

- `account_type`
- `effective_at`
- `expired_at`

### 3.7 `quota_buckets`

用途：承载按周期统计的额度窗口，例如日额度、周额度。

建议核心字段：

- `id`
- `quota_account_id`
- `bucket_type`
- `period_start`
- `period_end`
- `quota_limit`
- `used_amount`
- `refunded_amount`
- `remaining_amount`
- `created_at`
- `updated_at`

建议冗余字段：

- `used_amount`
- `refunded_amount`
- `remaining_amount`

这些字段建议直接存储，避免每次查询配额时从 `usage_ledgers` 全量聚合。

后续可选字段：

- `reset_policy`
- `granted_by`
- `policy_snapshot`

### 3.8 `usage_ledgers`

用途：记录额度消耗、确认与返还的流水，作为对账与审计基础。

建议核心字段：

- `id`
- `user_id`
- `quota_account_id`
- `quota_bucket_id`
- `ai_execution_id`
- `ledger_type`
- `amount`
- `balance_after`
- `reason_code`
- `created_at`

说明：

- `ledger_type` 第一版可先覆盖 `reserve`、`capture`、`refund` 等动作语义。
- `balance_after` 建议直接冗余，便于对账与问题追踪。

后续可选字段：

- `job_id`
- `job_item_id`
- `operator_type`
- `remark`

### 3.9 `artifacts`

用途：保存对象存储产物元数据，数据库仅保留摘要与引用，不直接存大结果。

建议核心字段：

- `id`
- `job_id`
- `job_item_id`
- `ai_execution_id`
- `user_id`
- `artifact_type`
- `storage_provider`
- `object_key`
- `file_name`
- `mime_type`
- `size_bytes`
- `result_summary`
- `created_at`

说明：

- `artifact_type` 用于区分文本结果、导出包、日志附件等类型。
- `result_summary` 可与对象存储内容形成展示层摘要映射。

后续可选字段：

- `sha256`
- `expires_at`
- `deleted_at`

### 3.10 `job_events`

用途：记录统一事件流，供轮询接口与后续 WebSocket 推送复用。

建议核心字段：

- `id`
- `job_id`
- `job_item_id`
- `ai_execution_id`
- `user_id`
- `event_type`
- `event_payload`
- `created_at`

说明：

- `event_payload` 建议使用 `jsonb`。
- 事件流第一版可覆盖任务创建、启动、进度变更、额度耗尽、取消、结束等核心状态变化。

后续可选字段：

- `event_version`
- `trace_id`
- `visible_to_user`

## 四、关系与外键建议

建议关系如下：

- `jobs.user_id -> users.id`
- `job_items.job_id -> jobs.id`
- `job_items.user_id -> users.id`
- `ai_executions.job_id -> jobs.id`
- `ai_executions.job_item_id -> job_items.id`
- `ai_executions.user_id -> users.id`
- `execution_charges.ai_execution_id -> ai_executions.id`
- `execution_charges.user_id -> users.id`
- `quota_accounts.user_id -> users.id`
- `quota_buckets.quota_account_id -> quota_accounts.id`
- `usage_ledgers.user_id -> users.id`
- `usage_ledgers.quota_account_id -> quota_accounts.id`
- `usage_ledgers.quota_bucket_id -> quota_buckets.id`
- `usage_ledgers.ai_execution_id -> ai_executions.id`
- `artifacts.job_id -> jobs.id`
- `artifacts.job_item_id -> job_items.id`
- `artifacts.ai_execution_id -> ai_executions.id`
- `artifacts.user_id -> users.id`
- `job_events.job_id -> jobs.id`
- `job_events.job_item_id -> job_items.id`
- `job_events.ai_execution_id -> ai_executions.id`
- `job_events.user_id -> users.id`

建议约束说明：

- `execution_charges.ai_execution_id` 建议唯一，表达一条执行对应一条主结算记录。
- `job_items.job_id + item_index` 建议唯一，避免同一任务内子项序号重复。
- `quota_buckets` 应保证同一 `quota_account_id + bucket_type + period_start + period_end` 组合唯一。
- `artifacts`、`job_events` 中涉及 `job_item_id`、`ai_execution_id` 的外键，可按业务允许为空，用于兼容仅任务级事件 / 产物。

## 五、状态字段建议

### 5.1 `jobs.status`

建议枚举值：

- `draft`
- `running`
- `cancelling`
- `cancelled`
- `succeeded`
- `partial_quota_exhausted`
- `failed_business`
- `failed_system`

说明：

- `draft` 用于“已创建并展开子项，但尚未开始执行”。
- `partial_quota_exhausted` 用于表达部分执行成功，但后续因额度不足停止。
- 第一版不强求拆更细的排队态，可待调度器实现细化后再补。

### 5.2 `job_items.status`

建议枚举值：

- `pending`
- `running`
- `succeeded`
- `failed_business`
- `failed_system`
- `skipped_quota_exhausted`
- `skipped_cancelled`

说明：

- `pending` 可覆盖待执行与待调度状态。
- `skipped_quota_exhausted` 用于表达未启动即因额度不足跳过。
- `skipped_cancelled` 用于表达未启动即因上层取消而跳过。

### 5.3 `ai_executions.status`

建议枚举值：

- `pending`
- `running`
- `succeeded`
- `failed_business`
- `failed_system`

说明：

- 第一版仅保留最小必需状态，不额外拆分 dispatching / refunded 等细态。
- 是否发生返还不放进 `status`，而通过 `execution_charges` 与 `usage_ledgers` 体现。

### 5.4 `execution_charges.charge_status`

建议枚举值：

- `reserved`
- `captured`
- `refunded`

说明：

- `reserved`：已预留或已写入待确认扣减
- `captured`：实际扣减已确认
- `refunded`：系统异常或其他规则触发返还

## 六、索引建议

建议第一版至少建立以下索引：

- `jobs(user_id, created_at desc)`
- `jobs(status, created_at desc)`
- `job_items(job_id, item_index)`
- `ai_executions(job_item_id, created_at)`
- `execution_charges(ai_execution_id)` 唯一索引
- `quota_buckets(quota_account_id, bucket_type, period_start, period_end)`
- `usage_ledgers(user_id, created_at desc)`
- `job_events(job_id, created_at)`

补充建议：

- 若后续管理员频繁按 `ai_executions.user_id` 或 `status` 检索，可再补组合索引。
- 若 `artifacts` 后续出现高频按 `job_item_id` 查询的场景，可补充对应索引，但第一版可先按实际接口压力决定。

## 七、第一版最小闭环落表顺序

建议按“先主链、再额度、再结果、再事件”的顺序推进：

1. `users`
   - 若已有用户表，则确认复用方式，不重复建表。
2. `jobs`
   - 先落用户任务主表与聚合统计字段。
3. `job_items`
   - 先打通任务展开与子项状态流转。
4. `ai_executions`
   - 打通最小 AI 调用与管理员审计主链。
5. `execution_charges`
   - 绑定最小计费单元，形成一对一结算记录。
6. `quota_accounts`
   - 建立用户额度账户主体。
7. `quota_buckets`
   - 落日 / 周周期额度窗口。
8. `usage_ledgers`
   - 打通预扣、确认、返还流水。
9. `artifacts`
   - 打通摘要入库 + 对象存储引用。
10. `job_events`
   - 打通轮询事件流，为后续 WebSocket 复用做准备。

按这个顺序，第一版可以先完成“创建任务 -> 展开子项 -> 启动执行 -> 按次检查额度 -> 记录执行 -> 结算 / 返还 -> 聚合展示 -> 查询事件”的最小业务闭环。

## 八、延后设计项

以下内容建议明确延后，不阻塞第一版落表：

- 团队 / 组织 / 工作区维度的数据归属与配额隔离。
- 月付套餐、权益包、复杂价格版本等商业化结构。
- 更细颗粒度的模型调用成本字段，如详细 token 成本拆分。
- BYOK 本地数据库与本地执行记录建模。
- 更复杂的任务重试、人工重跑、人工改判返还能力。
- 更细的错误码字典、事件版本治理、跨系统链路追踪字段。
- 任务模板、提示词版本、资源引用快照等高阶审计能力。

其中以下字段 / 能力可视为“后续可选项”，不建议阻塞第一版：

- `ai_executions.prompt_template_version`
- `ai_executions.token_input`
- `ai_executions.token_output`
- `execution_charges.pricing_version`
- `quota_buckets.policy_snapshot`
- `artifacts.sha256`
- `job_events.trace_id`

## 附：第一版实现提醒

- 本文是数据库表设计草案，不是最终 DDL；实现阶段仍需进一步补充字段类型、默认值、非空约束、删除策略与迁移方案。
- 聚合计数建议优先在 `jobs` 与 `job_items` 冗余存储，不建议第一版完全依赖联表实时计算。
- 摘要字段建议优先满足列表页与详情页展示；大结果统一走对象存储。
- 若后续仓库内已经存在用户体系或审计体系，应优先复用既有主表与命名约定，再映射本文建议结构。
