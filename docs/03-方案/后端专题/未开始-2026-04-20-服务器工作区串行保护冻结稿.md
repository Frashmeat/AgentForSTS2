# 2026-04-20 服务器工作区串行保护冻结稿

## 1. 冻结目标

本文只解决 4 件事：

1. 冻结服务器工作区并发保护的资源粒度
2. 冻结哪些执行链属于“写型链路”，必须串行
3. 冻结锁的持有边界、忙时语义和最小可观测信息
4. 选定第一版推荐实现方向，避免后续在“内存锁 / 文件锁 / 数据库租约”之间反复摇摆

本文不解决：

- 全局任务排队系统的完整实现
- 多实例分布式调度
- 更细粒度的步骤级资源池
- 具体前端提示文案与页面交互

## 2. 当前代码事实

基于当前仓库代码，以下事实已经明确：

- 前端当前可先通过 `POST /api/me/server-workspaces` 获取 `server_project_ref`，后续平台任务会显式携带该引用。
- 服务器工作区当前按 `runtime/platform-workspaces/{user_id}/{token}/{project_name}` 落盘。
- `code.generate`、`asset.generate`、`build.project` 当前都会直接消费 `server_workspace_root`，在该目录内写源码、生成资源或执行构建。
- 当前平台执行链仍主要在 `web` 进程内直接组装 runner 执行，而不是已经稳定落成 `web -> 独立 server workstation` 的远程调用模型。

因此，本稿冻结的保护语义不能依赖“当前恰好是单进程”这一事实，而应尽量面向后续可迁移的形态设计。

## 3. 资源粒度冻结

第一版工作区并发保护的最小资源粒度冻结为：

- `server_project_ref`

原因：

- 它已经是平台链路显式携带的稳定服务器项目引用。
- 它比 `project_name` 更精确，能够区分同一用户或不同用户创建的多个同名服务器工作区。
- 它比 `workspace_root` 字符串更适合作为跨层 contract，不要求上层了解实际物理路径。

第一版明确不使用以下粒度作为主锁键：

- `user_id`
- `project_name`
- `workspace_root`
- `job_id`

## 4. 写型链路冻结

只有会真实修改服务器工作区内容或消费其构建状态的链路，才纳入工作区串行保护。

### 4.1 必须串行的步骤

- `code.generate`
- `asset.generate`
- `build.project`

### 4.2 当前允许并行的步骤

- `single.asset.plan`
- `batch.custom_code.plan`
- `log.analyze`
- 其他只读取输入、产出文本方案、不写服务器工作区的步骤

### 4.3 冻结原则

- 保护的目标不是“让所有服务器任务都串行”，而是防止多个任务同时写同一服务器工作区。
- 后续若新增步骤会写 `server_workspace_root`，必须默认纳入同一保护集合，而不是另起一套规则。

## 5. 锁持有边界冻结

第一版冻结以下语义：

- 同一 `server_project_ref` 在同一时刻只允许一条写型执行链运行。
- 锁应在进入第一步写型步骤前获取，而不是在每个步骤开始时临时获取。
- 锁应在该条写型执行链到达终态后统一释放，避免 `code.generate -> build.project` 之间出现可插队窗口。

当前推荐的持有边界：

1. 命中写型 workflow
2. 获取 `server_project_ref` 级别写锁
3. 依次执行 `code.generate / asset.generate / build.project`
4. 完成、失败、取消或 deferred 后统一释放

第一版明确不建议：

- 每个步骤单独加锁再立即释放
- 只对 `build.project` 加锁
- 允许 `code.generate` 与 `build.project` 被不同任务交错执行

## 6. 忙时语义冻结

当第二条任务命中同一 `server_project_ref` 且需要进入写型链路时，第一版应遵守以下原则：

- 不允许静默并行执行
- 不允许继续假装“当前无冲突”

### 6.1 推荐主语义

若平台排队语义已落地：

- 第二条任务进入 `queued`
- 并显式标注其等待原因属于 `server_project_ref` 被占用

### 6.2 允许的过渡语义

若工作区串行保护先于统一排队落地：

- 可以先返回稳定的 `busy/conflict` 语义
- 但必须保证同一工作区不会被并发写入

这意味着：

- “先有锁、后补排队”是允许的
- “先放开并发、等以后再治理冲突”是不允许的

## 7. 可观测性冻结

第一版至少要能记录以下信息：

- `server_project_ref`
- `job_id`
- `job_item_id`
- `ai_execution_id`（若已创建）
- `lock_acquired_at`
- `lock_released_at`
- `blocked_reason`

最小可观测目标：

- 让后端日志或事件能回答“哪条任务正在占用哪个服务器工作区”
- 让排障时能区分“凭据不足 / 排队等待 / 工作区忙”

## 8. 锁服务最小接口草案冻结

为了避免后续把锁逻辑散落到 router、service 与 runner 之间，第一版建议冻结一个最小工作区锁服务抽象。

### 8.1 推荐服务职责

最小职责只包含：

1. 尝试获取指定 `server_project_ref` 的写锁
2. 释放指定 `server_project_ref` 的写锁
3. 返回当前持有者的最小占用信息，用于日志和冲突解释

### 8.2 推荐输入字段

尝试获取锁时，建议至少携带：

- `server_project_ref`
- `job_id`
- `job_item_id`
- `user_id`
- `owner_scope`

其中：

- `owner_scope` 第一版建议冻结为固定值 `workspace_write`

### 8.3 推荐返回结构

获取锁的最小返回语义建议冻结为：

- `acquired`
- `busy`
- `current_holder`

`current_holder` 当前建议最少包含：

- `job_id`
- `job_item_id`
- `user_id`
- `locked_at`
- `resource_key`

### 8.4 明确不要求的第一版能力

- 不要求支持锁重入
- 不要求支持锁续租
- 不要求支持复杂优先级抢占
- 不要求支持一个任务同时持有多个工作区写锁

## 9. 状态与事件草案冻结

第一版应优先复用现有平台主状态与事件，不建议为“工作区忙”单独发明一套全新状态机。

### 9.1 状态流转建议

当前推荐的工作区忙状态流转为：

1. `draft`
2. `queued`
3. `running`
4. 终态

其中：

- 因工作区忙而等待的任务，状态仍为 `queued`
- 不应因为工作区忙而进入 `deferred`

### 9.2 `deferred` 的边界冻结

`deferred` 仍只用于：

- workflow 未注册
- 当前 payload 仍依赖本地语义
- 当前运行时能力明确无法消费该任务

第一版明确禁止把以下情况写成 `deferred`：

- 同一 `server_project_ref` 正被别的任务占用
- 当前只是在等锁
- 当前只是在等前序写型任务释放工作区

### 9.3 事件复用建议

第一版优先复用：

- `job.queued`
- `job.cancel_requested`
- `ai_execution.started`
- `ai_execution.finished`

并建议对 `job.queued` 的 payload 增补以下可选字段：

- `queue_reason_code`
- `queue_reason_message`
- `resource_type`
- `resource_key`

工作区等待场景建议固定为：

- `queue_reason_code = server_workspace_busy`
- `queue_reason_message = 当前服务器工作区正在被另一条写型任务占用，请等待其完成后再继续执行。`
- `resource_type = server_workspace`
- `resource_key = server_project_ref`

### 9.4 可选新增事件

若后续认为仅靠 `job.queued` 不足以表达“已从等待进入可执行”，允许新增：

- `job.queue_promoted`

但第一版不强制要求该事件，避免在排队语义未完全落地前扩大事件面。

## 10. 第一版推荐实现方向

当前推荐实现方向冻结为：

- **文件锁优先**

推荐原因：

- 当前服务器工作区真源就是文件系统，文件锁更贴近真实资源。
- 比纯内存锁更容易迁移到未来“独立 workstation 进程”或多进程形态。
- 比数据库租约锁更轻量，适合作为第一版运行时保护。

### 8.1 推荐落点

可选的第一版落点示意：

- 工作区目录下固定锁文件
- 或 `runtime/platform-workspace-locks/{token}.lock`

### 8.2 当前不推荐作为第一版主方案

- 仅进程内内存锁
  - 只适用于当前单进程事实，未来易失效
- 数据库租约锁
  - 成本较高，不适合作为第一版起步方案

## 11. 无排队前的冲突返回口径冻结

若工作区锁先于统一排队语义落地，第一版允许临时返回同步冲突，但需要固定口径。

推荐返回约束：

- HTTP：`409`
- detail：稳定机器可读短语，不使用临时文案拼接

推荐 detail：

- `server workspace is busy`

同时建议在错误 payload 或日志中补充：

- `reason_code = server_workspace_busy`
- `server_project_ref`

冻结原则：

- 该同步冲突只应作为“排队未落地前的过渡语义”
- 一旦统一排队主链落地，应优先收口到 `queued`

## 12. 代码落点建议

后续若进入实现，优先关注：

- `backend/app/modules/platform/application/services/job_application_service.py`
- `backend/app/modules/platform/application/services/execution_orchestrator_service.py`
- `backend/app/modules/platform/application/services/server_workspace_service.py`
- 新增的工作区锁服务或等价抽象

冻结要求：

- 锁服务不应散落在 runner handler 内部各自实现
- 锁服务应成为平台执行编排层的显式依赖，而不是隐藏在某个单步骤函数里

## 13. 验收标准

当以下条件满足时，可认为“服务器工作区串行保护冻结稿”完成：

- 明确了并发保护的资源粒度是 `server_project_ref`
- 明确了哪些步骤必须串行、哪些步骤允许继续并行
- 明确了锁获取与释放的边界
- 明确了忙时不能继续并行写入同一工作区
- 明确了锁服务的最小接口、`job.queued` 的复用方式与 `deferred` 边界
- 明确了第一版优先采用文件锁，而不是继续保持实现开放状态

## 14. 当前建议

本稿完成后，下一步才适合进入：

1. 先决定排队语义与工作区忙时语义如何衔接
2. 再补最小工作区锁服务
3. 最后补对应测试与事件可观测性

在此之前，不建议继续扩大服务器写型 workflow 覆盖面。
