# 2026-03-27 后端设计详情

> 适用范围：本文为平台模式、用户维度第一版后端设计详情基线文档，用于指导后续 service、runner、repository、migration、接口与事件模型实现。本文是设计详情与实现基线，不是最终规格书；内容按“已确认结论 / 推荐实现 / 后续待定”区分。BYOK 模式仅作为系统边界说明，不进入本版服务器持久化与任务主链。

## 一、设计目标

### 1.1 已确认结论

- 当前后端继续采用 Python 单体 `FastAPI + Uvicorn`。
- 服务端同时承接 HTTP 与 WebSocket，但第一版以 HTTP 轮询跑通闭环，WebSocket 后补且必须复用同一事件模型。
- 第一版只覆盖平台模式、用户维度；BYOK 不进入服务器数据库体系。
- 当前真正需要补齐的是任务系统、执行分层、持久化、配额与审计，而不是切换 Web 框架。

### 1.2 为什么继续 FastAPI

推荐继续沿用 `FastAPI`，原因如下：

- 当前系统已经存在 Python 本地工作流、CLI、`subprocess` 执行链，继续沿用 Python 单体可以直接复用现有执行资产与工程经验。
- 第一版关键问题在于任务编排、状态持久化、配额结算和审计留痕，这些问题与是否更换 Web 框架关联很弱。
- `FastAPI` 已能覆盖当前需要的 HTTP API、后台编排入口、依赖注入、Pydantic 模型与 WebSocket 能力，足以支撑第一阶段落地。
- 在产品边界和任务模型尚未完全冻结前，切换框架会扩大迁移面，却不能实质降低核心设计风险。

结论：短期最优策略是继续基于 `FastAPI` 演进，在现有单体内补齐任务、执行、持久化与审计分层。

### 1.3 本文目标

- 把已确认讨论结论整合为一份可以直接指导实现拆分的基线文档。
- 明确运行模式、执行边界、模块划分、核心状态机、计费配额、接口模型与仓储边界。
- 为后续数据库迁移、ORM 模型、应用服务、执行器适配和管理台接口提供统一参照。

## 二、核心结论总览

### 2.1 已确认结论

- 产品主形态仍是工作站，Web 是平台控制台。
- 平台模式由服务器承接 AI 调用、保存任务结果并纳入计费、限额、返还与审计体系。
- BYOK 模式为本地执行、本地存储，不接收服务器下发步骤，不进入服务器任务库。
- 本地与服务器共享同一套 Python 核心工作流定义，不维护两套长期分叉的生成逻辑。
- 版本同步采用 `workflow_version`、`step_protocol_version`、`result_schema_version`。
- 第一版不做远程热更新，协议与工作流版本跟随应用版本发布。
- 核心任务模型为 `job -> job_item -> ai_execution`。
- 计费最小单位是 `ai_execution`。
- 创建任务与开始执行分离。
- 批量任务按“边执行边检查配额”推进，不预占整批额度。
- 用户取消后，已开始的 `ai_execution` 尽量跑完，未开始项不再启动。
- 普通用户只看 `job` / `job_item` 聚合结果，`ai_execution` 仅管理员与内部服务可见。
- `admin` 接口纳入第一阶段，第一版即提供管理员执行、返还与审计查询能力。
- 幂等策略为 `ai_executions(user_id, job_item_id, request_idempotency_key)` 的 scoped partial unique，且仅在 key 非空时生效。
- 主链、计费链、审计链默认不做物理删除，统一预留 `deleted_at / archived_at`。
- `artifacts` 采用“元数据保留、文件异步清理”策略。
- 第一版状态枚举按细颗粒度落库，不采用“先少后多”的收敛策略。
- 双路径迁移阶段采用 `dual-write + read by flag`；第一阶段先 `read legacy`，待校验稳定后再切换新读路径。
- 第一版退款矩阵按本文第七章冻结，超出矩阵的扩展场景放到后续版本单独评审。

### 2.2 推荐实现

- 在单体内拆分明确的应用服务与执行模块，而不是继续把 HTTP 入口、执行编排、计费逻辑和数据库访问混在一起。
- 采用“用户可见聚合模型”和“内部执行审计模型”分离设计，避免前台接口直接耦合到底层执行事实。
- 把轮询接口、未来 WebSocket 推送、管理员审计查询统一到一条事件主链。

### 2.3 后续待定

- `job_type`、`item_type`、`event_type` 的最终稳定枚举。
- 第二阶段是否引入更强的后台调度器或独立任务进程。

## 三、运行模式与执行边界

### 3.1 已确认结论

#### 平台模式

- 服务器既是控制面，也是 AI 服务提供方。
- 服务器可以下发步骤给工作站，本地执行依赖本地环境的动作。
- 服务器持久化任务、结果摘要、计费、额度、返还、审计与事件。

#### BYOK 模式

- 用户使用自带 key，本地完整执行，本地存储。
- 服务器不接收 BYOK 执行记录、不保存任务结果，也不下发远程步骤。

### 3.2 推荐实现

- 服务端代码只为平台模式建设任务与持久化主链，BYOK 只保留协议兼容和产品边界说明。
- 平台模式下，将“服务端控制流程”和“工作站本地执行步骤”通过版本化步骤协议连接，而不是复制两套工作流逻辑。
- 当某些步骤必须依赖本地环境时，服务端负责生成步骤描述、记录状态与接收结果，不把本地细节直接写死在 Web 层接口中。

### 3.3 边界约束

- 服务器数据库中不混入 BYOK 任务、BYOK 结果和 BYOK 配额数据。
- 控制台接口默认只返回平台模式数据。
- 任何需要本地依赖、本地文件系统或本地 CLI 的执行动作，都应通过“工作流步骤 + 协议版本”表达，不应在后端接口层直接耦合具体桌面实现。

## 四、系统分层与模块划分

### 4.1 推荐实现

建议在单体服务内按职责拆分如下模块：

#### API / Transport 层

- 提供用户侧 HTTP API、管理员 HTTP API、轮询事件接口、未来 WebSocket 推送入口。
- 只负责鉴权、参数校验、调用应用服务与返回 DTO，不承载任务编排和数据库拼装逻辑。

#### Application Service 层

- `JobApplicationService`
- `JobQueryService`
- `ExecutionOrchestratorService`
- `QuotaBillingService`
- `ArtifactService`
- `EventService`
- `AdminQueryService`

职责：

- 编排任务创建、确认开始、取消、执行推进、结果回写、返还和查询聚合。
- 管理事务边界和跨仓储协作。

#### Runner / Workflow 层

- `WorkflowRegistry`
- `WorkflowRunner`
- `StepDispatcher`
- `ExecutionAdapter`

职责：

- 把 `job_type` / `item_type` 转换为共享工作流定义。
- 驱动步骤执行、收集步骤结果、映射为 `ai_execution` 与产物写回。
- 适配本地 CLI / `subprocess` / 外部 AI 调用等执行手段。

#### Domain / Repository 层

- `JobRepository`
- `AIExecutionRepository`
- `ExecutionChargeRepository`
- `QuotaAccountRepository`
- `UsageLedgerRepository`
- `ArtifactRepository`
- `JobEventRepository`
- 各类 Query Repository

职责：

- 承接聚合根持久化、内部事实实体落库和查询投影。

### 4.2 已确认方向

- 单体部署，不拆微服务。
- HTTP 与 WebSocket 共存，但共享同一事件模型。
- 已有本地工作流/CLI/`subprocess` 执行链需要被复用，而不是重造。

### 4.3 模块边界建议

- `service` 负责业务编排。
- `runner` 负责工作流步骤执行与状态映射。
- `repository` 负责数据库读写与查询投影。
- `migration` 负责主链、额度链、事件链和产物链的表结构演进。

这也是后续实现拆目录的推荐基线。

## 五、共享工作流核心与版本同步策略

### 5.1 已确认结论

- 本地与服务器共享同一套 Python 核心模块与工作流定义。
- 不维护两套长期分叉的生成逻辑。
- 使用版本化协议：`workflow_version` / `step_protocol_version` / `result_schema_version`。
- 第一版不做远程热更新，跟随应用版本发布同步。

### 5.2 推荐实现

- 把工作流抽象为稳定的 Python 核心模块，服务端与工作站都依赖同一套 workflow definition。
- 服务端只负责：
  - 根据 `job_type` 选择工作流。
  - 根据输入展开 `job_items`。
  - 记录工作流版本与步骤协议版本。
  - 下发步骤描述或触发平台内执行。
- 工作站负责：
  - 执行依赖本地环境的步骤。
  - 按 `step_protocol_version` 回传步骤结果。
- 结果回传时按 `result_schema_version` 校验并规范化写库。

### 5.3 设计建议

- `jobs` 或 `job_items` 上建议保留工作流版本快照字段。
- `ai_executions` 建议保留请求模板 / 执行协议版本快照，便于审计与回放。
- 版本字段应参与日志、事件和管理员查询输出，避免后续问题无法定位到具体工作流版本。

### 5.4 后续待定

- 工作流版本信息最终落在 `job`、`job_item` 还是 `ai_execution` 的完整策略。
- 不同任务类型是否需要统一的步骤 DSL，还是允许在统一协议下保留少量类型化 payload。

## 六、核心任务模型与状态机

### 6.1 三层模型职责

#### `job`

已确认结论：

- 用户视角的大任务。
- 负责聚合状态、聚合结果、统计计数和生命周期时间。

推荐实现：

- 作为用户侧列表、详情与取消操作的主聚合根。
- 负责保存任务摘要、总数统计、是否因额度中止、是否进入取消收尾等聚合语义。

#### `job_item`

已确认结论：

- `job` 下的可独立观察子项。
- 是普通用户可见的最小工作单元。

推荐实现：

- 每个 `job_item` 对应一个可执行单元或可观察对象。
- 保存规范化输入、子项级结果摘要、最近一次执行引用与错误摘要。

#### `ai_execution`

已确认结论：

- 单次实际 AI 调用。
- 最小计费单元。
- 仅管理员与内部服务可见。

推荐实现：

- 只在真实发起 AI 调用前创建。
- 绑定幂等键、计费上下文、模型上下文、执行状态和输入输出摘要。

### 6.2 第一版确认状态机

#### `job.status`

第一版 `job.status` 使用以下细颗粒度聚合态：

- `draft`
- `queued`
- `running`
- `partial_succeeded`
- `succeeded`
- `failed`
- `quota_exhausted`
- `cancelling`
- `cancelled`

说明：

- `draft` 表示已创建并展开子项，但尚未开始。
- `quota_exhausted` 或 `partial_succeeded` 用于表达“已完成部分，但后续因额度或异常未全量完成”。

#### `job_item.status`

第一版 `job_item.status` 使用：

- `pending`
- `ready`
- `running`
- `succeeded`
- `failed_business`
- `failed_system`
- `quota_skipped`
- `cancelled_before_start`
- `cancelled_after_start`

#### `ai_execution.status`

第一版 `ai_execution.status` 使用：

- `created`
- `dispatching`
- `running`
- `succeeded`
- `failed_business`
- `failed_system`
- `completed_with_refund`

### 6.3 第一版落地建议

第一版不再收敛状态枚举，数据库与实现直接按细颗粒度状态落地，只要满足以下原则：

- `job` 能明确区分“未开始 / 运行中 / 部分完成 / 成功 / 失败 / 取消 / 配额中止”。
- `job_item` 能明确区分“未启动 / 运行中 / 成功 / 业务失败 / 系统失败 / 未启动即跳过”。
- `ai_execution` 能明确区分“创建 / 分发中 / 运行中 / 成功 / 业务失败 / 系统失败 / 已返还完成”。
- 返还信息既体现在计费链，也允许通过 `completed_with_refund` 这类执行终态表达最终结算结果。

### 6.4 状态推进主流程

1. 创建 `job`，展开 `job_items`，状态进入 `draft`。
2. 用户确认开始后，`job` 进入 `queued` 或 `running`。
3. 每次准备启动新的 `ai_execution` 前，检查配额与取消标记。
4. 可执行则创建 `ai_execution`，推进 `job_item` / `job` 为运行态。
5. 执行结束后回写 `ai_execution`、`job_item`、`job` 聚合结果与事件。
6. 若额度不足，则后续未开始项进入 `quota_skipped`，`job` 进入 `quota_exhausted` 或 `partial_succeeded`。
7. 若用户取消，则未开始项停止启动，已开始项尽量收尾，最后 `job` 进入 `cancelled`。

## 七、计费、配额与返还规则

### 7.1 已确认结论

- 计费最小单位为 `ai_execution`。
- 非 AI 操作不计费。
- 批量任务边执行边检查配额，不预占整批额度。
- 系统异常自动返还，由系统自动判定，管理员只查看记录。
- 第一版退款矩阵已冻结，先按明确列出的返还 / 不返还边界执行，不在实施期继续漂移。

### 7.2 推荐实现

#### 计费链

- `ai_execution` 记录事实。
- `execution_charge` 记录单次执行结算状态。
- `usage_ledger` 记录 reserve / capture / refund 流水。

#### 配额链

- `quota_account` 表达用户额度账户。
- `quota_bucket` 表达日 / 周窗口。
- 当前阶段先支持日额度、周额度。

### 7.3 扣减与返还时机

推荐流程：

1. 启动前锁定当前有效 `quota_bucket`。
2. 校验剩余额度。
3. 通过后写入 `reserve` 流水，并创建 `execution_charge(reserved)`。
4. 执行成功或确认为有效消耗后，写 `capture` 并推进结算状态。
5. 若判定为系统异常，写 `refund` 并推进为 `refunded`。

### 7.4 已确认与推荐的异常语义

已确认结论：

- 系统异常自动返还。
- 第一版退款矩阵冻结如下：
  - 返还：平台内部异常、平台基础设施异常、平台 AI 网关异常、平台 worker/runner 崩溃、平台内部超时、上游 `429` / 限流、上游超时 / 网络不可达、响应解析失败、平台下发步骤错误或协议不兼容未拦住。
  - 不返还：用户输入错误、业务规则失败、用户主动取消后已启动的 execution、模型正常返回但结果不满意、提供商返回请求本身非法、本地工作站环境问题导致的执行失败。

推荐实现：

- `failed_business`：输入不合法、业务规则不通过、模型返回不满足业务要求，通常不返还。
- `failed_system`：模型服务异常、超时、网关错误、内部调度错误，通常返还。
- 用户取消后若调用已真实发出，是否计费取决于最终是否形成有效 AI 调用且是否被判定为系统异常。

### 7.5 后续待定

- 是否在未来增加套餐权益、月付权益与优惠抵扣，但这些扩展应复用同一额度与流水主链。

## 八、持久化设计摘要

### 8.1 已确认结论

- 当前后端持久化范围只覆盖平台模式。
- 数据库存摘要，对象存储存大结果或文件。
- 默认不做物理删除，预留 `deleted_at / archived_at`。
- `artifacts` 只清理文件，不轻易清理元数据。

### 8.2 推荐主表

- `jobs`
- `job_items`
- `ai_executions`
- `execution_charges`
- `quota_accounts`
- `quota_buckets`
- `usage_ledgers`
- `artifacts`
- `job_events`

### 8.3 字段与建模原则

- `jobs`、`job_items` 保留聚合计数与摘要字段，优先服务列表页、详情页与轮询查询。
- `ai_executions` 保留模型、提供方、幂等键、输入输出摘要与执行时间。
- `job_events.event_payload` 使用可扩展结构，兼容轮询与 WebSocket 共模。
- 各主链表统一具备 `created_at`、`updated_at` 与关键业务时间字段。
- 索引优先覆盖：
  - 用户任务列表查询
  - 子项分页
  - execution 按子项回查
  - 幂等唯一约束
  - 配额窗口查找
  - 事件流读取

### 8.4 删除与归档策略

已确认结论：

- 主链、计费链、审计链默认不物理删除。

推荐实现：

- 面向用户的“删除任务”优先理解为软删除或不可见。
- 真正归档可在第二阶段补充冷热分层。
- 物理文件清理采用异步任务，元数据保留用于对账、下载失败解释和审计。

## 九、接口与通信模型（HTTP / WS / 步骤协议）

### 9.1 已确认结论

- 第一版先用 HTTP 轮询跑通。
- 后续补 WebSocket，但必须共享同一事件模型。
- 创建与开始执行必须分开。

### 9.2 推荐用户侧 HTTP 接口

- `POST /api/platform/jobs`
  - 创建任务并展开 `job_items`
- `POST /api/platform/jobs/{jobId}/start`
  - 确认并开始执行
- `POST /api/platform/jobs/{jobId}/cancel`
  - 请求取消
- `GET /api/platform/jobs`
  - 查询任务列表
- `GET /api/platform/jobs/{jobId}`
  - 查询任务详情
- `GET /api/platform/jobs/{jobId}/items`
  - 分页查询子项
- `GET /api/platform/jobs/{jobId}/events`
  - 轮询统一事件流
- `GET /api/platform/quota`
  - 查询用户额度视图

### 9.3 推荐管理员接口

- `GET /api/admin/jobs/{jobId}/executions`
- `GET /api/admin/executions/{executionId}`
- `GET /api/admin/quota/refunds`
- `GET /api/admin/audit/events`

已确认结论：

- 上述 `admin` 接口纳入第一阶段范围，不推迟到第二阶段。

### 9.4 推荐统一事件模型

建议事件主类型：

- `job.created`
- `job.started`
- `job.progress_updated`
- `job.partial_blocked_by_quota`
- `job.cancel_requested`
- `job.finished`
- `job_item.started`
- `job_item.finished`
- `ai_execution.started`
- `ai_execution.finished`
- `quota.refunded`

每条事件建议至少包含：

- `event_id`
- `event_type`
- `job_id`
- `job_item_id`（可选）
- `ai_execution_id`（管理员场景可选）
- `occurred_at`
- `payload`

### 9.5 步骤协议建议

推荐步骤协议至少包含：

- `step_protocol_version`
- `workflow_version`
- `step_type`
- `step_id`
- `job_id`
- `job_item_id`
- `input_payload`
- `result_schema_version`

原则：

- HTTP / WebSocket 是控制面通信协议。
- 步骤协议是服务端与执行端之间的工作流协议。
- 两者不能混成一套随业务页面变化的临时 JSON 结构。

## 十、应用服务与仓储边界

### 10.1 推荐应用服务

#### `JobApplicationService`

- 创建任务
- 展开 `job_items`
- 确认开始
- 受理取消

#### `ExecutionOrchestratorService`

- 驱动单个 `job_item` 的执行推进
- 调用配额服务做执行前检查
- 创建 `ai_execution`
- 处理成功、失败、返还与聚合状态回写

#### `QuotaBillingService`

- 锁定配额账户与额度桶
- 写入 reserve / capture / refund
- 维护结算状态与额度聚合值

#### `JobQueryService` / `AdminQueryService`

- 面向用户或管理员聚合查询，不混入写流程。

### 10.2 推荐仓储边界

用户侧：

- `JobRepository`
- `JobQueryRepository`
- `QuotaQueryRepository`

内部调度 / 计费侧：

- `AIExecutionRepository`
- `ExecutionChargeRepository`
- `QuotaAccountRepository`
- `UsageLedgerRepository`
- `ArtifactRepository`
- `JobEventRepository`

管理员侧：

- `AdminJobQueryRepository`
- `AdminExecutionQueryRepository`
- `AdminBillingQueryRepository`
- `AdminEventQueryRepository`

### 10.3 事务边界建议

建议至少有以下关键事务：

1. 创建任务事务
   - 创建 `job`
   - 展开并插入 `job_items`
   - 初始化聚合计数
   - 写 `job.created` 事件
2. 启动单个执行事务
   - 锁定 `job` / `job_item`
   - 幂等查找
   - 锁定 `quota_account` / `quota_bucket`
   - 写 `reserve`
   - 创建 `ai_execution`
   - 回写运行态与事件
3. 执行完成事务
   - 更新 `ai_execution`
   - 写 `artifact`
   - 推进 `execution_charge`
   - 写 `capture` 或 `refund`
   - 回写 `job_item` / `job`
   - 追加完成事件
4. 取消受理事务
   - 标记 `cancel_requested_at`
   - 推进 `job` 到取消中
   - 写取消事件

### 10.4 实现提醒

- 不让 Query Repository 承担状态写入。
- 不让用户侧接口直接暴露 `ai_execution`。
- 不让 API 层直接调用多个仓储拼事务。
- 不只靠内存锁做幂等和配额控制，最终一致性兜底必须落在数据库约束和事务上。

## 十一、安全与审计边界

### 11.1 已确认结论

- 普通用户只看 `job` / `job_item` 聚合结果。
- `ai_execution` 仅管理员 / 内部可见。
- 主链、计费链、审计链需要可追溯。

### 11.2 推荐实现

- 用户接口返回脱敏错误码、摘要和建议动作，不返回原始提示词、底层请求体或系统堆栈。
- 管理员接口可查看执行摘要、错误分类、模型信息、返还记录和事件链路。
- 下载类接口返回短时有效凭证，不直接暴露对象存储底层地址。
- 所有关键操作应写事件或审计记录：
  - 创建任务
  - 开始执行
  - 取消请求
  - 配额中止
  - 自动返还
  - 管理员查询敏感执行详情（如后续需要）

### 11.3 后续待定

- 是否在数据库层增加 `visible_to_user` 一类事件字段，还是完全在查询层裁剪。
- 管理员是否需要重试某个 `job_item` 或手动补偿的操作能力。

## 十二、分阶段实现路线

### 12.1 第一阶段：最小闭环

已确认范围：

- 平台模式
- 用户维度
- 单体 `FastAPI`
- HTTP 轮询
- `job -> job_item -> ai_execution`
- 自动返还
- 第一版细颗粒度状态枚举
- 用户侧与管理员侧基础接口同时交付
- 双路径迁移先 `dual-write + read by flag`，并以 `read legacy` 作为第一阶段读策略

建议交付：

- 核心表迁移
- ORM 实体与基础仓储
- 创建任务 / 开始执行 / 取消 / 列表 / 详情 / 事件轮询
- 执行前额度检查与单次执行计费
- 结果摘要入库与 `artifacts` 元数据
- 自动返还链路

### 12.2 第二阶段：可观测性与体验增强

- WebSocket 推送，复用同一事件模型
- 更完整的管理员执行详情与返还检索
- 更细的错误分类与事件过滤
- 更稳定的执行器适配层

### 12.3 第三阶段：扩展与商业化预留

- 更抽象的额度 / 权益模型
- 月付套餐能力接入
- 更复杂任务类型与步骤协议演进
- 归档、冷热分层与后台清理能力

## 十三、开放问题与后续决策点

### 13.1 后续待定

1. `job_type`、`item_type`、`event_type` 的最终冻结范围。
2. `workflow_version`、`step_protocol_version`、`result_schema_version` 的最终落表位置与回传策略。
3. 是否需要任务草稿过期清理机制，避免长期停留在未开始状态。
4. 管理员是否需要人工重试、人工补偿、人工冻结额度等高权限操作。
5. 第二阶段是否需要把执行编排从 Web 进程中进一步隔离为独立 worker 进程。

### 13.2 当前基线结论

在这些开放问题未完全收敛前，第一版仍可以按以下基线实现：

- 不换框架，继续 `FastAPI` 单体。
- 围绕 `job -> job_item -> ai_execution` 建立主链。
- 用共享 Python 工作流核心 + 版本化步骤协议连接服务端与执行端。
- 用配额链、结算链、事件链和产物链补齐平台模式闭环。

这份文档可作为后续拆分 `service`、`runner`、`repository`、`migration` 的统一参考基线。
