# 2026-04-12 复杂 Mod 执行策略决策引导升级实施计划

## 目标

- 把现有 `review_bundles` 从“bundle 预览页”升级为“执行策略决策工作台”。
- 解决当前复杂 Mod 执行前缺少明确引导的问题，补齐“为什么被拦、该怎么处理、处理后会发生什么”的完整闭环。
- 在不推翻现有 `plan_mod -> review_items/review_bundles -> confirm_plan/start_with_plan -> batch_workflow` 主链的前提下，引入显式的 bundle 决策模型。

## 当前问题归纳

- 当前页面能展示 `bundle`、分组理由、风险标签，但缺少“接受当前分组 / 要求拆分 / 返回补充说明”的明确动作。
- 当前风险信息直接暴露内部 `risk_code`，缺少用户可理解的解释与操作建议。
- 当前 `needs_confirmation` 与 `split_recommended` 在交互上都被统一阻塞，用户无法分辨两者差异。
- 当前后端 gate 要求所有 bundle 先变成 `clear`，这意味着前端即使新增动作，也无法把“用户已做决策”传递到执行放行。

## 范围冻结

- 只覆盖复杂 Mod 批量生成链路，不改单资产工作流。
- 保留现有 `review_items` 阶段，不回退为单段确认流。
- 第一版不做任意拖拽式 bundle 编排。
- 第一版不让用户自由手工重组 item，只支持：
  - 接受系统建议的分组
  - 要求系统把某个 bundle 拆开
  - 返回 item 层补充依赖说明后重算
- 默认不自动执行 build、全量测试、重型验证，只做每个任务对应的最小定向验证。

## 设计原则

1. 风险展示必须伴随可执行动作，不能只告诉用户“有问题”。
2. 风险码必须翻译为面向用户的解释，不直接暴露内部枚举作为主要文案。
3. `bundle` 状态与“用户决策状态”分离建模，避免把“系统风险评估”和“用户是否已处理”混成一个字段。
4. 后端继续保留最终 gate，但 gate 的判断口径应升级为“风险 bundle 是否已完成有效决策”，而不是“所有 bundle 必须天然 clear”。
5. 恢复场景、重新检查、严格程度切换都必须保持决策状态一致或显式失效。

## 目标交互

### 顶部全局引导

- 标题从“执行策略确认”升级为“执行策略决策”。
- 顶部固定说明当前阶段目标：
  - 先决定哪些 item 必须一起执行
  - 再决定哪些 item 应拆开执行
  - 所有待决策 bundle 处理完后才进入真正执行
- 当存在阻塞 bundle 时，页面顶部直接显示结构化摘要：
  - 还有多少个 bundle 待接受
  - 还有多少个 bundle 建议拆分
  - 是否存在需要返回 item 层补充说明的风险

### bundle 卡片

每个 bundle 卡片至少包含：

- bundle 名称
- 包含 item
- 分组理由
- 风险解释
- 建议动作
- 当前决策状态

### 风险解释层

每个风险码需要映射为：

- 中文标题
- 这意味着什么
- 为什么系统这样判断
- 推荐处理方式

首批覆盖至少：

- `unclear_coupling`
- `bundle_size_threshold`
- `mixed_item_types`
- `affected_targets_spread`

### bundle 用户动作

- 对 `clear` bundle：
  - 默认直接视为可执行，不要求额外动作
- 对 `needs_confirmation` bundle：
  - `接受当前分组`
  - `返回补充依赖说明`
- 对 `split_recommended` bundle：
  - `要求拆分`
  - `仍接受当前分组`
  - `返回补充依赖说明`

## 数据契约升级

### 后端 review payload 新增字段

现有 `execution_bundles[]` 需要从静态预览升级为可决策对象。首版建议新增：

- `bundle_id`
- `status`
- `reason`
- `risk_codes`
- `risk_details`
- `recommended_actions`
- `blocking_reason`

其中：

- `risk_details` 为结构化解释列表
- `recommended_actions` 为后端建议动作集合
- `blocking_reason` 为当前 bundle 为什么会阻塞执行的用户可读说明

### 前端本地决策状态

前端新增 bundle 决策状态，不复用后端 `status`：

- `unresolved`
- `accepted`
- `split_requested`
- `needs_item_revision`

### 执行确认提交

`confirm_plan` / `start_with_plan` 需要允许携带 bundle 决策结果，例如：

- `bundle_decisions`

后端基于“review 结果 + bundle 决策”联合判断是否放行。

## 状态与放行规则

### 系统评估状态

- `clear`
- `needs_confirmation`
- `split_recommended`

### 用户决策状态

- `unresolved`
- `accepted`
- `split_requested`
- `needs_item_revision`

### 放行规则

- 所有 item 校验必须通过
- 所有 bundle 必须满足以下之一：
  - 系统状态为 `clear`
  - 用户已显式 `accepted`
  - 用户已 `split_requested` 且重算后不再阻塞
- 只要存在 `needs_item_revision` 或 `unresolved`，执行按钮保持阻塞

## 风险

- 前端一次性增加过多解释，可能让页面过载。
- 后端若不支持决策协议，前端动作会变成假交互。
- 接受高风险 bundle 时，如果解释不够清楚，会把黑箱改成“误点放行”。
- 恢复逻辑若未同步升级，可能出现页面显示已处理但后端仍拦截的状态漂移。

## 防线

- 默认收起详细解释，只在风险 bundle 展开更多内容。
- 前后端统一基于 `bundle_id` 传递决策，不靠索引位置推断。
- 保留后端最终 gate，但把拒绝原因做成结构化返回。
- 每个任务完成后跑对应最小测试，不积压到最后一次性排查。

## 最终验收标准

- 复杂 Mod 执行前的 P1 页面不再只是展示 bundle，而是完整的决策工作台。
- 风险 bundle 都有用户可读解释和明确动作。
- `needs_confirmation` 与 `split_recommended` 有差异化交互。
- 用户做出的 bundle 决策能够进入后端放行判断，而不是停留在前端。
- 恢复、重查、严格程度切换不会绕过或破坏 bundle 决策流程。
