# 平台后端改造实施计划 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在现有 `FastAPI` 单体中建立平台模式的持久化任务主链、服务层、步骤协议和用户/管理员 API，并在不重写现有入口的前提下逐步把旧路由迁移为薄传输层。

**Architecture:** 采用“单写新主链 + flag 控制入口与兼容投影”的迁移方式。新的 `job -> job_item -> ai_execution` 是平台模式唯一事实源，兼容性通过 router 委托、DTO 适配和统一事件模型处理，不做旧/新数据库事实双写。当前仓库缺少数据库与 migration 基础设施，因此先补 ORM / migration / session 基座，再落 schema、repository、service、runner 和平台 API。

**Tech Stack:** Python, FastAPI, Uvicorn, Pydantic, SQLAlchemy 2.x, Alembic, PostgreSQL, pytest, pytest-asyncio

---

## 审阅后修订结论

- 当前仓库没有成熟的旧任务数据库主链，也没有现成 `migrations` 体系，因此不采用旧方案中的 `dual-write + read by flag`。
- 当前后端已经形成 `app/modules/<domain>/application|domain|infra` 的分层习惯；新平台模块必须沿用这一结构，避免另起一套风格。
- 平台模式第一阶段直接纳入管理员接口和步骤协议，不推迟到第二阶段。
- 细颗粒度状态机直接冻结到数据库和代码，不再在实现期收缩为简版枚举。
- 第一阶段不落 `job_item.last_ai_execution_id` 实体字段，避免制造难以保证一致性的“伪最新引用”。
- 当前阶段计费和配额按“次数”实现，底层字段命名保持可扩展，但数据库数值类型优先使用整数。

## Decision Freeze

- 平台模式唯一事实主链为 `job -> job_item -> ai_execution`。
- 第一阶段状态枚举采用细颗粒度版本：
  - `job.status`: `draft`, `queued`, `running`, `partial_succeeded`, `succeeded`, `failed`, `quota_exhausted`, `cancelling`, `cancelled`
  - `job_item.status`: `pending`, `ready`, `running`, `succeeded`, `failed_business`, `failed_system`, `quota_skipped`, `cancelled_before_start`, `cancelled_after_start`
  - `ai_execution.status`: `created`, `dispatching`, `running`, `succeeded`, `failed_business`, `failed_system`, `completed_with_refund`
- 创建任务与开始执行分离；`draft -> queued/running` 是真实业务阶段，不用简化状态替代。
- 管理员接口属于第一阶段基线范围。
- 步骤协议属于第一阶段基线范围。
- `workflow_version` 落在 `jobs`。
- `step_protocol_version`、`result_schema_version` 落在 `ai_executions`。
- `job_item.last_ai_execution_id` 第一阶段不落库。
- 配额、计费、返还第一阶段使用整数次数口径。
- 兼容迁移方式冻结为：
  - 新平台表单写
  - 旧 router 保留
  - 新旧入口通过 flag 控制
  - 旧响应若需兼容，从新主链投影，不反向维护旧数据库事实

## Decision Gates

- Gate A: 未补齐数据库基础设施前，不开始 schema 和 repository 实现。
- Gate B: 未冻结平台 DTO、事件 contract、步骤协议字段前，不开始 router 接口实现。
- Gate C: 未跑通平台用户侧查询和管理员查询前，不把旧 `workflow` / `batch_workflow` 改成新委托。
- Gate D: 未稳定轮询事件流前，不启用统一 WebSocket 事件 contract。

## 风险与防线

- 风险：当前仓库没有既有 DB 栈，首批就要补基础设施。
  防线：先只打通基础设施和 contract，不在同一批引入 schema/service/router 逻辑。

- 风险：细颗粒度状态机会放大 service 和事件映射复杂度。
  防线：先把状态枚举冻结到 `contracts` 和 `domain/models/enums.py`，再写 schema。

- 风险：步骤协议和 legacy WebSocket 合同并存，容易出现事件顺序漂移。
  防线：统一事件模型先落 `contracts/events.py`，legacy router 只做适配，不重复发明结构。

- 风险：管理员接口一期纳入后，查询边界容易和用户接口混淆。
  防线：管理员 DTO、查询仓储、router 单独成组，不复用用户视图对象。
