# 2026-04-10 知识库资产化改造计划

## 1. 目标

把当前 STS2 / BaseLib 知识来源从“运行时临时拼装”改造成“独立资产目录 + 统一消费 facade + 工作流只读不构建”的结构。

## 2. 当前问题

- 游戏反编译源码、Baselib 反编译源码、仓库内静态参考和 runtime 知识目录同时存在，多头决策分散在不同模块。
- `codegen.api` 仍直接拼装多种来源路径，工作流消费层与知识来源决策耦合。
- `sts2_api_reference.md` 是仓库静态参考，但过去缺少明确版本标注，容易被误用成当前游戏版本真源。
- 知识库更新依赖 `ilspycmd`，但工具存在方式和查找顺序不清晰。

## 3. 冻结后的结构

### 3.1 运行时知识资产目录

- `runtime/knowledge/knowledge-manifest.json`
- `runtime/knowledge/game_decompiled/`
- `runtime/knowledge/baselib_decompiled/`
- `runtime/knowledge/cache/`

### 3.2 仓库静态 fallback

- `backend/agents/sts2_api_reference.md`
- `backend/agents/baselib_src/BaseLib.decompiled.cs`
- `backend/app/modules/knowledge/resources/sts2/*.md`

### 3.3 统一消费原则

- 工作流执行时不再触发知识库构建。
- Planning / Codegen / Analyzer 只读取知识状态与知识来源。
- 运行时知识目录优先于仓库静态 fallback。

## 4. 来源优先级

### 4.1 游戏源码

1. `runtime/knowledge/game_decompiled/`
2. `backend/agents/sts2_api_reference.md`
3. `missing`

### 4.2 Baselib

1. `runtime/knowledge/baselib_decompiled/BaseLib.decompiled.cs`
2. `backend/agents/baselib_src/BaseLib.decompiled.cs`
3. `missing`

## 5. 状态模型

- `fresh`
- `stale`
- `missing`
- `refreshing`
- `error`

补充来源字段：

- `game.source_mode`
  - `runtime_decompiled`
  - `repo_reference`
  - `missing`
- `baselib.source_mode`
  - `runtime_decompiled`
  - `repo_fallback`
  - `missing`

## 6. 第一阶段实施范围

### 6.1 新增统一 facade

新增：

- `backend/app/modules/knowledge/application/knowledge_facade.py`

负责：

- 统一决策当前游戏知识来源
- 统一决策当前 Baselib 知识来源
- 输出 codegen prompt 所需的 API lookup context

### 6.2 收口 codegen 入口

修改：

- `backend/app/modules/codegen/api.py`

要求：

- 不再直接组合 `get_active_game_decompiled_src_path()`、`get_decompiled_src_path()`、`get_active_baselib_src_path()`
- 统一改为调用 knowledge facade

### 6.3 补测试

新增：

- `backend/tests/test_knowledge_facade.py`

覆盖：

- runtime game 目录优先
- repo 静态参考回退
- runtime baselib 优先
- repo baselib 回退
- missing 场景
- `build_api_lookup_context()` 输出符合 codegen 预期

### 6.4 文档与工具前提

- `sts2_api_reference.md` 明确标注当前游戏 build 版本
- `install.ps1` 确保 `ilspycmd`
- `knowledge_runtime.py` 查找 `ilspycmd` 时先查项目目录，再查 PATH

## 7. 第二阶段实施范围

- 统一设置页、知识库说明和工作流 banner 的来源展示
- 让前端明确看到当前是 runtime knowledge 还是 repo fallback

## 8. 第三阶段实施范围

- 提供独立知识库刷新脚本
- 把知识库更新彻底从工作流执行链中剥离

## 9. 验收标准

- `codegen.api` 只从 knowledge facade 读取知识来源
- 工作流执行不再临时构建知识库
- `sts2_api_reference.md` 只作为 `repo_reference`
- Baselib / 游戏源码当前实际来源可被前后端统一表达
- 第一阶段定向测试通过
