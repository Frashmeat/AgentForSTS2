# AI协作文档规范迁移模板（规范）

> 文档定位：本文用于把“统一项目规范入口 + 统一知识索引 + 专题文档下钻 + 文档同步门禁”的协作规则迁移到其他项目。
>
> 适用范围：适用于需要 Claude / Codex / 其他 AI 编码助手共同参与维护的代码仓库。
>
> 使用方式：复制本文的规则框架到目标项目后，必须按目标项目的技术栈、目录结构、构建方式和发布要求做本地化调整。
>
> 最后更新：2026-04-26

## 一、目标

迁移这套规则时，先实现四个目标：

1. 让项目只有一个默认规范入口，负责说明“怎么做事”。
2. 让文档体系只有一个默认知识索引，负责说明“事实去哪看”。
3. 让 AI 修改代码、接口、数据库、配置或流程后，能同步更新对应文档。
4. 让长任务中的规范恢复、检查和收口成为固定流程，而不是依赖记忆。

## 二、推荐目录结构

目标项目建议至少具备以下文档入口：

```text
.
├── AGENTS.md                 # Codex 或通用 AI 助手跳板入口
├── CLAUDE.md                 # Claude 跳板入口，可选
├── PROJECT_SPEC.md           # 项目唯一规范入口
├── README.md                 # 系统概览、启动方式、脚本入口
└── docs/
    ├── README.md             # 知识库总索引
    ├── 需求文档/             # 业务规则、需求口径、活跃事项
    ├── 系统流程/             # 页面、操作流、业务闭环
    ├── 技术文档/             # API、部署、权限、集成说明
    ├── 数据库设计/           # 表结构、字段、初始化数据
    └── 审查记录/             # 审查、风险、整改、复盘
```

如果目标项目不需要某些目录，可以删除；但不要把“规范正文”和“知识索引”混在同一份文档里。

## 三、文件职责

### 1. `PROJECT_SPEC.md`

这是 AI 和人协作时的第一入口，只写规则，不写具体业务事实。

建议包含：

- 默认阅读顺序
- 协作流程
- 多 Agent 规则
- 构建、测试、启动服务、部署等高风险命令的执行边界
- 文档同步规则
- 代码实现约束
- 数据库变更约束
- Git 规则
- 项目级硬约束
- 文档命名规则

### 2. `docs/README.md`

这是知识库总索引，只回答“当前事实去哪看”，不承载规范正文。

建议包含：

- 默认使用方式
- 快速理解项目时的阅读顺序
- 文档目录导航
- 当前权威入口
- 历史材料使用方式

### 3. `AGENTS.md` / `CLAUDE.md`

这类宿主入口应尽量薄，只做跳转，不重复大段规则。

推荐正文：

```md
# AGENTS.md

本文件是 AI 助手在本仓库内的跳板入口。

默认阅读顺序：

1. `PROJECT_SPEC.md`
2. `docs/README.md`
3. 按任务进入对应专题文档
```

### 4. `README.md`

`README.md` 面向项目使用者，负责系统说明、启动方式、脚本入口、技术栈和部署提示。它可以链接规范入口，但不应替代 `PROJECT_SPEC.md`。

## 四、默认阅读顺序

AI 进入项目后，默认按以下顺序读取：

1. `PROJECT_SPEC.md`
   先理解协作规则、验证限制、文档同步要求和项目硬约束。
2. `docs/README.md`
   再定位需求、流程、接口、数据库、审查记录分别去哪看。
3. 专题文档
   按任务类型继续下钻。

推荐专题入口：

| 任务类型 | 优先阅读 |
| -------- | -------- |
| 业务规则、需求口径 | `docs/需求文档/` |
| 页面行为、流程闭环 | `docs/系统流程/` |
| 接口、部署、权限 | `docs/技术文档/` |
| 数据库表结构、字段、初始化数据 | `docs/数据库设计/` |
| 历史风险、整改、复盘 | `docs/审查记录/` |

## 五、协作流程

### 1. 默认原则

1. 默认先调查现状，再给计划，再进入实现。
2. 中等及以上复杂任务，先说明：
   - 当前理解
   - 计划
   - 影响范围
   - 风险
   - 验收标准
3. 用户确认前，不直接进入复杂实现。
4. 对已确认、无关键歧义、无重验证、无高破坏动作的任务，可以连续推进到：
   - 实现完成
   - 必要文档同步
   - 局部验证
   - 收口说明

### 2. 必须暂停确认的场景

遇到以下情况，应先暂停并请用户确认：

1. 需求边界存在关键未知项。
2. 需要执行 build、全量测试、启动服务、部署、数据重置等重型或高影响动作。
3. 涉及高破坏性操作，或影响范围明显超出原计划。
4. 用户即时要求与项目规则冲突。

### 3. 多 Agent 规则

默认关闭多 Agent / 多子进程。

如果任务适合并行拆分，必须先询问用户：

> 这个任务适合拆成多个子任务并行处理，是否开启多 Agent？默认不开启。

只有用户明确同意后，才开启子 Agent。

## 六、验证规则

目标项目应把验证命令分级写清楚。

推荐分级：

| 验证级别 | 示例 | 默认策略 |
| -------- | ---- | -------- |
| 静态阅读 | 查看文件、搜索引用、检查配置 | AI 可直接执行 |
| 局部轻量验证 | 类型检查、单个单元测试、格式检查 | 按项目规则决定是否可执行 |
| 重型验证 | build、全量测试、启动服务、部署、数据库重置 | 默认先建议命令，由用户确认 |

如果项目有特殊限制，例如“所有构建必须在 Docker 容器中执行”或“禁止本机安装工具链”，必须写入 `PROJECT_SPEC.md`。

## 七、文档同步规则

实际操作完成后，应立即回看相关文档，不把文档同步拖到最后。

建议同步矩阵：

| 改动类型 | 必须回看或更新 |
| -------- | -------------- |
| 后端接口 | API 接口文档、前端调用说明、变更记录 |
| 前端页面行为 | 系统流程文档、页面操作说明、需求口径 |
| 数据库表结构 | 建库脚本、测试数据、数据库设计文档 |
| 配置或脚本 | README、部署文档、脚本说明 |
| 权限或认证 | 权限说明、接口文档、流程文档 |
| 文档目录调整 | `docs/README.md` 和对应目录 `README.md` |
| 破坏性变更 | 相关规范、迁移说明、旧口径删除说明 |

如果不确定该改哪份文档，先读 `docs/README.md` 和相关目录 `README.md`，再决定。

## 八、文档命名规则

### 1. 状态归目录

文档状态由目录表达，不在文件名里重复表达。

推荐目录：

```text
现行口径/
活跃事项/
未开始/
进行中/
归档/
archive/
```

新增文档不建议继续使用以下状态词：

- `（当前）`
- `（已完成）`
- `（未开始）`
- `（进行中）`
- `（归档）`
- `（历史）`
- `（现状）`

### 2. 文件名只表达主题与用途

推荐格式：

```text
主题（用途）.md
```

示例：

```text
当前需求口径汇总（规范）.md
文档总览（索引）.md
登录流程说明（说明）.md
库存重构实施计划（计划）.md
权限模型调整（方案）.md
数据库结构（设计）.md
全工程风险复查（审查）.md
```

### 3. 用途词统一

新增正式文档时，用途词建议只使用以下 7 类：

1. `规范`
2. `索引`
3. `说明`
4. `计划`
5. `方案`
6. `设计`
7. `审查`

一个文档只选择一个主要用途，不使用复合用途。

### 4. 日期不是默认命名主结构

默认不要求在文件名前加日期。

只有在以下场景，才建议保留日期前缀：

1. 同主题文档会反复出现，需要按时间追溯。
2. 会议纪要、专项复盘、阶段审查等强时间序列材料。

## 九、文首格式

当前有效文档顶部建议统一写清：

```md
# 文档标题

> 文档定位：说明本文负责什么，不负责什么。
>
> 事实依据：说明是否基于当前代码、当前系统、历史讨论或人工确认。
>
> 权威入口：说明相关上级入口或专题目录。
>
> 最后更新：YYYY-MM-DD
```

计划、方案、审查类文档还应补充：

- 当前状态
- 适用范围
- 不纳入范围
- 风险与约束
- 完成标准

## 十、当前文档写法

1. 当前有效文档优先写成完整正文，直接描述“当前系统是什么、怎么工作、接口/流程/字段是什么”。
2. 已经沉淀为当前口径的内容，应并回正文，不在文末追加按日期排列的变更流水账。
3. 历史变化放入归档目录。
4. 活跃事项放入活跃事项目录。
5. 审查结论放入审查记录目录。
6. 索引文档只负责定位资料，不承担阶段总结、代码核对结论或长篇规则正文。

## 十一、AI 规范漂移治理

长任务中，AI 容易因为上下文变长、工具输出变多、报错信息挤占注意力而弱化项目规则。治理方式不是依赖记忆，而是建立可重复的规则恢复动作。

建议在以下时机执行一次规则恢复：

1. 对话明显变长。
2. 上下文发生压缩或恢复。
3. 任务范围扩大。
4. 从调查进入实现。
5. 从实现进入复查。
6. 从复查进入交付。

规则恢复动作建议包括：

1. 重读 `PROJECT_SPEC.md`。
2. 重读 `docs/README.md`。
3. 重读当前任务相关专题文档。
4. 列出当前必须遵守的规则。
5. 列出禁止自动执行的动作。
6. 列出需要同步的文档。
7. 明确下一步。

## 十二、落地清单

把本文迁移到其他项目时，按以下清单执行：

- [ ] 创建或收口 `PROJECT_SPEC.md`
- [ ] 创建或收口 `docs/README.md`
- [ ] 把 `AGENTS.md` / `CLAUDE.md` 改成薄跳板入口
- [ ] 明确 README 只承担系统说明和脚本入口职责
- [ ] 建立需求、流程、技术、数据库、审查等专题目录
- [ ] 为每个专题目录补 `README.md`
- [ ] 写清构建、测试、启动服务、部署命令的执行边界
- [ ] 写清代码改动后的文档同步矩阵
- [ ] 写清文档命名规则和文首格式
- [ ] 处理历史文档，把归档材料与当前口径分开
- [ ] 在一次真实任务中试运行，并根据遗漏点修正规范

## 十三、最小模板

如果目标项目暂时不想建立完整体系，可以先落地最小版本：

```text
.
├── AGENTS.md
├── PROJECT_SPEC.md
├── README.md
└── docs/
    └── README.md
```

`PROJECT_SPEC.md` 最少写清：

1. 默认阅读顺序。
2. 开发前是否需要计划和确认。
3. 哪些命令禁止 AI 自动执行。
4. 改代码后必须同步哪些文档。
5. 文档命名和归档规则。

`docs/README.md` 最少写清：

1. 当前需求去哪看。
2. 当前接口去哪看。
3. 当前数据库去哪看。
4. 当前流程去哪看。
5. 历史材料去哪看。

这个最小版本稳定后，再逐步补齐专题目录和检查流程。
