# AgentTheSpire 模块化解耦重构计划书

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将当前以技术分层为主的实现，重构为“领域模块 + 应用层 + 基础设施适配层 + 统一工作流编排内核”的结构，降低路由、工作流、LLM、图像、审批之间的直接耦合，提升可扩展性与可维护性。

**Architecture:** 现有后端主要按 `routers / agents / image / llm / approval` 划分，工作流编排逻辑分散在路由和 agent 中，前端 WebSocket 事件处理也存在按页面分裂的问题。本方案采用较大重构路线：后端新增 `shared` 与 `modules` 分层，所有业务模块只依赖端口接口，不直接依赖具体 CLI、图像供应商和执行器；单资产与批量流程统一收口到一个可扩展的工作流编排内核；前端同步收口为 `features + shared transport`，与后端共享标准化事件契约。

**Tech Stack:** Python FastAPI、pytest、Pydantic/dataclasses、React、TypeScript、WebSocket、dotnet、Claude CLI、Codex CLI、LiteLLM

## 实施顺序建议

1. 先做 `Chunk 1`，冻结公共事件契约和配置入口。
2. 再做 `Chunk 2`，把最容易扩展的外部能力先端口化。
3. 接着做 `Chunk 3`，拆清业务模块职责边界。
4. 再做 `Chunk 4`，引入统一工作流引擎，先单资产、后批量。
5. 然后做 `Chunk 5`，同步前端传输层与 feature 化改造。
6. 最后做 `Chunk 6`，通过迁移开关稳态切换并清理旧代码。

## 里程碑定义

- **M1:** 共享契约、配置、registry 稳定，可支撑新模块挂接。
- **M2:** LLM / 图像 / 审批 / 构建全部完成端口化，旧入口仍兼容。
- **M3:** 单资产工作流迁移到统一引擎并通过回归。
- **M4:** 批量工作流迁移到统一引擎并通过回归。
- **M5:** 前端切换到统一 transport 与 feature 装配层。
- **M6:** 删除旧直连路径，更新架构文档，完成收尾。

## 非目标

- 本次不改动业务功能语义，不额外新增新的资产类型。
- 本次不重写外部 CLI 或图像供应商实现，只重构接入边界。
- 本次不一次性移除全部旧代码，删除动作放到迁移完成后执行。
