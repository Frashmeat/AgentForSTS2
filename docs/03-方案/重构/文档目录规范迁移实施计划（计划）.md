# 文档目录规范迁移实施计划

> 文档定位：本文记录本仓库参考 `AI协作文档规范迁移模板（规范）.md` 后的文档治理改造计划，负责说明迁移目标、范围、步骤与待决策问题。
>
> 事实依据：基于 2026-04-26 对仓库根目录、`docs/README.md`、`AGENTS.md`、`AGENTS_CODEX.md`、`.trellis/` 与当前 `docs/` 目录结构的只读调查。
>
> 权威入口：迁移完成后以计划新增的 `PROJECT_SPEC.md` 与 `docs/README.md` 为默认入口；迁移前仍以现有 `AGENTS.md` 与 `docs/README.md` 为准。
>
> 当前状态：讨论中；`AGENTS_CODEX.md` 已确认采用彻底收口方案。
>
> 最后更新：2026-04-26

## 一、当前理解

本仓库已经存在一套有效的文档目录结构：

```text
docs/
  README.md
  01-总览/
  02-现状/
  03-方案/
  04-决策/
  05-审查与验证/
  90-归档/
  superpowers/
```

这套结构已经能表达项目文档的主要生命周期与阅读路径，不应为了套用模板而整体替换为 `需求文档/系统流程/技术文档/数据库设计/审查记录` 等新目录。

本轮迁移的核心不是重命名目录，而是收口入口职责：

1. `PROJECT_SPEC.md` 负责“怎么协作、怎么执行、哪些动作受限”。
2. `docs/README.md` 负责“事实去哪看、专题怎么下钻”。
3. `AGENTS.md` 负责跳转，不再承载大段规则正文。
4. 一级文档目录补齐自己的 `README.md`，降低查找成本。
5. 新文档逐步采用模板里的命名与文首格式，旧文档暂不批量移动或重命名。

## 二、迁移目标

1. 保留当前 `docs/01-总览` 到 `docs/90-归档` 的目录结构。
2. 新增或收口唯一项目规范入口 `PROJECT_SPEC.md`。
3. 将 `docs/README.md` 调整为知识库总索引，而不是规则正文集合。
4. 将 `AGENTS.md` 改造成薄跳板入口。
5. 为 `docs/` 下主要一级目录补充目录级索引。
6. 明确新文档命名、文首元信息、状态表达、归档规则。
7. 删除 `AGENTS_CODEX.md` 及其后端自动注入逻辑，运行时全局提示词统一交给 `llm.custom_prompt`。

## 三、非目标

本轮不做以下事情：

1. 不整体替换现有 `docs/` 一级目录命名。
2. 不按模板改成 `需求文档/系统流程/技术文档/数据库设计/审查记录`。
3. 已按用户确认，将带 `未开始-`、`进行中-`、`已经完成-` 前缀的旧文件迁移到同级状态子目录。
4. 不改变业务功能逻辑；仅允许为移除历史提示词注入链路做最小后端收口。
5. 不自动执行 build、全量测试或启动服务。
6. 不改变 Trellis 作为当前项目工作流入口之一的事实。

## 四、计划步骤

### 阶段 1：建立规范入口

新增 `PROJECT_SPEC.md`，收口以下内容：

- 默认阅读顺序。
- Trellis 工作流要求。
- 默认沟通语言与协作方式。
- 多 Agent 默认关闭规则。
- 构建、全量测试、启动服务、部署等高影响命令的执行边界。
- 文档同步规则。
- 代码、接口、数据库、配置、脚本、页面行为变更后的文档回看矩阵。
- Git 与提交边界。
- 文档命名规则与文首格式。
- 本项目继续保留现有 `docs/01-总览` 等目录结构的说明。

### 阶段 2：瘦身 AI 入口

调整 `AGENTS.md`：

- 保留为 AI 助手跳板入口。
- 默认阅读顺序改为：
  1. `PROJECT_SPEC.md`
  2. `docs/README.md`
  3. `.trellis/workflow.md`
  4. 当前任务相关专题文档
- 不再在 `AGENTS.md` 中重复大段规则正文。

`AGENTS_CODEX.md` 不再保留为后端运行时输入；相关规则迁移到 `PROJECT_SPEC.md` 或运行时自定义提示词说明中。

### 阶段 3：重写知识库总索引

调整 `docs/README.md`：

- 保留当前一级目录结构。
- 明确每个目录的职责。
- 给出不同任务类型的优先阅读路径。
- 标注当前权威文档入口。
- 标注历史材料和归档材料的使用方式。
- 删除或压缩与 `PROJECT_SPEC.md` 重复的协作规则。

### 阶段 4：补齐目录级索引

为以下目录新增 `README.md`：

- `docs/01-总览/README.md`
- `docs/02-现状/README.md`
- `docs/03-方案/README.md`
- `docs/04-决策/README.md`
- `docs/05-审查与验证/README.md`
- `docs/90-归档/README.md`

目录级索引只做下钻说明，不写长篇规则。

如需要，可后续再为专题子目录补索引，例如：

- `docs/03-方案/后端专题/README.md`
- `docs/03-方案/前端稳定化/README.md`
- `docs/03-方案/重构/README.md`
- `docs/04-决策/后端/README.md`

### 阶段 5：建立新旧文档过渡规则

迁移完成后执行以下规则：

1. 新增正式文档优先使用 `主题（用途）.md`。
2. 用途词优先限制在：`规范`、`索引`、`说明`、`计划`、`方案`、`设计`、`审查`。
3. 新增计划、方案、审查等存在推进状态的文档时，优先在对应专题目录下使用 `未开始/`、`进行中/`、`已经完成/` 子文件夹表达状态。
4. 文件名不再使用 `未开始-`、`进行中-`、`已经完成-` 等状态前缀，只表达主题与用途。
5. 当文档状态变化时，优先移动到对应状态子文件夹，并同步更新上级目录 `README.md` 或相关索引链接。
6. 旧文档暂不批量重命名或移动，避免破坏链接和历史上下文。
7. 当旧专题完成归档或被重新整理时，再顺手按新规则收口。

预计直接改动：

- `PROJECT_SPEC.md`
- `AGENTS.md`
- `docs/README.md`
- `docs/01-总览/README.md`
- `docs/02-现状/README.md`
- `docs/03-方案/README.md`
- `docs/04-决策/README.md`
- `docs/05-审查与验证/README.md`

`AGENTS_CODEX.md` 彻底收口直接改动：

- `backend/tests/test_agent_runner_selection.py`
- 与 `AGENTS_CODEX.md` 注入行为相关的历史文档或接口文档

## 六、风险

1. 入口文件规则重复：如果 `PROJECT_SPEC.md`、`AGENTS.md`、`docs/README.md` 同时写规则，会再次分散真源。
2. 链接断裂：批量移动或重命名旧文档会影响现有索引和历史引用，因此本轮避免大规模移动。
3. Trellis 与项目规范冲突：`PROJECT_SPEC.md` 需要明确 Trellis 是当前项目工作流的一部分，避免新入口覆盖既有流程。
4. `AGENTS_CODEX.md` 行为不透明：已选择删除文件和自动注入逻辑，需要同步改测试和文档。

## 七、验收标准

本轮文档迁移完成后，应满足：

- [ ] 当前 `docs/` 一级目录结构保持不变。
- [ ] `PROJECT_SPEC.md` 成为唯一项目规范入口。
- [ ] `AGENTS.md` 成为薄跳板入口。
- [ ] `docs/README.md` 成为知识库总索引。
- [ ] 主要一级目录都有自己的 `README.md`。
- [ ] 新文档命名规则、文首格式、状态表达方式写清楚。
- [ ] 旧文档不因本轮迁移发生大规模断链。
- [x] `AGENTS_CODEX.md` 的去留和后端注入行为有明确结论：彻底删除文件与后端自动注入逻辑。

## 八、`AGENTS_CODEX.md` 收口决策

当前调查结论：

- `backend/llm/agent_runner.py` 中存在 Codex 专属注入逻辑。
- 当 `agent_backend == "codex"` 时，后端会读取仓库根目录的 `AGENTS_CODEX.md` 并拼接到 agent prompt 前面。
- `backend/tests/test_agent_runner_selection.py` 中有测试覆盖这一行为。
- 当前后端同时已经支持 `llm.custom_prompt`，并通过 `append_global_ai_instructions()` 注入全局自定义提示词。

因此 `AGENTS_CODEX.md` 现在存在两类职责重叠：

1. 仓库内 AI 协作入口文档。
2. 后端运行时 Codex 专属默认提示词。

已确认采用选项 B：删除 `AGENTS_CODEX.md` 并同步删除注入逻辑。

落地要求：

- 删除 `AGENTS_CODEX.md`。
- 删除 `agent_runner.py` 中对 `AGENTS_CODEX.md` 的自动读取。
- 删除或改写对应测试。
- 将运行时提示词统一交给 `llm.custom_prompt`。
- 在当前接口或配置说明中明确 `custom_prompt` 是后端运行时全局提示词来源。
- 后续 `PROJECT_SPEC.md` 只作为项目协作规范入口，不自动注入后端生成 prompt。

以下为讨论时保留的备选记录：

### 选项 A：保留文件，取消后端自动注入

做法：

- 删除 `agent_runner.py` 中对 `AGENTS_CODEX.md` 的自动读取。
- 将运行时提示词统一交给 `llm.custom_prompt`。
- `AGENTS_CODEX.md` 可以改成历史说明，或改成只给本地 Codex 会话阅读的跳板文档。

优点：

- 后端运行时规则来源清晰。
- 与 `PROJECT_SPEC.md` 唯一规范入口不冲突。
- 不再有 Codex 专属暗中注入。

风险：

- 如果用户未配置 `custom_prompt`，Codex CLI 代码生成不再自动携带旧规则。

### 选项 B：删除 `AGENTS_CODEX.md` 并同步删除注入逻辑

做法：

- 删除 `AGENTS_CODEX.md`。
- 删除后端注入逻辑和对应测试。
- 将必要规则迁移到 `PROJECT_SPEC.md` 或运行时自定义提示词说明中。

优点：

- 最彻底，避免双入口。
- 文件职责最清晰。

风险：

- 需要确认没有外部脚本、人工流程或发布包仍依赖该文件。

### 选项 C：保留自动注入，但改为注入 `PROJECT_SPEC.md`

做法：

- 后端不再读取 `AGENTS_CODEX.md`。
- Codex 后端改为读取 `PROJECT_SPEC.md` 或一个专门的运行时提示词文件。

优点：

- 仍保留默认规则兜底。

风险：

- `PROJECT_SPEC.md` 是协作规范，不一定适合作为每次代码生成的运行时 prompt。
- 可能让后端生成逻辑和仓库协作规范再次耦合。

最终结论：

采用选项 B。彻底删除 `AGENTS_CODEX.md`，并把 Codex / Claude 的运行时全局行为规则统一收口到 `llm.custom_prompt`。
