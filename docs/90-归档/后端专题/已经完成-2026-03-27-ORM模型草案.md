# 2026-03-27 ORM 模型草案

> 适用范围：本文面向平台模式、用户维度第一版后端 ORM 建模草案，用于指导后续实体建模、仓储实现与数据库映射。本文不是最终代码，也不绑定具体 ORM 框架；设计目标是让模型边界、实体职责、字段暴露策略与仓储接口先收敛，再按具体技术栈落地。

## 一、建模目标与原则

### 1.1 建模目标

- 围绕已确认核心链路 `job -> job_item -> ai_execution` 建立第一版统一领域模型。
- 在 ORM 层明确“用户可见聚合数据”和“系统内部审计 / 计费数据”的边界。
- 为后续仓储实现提供稳定的实体命名、关系方向、字段分层与查询边界。
- 保持对 PostgreSQL 表结构友好，同时不过早绑定某个具体 ORM 框架语法。

### 1.2 设计前提

- 第一版只做用户维度，不做团队 / 组织 / 工作区维度。
- 第一版只覆盖平台模式；BYOK 不进入服务器数据库体系，因此不进入本文 ORM 建模范围。
- 计费最小单位是 `ai_execution`。
- 普通用户只看 `job` / `job_item` 聚合结果，`ai_execution` 仅管理员与系统内部可见。
- 主链、计费链、审计链默认不做物理删除，统一预留 `deleted_at` / `archived_at`。
- `artifacts` 以“元数据尽量保留、文件异步清理”为默认策略。

### 1.3 建模原则

- 聚合优先：用户侧读模型优先围绕 `Job` 聚合与 `JobItem` 子实体设计，不要求前端下钻到底层执行实体。
- 审计隔离：`AIExecution`、计费流水、额度流水、事件流属于内部域对象，不应与用户视图实体混用。
- 字段分层：同一 ORM 模型内区分“业务公开字段”“内部控制字段”“审计 / 冗余字段”，避免误暴露。
- 可映射性优先：实体关系、枚举与值对象设计应容易映射到常见 ORM 的一对多、一对一、组合唯一约束与软删除机制。
- 草案导向：优先定义边界和职责，不在本文中冻结所有字段类型、懒加载策略、级联策略和框架注解写法。

## 二、核心实体清单

### 2.1 聚合根

#### `Job`

- 定位：用户视角的大任务，是第一版最核心的聚合根。
- 职责：
  - 承载任务主状态与生命周期时间。
  - 承载 `job_items` 聚合统计与展示摘要。
  - 作为用户查询、任务列表、任务详情的主入口。
- 对外暴露建议：
  - 应直接作为用户侧仓储 / 查询的主模型输出基础。

#### `QuotaAccount`

- 定位：额度域的聚合根。
- 职责：
  - 表达用户额度账户。
  - 作为 `quota_buckets` 与 `usage_ledgers` 的归属主体。
- 对外暴露建议：
  - 可向用户暴露聚合后的额度视图，但不建议直接把完整 ORM 实体透给控制器 / API。

### 2.2 子实体

#### `JobItem`

- 定位：`Job` 聚合下的子实体。
- 职责：
  - 表达一个可独立观察、可独立执行的工作单元。
  - 承载子项级状态、摘要与最近执行结果引用。
- 对外暴露建议：
  - 可作为 `Job` 详情的一部分直接暴露给用户。
  - 不建议脱离 `Job` 聚合单独承载复杂写操作。

#### `QuotaBucket`

- 定位：`QuotaAccount` 聚合下的子实体或近似子实体。
- 职责：
  - 表达按周期划分的额度窗口。
  - 提供当前周期的额度统计基础。

### 2.3 审计 / 流水实体

#### `AIExecution`

- 定位：内部执行实体，既是最小计费单元，也是最小 AI 调用审计单元。
- 职责：
  - 记录一次真实 AI 调用的状态、上下文、摘要与幂等信息。
  - 连接任务域、计费域、返还域与排障域。
- 暴露策略：
  - 默认仅管理员与内部服务可见。
  - 不应直接进入普通用户仓储接口。

#### `ExecutionCharge`

- 定位：`AIExecution` 的一对一结算实体。
- 职责：
  - 记录单次执行的预扣、确认、返还状态与金额。

#### `UsageLedger`

- 定位：额度消耗 / 返还流水实体。
- 职责：
  - 作为配额对账、返还审计与问题回溯依据。

#### `Artifact`

- 定位：产物元数据实体。
- 职责：
  - 保存对象存储引用、文件摘要与清理状态。
  - 作为结果域和审计域之间的桥梁。

#### `JobEvent`

- 定位：事件流水实体。
- 职责：
  - 为轮询与后续 WebSocket 提供统一事件源。
  - 兼容任务级、子项级和执行级事件记录。

## 三、实体关系说明

### 3.1 主任务链

- `Job 1 -> N JobItem`
- `JobItem 1 -> N AIExecution`
- `Job 1 -> N AIExecution`

建模建议：

- ORM 层以 `Job` 持有 `jobItems` 集合作为主聚合关系。
- `AIExecution` 应同时保留 `jobId` 与 `jobItemId` 引用，便于计费、审计、排障直接定位。
- 普通用户读场景应默认停留在 `Job` 与 `JobItem`，不要通过 ORM 导航属性自动暴露 `AIExecution` 集合。

### 3.2 计费与配额链

- `AIExecution 1 -> 1 ExecutionCharge`
- `QuotaAccount 1 -> N QuotaBucket`
- `QuotaAccount 1 -> N UsageLedger`
- `AIExecution 1 -> N UsageLedger`（通常至少对应 reserve / capture / refund 中的一部分）

建模建议：

- `ExecutionCharge` 可建模为 `AIExecution` 的一对一从属实体。
- `UsageLedger` 更适合独立实体，不建议作为 `QuotaAccount` 内嵌集合长期整体加载，避免热点账户出现过大对象图。

### 3.3 结果与事件链

- `Job 1 -> N Artifact`
- `JobItem 1 -> N Artifact`
- `AIExecution 1 -> N Artifact`
- `Job 1 -> N JobEvent`
- `JobItem 0..1 -> N JobEvent`
- `AIExecution 0..1 -> N JobEvent`

建模建议：

- `Artifact` 与 `JobEvent` 建议作为独立实体仓储管理，而不是强耦合在 `Job` 聚合根上做全量级联加载。
- 对用户侧结果查询，可基于 `Job` / `JobItem` 投影出摘要；只有下载或审计时再查询 `Artifact`。

## 五、枚举与值对象建议

### 5.1 建议作为枚举建模的字段

- `JobStatus`
- `JobItemStatus`
- `AIExecutionStatus`
- `ChargeStatus`
- `LedgerType`
- `BucketType`
- `JobType`
- `ItemType`
- `ArtifactType`
- `EventType`

建议：

- ORM 层优先使用语言级枚举或受限常量集合，不直接在业务层散落裸字符串。
- 由于数据库草案倾向 `text + check constraint`，ORM 层应保留“枚举值可扩展”的适配空间。

### 5.2 建议作为值对象建模的内容

- `SummaryText`
  - 统一承载 `inputSummary`、`resultSummary`、`errorSummary` 这类面向展示的摘要文本。
- `TimeRange`
  - 用于 `QuotaBucket.periodStart + periodEnd`。
- `QuotaAmount`
  - 若后续额度口径从“次数”扩展到更一般的数值体系，建议单独抽值对象。
- `StorageRef`
  - 用于封装 `storageProvider + objectKey + fileName + mimeType`。

### 5.3 状态映射建议

- `Job` 与 `JobItem` 的状态不要求一一映射。
- `Job` 允许存在聚合态，例如 `partial_succeeded`、`quota_exhausted`、`cancelling`，这些状态应由子项分布和流程阶段共同推导。
- `AIExecution` 状态应保持偏底层、偏事实，不建议承载面向用户的聚合语义。

## 六、仓储 / 查询边界建议

### 6.1 用户侧仓储

建议优先提供：

- `JobRepository`
- `JobQueryRepository` 或等价读写分离接口

建议职责：

- 创建 `Job` 并展开 `JobItem`
- 按用户查询 `Job` 列表与详情
- 查询 `Job` 下的 `JobItem` 分页
- 更新 `Job` 聚合状态与计数

不建议职责：

- 直接暴露 `AIExecution` 明细
- 在用户接口内混入 `UsageLedger`、`ExecutionCharge` 原始实体查询

### 6.2 执行与计费侧仓储

建议拆分为独立仓储：

- `AIExecutionRepository`
- `ExecutionChargeRepository`
- `QuotaAccountRepository`
- `UsageLedgerRepository`

建议职责：

- 在启动执行前进行幂等查询与唯一约束兜底
- 创建执行记录、结算记录与额度流水
- 处理系统异常返还链路

### 6.3 结果与事件侧仓储

建议独立：

- `ArtifactRepository`
- `JobEventRepository`

原因：

- 结果元数据和事件流通常是高频追加、按条件分页查询，不适合总挂在 `Job` 聚合根上做大对象图加载。

### 6.4 管理员查询边界

建议建立单独管理员查询模型，而不是直接复用用户 ORM 实体：

- 管理员可看 `AIExecution`、`ExecutionCharge`、`UsageLedger`、execution 级 `JobEvent`
- 普通用户只看 `Job`、`JobItem` 和必要的 `Artifact` 摘要投影

## 七、软删除与归档建模建议

### 7.1 统一原则

- `jobs`、`job_items`、`ai_executions`、`execution_charges`、`usage_ledgers`、`job_events` 默认不做物理删除。
- ORM 层统一预留 `deletedAt`，并为后续冷热分层预留 `archivedAt`。
- 对于默认查询，仓储层应隐式过滤已软删除数据；管理员 / 审计查询可按需要显式包含。

### 7.2 `Artifact` 特殊处理

- `Artifact` 的 ORM 实体删除不应等价于对象存储文件立即删除。
- 文件删除应通过异步清理流程处理，可通过 `fileCleanupRequestedAt`、`fileCleanedAt` 建模。
- 即使文件已清理，元数据仍建议保留，以便对账、审计和结果解释。

### 7.3 归档建议

- 第一版可以先只实现 `deletedAt` 语义，`archivedAt` 仅保留字段与接口空间。
- 若后续实现冷热分层，建议按仓储查询策略而非业务实体 API 暴露归档明细。

## 八、第一版实现优先级

### 8.1 第一优先级：先落主链聚合

- `Job`
- `JobItem`
- `AIExecution`

目标：

- 先打通任务创建、子项展开、启动执行、执行完成后的状态回写与聚合展示。

### 8.2 第二优先级：补齐计费与额度最小闭环

- `ExecutionCharge`
- `QuotaAccount`
- `QuotaBucket`
- `UsageLedger`

目标：

- 打通按 `ai_execution` 计费、异常返还和额度窗口统计。

### 8.3 第三优先级：补结果与事件

- `Artifact`
- `JobEvent`

目标：

- 支撑结果下载、轮询事件流与后续 WebSocket 共模。

### 8.4 第一版不建议过早实现

- 团队 / 工作区维度归属
- BYOK 服务端模型
- 复杂价格版本体系
- 大而全的 ORM 级级联图
- 与具体 ORM 框架深度绑定的高级技巧

## 九、开放问题

1. `Job` 的“待确认”状态最终在 ORM 层命名为 `draft`、`created` 还是通过附加时间字段表达，仍需与最终状态机统一。
2. `JobItem.lastAIExecutionId` 是否需要真实落库，还是只保留为查询层派生能力，仍需结合性能与一致性权衡。
3. `AIExecution` 是否在第一版就引入更细的 `dispatching`、`completed_with_refund` 等中间态，尚未冻结。
4. `ExecutionCharge` 与 `UsageLedger` 的职责是否需要进一步收敛，避免双链路信息重叠。
5. `Artifact` 是否需要在 ORM 层区分“展示产物”和“审计附件”两个子类型实体，暂未决定。
6. `JobEvent` 是否需要数据库层直接携带 `visibleToUser` 一类字段，当前仍以查询层裁剪为主。
7. 未来若引入团队 / 组织维度，`QuotaAccount` 是否仍作为独立聚合根，还是需要上提到账户主体抽象，尚待后续设计。
8. 是否需要在 ORM 层显式建模并发版本字段（如 `version`）来保护聚合计数更新，第一版尚未确定。
