# 2026-03-27 SQL DDL 草案

> 适用范围：本文为平台模式、用户维度第一版数据库 SQL DDL 草案，目标数据库为 PostgreSQL。本文用于指导后续迁移脚本、ORM 模型与接口实现，不等同于最终可直接上线的迁移文件；字段命名、默认值、约束颗粒度、索引策略与归档策略后续仍可能调整。

## 一、说明与假设

### 1.1 当前范围

- 第一版只覆盖用户维度，不覆盖团队 / 组织 / 工作区维度。
- 第一版只覆盖平台模式；BYOK 不进入服务器数据库体系。
- 第一版核心任务模型为 `job -> job_item -> ai_execution`。
- 结果文件放对象存储，数据库只存元数据与摘要。
- 字符串枚举优先使用 `text check (...)`，暂不引入 PostgreSQL enum type。
- `event_payload` 使用 `jsonb`。
- `jobs` 与 `job_items` 保留聚合计数字段，优先服务列表页、详情页与轮询查询。

### 1.2 users 表假设

- 假设已有 `users` 表，以下表通过 `user_id` 关联。
- 本文不重复定义完整用户资料字段，仅假设 `users.id` 为主键。
- 若后续仓库确认尚无统一用户表，可再单独补一版用户表设计，不影响本文其他表的主链结构。

### 1.3 状态与结算枚举约定

- `jobs.status`: `created`, `running`, `cancelling`, `cancelled`, `succeeded`, `partial_quota_exhausted`, `failed_business`, `failed_system`
- `job_items.status`: `pending`, `running`, `succeeded`, `failed_business`, `failed_system`, `skipped_quota_exhausted`, `skipped_cancelled`
- `ai_executions.status`: `running`, `succeeded`, `failed_business`, `failed_system`
- `execution_charges.charge_status`: `reserved`, `captured`, `refunded`

### 1.4 删除策略约定

- 本草案优先保证审计、计费与排障链路可追溯，`jobs -> job_items -> ai_executions -> execution_charges / usage_ledgers / job_events` 默认不使用危险的物理级联删除。
- 若业务上需要“删除任务”，优先理解为软删除、归档或用户侧不可见，而不是直接删除主链数据。
- 建议主链与审计链统一预留 `deleted_at`；若后续确认需要冷热分层或后台归档，可再补 `archived_at`。
- `artifacts` 以“元数据尽量保留、对象存储文件异步清理”为默认方向，不把对象删除与主链删除强绑定。
- 因此以下外键草案默认省略 `on delete cascade`；是否允许物理删除留待正式迁移阶段结合归档策略再定。

## 二、建表顺序建议

建议按以下顺序推进，先落业务主链，再补配额、结果与事件：

1. `users`
   - 若已有统一用户表则直接复用，不重复建表。
2. `jobs`
3. `job_items`
4. `ai_executions`
5. `execution_charges`
6. `quota_accounts`
7. `quota_buckets`
8. `usage_ledgers`
9. `artifacts`
10. `job_events`

这样可以先跑通“创建任务 -> 展开子项 -> 启动执行 -> 记录执行 -> 计费 / 返还 -> 产物落库 -> 事件轮询”的最小闭环。

## 三、SQL DDL 草案

### 3.1 jobs

```sql
create table if not exists jobs (
    id                  bigserial primary key,
    user_id             bigint not null references users(id),
    job_type            text not null,
    status              text not null
                            check (status in (
                                'created',
                                'running',
                                'cancelling',
                                'cancelled',
                                'succeeded',
                                'partial_quota_exhausted',
                                'failed_business',
                                'failed_system'
                            )),
    input_summary       text,
    result_summary      text,
    error_summary       text,
    total_item_count    integer not null default 0 check (total_item_count >= 0),
    pending_item_count  integer not null default 0 check (pending_item_count >= 0),
    running_item_count  integer not null default 0 check (running_item_count >= 0),
    succeeded_item_count integer not null default 0 check (succeeded_item_count >= 0),
    failed_business_item_count integer not null default 0 check (failed_business_item_count >= 0),
    failed_system_item_count integer not null default 0 check (failed_system_item_count >= 0),
    skipped_quota_exhausted_item_count integer not null default 0 check (skipped_quota_exhausted_item_count >= 0),
    skipped_cancelled_item_count integer not null default 0 check (skipped_cancelled_item_count >= 0),
    created_at          timestamptz not null default now(),
    updated_at          timestamptz not null default now(),
    started_at          timestamptz,
    finished_at         timestamptz,
    cancel_requested_at timestamptz,
    deleted_at          timestamptz,
    archived_at         timestamptz,
    constraint uq_jobs_id_user_id unique (id, user_id)
);
```

补充说明：

- `jobs` 用 `created_at` + `status='created'` 表达“已创建但未开始”，不再单独保留 `confirmed_at`，避免与“创建和开始分开”的状态机重复表达。
- 删除任务默认指软删除或归档；`deleted_at` 用于用户侧不可见，`archived_at` 预留给后续冷热分层，不在本草案强制要求两者都立即启用。

### 3.2 job_items

```sql
create table if not exists job_items (
    id                  bigserial primary key,
    job_id              bigint not null,
    user_id             bigint not null references users(id),
    item_index          integer not null check (item_index >= 0),
    item_type           text not null,
    status              text not null
                            check (status in (
                                'pending',
                                'running',
                                'succeeded',
                                'failed_business',
                                'failed_system',
                                'skipped_quota_exhausted',
                                'skipped_cancelled'
                            )),
    input_summary       text,
    result_summary      text,
    error_summary       text,
    attempt_count       integer not null default 0 check (attempt_count >= 0),
    created_at          timestamptz not null default now(),
    updated_at          timestamptz not null default now(),
    started_at          timestamptz,
    finished_at         timestamptz,
    deleted_at          timestamptz,
    archived_at         timestamptz,
    constraint fk_job_items_job_user
        foreign key (job_id, user_id)
        references jobs(id, user_id),
    constraint uq_job_items_job_id_item_index unique (job_id, item_index),
    constraint uq_job_items_job_id_id unique (job_id, id),
    constraint uq_job_items_id_job_id_user_id unique (id, job_id, user_id),
    constraint uq_job_items_job_id_id_user_id unique (job_id, id, user_id)
);
```

补充说明：

- 本版先移除 `last_ai_execution_id`。仅靠数据库外键无法安全保证它一定指向“同一 `job_item` 下的最后一次 execution”，容易误导实现。
- 若后续确有性能需要，优先通过 `(job_item_id, created_at desc)` 查询最新 execution，或在应用层维护缓存字段并配套强校验。
- `job_items` 默认跟随主链走软删除/归档导向，不建议因为用户删除视图而物理移除子项记录。

### 3.3 ai_executions

```sql
create table if not exists ai_executions (
    id                      bigserial primary key,
    job_id                  bigint not null,
    job_item_id             bigint not null,
    user_id                 bigint not null references users(id),
    status                  text not null
                                check (status in (
                                    'running',
                                    'succeeded',
                                    'failed_business',
                                    'failed_system'
                                )),
    provider                text not null,
    model                   text not null,
    request_idempotency_key text,
    input_summary           text,
    result_summary          text,
    error_summary           text,
    started_at              timestamptz,
    finished_at             timestamptz,
    created_at              timestamptz not null default now(),
    updated_at              timestamptz not null default now(),
    deleted_at              timestamptz,
    archived_at             timestamptz,
    constraint fk_ai_executions_job_user
        foreign key (job_id, user_id)
        references jobs(id, user_id),
    constraint fk_ai_executions_job_item_user
        foreign key (job_id, job_item_id, user_id)
        references job_items(job_id, id, user_id),
    constraint uq_ai_executions_id_user_id unique (id, user_id)
);
```

补充说明：

- `ai_executions` 只在真正开始一次 AI 调用时创建，因此状态集合不再包含 `pending`。
- 用 `(job_id, user_id)` 与 `(job_id, job_item_id, user_id)` 复合外键同时收紧“同一用户、同一任务、同一子项”链路，避免跨租户 / 跨任务错绑。
- 第一版不单独引入幂等表；`request_idempotency_key` 仅在非空时参与约束，建议以 `user_id + job_item_id + request_idempotency_key` 作为作用域建立 partial unique index。
- `ai_executions` 属于审计与计费主链，默认也走软删除/归档导向，不建议通过物理删除清理历史执行记录。

### 3.4 execution_charges

```sql
create table if not exists execution_charges (
    id                  bigserial primary key,
    ai_execution_id     bigint not null,
    user_id             bigint not null references users(id),
    charge_status       text not null
                            check (charge_status in (
                                'reserved',
                                'captured',
                                'refunded'
                            )),
    charge_unit         text not null default 'execution',
    charge_amount       numeric(18, 6) not null check (charge_amount >= 0),
    refund_reason       text,
    reserved_at         timestamptz,
    captured_at         timestamptz,
    refunded_at         timestamptz,
    created_at          timestamptz not null default now(),
    updated_at          timestamptz not null default now(),
    deleted_at          timestamptz,
    archived_at         timestamptz,
    constraint fk_execution_charges_execution_user
        foreign key (ai_execution_id, user_id)
        references ai_executions(id, user_id)
        on delete restrict,
    constraint uq_execution_charges_ai_execution_id unique (ai_execution_id)
);
```

补充说明：

- 当前已确认“计费最小单位是 `ai_execution`”，因此 `execution_charges` 仍保持一条 execution 最多一条主计费记录的草案表达。
- `execution_charges` 默认保留完整历史，删除导向以软删除/归档为主，避免影响对账与返还追踪。

### 3.6 quota_buckets

```sql
create table if not exists quota_buckets (
    id                  bigserial primary key,
    quota_account_id    bigint not null references quota_accounts(id),

                            check (bucket_type in (
                                'daily',
                                'weekly'
                            )),
    period_start        timestamptz not null,
    period_end          timestamptz not null,
    quota_limit         numeric(18, 6) not null check (quota_limit >= 0),
```

### 3.7 usage_ledgers

```sql
create table if not exists usage_ledgers (
    id                  bigserial primary key,
    user_id             bigint not null references users(id),
    quota_account_id    bigint not null,
    quota_bucket_id     bigint,
    ai_execution_id     bigint,
    ledger_type         text not null
                            check (ledger_type in (
                                'reserve',
                                'capture',
                                'refund'
                            )),
    amount              numeric(18, 6) not null check (amount >= 0),
    balance_after       numeric(18, 6) not null check (balance_after >= 0),
    reason_code         text,
    created_at          timestamptz not null default now(),
    deleted_at          timestamptz,
    archived_at         timestamptz,
    constraint fk_usage_ledgers_account_user
        foreign key (quota_account_id, user_id)
        references quota_accounts(id, user_id)
        on delete restrict,
    constraint fk_usage_ledgers_bucket_account
        foreign key (quota_bucket_id, quota_account_id)
        references quota_buckets(id, quota_account_id)
        on delete restrict,
    constraint fk_usage_ledgers_execution_user
        foreign key (ai_execution_id, user_id)
        references ai_executions(id, user_id)
        on delete restrict
);
```

补充说明：

- 批量任务会边执行边检查配额，`usage_ledgers` 仍保留为执行级流水；是否需要额外冗余聚合表留待实现阶段再定。
- `usage_ledgers` 默认视作不可轻易物理删除的对账链；如确需清理，优先先归档再做离线物理回收。

### 3.8 artifacts

```sql
create table if not exists artifacts (
    id                  bigserial primary key,
    job_id              bigint not null,
    job_item_id         bigint,
    ai_execution_id     bigint,
    user_id             bigint not null references users(id),
    artifact_type       text not null,
    storage_provider    text not null,
    object_key          text not null,
    file_name           text,
    mime_type           text,
    size_bytes          bigint check (size_bytes is null or size_bytes >= 0),
    result_summary      text,
    created_at          timestamptz not null default now(),
    deleted_at          timestamptz,
    file_cleanup_requested_at timestamptz,
    file_cleaned_at     timestamptz,
    constraint fk_artifacts_job_user
        foreign key (job_id, user_id)
        references jobs(id, user_id)
        on delete restrict,
    constraint fk_artifacts_job_item_chain
        foreign key (job_item_id, job_id, user_id)
        references job_items(id, job_id, user_id)
        on delete restrict,
    constraint fk_artifacts_execution_user
        foreign key (ai_execution_id, user_id)
        references ai_executions(id, user_id)
        on delete restrict
);
```

补充说明：

- 产物元数据默认也按“可追溯优先”处理：任务主链若需对用户不可见，优先软删除或归档，不通过物理删除把 `artifacts` 反向打成孤儿记录。
- 真正的对象存储文件删除建议走异步清理；`file_cleanup_requested_at` / `file_cleaned_at` 仅作为草案阶段的预留表达。

### 3.9 job_events

```sql
create table if not exists job_events (
    id                  bigserial primary key,
    job_id              bigint not null,
    job_item_id         bigint,
    ai_execution_id     bigint,
    user_id             bigint not null references users(id),
    event_type          text not null,
    event_payload       jsonb not null default '{}'::jsonb,
    created_at          timestamptz not null default now(),
    deleted_at          timestamptz,
    archived_at         timestamptz,
    constraint fk_job_events_job_user
        foreign key (job_id, user_id)
        references jobs(id, user_id)
        on delete restrict,
    constraint fk_job_events_job_item_chain
        foreign key (job_item_id, job_id, user_id)
        references job_items(id, job_id, user_id)
        on delete restrict,
    constraint fk_job_events_execution_user
        foreign key (ai_execution_id, user_id)
        references ai_executions(id, user_id)
        on delete restrict
);
```

补充说明：

- 用户侧默认只返回 `job` / `job_item` 聚合事件；带 `ai_execution_id` 的 execution 级事件默认仅管理员可见，用于排障、审计与计费核对。
- 因此 `job_events` 在草案阶段允许同时挂 `job_item_id` 与 `ai_execution_id`，但接口层不应把 execution 级细粒度事件直接暴露给普通用户。
- `job_events` 属于关键审计链，草案默认以 `restrict / no action` 导向保护事件记录；正式迁移若要支持物理清理，应先补归档链路而不是依赖 `set null`。

## 四、索引草案

以下索引优先覆盖已讨论的核心查询路径：

```sql
create index if not exists idx_jobs_user_id_created_at_desc
    on jobs (user_id, created_at desc);

create index if not exists idx_jobs_status_created_at_desc
    on jobs (status, created_at desc);

create index if not exists idx_ai_executions_job_item_id_created_at
    on ai_executions (job_item_id, created_at);

create unique index if not exists uq_ai_executions_user_item_idempotency_key
    on ai_executions (user_id, job_item_id, request_idempotency_key)
    where request_idempotency_key is not null;

create index if not exists idx_usage_ledgers_user_id_created_at_desc
    on usage_ledgers (user_id, created_at desc);

create index if not exists idx_job_events_job_id_created_at
    on job_events (job_id, created_at);
```

补充说明：

- `uq_job_items_job_id_item_index`、`uq_execution_charges_ai_execution_id`、`uq_quota_buckets_period` 已分别提供唯一约束，草案中不再重复列出同列索引。
- 第一版幂等策略明确为“不做独立幂等表”，先在 `ai_executions` 上落 scoped partial unique；当前建议作用域为 `user_id + job_item_id + request_idempotency_key`，且仅对非空 key 生效。
- 若管理员后续高频按 `ai_executions.user_id`、`ai_executions.status` 检索，可再补组合索引。
- 若前端高频按 `artifacts.job_item_id` 拉取结果列表，可再补 `artifacts(job_item_id, created_at)`。

## 五、后续可能调整项

以下内容在进入正式迁移脚本前仍可能调整：

1. 主键类型
   - 当前草案统一使用 `bigserial`，后续可能按仓库既有规范改为 `uuid` 或 `generated always as identity`。
2. users 复用方式
   - 当前假设已有 `users` 表；若实际用户体系字段或主键类型不同，需要统一调整所有 `user_id` 外键类型。
3. job_type / item_type / event_type 枚举集合
   - 当前仅保留 `text` 字段，不在本草案中冻结完整枚举表。
4. 配额数值类型
   - 当前 `charge_amount`、`quota_limit`、`used_amount` 等采用 `numeric(18, 6)`，若最终是“纯次数”口径，可收敛为整数。
5. 状态机细分
   - 当前状态值按已知结论收敛，后续如果需要区分 `queued`、`ready`、`dispatching` 等中间态，可补充字段或扩展 check 集合。
6. 删除策略
   - 当前草案已明确主链与审计链默认走软删除/归档导向，并预留 `deleted_at` / `archived_at` 类字段；最终归档表设计与后台物理清理节奏仍需结合审计要求确认。
7. updated_at 自动维护
   - 草案只声明字段，不包含触发器或应用层自动更新时间方案。
8. 对象存储元数据
   - `artifacts` 目前只保留最小元数据与异步清理预留字段，后续可能补 `sha256`、`expires_at` 等扩展信息。
9. 审计与幂等
   - `request_idempotency_key` 已按“`user_id + job_item_id + request_idempotency_key` 的 scoped partial unique”写入草案；正式迁移前仍需确认该作用域是否最终冻结。
10. `job_events` 可见性边界
   - 当前仅在文档层明确“用户看聚合事件、管理员看 execution 事件”；若后续需要在数据库层直接辅助鉴权，可再考虑增加事件可见性字段。
11. 迁移拆分方式
   - 本文是 DDL 草案，不是最终迁移脚本；正式落地时建议拆成多份 migration，并补数据修复、回填与回滚策略。
