## 1. 总体边界

- 本次改造只覆盖平台模式后端。
- BYOK 模式不进入服务端数据库主链，不进入本次持久化、配额、计费、返还、审计实现。
- 当前后端继续采用 Python 单体 `FastAPI + Uvicorn`，不做服务拆分。
- 当前仓库没有成熟旧任务数据库主链，因此不采用“旧/新数据库事实双写”。

## 2. 迁移策略

- 采用“单写新主链 + flag 控制入口与兼容投影”。
- 新的 `job -> job_item -> ai_execution` 是平台模式唯一事实源。
- 旧 router 保留，但后续逐步委托给新的 service / runner。
- 平台用户侧 API 通过 `platform_jobs_api_enabled` 控制挂载，平台管理员侧 API 通过 `platform_service_split_enabled` 控制挂载。
- 若需要兼容旧响应结构，应从新主链投影，不反向维护旧数据库事实。
- 第一阶段不做全量切流，不移除旧入口。
- 当前仓库尚未提供统一 `users` 表与对应 ORM 元数据；平台 schema 第一版先保留 `user_id` 字段和平台内链路约束，不在本批 migration 中直接建立 `users` 外键，待后续用户体系基线明确后再补迁移。

## 3. 冻结的核心模型

### 3.1 任务主链

- `job`：用户视角的大任务，负责聚合状态、聚合计数、聚合摘要。
- `job_item`：普通用户可见的最小工作单元。
- `ai_execution`：单次实际 AI 调用，最小计费单元，仅管理员与内部服务可见。

### 3.2 计费与配额

- 第一阶段计费和配额按“整数次数”实现。
- 扣减最小单位为 `ai_execution`。
- 批量任务采用边执行边检查配额，不预占整批。
- 系统异常按冻结的 v1 refund matrix 自动返还。

### 3.3 版本字段

- `workflow_version` 落在 `jobs`。
- `step_protocol_version` 落在 `ai_executions`。
- `result_schema_version` 落在 `ai_executions`。

## 5. 冻结的行为规则

- 创建任务与开始执行分离，`draft -> queued/running` 是真实阶段，不简化掉。
- 用户取消后，未开始项不再启动；已开始的执行尽量收尾。
- 用户侧只看 `job` / `job_item` 聚合视图。
- 管理员接口一期纳入范围，可看 `ai_execution`、refund、audit。
- 平台用户/管理员 HTTP 路由一期直接消费平台 service，不依赖先迁移旧 WebSocket 入口。
- 步骤协议一期纳入范围，先定义最小字段集，再接入 runner。

### 5.1 统一错误契约

- HTTP 非成功响应统一返回 `error envelope`，结构固定为 `{"error":{"code","message","detail","request_id?"}}`。
- `message` 面向前端直接展示，要求可读、可落地；`detail` 用于补充诊断信息，可为空；`code` 作为稳定机器码供前端和日志归类。
- 业务型 `HTTPException` 允许把明确可展示的信息写入 `message/detail`；未知异常统一收口为 `internal_server_error`，前端只展示通用提示，不直接暴露 traceback 或敏感内部错误。
- 工作站 WebSocket `error` / `item_error` 事件统一补齐 `code`、`detail`、`request_id?`、`traceback?` 字段命名；其中 `message` 仍作为首要展示文案，`detail` 用于补充诊断信息。
- 前端统一按“结构化 `error.message` / `message` -> `detail` -> 友好兜底文案”的优先级解析，禁止把原始 JSON 或 `[object Object]` 直接展示给用户。

### 5.2 统一工作流日志展示契约

- 工作站各类工作流日志统一采用“单面板双标签”展示，固定提供 `优化后输出` 与 `原始输出` 两个视图。
- 日志面板顶部固定显示“当前实际调用模型”；若当前阶段没有模型信息，则显示统一兜底文案，而不是暴露底层配置对象。
- 后端 `stream` / `agent_stream` / `item_agent_stream` 事件统一允许附带 `source`、`channel`、`model` 元信息，用于前端做轻量展示优化，不改变原有主事件名。
- 前端“优化后输出”只允许做轻量规则处理，例如相邻重复折叠、`stderr` 前缀提示、连续分片合并；不得对模型原始输出做 AI 改写。
- 前端“原始输出”保留原始文本顺序与内容，不额外润色，但允许把连续同源分片合并成一个连续文本块，避免把 websocket 分片边界暴露给用户。
- 2026-04-09 起，单资产 `Code Agent` 面板的“优化后输出”允许进一步收口为 `执行播报` 布局；该布局仍然从共享日志条目规则派生，不改变原始视图边界。
- 单资产 `Code Agent` 的首轮 `执行播报` 固定包含：
  - 顶部状态头：`当前模型`、模式切换、`X / 6 已完成`
  - 单段 `当前状态` 说明
  - 代码生成 6 步固定进度
  - `执行摘要`
  - `技术细节` 折叠区
- 上述 `X / 6` 仅表示代码生成展示层的固定步骤推进，不映射数据库任务状态，也不要求与其他工作流共用同一组步数。
- 单资产 `Code Agent` 的固定 6 步冻结为：`理解任务`、`扫描项目`、`定位改动点`、`生成修改方案`、`写入代码`、`自检收尾`。

## 6. 第一阶段明确不做

- 不做旧/新数据库事实双写。
- 不做 BYOK 服务端建模。
- 不落 `job_item.last_ai_execution_id`。
- 不在第一阶段引入团队 / 工作区 / 组织维度。
- 不自动执行 build、全量测试、重型验证。

## 7. 实现顺序

1. 先补数据库基础设施与 migration 框架。
2. 再落平台 contracts、枚举和模块骨架。
3. 再落 schema 与 repositories。
4. 再落 services、runner、步骤协议。
5. 再落平台用户/管理员 API。
6. 最后把 legacy routers 逐步改成薄传输层。

## 8. 2026-04-08 本地 mixed deploy 生命周期基线

- `tools/latest/deploy-docker.ps1` 在 `hybrid` / `workstation` / `frontend` / `full` 下拉起的本地服务属于后台进程，不与日志终端窗口绑定生命周期。
- 本地服务启动后必须写入 `release/runtime/local-deploy-state.json`，至少记录目标、release 根目录、服务名、PID、端口。
- 停止这些本地服务时，统一使用显式脚本 `tools/latest/stop-deploy.ps1` 读取状态文件并停止；不依赖用户手工关闭日志终端。
- `tools/latest/deploy-docker.ps1 hybrid` 默认联动本机 `web-backend` 时，如需刷新默认 `agentthespire-web-release`，必须先停止同名 Compose 项目，避免直接改写仍在使用的 release 目录。
- `tools/latest/deploy-docker.ps1 web` 必须把 `SPIREFORGE_AUTH_SESSION_SECRET` 与 `SPIREFORGE_SERVER_CREDENTIAL_SECRET` 持久化到 `release/runtime/.env`，并通过 Compose 环境变量注入；生成后的 `runtime/web.config.json` 不再继续保留 `auth.session_secret`。
- 同名 `web release` 重新打包时必须保留既有 `runtime/.env`，避免 secret 漂移；正式部署前还必须把 release 内 `docker-compose.yml` 刷新为当前模板，避免旧 release 遗留的 Compose 注入方式失效。
- `workstation` 本地 Python 运行时缓存固定保存在 `release/runtime/python-runtime/workstation`，并以 `requirements.txt` 指纹和启动 Python 路径作为缓存有效性判断依据。
- `package-release.ps1` 生成 zip 时默认排除 `release/runtime/python-runtime` 本地缓存；缓存继续留在 release 目录供本地 mixed/workstation 复用，但不再进入归档发布内容。
- 本地 `hybrid` / `frontend` 独立静态前端若显式改用非默认端口，部署脚本必须把实际前端端口同步补入 `runtime/workstation.config.json` 与 linked `web` 的 `runtime/web.config.json` 的 `cors_origins`，避免前端端口与后端白名单脱节。
- 本地 `workstation` 与 linked `web` 运行时允许额外开启 loopback origin 兜底，仅用于 `localhost` / `127.0.0.1` / `::1` 下的本机联调；该兜底不得直接替代正式公网 `web` 部署的显式来源白名单。

## 9. 2026-04-08 工作站配置真源基线

- 工作站应用级配置唯一真源冻结为 `runtime/workstation.config.json`。
- 仓库根目录 `config.json` 仅允许承担开发输入、迁移来源或模板相关职责，不再被定义为当前运行实例的工作站配置真源。
- `services/workstation/config.json` 如在迁移期仍然存在，只能作为兼容占位文件或由真源派生的镜像，不得继续作为独立运行时真源。
- 历史 Docker / mixed deploy 配置链路不得再拥有新的工作站应用配置所有权，只能消费唯一真源生成运行参数。
- Codex CLI 与 Claude CLI 的用户全局配置属于外部环境输入，不属于项目配置真源；后续实现只能把它们作为风险来源、兼容来源或隔离对象对待。
- Windows 下图片后处理在探测 ONNX Runtime CUDA provider 前，必须先做 provider DLL 可加载性预检；若缺少 `cublasLt64_12.dll` 等 CUDA 依赖，则直接回退 `CPUExecutionProvider`，不得把该场景继续打印为工作流主错误。
- Windows 下 Claude CLI 的代码生成链路不得把长 prompt 直接放入 `claude.cmd ... -p <prompt>` 批处理命令行；实现必须绕过 `.cmd` 包装层，优先调用 `node + cli.js` 等非 `cmd.exe` 路径，避免因命令行长度限制造成秒退。
- 设置面板中的 `CLI 模型（可选）` 在 `agent_cli` 模式下属于有效运行参数；当用户选择 Codex CLI 或 Claude CLI 时，后端都必须把 `llm.model` 透传给对应 CLI。

## 10. 2026-04-09 工作站知识库版本治理补充基线

- 工作站知识库状态必须显式区分 `checking`、`fresh`、`stale`、`missing`、`refreshing`、`error`。
- 游戏知识库版本真源冻结为：当前自动检测到的 `sts2_path` 对应 Steam `app manifest / 安装版本文本`。
- Baselib 知识库版本真源冻结为：GitHub latest release tag。
- 第一版 stale 判定只比较版本/tag，不严格比较 hash。
- 设置页必须同时提供：
  - `检查更新`
  - `更新知识库`
  - `查看知识库说明`
- “知识库说明”只负责展示来源与工程内位置，不承担更新动作。
- Baselib 知识库刷新在下载 GitHub release 资产时必须显式跟随 3xx 跳转；不能把 `302 Found` 直接当成下载失败。
- 若最新 Baselib release 的资产命名不再匹配当前 DLL 选择规则，错误信息必须包含 release tag、资产名列表以及当前匹配规则，便于用户和开发者快速判断是否需要调整发布兼容逻辑。
- 发行包必须直接包含 `runtime/knowledge/`；该目录冻结为用户可查看、可编辑、且修改后直接生效的运行时唯一真源。
- 工作站本地知识目录冻结到：
  - `runtime/knowledge/knowledge-manifest.json`
  - `runtime/knowledge/game/`
  - `runtime/knowledge/baselib/`
  - `runtime/knowledge/resources/sts2/`
  - `runtime/knowledge/cache/`
- 仓库内 `backend/agents/*` 与 `backend/app/modules/knowledge/resources/sts2/*` 只允许承担开发期与打包期种子来源职责，不得继续作为运行时并列知识真源。
- codegen prompt 与文档说明不得再把仓库内静态 Baselib 副本描述为长期唯一权威真源；实现必须直接消费上述运行时知识目录，并在缓存 stale 时提示风险。

## 11. 2026-04-10 STS2 项目路径配置同步补充基线

- 设置页中的 `sts2_path` / `godot_exe_path` 属于工作站运行时真源，项目构建不得继续把历史 `local.props` 视为独立真源。
- `local.props` 允许作为 STS2 mod 项目的本地构建输入，但其托管字段必须可由当前运行时设置同步更新，而不是“仅首次生成”。
- 工作站自动生成/同步的 `local.props` 必须与 STS2 模板 `.csproj` 实际消费的 MSBuild 字段一致；本轮冻结为：
  - `SteamLibraryPath`
  - `GodotPath`
- 已有项目进入 Build & Deploy 或同类构建链路前，后端必须先执行一次低风险 `local.props` 同步，覆盖“设置已更新但项目保留旧路径”的场景。
- 对 `local.props` 的同步属于受控更新：仅允许工具托管字段被覆盖，未知自定义字段应保留。

## 12. 2026-04-20 服务器运行时锁真源补充基线

- `server_project_ref` 工作区锁与 `project_name` 部署锁当前统一冻结为锁文件 `v2` 协议。
- 锁文件只负责表达“当前占用”，至少包含：
  - `lock_version`
  - `job_id`
  - `job_item_id`
  - `user_id`
  - `owner_scope`
  - `locked_at`
  - `expires_at`
  - 最小资源上下文字段
- 当前陈旧锁恢复统一采用“短期 `recovering` claim + 接管旧锁”口径，避免多个恢复者同时覆盖同一把锁。
- 对旧版未写 `expires_at` 的锁文件，当前按 `locked_at + lease` 口径做兼容判断，避免升级瞬间误接管仍在执行中的旧任务。
- `.server-deploy.json` 当前继续只表达“最近一次成功部署事实”，不承担“当前占用”真源职责。
- 本补充基线只冻结锁真源协议与恢复口径，不等同于多实例 queue worker 协调已经完成；多实例消费归属、去重与抢占仍需后续实现。

## 13. 2026-04-20 queued job 消费归属补充基线

- `queued` job 的启动归属当前冻结为 `job_id` 级 claim 第一版。
- 任何进入实际启动链的入口，包括：
  - 显式 `start_job`
  - `system_queue_worker`
  - `system_workspace_resume`
  当前都必须先尝试获取同一个 `job_id` claim，再进入 `_start_job_internal`。
- `job_id` claim 的职责仅限于“同一个 job 只能被一个消费者拉起”，不得替代：
  - 工作区写锁
  - 部署目标锁
  - 多实例 leader / scan 协调
- 若 `job_id` claim 已被其他消费者持有，后续消费者必须跳过，不继续进入实际启动链。
- 本补充基线只收口单 job 的启动归属，不等同于多实例 queue worker 已具备完整的扫描、选主、抢占与去重能力。

## 14. 2026-04-20 queue scan 协调补充基线

- queue worker 当前已新增 leader / 租约第一版。
- 任一 `web` 实例当前都可竞争 queue worker leader；持有 leader lease 的实例负责持续扫描，未持有 lease 的实例必须跳过后台扫描。
- leader lease 当前补充 `leader_epoch` 代际字段；续租不变更 epoch，接管时递增。
- leader lease 的职责仅限于“谁可以持续扫描 queued 列表”，不得替代：
  - `job_id` claim
  - 工作区锁
  - 部署目标锁
- 本补充基线只收口 leader / 租约第一版，不等同于已经具备稳定主备切换、完整长期选主或抢占策略。
