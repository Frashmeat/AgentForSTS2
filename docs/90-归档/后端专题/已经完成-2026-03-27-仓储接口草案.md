# 2026-03-27 仓储接口草案

> 适用范围：本文面向平台模式、用户维度第一版后端仓储接口设计草案，基于现有 ORM 模型草案、后端设计草案与 SQL DDL 草案整理而成。本文目标是为后续 repository / service 实现提供边界、职责分层与事务建议；本文不是最终代码接口，也不绑定具体语言、ORM 或框架。

## 一、设计目标与分层原则

### 1.1 设计目标

- 围绕已确认主链 `job -> job_item -> ai_execution` 收敛第一版仓储边界。
- 让用户侧、管理员侧、内部调度 / 计费侧各自拥有独立仓储接口，不混成一套“大而全”的数据访问层。
- 明确哪些仓储返回聚合根，哪些仓储只返回轻量投影、列表项或审计明细。
- 为后续 service 编排提供稳定的事务边界与并发控制建议。

### 1.2 已采用的前提

- 第一版只覆盖平台模式、用户维度。
- BYOK 不进入服务端持久层，因此不进入本文仓储设计范围。
- 普通用户仅查看 `Job` / `JobItem` 聚合结果，`AIExecution` 仅管理员与内部服务可见。
- 计费最小单位是 `AIExecution`。
- 幂等策略与软删除 / 归档策略已在 DDL 与 ORM 草案中确认，仓储层应直接承接这些约束，而不是另起一套实现语义。

### 1.3 分层原则

- 用户侧仓储面向“任务聚合查询与任务生命周期写入”，不暴露执行底账。
- 管理员侧仓储面向“审计、排障、对账、检索”，默认以读为主。
- 内部调度 / 计费仓储面向“创建执行、额度校验、记账、返还、事件落库”等系统流程，不直接承担前端展示模型输出。
- 聚合根优先：`Job`、`QuotaAccount` 更适合作为写入与一致性边界；`AIExecution` 适合作为内部事实实体，不应直接上浮为普通用户模型。
- 读写分离优先：列表、分页、管理员检索、事件流、结果摘要更适合轻量投影，不要求每次加载完整对象图。

## 二、仓储清单总览

| 分层 | 建议仓储 | 主要职责 | 主要返回 |
| --- | --- | --- | --- |
| 用户侧 | `JobRepository` | 创建任务、加载任务聚合、更新任务聚合状态、受理取消 | `Job` 聚合根 |
| 用户侧 | `JobQueryRepository` | 任务列表、详情投影、子项分页、用户可见事件 / 结果摘要查询 | `JobListItem` / `JobDetailView` / `JobItemListItem` |
| 用户侧 | `QuotaQueryRepository` | 查询用户当前额度视图 | `UserQuotaView` |
| 管理员侧 | `AdminJobQueryRepository` | 按任务查看主链聚合、管理员视角任务检索 | `AdminJobDetailView` / `AdminJobListItem` |
| 管理员侧 | `AdminExecutionQueryRepository` | 查询执行明细、执行列表、错误与耗时信息 | `AIExecutionDetailView` / `AIExecutionListItem` |
| 管理员侧 | `AdminBillingQueryRepository` | 查询计费记录、返还记录、额度流水 | `ExecutionChargeView` / `UsageLedgerView` |
| 管理员侧 | `AdminEventQueryRepository` | 查询审计事件、execution 级事件 | `JobEventView` |
| 内部调度 / 计费 | `AIExecutionRepository` | 幂等查找、执行记录创建、状态更新、按子项获取最近执行 | `AIExecution` |
| 内部调度 / 计费 | `ExecutionChargeRepository` | 创建 / 更新单次执行计费记录 | `ExecutionCharge` |
| 内部调度 / 计费 | `QuotaAccountRepository` | 加载额度账户、锁定当前 bucket、更新聚合额度 | `QuotaAccount` |
| 内部调度 / 计费 | `UsageLedgerRepository` | 写入 reserve / capture / refund 流水、对账查询 | `UsageLedger` |
| 内部调度 / 计费 | `ArtifactRepository` | 保存产物元数据、按链路查询结果引用 | `Artifact` / `ArtifactSummary` |
| 内部调度 / 计费 | `JobEventRepository` | 追加统一事件流 | `JobEvent` |

结论：

- 第一版不建议用单个 `PlatformRepository` 或单个 `TaskRepository` 混合承接任务、执行、计费、额度、事件、结果。
- 管理员侧与内部调度侧即便都可见 `AIExecution`，也应分成“管理员查询接口”和“内部写模型仓储”，避免把审计查询能力塞进流程写仓储。

## 三、用户侧仓储接口建议

### 3.1 `JobRepository`

定位：

- 面向任务聚合根的写仓储。
- 负责 `Job` 与 `JobItem` 在同一聚合边界内的创建和关键状态回写。

负责什么：

- 创建 `Job` 并展开初始 `JobItem` 集合。
- 按 `jobId + userId` 加载单个 `Job` 聚合根及必要子项。
- 在执行推进后回写 `Job` 聚合状态、聚合计数、摘要字段与关键时间字段。
- 受理用户取消请求并记录 `cancel_requested_at` / 聚合状态。

不负责什么：

- 不暴露 `AIExecution` 明细。
- 不提供管理员检索、计费底账或额度流水查询。
- 不直接负责对象存储下载、执行级事件检索或返还判定。

建议接口形态：

```text
createJobWithItems(draft: CreateJobDraft) -> Job
findByIdForUser(jobId, userId) -> Job?
findByIdForUpdate(jobId, userId) -> Job?
save(job: Job) -> void
markCancelRequested(jobId, userId, requestedAt) -> boolean
```

返回建议：

- 创建、单任务加载、状态回写前后的读取适合返回 `Job` 聚合根。
- 不建议用它做任务列表；列表应走轻量查询仓储。

### 3.2 `JobQueryRepository`

定位：

- 面向普通用户读模型的查询仓储。

负责什么：

- 按用户分页查询任务列表。
- 查询任务详情页所需的聚合视图。
- 分页查询某个 `Job` 下的 `JobItem` 列表。
- 查询用户可见的任务事件、结果摘要、产物摘要。

不负责什么：

- 不承接任务状态写入。
- 不回传内部实体对象给上层继续修改。
- 不返回 execution 级细节、计费底账、幂等键、内部错误堆栈。

建议接口形态：

```text
listJobs(userId, query) -> Page<JobListItem>
getJobDetail(userId, jobId) -> JobDetailView?
listJobItems(userId, jobId, query) -> Page<JobItemListItem>
listVisibleEvents(userId, jobId, cursor) -> EventStreamSlice<UserVisibleJobEvent>
listArtifactSummaries(userId, jobId) -> list<ArtifactSummary>
```

返回建议：

- 列表页返回 `JobListItem` 轻量投影。
- 详情页返回 `JobDetailView`，可包含 `Job` 摘要字段、聚合计数、第一页 `JobItem` 摘要、用户可见事件摘要与结果摘要。
- `JobItem` 分页应返回 `JobItemListItem` 或 `JobItemDetailView`，而不是直接返回 ORM 子实体集合。

### 3.3 `QuotaQueryRepository`

定位：

- 面向用户的额度只读仓储。

负责什么：

- 返回当前周期额度、已用、已返还、剩余与刷新时间等聚合视图。

不负责什么：

- 不暴露 `UsageLedger` 原始流水。
- 不承接扣减、返还、锁定等写操作。

建议接口形态：

```text
getCurrentQuotaView(userId, now) -> UserQuotaView?
```

## 四、管理员侧仓储接口建议

### 4.1 总体原则

- 管理员侧默认以查询仓储为主，不与内部调度写仓储复用同一接口。
- 管理员需要的是“事实明细与检索能力”，不是聚合写一致性入口。
- 普通用户可见模型与管理员可见模型应明确分离，避免错误复用用户 DTO。

### 4.2 `AdminJobQueryRepository`

负责什么：

- 按任务维度查看管理员视角的聚合详情。
- 支持按状态、任务类型、时间范围、用户、结果分布做检索。

不负责什么：

- 不直接下发执行动作或重试动作。
- 不负责写 `Job` 聚合状态。

建议接口形态：

```text
listJobs(query) -> Page<AdminJobListItem>
getJobDetail(jobId) -> AdminJobDetailView?
```

返回建议：

- 列表返回 `AdminJobListItem`。
- 详情可聚合 `Job`、`JobItem` 摘要、执行计数、返还计数、结果摘要，但仍以视图对象返回，而非可编辑聚合根。

### 4.3 `AdminExecutionQueryRepository`

负责什么：

- 查询某任务或某子项下的 `AIExecution` 列表。
- 查询单次执行的输入 / 输出摘要、模型、耗时、错误分类、幂等命中情况、关联计费状态。

不负责什么：

- 不创建或更新 `AIExecution`。
- 不承担幂等写入路径。

建议接口形态：

```text
listExecutions(query) -> Page<AIExecutionListItem>
getExecutionDetail(executionId) -> AIExecutionDetailView?
findLatestExecution(jobItemId) -> AIExecutionListItem?
```

返回建议：

- 管理员检索场景优先返回轻量列表项与详情视图。
- 除内部排障明确需要外，不建议把 ORM 实体直接暴露到控制器。

### 4.4 `AdminBillingQueryRepository`

负责什么：

- 查询 `ExecutionCharge`、`UsageLedger`、自动返还记录、额度消耗明细。

不负责什么：

- 不执行真实扣减 / 返还写入。

建议接口形态：

```text
listExecutionCharges(query) -> Page<ExecutionChargeView>
listUsageLedgers(query) -> Page<UsageLedgerView>
listRefunds(query) -> Page<RefundRecordView>
```

### 4.5 `AdminEventQueryRepository`

负责什么：

- 查询任务级、子项级、execution 级审计事件。

不负责什么：

- 不负责事件写入。

建议接口形态：

```text
listEvents(query) -> Page<JobEventView>
streamEvents(query, cursor) -> EventStreamSlice<JobEventView>
```

## 五、内部调度 / 计费仓储接口建议

### 5.1 `AIExecutionRepository`

定位：

- 内部流程写仓储，承接执行事实的创建、幂等判断、状态更新与子项级追踪。

负责什么：

- 基于 `userId + jobItemId + requestIdempotencyKey` 做幂等查找。
- 在真正准备调用 AI 时创建 `AIExecution`。
- 更新执行状态、摘要、时间字段。
- 查询某个 `job_item` 的最近执行，用于重试编排或状态回写。

不负责什么：

- 不负责普通用户查询。
- 不负责额度扣减聚合计算。
- 不负责管理员复杂组合检索。

建议接口形态：

```text
findByScopedIdempotency(userId, jobItemId, requestIdempotencyKey) -> AIExecution?
create(execution: NewAIExecution) -> AIExecution
findByIdForUpdate(executionId) -> AIExecution?
save(execution: AIExecution) -> void
findLatestByJobItem(jobItemId) -> AIExecution?
```

返回建议：

- 内部流程写入与状态机推进适合返回 `AIExecution` 实体。
- 给管理员或用户展示时，应经查询仓储转为视图模型。

### 5.2 `ExecutionChargeRepository`

负责什么：

- 按 `ai_execution` 创建一对一计费记录。
- 将状态从 `reserved` 推进到 `captured` 或 `refunded`。

不负责什么：

- 不承担额度窗口锁定。
- 不向用户提供原始计费记录。

建议接口形态：

```text
createReserved(charge: NewExecutionCharge) -> ExecutionCharge
findByExecutionIdForUpdate(executionId) -> ExecutionCharge?
save(charge: ExecutionCharge) -> void
```

### 5.3 `QuotaAccountRepository`

定位：

- 额度聚合根写仓储。

负责什么：

- 加载用户额度账户与当前有效 bucket。
- 在扣减 / 返还流程中锁定相关聚合数据。
- 更新 bucket 聚合值，如 `usedAmount`、`refundedAmount`。

不负责什么：

- 不保存每条流水明细。
- 不返回用户界面直接使用的额度展示模型。

建议接口形态：

```text
findAccountByUserId(userId) -> QuotaAccount?
findAccountByUserIdForUpdate(userId) -> QuotaAccount?
findActiveBucketForUpdate(quotaAccountId, now) -> QuotaBucket?
save(account: QuotaAccount) -> void
saveBucket(bucket: QuotaBucket) -> void
```

### 5.4 `UsageLedgerRepository`

负责什么：

- 写入 `reserve` / `capture` / `refund` 流水。
- 按 execution 或账户查询对账链路。

不负责什么：

- 不作为用户额度查询入口。
- 不代替 `QuotaAccount` 维护聚合余额。

建议接口形态：

```text
appendReserve(entry: NewUsageLedger) -> UsageLedger
appendCapture(entry: NewUsageLedger) -> UsageLedger
appendRefund(entry: NewUsageLedger) -> UsageLedger
listByExecutionId(executionId) -> list<UsageLedger>
```

### 5.5 `ArtifactRepository`

负责什么：

- 保存结果元数据与对象存储引用。
- 按 `job` / `job_item` / `ai_execution` 读取关联产物。

不负责什么：

- 不直接发放下载凭证。
- 不把文件物理删除与元数据删除耦合。

建议接口形态：

```text
create(artifact: NewArtifact) -> Artifact
listByJob(jobId) -> list<Artifact>
listByJobItem(jobItemId) -> list<Artifact>
listByExecution(executionId) -> list<Artifact>
markFileCleanupRequested(artifactId, requestedAt) -> boolean
```

### 5.6 `JobEventRepository`

负责什么：

- 统一追加任务级、子项级、execution 级事件。
- 为轮询与后续 WebSocket 提供共用事件源。

不负责什么：

- 不在仓储层判断“给谁看什么事件”，可见性裁剪应交给查询层或 service。

建议接口形态：

```text
append(event: NewJobEvent) -> JobEvent
listByJob(jobId, cursor, limit) -> EventStreamSlice<JobEvent>
```

## 六、查询对象与返回模型建议

### 6.1 适合返回聚合根的场景

- `JobRepository.findByIdForUser`、`JobRepository.findByIdForUpdate`
- `JobRepository.createJobWithItems`
- `AIExecutionRepository.findByScopedIdempotency`
- `AIExecutionRepository.findByIdForUpdate`
- `QuotaAccountRepository.findAccountByUserIdForUpdate`

原因：

- 这些场景通常需要承接状态迁移、并发控制、聚合计数回写或领域规则判断。

### 6.2 适合返回轻量投影 / 列表项的场景

- 任务列表、任务详情页摘要、子项分页
- 管理员执行列表、返还记录列表、事件列表
- 用户额度页
- 用户可见产物摘要列表

建议模型：

- `JobListItem`
- `JobDetailView`
- `JobItemListItem`
- `UserQuotaView`
- `ArtifactSummary`
- `AdminJobListItem`
- `AdminJobDetailView`
- `AIExecutionListItem`
- `AIExecutionDetailView`
- `ExecutionChargeView`
- `UsageLedgerView`
- `RefundRecordView`
- `JobEventView`

### 6.3 返回模型分层建议

- 领域实体：仅供内部写流程与一致性规则使用。
- 查询投影：面向列表、详情、筛选、导出。
- 事件切片：面向轮询 / 流式游标读取。

建议：

- 不让 service 把 ORM 实体直接序列化给 API。
- 不让管理员查询仓储返回可修改实体后再写回数据库。

## 七、事务边界与并发注意事项

### 7.1 建议在同一事务内完成的操作

#### 创建任务事务

- 创建 `Job`
- 批量插入初始 `JobItem`
- 初始化聚合计数字段
- 写入 `job.created` 事件（若第一版要求创建即产生日志）

说明：

- 目标是确保“任务存在”与“子项展开完成”同时成立，避免出现空任务或计数不一致。

#### 启动单个 `AIExecution` 事务

- 锁定 `Job` / 目标 `JobItem` 的必要行
- 校验任务未取消、子项仍可启动
- 幂等查询 `AIExecution`
- 锁定 `QuotaAccount` 与当前 `QuotaBucket`
- 写入 `ExecutionCharge.reserved` 与 `UsageLedger.reserve`
- 创建 `AIExecution`
- 回写 `JobItem` / `Job` 的运行中计数与状态
- 追加 `job_item.started` / `ai_execution.started` 事件

说明：

- “准备启动执行”是第一版最关键的一致性边界，应避免额度先扣未建 execution，或 execution 已建但聚合状态未推进。

#### 执行完成回写事务

- 更新 `AIExecution` 状态、摘要、结束时间
- 创建或更新 `Artifact` 元数据
- 将 `ExecutionCharge` 从 `reserved` 推进到 `captured` 或 `refunded`
- 写入 `UsageLedger.capture` 或 `UsageLedger.refund`
- 回写 `JobItem` 聚合状态
- 回写 `Job` 聚合计数、摘要与终态
- 追加完成类事件

说明：

- 这组写入应保持原子，避免用户界面、管理员审计、计费流水彼此看到不同步结果。

#### 用户取消受理事务

- 锁定 `Job`
- 更新 `cancel_requested_at` 与聚合状态
- 追加 `job.cancel_requested` 事件

说明：

- 已启动的 execution 可在事务外自然跑完，但“取消请求已受理”这件事应先稳定落库。

### 7.2 可拆分为最终一致的操作

- 对象存储大文件上传
- 下载凭证生成
- 文件异步清理
- WebSocket 推送
- 非关键统计缓存刷新

这些动作可以在主事务提交后异步执行，但要保证数据库中的元数据与事件先成立。

### 7.3 并发控制建议

- 对 `Job` 聚合计数和 `JobItem` 状态推进，优先使用行级锁或等价的悲观控制。
- 对 `QuotaAccount` / `QuotaBucket` 的扣减与返还，必须在同一事务内锁定热点账户，避免并发超扣。
- 对 `AIExecution` 幂等，优先依赖数据库 scoped partial unique 约束做最终兜底，应用层只做友好重试。
- 对 `ExecutionCharge` 一对一关系，依赖 `ai_execution_id` 唯一约束防止重复计费记录。
- 对软删除数据，默认查询应隐式过滤 `deleted_at`；管理员 / 审计查询可显式开启“包含已删除 / 已归档”。

### 7.4 不建议的实现方式

- 在仓储外层拼多次无事务写入再“希望最后一致”。
- 让用户查询仓储顺手改状态。
- 在 `JobQueryRepository` 中透传 `AIExecution` 实体。
- 只靠内存锁保证幂等与额度一致性。

## 八、第一版实现优先级

### 8.1 第一优先级

- `JobRepository`
- `JobQueryRepository`
- `AIExecutionRepository`
- `ExecutionChargeRepository`
- `QuotaAccountRepository`
- `UsageLedgerRepository`
- `JobEventRepository`

目标：

- 跑通“创建任务 -> 展开子项 -> 确认执行 -> 按子项检查额度 -> 创建执行 -> 回写聚合 -> 自动返还 / 完成 -> 用户可轮询查询”的最小闭环。

### 8.2 第二优先级

- `ArtifactRepository`
- `QuotaQueryRepository`
- `AdminExecutionQueryRepository`
- `AdminBillingQueryRepository`

目标：

- 完善结果查看、额度展示、管理员执行排障与返还核对。

### 8.3 第三优先级

- `AdminJobQueryRepository`
- `AdminEventQueryRepository`

目标：

- 完善管理员高级检索、审计事件过滤与运营侧排查能力。

## 九、开放问题

1. `JobRepository` 是否需要显式区分 `createDraft` 与 `confirmStart` 两类写接口，还是由 service 编排状态推进后统一 `save`，仍待落地时决定。
2. `JobDetailView` 是否在第一版就内联第一页 `JobItem` 摘要，还是由独立子项分页接口拉取，仍需结合前端查询模式权衡。
3. `AdminExecutionQueryRepository` 是否需要暴露“按幂等键检索”能力，当前草案倾向仅供内部排障使用。
4. `QuotaAccountRepository` 是否在第一版就承载 bucket 选择逻辑，还是由 service 先计算当前周期再传入 repository，仍待实现习惯决定。
5. `ArtifactRepository` 是否需要区分“展示产物摘要查询”和“内部元数据写仓储”两层接口，当前先保持一层草案。
6. `JobEventRepository` 的用户可见性是否需要数据库层辅助字段，当前草案仍建议在查询层裁剪。
7. 是否在仓储层统一引入并发版本字段（如 `version`）以配合乐观锁，当前草案未冻结，先以数据库行锁与唯一约束为主。
