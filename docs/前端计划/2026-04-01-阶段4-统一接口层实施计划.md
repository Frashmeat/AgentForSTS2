# 阶段 4：统一接口层实施计划

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把当前前端零散的 HTTP 请求入口收敛到统一 API 层，先统一 `config` 和 `approvals` 两个已经在多个页面复用的领域，减少组件直接 `fetch` 和重复错误处理。

**Architecture:** 先补一个轻量 `shared/api/http.ts` 作为 JSON 请求入口，再把 `config` 和 `approvals` 迁到 `shared/api/*`。旧的 `lib/approvals.ts` 保留为兼容薄层或直接转发，页面逐步迁到共享 API 层，避免一次性大迁移。

**Tech Stack:** React 18、TypeScript、浏览器 `fetch`、Node `node:test`

## 风险

- 如果兼容层处理不好，会同时出现 `shared/api/approvals.ts` 和 `lib/approvals.ts` 双源维护
- 如果把组件字段转换也一起做，会把“接口层收敛”演变成“大重构”
- `SettingsPanel.tsx` 当前本地表单状态较多，迁移时容易顺手扩大范围

## 完成标准

- 前端有统一 JSON HTTP client
- `config` 和 `approvals` 已进入统一 API 层
- `SettingsPanel.tsx` 不再直接 `fetch`
- 后续扩展 `workflow/platform/admin` 时可沿用同一套接口层模式
