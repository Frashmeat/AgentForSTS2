# 阶段 3：异常恢复与可恢复性实施计划

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 为前端补齐“刷新后可恢复、失败后可重试、审批态可重新同步”的基础能力，先把批量模式做稳，再评估单资产中间态恢复。

**Architecture:** 采用“本地 UI 快照恢复 + 后端真实状态补拉”的两段式方案。前端只持久化最小可恢复快照，不伪造 WebSocket 会话续传；凡是审批类真实状态，都在恢复后通过 HTTP 审批接口重新拉取并覆盖本地快照。

**Tech Stack:** React 18、TypeScript、Node `node:test`、浏览器 `localStorage`、现有 `frontend/src/lib/approvals.ts` 与 batch reducer 体系

## Chunk 1：范围与恢复原则

### 目标

先明确本轮恢复能力的边界，避免把“可恢复 UI”误做成“可恢复后端执行”。

### 本轮范围

- 新增 batch runtime 快照持久化与恢复
- 新增审批列表/详情查询能力
- 新增审批态恢复后的后端刷新
- 新增 item 重试前的状态清理规则
- 新增单资产草稿/运行态最小恢复

### 非目标

- [ ] 不实现 WebSocket/session resume
- [ ] 不伪造“后台仍在继续执行”的假象
- [ ] 不做全局 storage 框架重构
- [ ] 不实现单资产后台会话续传

### 恢复原则

- [ ] 本地快照只表达“上次 UI 看到什么”
- [ ] 真正需要信任的状态，以后端接口返回为准
- [ ] 页面恢复后，用户可以重新连接执行，但不能假设旧 socket 仍然有效
- [ ] 持久化数据必须可裁剪，不能把大体积日志和图片无限写入本地

## Chunk 4：审批态恢复与后端刷新

### 目标

当 batch item 在本地快照中处于 `approval_pending` 时，恢复页面后自动补拉真实审批状态，而不是盲信旧快照。

### 行为约定

- [ ] 快照恢复后先渲染本地状态，避免白屏
- [ ] 然后按 `approvalRequests[].action_id` 查询后端审批详情
- [ ] 若后端返回已执行/已失败/不存在，则覆盖本地审批状态
- [ ] 若该 item 全部审批请求已 `succeeded`，允许 reducer 收敛到可继续状态

### 验收标准

- [ ] 刷新页面后，审批区不会永久停留在旧的 `pending`
- [ ] 后端状态与本地快照冲突时，以后端为准

## 风险

- 如果快照字段收得过大，localStorage 容量和性能都会变差
- 如果恢复逻辑直接写进页面 effect，后续阶段 4 会继续发散
- 如果把“审批恢复完成”直接等同于“后端继续执行”，会制造错误的 UI 预期

## 完成标准

- batch 模式具备最小快照恢复
- 审批态刷新后可重新同步真实状态
- item 重试能清理旧残留
- 页面刷新后不会回到明显错误的阶段
