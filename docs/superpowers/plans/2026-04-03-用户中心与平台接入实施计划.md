# 用户中心与平台接入 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在现有工作站前端壳内补齐账号认证、用户中心和服务器 AI 服务入口分流，让用户在一个入口内完成本机链路与平台链路的切换和查询。

**Architecture:** 前端继续保留单应用入口，但引入轻量路由、认证状态层和用户中心页面；后端新增认证与当前用户接口，并在平台能力上补“当前用户视角”查询层。前端不再直传裸 `user_id`，但后端继续保留并完善 `user_id` 作为认证、任务归属、配额与返还的正式内部主键。BYOK 仍走本地工作站链路，不进入服务器历史；平台模式继续围绕 `job -> job_item` 聚合结果对普通用户可见。

**Tech Stack:** FastAPI、现有平台服务层、React 18、TypeScript、浏览器 `fetch`、Node `node:test`、pytest

说明：

- 因此，`/api/me/*` 不是普通增强项，而是本次方案成立的前置条件。

## 风险

- 若继续让前端直接依赖裸 `user_id` 查询平台接口，会把新认证链做成半成品。
- 若在改造中误把 `user_id` 当作应删除字段处理，会破坏平台任务、配额和返还的内部归属模型。
- 若没有显式任务约束“平台历史与本地历史分离”，前端实现时很容易直接复用现有恢复快照，导致边界污染。
- 若返还信息不进入普通用户聚合视图，用户中心将无法解释“失败后次数去哪了”。
- 若本机配置检测直接复用完整配置对象，容易扩大敏感信息暴露面。
- 若在多个工作流页面各自复制“本机/服务器”判断逻辑，后续极易分叉；需要公共弹窗和公共 flow helper。

## 残留风险与后续规划

以下风险不建议在本次硬解，转入后续规划：

- 真实邮件发送与投递稳定性
- 更完整的账户安全能力，如频控、锁定、异常登录告警
- 本机执行历史的本地化展示方案
- 独立 Web 前端站点与 `main_web.py` 是否继续托管前端
- 套餐、计费规则后台配置、商业化权益模型
- 更细粒度的取消/超时/第三方限流返还判定矩阵

## 验收标准

- 右上角账号入口能正确区分未登录与已登录状态。
- 登录 / 注册 / 邮箱验证 / 找回密码主链可达。
- 用户中心单页可展示当前用户信息、统一服务器次数池、平台任务列表。
- 任务详情可展示子项与返还摘要。
- 失败返还可在普通用户界面中看到原始扣减、返还次数和净消耗。
- 生成入口可根据本机配置与登录态正确分流到 BYOK 或服务器模式。
- 选择 BYOK 不进入服务器历史；选择服务器模式才进入平台记录。
- 新增的认证页、用户中心页与平台历史查询页不再依赖前端直传裸 `user_id`。
- 用户中心任务列表不混入工作站本地恢复快照、localStorage 记录或 Agent 本地日志。
