# Web 后端独立部署实施计划

> 目标：在不拆第二个前端应用、不更换技术栈的前提下，让 `web-backend` 能作为独立部署单元运行，方便维护、发布与排障，并将其作为后续用户中心与平台接入改造的先行步骤。

## 背景

当前后端已具备 `full / workstation / web` 三种运行角色雏形：

- `main.py` 运行完整工作站入口
- `main_web.py` 运行 `web` 角色入口
- `app_factory.py` 已按角色选择挂载不同路由

但现状仍存在以下问题：

- `web` 角色虽可单独启动，但缺少完整的独立部署约束与文档
- 配置、CORS、容器装配与路由边界尚未收口到“便于独立维护”的程度
- 之前文档把“独立 Web 部署”放入后续规划，已不符合当前优先级

因此，本计划将“web 后端独立部署”提升为本轮先行实施项。

## 实施原则

- 不拆第二个前端应用，前端仍保持单入口
- 不更换技术栈，继续使用现有 FastAPI / Python
- 本轮优先实现 `web-backend` 独立部署能力，而不是独立 Web 前端站点
- `workstation-backend` 与 `web-backend` 共享仓库，但运行入口、路由与配置边界要更清晰
- 独立部署优先级高于用户中心页面铺设，作为后续平台接入的运行基础

## 目标

1. `main_web.py` 能稳定作为独立服务运行
2. `web` 角色只暴露平台与认证相关能力，不误挂工作站本地能力
3. `web-backend` 具备独立配置、独立端口和独立部署说明
4. 后续 `/api/auth/*`、`/api/me/*`、平台用户中心查询链路默认落在 `web-backend`

## 非目标

- 不在本轮拆独立 Web 前端站点
- 不在本轮改为 Java / Spring 或 Rust
- 不在本轮拆分数据库
- 不在本轮拆成多个微服务

## 风险

- 若只是保留 `main_web.py` 入口，但不清理路由和配置边界，部署虽然分开，维护仍会互相牵连
- 若前端继续混用工作站接口与平台接口而不区分后端承接者，后续排障会持续混乱
- 若把“独立后端部署”和“独立前端站点”混为一谈，会导致范围膨胀
- 若本轮先铺用户中心页面、不先收口运行边界，后续会反复返工接口归属

## 验收标准

- `main_web.py` 可作为独立服务启动
- `web` 角色只挂载 web/平台/认证相关路由
- `web-backend` 具备独立端口、独立配置说明和独立启动命令
- 文档明确：本轮优先先做 web 后端独立部署，再继续用户中心与平台接入主链
- 后续新增的 `/api/auth/*`、`/api/me/*` 默认归属 `web-backend`

## 与用户中心计划的关系

- 本计划是“运行边界先行计划”
- 用户中心与平台接入计划继续有效
- 但实现顺序调整为：
  1. 先完成 `web-backend` 独立部署基础
  2. 再继续认证、`/api/me/*`、用户中心、平台历史与分流逻辑

## 执行门槛

- 若 Phase 0 未完成，不开始用户中心主链实现
- 若 `main_web.py` 仍未具备独立运行、独立配置、独立路由边界，本轮不得宣称“web 侧接入基础已完成”
