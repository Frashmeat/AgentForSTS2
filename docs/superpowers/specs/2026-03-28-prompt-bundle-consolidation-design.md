# Prompt Bundle Consolidation Design

## 背景与问题

当前仓库中的运行时 Prompt 资源分散在多个模块目录，以单个 `.txt` 文件为主要载体。它们虽然名义上是文本资源，实际上已经构成规划、代码生成、图像提示适配、分析、审批和全局 LLM 拼装链路的一部分。

这种布局带来几个持续性问题：

- 同一模块内的 Prompt 片段分散，维护时需要跨多个目录和文件来理解完整语义。
- 现有 `PromptLoader` 只支持“按文件名读取整文件”，不能表达“一个模块下多个具名片段”的稳定组织方式。
- 资源命名缺少统一收口点，容易出现模块内约定漂移、重复表达和新增资源落点不一致。
- 迁移和审查成本高，尤其是 `image` 与 `codegen` 模块，资源数量大、调用面广，最容易在改造时引入行为偏差。
- 旧 `.txt` 文件一旦与新资源并存过久，会形成双真源问题，增加维护风险。

本设计的目标是把主要逻辑相关的 Prompt 资源按模块收敛到统一目录 `backend/app/shared/resources/prompts/`，并以 Markdown 总文件作为唯一真实来源。

## 目标

- 将运行时 Prompt 资源统一收口到 `backend/app/shared/resources/prompts/`。
- 采用“每个模块一个 Markdown 总文件”的组织方式，而不是全仓库单一总文件。
- 覆盖以下六组模块：`codegen`、`image`、`analyzer`、`planning`、`approval`、`llm`。
- 让每个总文件中的资源片段通过 `## 键名` 作为唯一键被读取。
- 扩展 `PromptLoader`，使调用方能够从 Markdown 总文件中按键加载片段，并保留现有模板变量替换能力。
- 在迁移窗口期内支持新旧资源并存，但统一目录中的 Markdown 文件必须是唯一真实来源。

## 非目标

- 不在本设计中定义逐个调用点的改造顺序、责任分配或时间安排。
- 不改变各业务 Prompt 的语义内容、变量占位规则或调用时机。
- 不引入新的模板引擎，也不把控制流逻辑迁入资源文件。
- 不把所有 Prompt 合并为单一超大 Markdown 文件。
- 不把非运行时文档、备忘或第三方文本资产纳入这次收口范围。

## 目录与命名方案

统一目录定义为：

```text
backend/app/shared/resources/prompts/
```

该目录下以模块为边界维护六个 Markdown 总文件：

```text
backend/app/shared/resources/prompts/codegen.md
backend/app/shared/resources/prompts/image.md
backend/app/shared/resources/prompts/analyzer.md
backend/app/shared/resources/prompts/planning.md
backend/app/shared/resources/prompts/approval.md
backend/app/shared/resources/prompts/llm.md
```

命名原则如下：

- 文件名使用模块名，保持与领域语义一致，不再暴露历史 `.txt` 文件组织细节。
- 每个 Markdown 文件只承载本模块运行时 Prompt 资源，不混入跨模块内容。
- 原先的文件名、partial 名和逻辑片段名迁移后转为 Markdown 内部键名，而不是继续映射为独立文件。
- 对调用方而言，资源定位从“目录 + 文件名”逐步收敛为“模块文件 + 片段键名”。

这种结构保留了模块边界，同时把原本分散的片段组织成可通读、可审查、可校验的单源文档。

## Markdown 结构规则

每个模块总文件使用 Markdown，并遵循以下规则：

1. 一级标题用于模块说明，可选，但不参与运行时索引。
2. 运行时资源片段必须使用二级标题 `## 键名` 定义。
3. `键名` 在单个文件内必须唯一，大小写和字符规则需要固定并可校验。
4. 一个 `## 键名` 标题后直到下一个同级 `##` 标题之前的全部内容，视为该键对应的原始文本值。
5. 片段内容允许包含多行、代码块、列表和模板变量，但不允许为空片段。
6. 不允许通过更深层级标题继续拆分运行时键；`###` 及以下标题只能作为片段正文内容存在，不能参与索引。

推荐的逻辑键命名采用稳定的短横线或下划线风格，避免依赖历史路径。例如：

```md
# codegen prompts

## asset_prompt
...

## build_prompt
...

## lookup_title
...
```

这里的关键点不是 Markdown 可读性本身，而是把“标题即索引键”作为运行时契约。只要契约稳定，文档形式与加载行为就可以保持一致。

为避免“看起来像规则、但无法测试”的模糊空间，解析契约进一步收紧如下：

- 键名字符集限定为小写字母、数字、下划线，正则形式为 `^[a-z0-9_]+$`，不做大小写归一；出现大写字母、短横线或空格应直接判定为非法键。
- 标题识别只接受行首开始的精确 `## ` 前缀，`##` 后必须且只能跟一个空格，再接键名；标题末尾允许 Markdown 常规尾随空白，但在键名解析前需要去除首尾空白。
- fenced code block 内出现的 `##` 必须视为正文，不参与键提取；解析器不能仅靠正则全局扫描，而需要识别代码块进入和退出状态。
- 正文内容采用“从标题下一行开始，到下一个有效 `## 键名` 标题前一行结束”的截取方式；正文内部出现的 `###`、列表项或普通文本 `##` 只要不满足有效标题条件，都必须保留为正文。
- 片段正文保留原始换行和中间空白，不做折叠、trim 或 Markdown 渲染；唯一允许的归一化是去除紧邻文件结尾的编码差异与读取层 BOM 影响。
- 空片段定义为正文在保留换行语义后不包含任何非空白字符；仅有空行、空格或制表符的片段必须报错，而不是按空字符串返回。

这些规则的目的不是增加格式负担，而是让“同一份 Markdown 在不同时间、不同机器、不同调用链下都解析成同一组键和值”成为可验证契约。

## 加载器设计

当前 `PromptLoader` 的模型是“解析根目录 + 读取目标文件全文 + 可选变量替换”。迁移后，加载器需要支持两类资源寻址：

- 旧模式：按文件路径读取整文件，用于兼容迁移窗口期尚未切换的调用方。
- 新模式：按模块 Markdown 文件与片段键读取局部内容，用于新真源。

设计上建议扩展而不是替换 `PromptLoader` 的职责：

- 保留现有 `load()` / `render()` 风格，避免上层调用方式完全重写。
- 新增对“bundle + key”形式的解析能力，使调用方可显式声明模块和片段键。
- 在内部把 Markdown 文件解析为“键 -> 内容”的索引映射，并以缓存方式复用。
- 兼容期内由加载器持有一份中心化旧引用映射表，把历史“旧文件名/旧路径”统一解析到 `bundle + key`，而不是要求人工双写双维护。

加载器应承担以下责任：

- 解析模块 Markdown，提取所有 `## 键名` 片段。
- 在加载阶段执行结构校验，至少覆盖重复键、空片段、非法标题层级用法和缺失键。
- 对解析结果做内存缓存，避免每次调用重复扫描 Markdown 文件。
- 在文件内容变化后支持缓存失效或重新装载，避免长生命周期进程持续读取旧内容。
- 继续复用现有模板变量替换能力，对片段内容执行 `{{ name }}` 渲染。
- 接收旧调用方传入的历史模板名或历史路径，并通过中心化映射表将其重定向到统一目录中的 Markdown 真源。

加载器不应承担以下责任：

- 不负责业务分支选择，不在资源文件内部做条件判断。
- 不负责跨模块回退决策，回退策略仍由业务层控制。
- 不负责修改片段内容或自动修复 Markdown 结构。

从接口语义上看，迁移后的 `PromptLoader` 更像“受校验的 Prompt bundle reader”，而不再只是简单文件读取器。

## 兼容策略

统一目录是最终唯一真实来源，但迁移不应要求所有调用点一次性完成切换。兼容期需要遵循以下原则：

- 新增六个 Markdown 总文件后，先让加载器具备从 Markdown 读取片段的能力。
- 在调用点尚未切换前，旧 `.txt` 文件可以继续存在，但只能视为过渡兼容入口，不能再作为新增修改的落点。
- 一旦某模块切到 Markdown 真源，该模块后续 Prompt 变更只允许修改对应 Markdown 总文件。
- 旧路径兼容应尽量收敛在加载器或适配层，不把“双读逻辑”扩散到每个业务模块。
- 当全部调用点完成迁移后，旧 `.txt` 文件整体退役并删除，结束双轨状态。

兼容策略的核心不是长期保留双套资源，而是用最小范围的兼容壳降低迁移阶段风险。

这里的关键约束是：兼容期并不允许人工同步维护旧 `.txt` 与新 Markdown。旧路径之所以还能工作，只能依赖“旧引用 -> bundle + key”的中心化桥接，把读取请求导向 Markdown 真源。

这份桥接映射应当集中维护在加载层或其紧邻适配层，表现为稳定、可审计的旧引用索引，例如：

- `backend/app/modules/planning/resources/prompts/planner_prompt.txt` -> `planning.planner_prompt`
- `backend/approval/resources/prompts/action_prompt.txt` -> `approval.action_prompt`
- `backend/app/modules/analyzer/resources/prompts/log_analyzer_system.txt` -> `analyzer.log_analyzer_system`
- `backend/app/modules/codegen/resources/prompts/partials/api_lookup/title.txt` -> `codegen.lookup_title`
- `backend/app/modules/image/resources/prompts/guide_flux2_rules.txt` -> `image.guide_flux2_rules`

映射的职责边界也需要明确：

- 历史调用方继续传入旧文件名、旧相对路径甚至旧根目录时，兼容桥只做一次查表并跳转到 Markdown 真源。
- 兼容桥不再回读旧 `.txt` 内容，也不允许“找不到映射时退回旧文件磁盘读取”，否则会重新引入双真源。
- 每个旧路径必须映射到且仅映射到一个 `bundle + key`；一对多或多对一都应视为设计错误，必须在桥接层显式处理。

`llm` 模块需要额外说明。当前 `backend/llm/resources/global_prompt_header.txt` 由 `backend/llm/prompt_builder.py` 直接读取，绕过了 `PromptLoader`。迁移后它不能继续作为旁路存在，而应分两步收口：

- 在兼容期先把 `global_prompt_header.txt` 纳入中心化映射，使其读取结果同样来自 `llm.md` 中对应键，而不是物理 `.txt` 文件内容。
- 在调用点切换完成后，`backend/llm/prompt_builder.py` 也应改为通过统一加载入口读取 `llm` bundle，结束对磁盘 `.txt` 的直读。

只有当 `llm` 这条旁路也被收口后，“Markdown 是唯一真源”这一要求才算在全仓层面真正成立。

## 迁移步骤

迁移的逻辑顺序应固定为四个阶段：

1. 在统一目录新增六个模块 Markdown 总文件，形成新真源。
2. 扩展 `PromptLoader`，支持按 Markdown 键读取、校验和缓存。
3. 按模块逐步把调用点切到新加载方式，直到各模块不再依赖旧 `.txt`。
4. 删除旧分散 `.txt` 文件，完成真源收口。

之所以采用这个顺序，是因为真源、加载能力和调用切换是三层不同责任。先建立真源和读取能力，才能让调用方迁移时保持可控；先删旧文件则会把迁移风险前置到所有调用链上。

第 4 阶段不是“感觉差不多就删除”，而是只有在满足明确退役门槛后才允许发生。允许删除旧 `.txt` 的硬条件如下：

- 仓库内已不存在任何运行时代码对旧 prompt 路径或旧文件名的直接引用，包括 `PromptLoader(root=旧目录).load(...)`、`Path(...).read_text()` 和等价直读方式。
- 中心化兼容映射表中，与旧 `.txt` 相关的映射项已经全部下线，不再承担旧引用转发职责。
- `llm` 的 `global_prompt_header.txt` 直读路径已迁移到统一加载入口，仓库中不存在绕过加载器读取该资源的残留路径。
- `image` 与 `codegen` 两个高风险模块在迁移后已完成关键 prompt 快照或等价文本对比，确认主路径输出未发生非预期漂移。
- 以统一目录为输入源的回归验证已经通过，且零旧路径引用扫描结果为真。

这些条件的作用是把“可删除”定义为一个可验证状态，而不是依赖维护者主观判断。

## 测试策略

本设计要求测试重点落在“结构正确性”和“运行时语义不漂移”两个层面。

首先，针对加载器增加专门校验：

- 能正确从模块 Markdown 中提取多个 `## 键名` 片段。
- 重复键会明确报错，而不是静默覆盖。
- 空片段会被识别并阻断。
- 缺失键会抛出稳定异常。
- 非法键名、标题空白不合法、代码块内伪标题等解析边界都能被稳定处理。
- 正文首尾空白和换行保真规则符合契约，不因解析过程被意外裁剪。
- 模板变量替换行为与现有 `.txt` 方案保持一致。
- 缓存命中与缓存失效行为符合预期。
- 旧路径经由中心化兼容映射读取时，结果与直接按 `bundle + key` 读取一致。

其次，针对业务模块保持现有调用结果稳定：

- `planning`、`approval`、`analyzer` 的关键 Prompt 读取测试应确认切换后正文未丢段。
- `codegen` 的主 Prompt 与 API lookup 片段装配测试需要覆盖最常用路径。
- `image` 需要覆盖适配主模板、guide 系列、fallback 系列和透明背景规则的组合读取。
- `llm` 需要覆盖全局提示头在迁移后的读取与拼装。

最后，针对迁移窗口期保留回归验证：

- 同一模块切换前后，关键调用点产出的 Prompt 文本应在语义上等价。
- 删除旧 `.txt` 前，应有一轮以新真源为基础的全链回归，确认不存在残留旧路径读取。

## 风险与回滚策略

这次整合的主要风险集中在结构完整性与高频调用模块。

结构性风险包括：

- Markdown 结构损坏导致片段边界解析错误。
- 同一文件内重复键导致覆盖或读取不确定。
- 空片段进入运行时，造成 Prompt 缺段或退化。
- 人工编辑 Markdown 时误把正文标题写成新的运行时键。

迁移性风险包括：

- `image` 模块资源数量最多，且存在 guide、fallback、规则片段混合，最容易出现键映射遗漏。
- `codegen` 模块调用面最广，且包含主 Prompt 与 API lookup 片段，最容易在切换时引发行为回归。
- 兼容桥设计不完整会让旧路径回退到磁盘 `.txt` 直读，重新引入隐性双真源。
- `llm` 若继续保留绕过 `PromptLoader` 的直读旁路，会使统一真源在全局层面失效。
- 兼容期过长会形成桥接层长期滞留，增加旧引用残留与删除门槛失真的风险。

为降低风险，设计上必须具备两个安全阀：

- 加载器结构校验：在读取阶段尽早失败，而不是把损坏的片段静默送入运行时。
- 缓存策略可控：既要避免重复解析，也要避免因为缓存不失效而长期使用错误内容。

回滚策略应保持简单直接：

- 在模块级切换期间，若发现 Markdown 读取导致行为异常，应优先回退该模块调用点到兼容层，而不是临时修改运行时逻辑。
- 在旧 `.txt` 仍未删除前，回滚可以通过恢复调用路径完成，而不需要反向重建资源内容。
- 一旦旧 `.txt` 已退役，回滚应以恢复 Markdown 真源内容和加载器行为为主，不再重新引入分散文件布局。

换句话说，回滚的对象应是“切换行为”或“真源内容”，而不是重新放大历史结构复杂度。

## 结论

本设计采用“统一目录 + 每模块一个 Markdown 总文件 + `## 键名` 片段索引”的方式，把六组核心 Prompt 资源收敛到共享目录，并以扩展后的 `PromptLoader` 承接运行时读取、校验和缓存。

这个方案的价值不在于把 `.txt` 改成 `.md`，而在于建立统一真源、稳定键契约和可校验加载边界。只要这三点成立，后续逐模块迁移与旧资源退役就能以较低风险推进。
