# 用户中心与平台接入设计

## 背景

当前前端已接入工作站主链，包含单资产生成、批量生成、Mod 修改、日志分析等本地工作流；平台侧 `platform/admin` API 已有基础后端实现与前端 API 封装，但尚未形成面向用户的入口、账号体系和用户中心。

本设计目标是在不拆分独立 Web 站点的前提下，把“账号登录、平台次数、平台历史记录、服务器 AI 服务入口”轻量接入现有前端壳，保持用户仍在同一个应用入口内完成全部操作。

本设计以 [2026-03-27-后端设计草案](../../03-方案/后端专题/2026-03-27-后端设计草案.md) 为主，优先遵守其中已确认边界。

## 当前实现状态（2026-04-04）

当前实现已落地并合并到主工作区，关键事实如下：

- 同一前端壳内已补 `/auth/*`、`/me`、`/me/jobs/:jobId`
- 用户中心与任务详情默认消费 `/api/me/*`
- 生成入口已接统一“本机执行 / 服务器模式”分流弹窗
- 服务器模式当前实际流程为：
  - 先创建当前用户视角平台任务草稿
  - 再确认是否开始
  - 确认后跳转用户中心任务详情页
- BYOK / 本机执行仍不进入平台历史

本设计文档以下内容以“当前落地实现”为准；对未来可再收紧的策略会显式标注。

## 已确认边界

- 产品主入口仍是当前工作站前端，不做独立 Web 前端站点。
- 右上角增加账号入口；已登录显示头像，未登录显示“登录 / 注册”。
- 头像点击进入独立“用户中心”页面。
- 新增一套账号密码登录、公开注册、邮箱验证、找回密码流程。
- 用户可以继续使用本机自配 AI；只有选择服务器 AI 服务时才要求登录。
- 本机配置只参与“生成前分流”，不进入服务器历史记录。
- 用户中心只展示平台模式数据，不展示 BYOK / 本机执行历史。
- 平台历史默认展示任务列表，点击后再查看子项详情。
- 用户中心中的次数是一个统一服务器次数池，不按模块拆余额；不同模块只体现单次扣减规则不同。
- 失败返还必须在用户中心历史与详情中清楚展示。

## 非目标

- 不在本轮拆分独立的 Web 前端站点或第二个前端应用。
- 不在本轮把 BYOK 模式任务同步到服务器数据库。
- 不在本轮暴露管理员视角的 `ai_execution` 明细给普通用户。
- 不在本轮引入手机验证码或第三方登录。
- 不在本轮扩展套餐、支付、月费订阅等商业化模型。

补充说明：

- 本轮允许并优先推进 `web-backend` 独立部署能力，用于方便维护与发布。
- 这里的“非目标”仅指不拆第二个 Web 前端站点，不等于后端不能独立部署。

## 与后端草案的对齐原则

按 2026-03-27 草案：

- BYOK 模式本地执行、本地存储，不经过服务器，服务器不保存其任务记录与结果。
- Web 控制台只查看平台模式数据。
- 平台任务遵循 `job -> job_item -> ai_execution` 三层模型。
- 普通用户只看 `job` 与 `job_item` 聚合结果。
- 只有实际调用 AI 的操作才计费。
- 批量任务按“边执行边检查配额”处理，不预占整批额度。
- 系统异常自动返还次数。

因此，本设计中的“用户中心”是平台控制台用户中心，不是工作站全局活动中心。

## 用户入口与导航

### 顶部入口

- 未登录：右上角显示“登录 / 注册”按钮。
- 已登录：右上角显示用户头像、昵称缩写或用户名，点击进入用户中心。

### 页面组织

在现有单页应用内引入轻量路由，至少包含以下页面：

- `/auth/login`
- `/auth/register`
- `/auth/verify-email`
- `/auth/forgot-password`
- `/auth/reset-password`
- `/me`
- `/me/jobs/:jobId`

原有工作站页面继续保留在同一前端壳内，不拆成第二个应用。

## 认证主链

### 认证方式

- 注册方式：公开注册。
- 登录方式：账号密码登录，支持用户名或邮箱登录。
- 激活条件：注册后需要完成邮箱验证。
- 邮件方案：第一阶段按“伪邮件 / 控制台验证码”落地流程，但接口与状态机按未来真实邮件发送设计。

### 访问控制

- 未登录用户：
  - 可以使用本机配置链路。
  - 不能使用服务器 AI 服务。
- 已登录用户：
  - 可以查看用户中心。
  - 可以发起平台模式任务并查看历史。

当前实现补充：

- 用户中心已展示 `email_verified` 状态
- 服务器模式前置分流当前只强制检查登录态，尚未在前端额外阻断“邮箱未验证”的开始动作
- 若后续需要收紧，可在前端分流层和后端命令接口上补双重校验

### 会话策略

前端不以手动拼装裸 `user_id` 作为主认证入口，而是通过服务端登录态获取当前用户信息。前端应基于“当前用户接口”判断登录态、邮箱验证状态和用户中心展示数据。

后端内部仍保留并完善 `user_id` 作为正式用户主键，用于：

- 用户表主键与认证归属
- 登录态与会话归属
- `job.user_id` 的任务归属
- 配额、返还、历史查询的归属依据

也就是说，本轮不是移除 `user_id`，而是把它从“前端直传参数”收敛为“后端内部正式身份标识”。

## 生成前分流流程

### 核心原则

点击“生成”后，前端先决定“本次走本机还是走服务器”，而不是立即创建平台任务或立即扣减次数。

### 流程

1. 用户点击生成。
2. 前端检测当前设备是否存在可用本机 AI 配置。
3. 若检测到本机配置：
   - 弹出二选一弹窗：
     - 使用本机配置执行
     - 使用服务器服务执行
4. 若未检测到本机配置：
   - 弹出提示，仅提供“使用服务器服务执行”路径。
5. 选择本机配置执行：
   - 继续现有工作站链路。
   - 不创建平台 `job`。
   - 不写入用户中心历史。
   - 不消耗服务器次数。
6. 选择服务器服务执行：
   - 未登录：跳转登录页，登录成功后回到原页面并恢复本次服务器执行动作。
   - 已登录：进入平台模式创建任务。

### 当前实现补充

当前前端已将平台模式具体落地为：

1. 先调用 `POST /api/me/jobs` 创建草稿任务
2. 再由用户确认是否开始
3. 确认后调用 `POST /api/me/jobs/{job_id}/start`
4. 跳转 `/me/jobs/:jobId`

### 平台模式下的前端文案约束

前端不应把提示写成“点击生成立即扣费”。应使用以下语义：

- 服务器将在实际发起 AI 调用前检查并扣减次数。
- 系统异常可能自动返还。
- 返还结果会出现在用户中心历史与详情中。

## 用户中心信息架构

用户中心采用单页总览，不使用左侧导航。

### 1. 用户信息区

展示：

- 头像
- 用户名
- 邮箱
- 邮箱验证状态
- 账号创建时间
- 退出登录入口

这里最多补一条轻提示：

- 当前设备可检测到本机 AI 配置。
- 本机执行不会进入平台历史，也不会消耗服务器次数。

### 2. 次数总览区

展示统一服务器次数池，而不是模块拆分余额。建议展示：

- 当前周期总额度
- 已使用次数
- 剩余次数
- 刷新时间

同时补一段计费规则说明，例如：

- 单资产生成：扣 X 次
- 批量生成：每个实际 AI 子调用按规则扣减
- 日志分析：扣 X 次
- Mod 修改：扣 X 次

### 3. 平台历史记录区

默认按 `job` 展示任务列表，不直接展开 `job_item`。每行建议展示：

- 模块类型
- 任务标题或输入摘要
- 当前状态
- 创建时间
- 结果摘要
- 净消耗次数
- 返还标签或异常标签

只展示平台模式任务，不展示本机执行历史。

### 4. 说明区

集中说明：

- 用户中心只展示平台模式数据。
- 本机配置执行不会进入平台历史。
- 系统异常会按平台规则自动返还。
- 用户取消后，已发出的 AI 调用可能仍产生消耗，最终以任务详情为准。

## 平台历史与返还展示

### 任务列表

任务列表默认展示 `job`。状态需覆盖草案中的关键用户态，例如：

- 待确认
- 排队中
- 执行中
- 已完成
- 部分成功
- 额度不足中止
- 已取消
- 失败

列表中的消耗信息不只显示“扣了多少”，而是显示最终净结果，例如：

- `净消耗 1 次`
- `净消耗 0 次`

若发生返还，应同时有标签提示，例如：

- `系统异常已返还 1 次`

### 任务详情

点击任务后，展示 `job` 详情与其下 `job_item` 聚合视图，不展示 `ai_execution` 细节。

建议分为三块：

- 任务概览：类型、状态、输入摘要、结果摘要、关键时间点
- 次数与返还卡片：
  - 原始扣减次数
  - 已返还次数
  - 最终净消耗
  - 返还原因摘要
- 子项列表：
  - 子项序号
  - 子项类型
  - 状态
  - 结果摘要
  - 消耗/返还标记

### 取消与失败

任务取消或失败时，也必须明确展示最终计费结果：

- 已开始执行的调用若实际发出，可能仍计费。
- 系统异常导致的失败可自动返还。
- 用户侧必须能看到“原始扣减、已返还、最终净消耗”三项，而不是只看到一个失败状态。

## 前后端接口方向

### 认证与当前用户接口

建议新增：

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `POST /api/auth/verify-email`
- `POST /api/auth/resend-verification`
- `POST /api/auth/forgot-password`
- `POST /api/auth/reset-password`
- `GET /api/auth/me`

### 当前用户视角平台接口

建议在现有平台服务之上补一层“当前用户视角”接口，避免前端继续依赖裸 `user_id` 参数：

- `GET /api/me/profile`
- `GET /api/me/quota`
- `GET /api/me/jobs`
- `POST /api/me/jobs`
- `GET /api/me/jobs/{job_id}`
- `POST /api/me/jobs/{job_id}/start`
- `GET /api/me/jobs/{job_id}/items`

底层可以继续复用现有 `platform_jobs` / `platform_admin` 能力或服务，并继续以 `user_id` 作为内部查询归属字段；但对前端暴露成登录态驱动接口。

### 接口迁移原则

本轮接口改造遵循以下顺序：

1. 保留现有 `platform` 领域内部的 `user_id` 归属模型。
2. 先补 `auth/me` 与 `/api/me/*` 接口，形成登录态驱动查询层。
3. 前端用户中心、新的认证链和服务器服务入口优先只消费 `/api/me/*`。
4. 旧的 `platform.ts` 中显式 `user_id` 版本接口保留为兼容层，供存量平台测试或过渡调用使用。
5. 待前端完成迁移后，再决定是否进一步缩减前端侧对显式 `user_id` 版本接口的使用面。

也就是说，本轮做的是“前端接口入口迁移”，不是“后端内部删除 `user_id`”。

## 历史边界隔离

### 原则

用户中心只展示平台模式历史；工作站本地历史若未来需要展示，必须留在本地工作站视图，不进入平台用户中心。

### 边界定义

- 平台历史：
  - 来源于服务器侧 `job` / `job_item`
  - 与登录用户、次数、返还直接关联
  - 进入用户中心
- 本机历史：
  - 来源于 BYOK / 本机配置执行链路
  - 不进服务器数据库
  - 不进入用户中心
  - 若未来需要展示，应作为工作站本地能力单独设计

### 前端落地要求

- 用户中心页面不得直接复用本地工作流的恢复快照、localStorage 快照或 Agent 日志作为“历史记录”来源。
- 生成入口在用户选择本机配置后，不得创建平台 `job` 或向用户中心写入任何服务器历史占位数据。
- 平台任务列表组件和工作站本地恢复/快照组件必须保持数据源隔离，避免后续误合并。

## 本机 AI 配置检测

前端需要一个稳定的“本机 AI 配置是否可用”判断来源。该判断只用于生成前分流，不上传实际密钥内容。建议通过以下方式之一提供：

- 由配置接口返回已脱敏的“本机文本 AI 可用 / 本机图像 AI 可用”布尔标记。
- 或新增一个专门的“本机配置健康状态”接口，仅返回是否可用于本地链路。

该能力不应把敏感配置直接暴露给用户中心。

## 风险

- 当前平台接口仍以 `user_id` 归属模型为主，若不补“当前用户视角”接口，前端认证链会与平台接口风格冲突。
- 用户中心容易被误做成“工作站全历史中心”，与后端草案冲突，必须坚持平台历史和 BYOK 历史分离。
- 若在无路由基础的前端内直接硬塞多页面，导航状态会迅速复杂化；建议尽早引入轻量路由。
- 返还展示若只落在管理员接口而不进入普通用户聚合视图，用户将无法判断次数是否退回。
- 本机 AI 配置检测若直接复用现有完整配置对象，存在把过多配置细节带入用户中心的风险。

## 验收标准

- 右上角未登录显示“登录 / 注册”，已登录显示头像并可进入用户中心。
- 用户中心单页可展示当前用户信息、统一服务器次数池、平台历史任务列表。
- 用户中心只展示平台模式数据，不展示本机执行历史。
- 点击生成时可根据本机配置状态正确分流到“本机执行”或“服务器执行”。
- 选择服务器执行但未登录时，能跳转登录并成功回到原工作台继续动作。
- 平台历史默认展示任务列表，点击后可查看子项。
- 任务列表与任务详情都能展示返还结果，至少包含原始扣减、已返还、最终净消耗。
- 前端文案不把“点击生成”错误表达为立即扣费，而是与服务端实际 AI 调用扣减逻辑一致。
