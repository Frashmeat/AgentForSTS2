# 2026-04-06 独立前端双后端验收记录

## 1. 目标

验证以下场景已经具备可交付的最小闭环：

- 独立静态前端站点承接页面入口
- `workstation-backend` 承接本地工作流、配置、审批、构建和工作台 WebSocket
- `web-backend` 承接认证、用户中心、平台任务与管理接口

本轮约束：

- 第一版只支持本机或 LAN 可达的 `workstation-backend`
- 不支持公网前端直接访问任意用户本机 `workstation-backend`
- 不做第二个前端应用，只保留一个前端壳

## 2. 验收前提

- 已存在 `config.json`
- 如果需要验证平台接口，`config.json` 中已配置有效的 `database.url`
- 已安装 Python / Node.js，本地仓库已有可用依赖
- `frontend/dist` 已存在；若要验证打包产物，可使用 `tools/latest/package-release.ps1`

## 3. 运行形态说明

### 3.1 兼容态 `full`

- 启动方式：`tools\start.bat`
- 前端托管：由 `full` 后端挂载 `frontend/dist`
- 适用场景：过渡期兼容、同机联调、历史脚本复用

### 3.2 工作站托管态

- 启动方式：`tools\start_workstation.bat`
- 前端托管：由 `workstation-backend` 挂载 `frontend/dist`
- 适用场景：单机本地创作、BYOK、本机构建部署

### 3.3 独立前端双后端态

- 启动方式：
  - `tools\start_split_local.bat`
  - 或分别启动独立静态前端、`workstation-backend`、`web-backend`
- 前端托管：由独立静态站点提供
- 适用场景：前端站点独立发布，工作台能力与平台能力分离部署

## 4. 独立前端双后端最小验收步骤

### 4.1 本地启动器验收

执行：

```powershell
pwsh -NoProfile -File .\tools\start_split_local.ps1 -DryRun -NoBrowser
```

预期：

- 输出仓库模式、端口、前端目录、后端命令、Python 路径和 `runtime-config.js` 生成位置
- 默认工作站 HTTP 指向 `http://127.0.0.1:7860`
- 默认本地前端入口指向 `http://127.0.0.1:8080`

### 4.2 运行时配置验收

检查 `runtime-config.js` 包含以下键：

- `window.__AGENT_THE_SPIRE_API_BASES__.workstation`
- `window.__AGENT_THE_SPIRE_API_BASES__.web`
- `window.__AGENT_THE_SPIRE_WS_BASES__.workstation`

预期：

- 工作台 HTTP 请求使用 `workstation`
- 平台认证、用户中心、管理接口使用 `web`
- 工作台 WebSocket 使用 `workstation`

### 4.3 平台接口边界验收

启动 `tools\start_web.bat` 或等价 `web-backend` 进程后，验证：

- `/api/auth/*` 由 `web-backend` 承接
- `/api/me/*` 由 `web-backend` 承接
- `/api/admin/*` 由 `web-backend` 承接
- 登录态 Cookie 配置与 CORS 来源允许独立前端站点访问

### 4.4 工作台接口边界验收

启动 `tools\start_workstation.bat` 或 `start_split_local` 拉起的本地 workstation 后，验证：

- `/api/config`
- `/api/plan`
- `/api/approvals/*`
- `/api/ws/create`
- `/api/ws/build-deploy`

以上请求均由 `workstation-backend` 承接。

## 7. 结论

在当前约束下，仓库已经具备以下交付方式：

- 用户拿到一个安装包，其中包含独立静态前端、`workstation-backend` 和本地 launcher
- 平台侧单独部署 `web-backend`
- 前端通过运行时配置在不重构建的前提下同时访问 `workstation-backend` 与 `web-backend`

这意味着第一版可以采用“一个前端站点 + 两个后端职责域 + 用户侧单安装包”的发放方式，而不需要额外拆第二个前端工程。
