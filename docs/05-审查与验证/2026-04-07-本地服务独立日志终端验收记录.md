# 2026-04-07 本地服务独立日志终端验收记录

## 1. 目标

让 `tools/latest/deploy-docker.ps1` 在本地启动 `workstation`、`frontend` 服务时，不再把两路日志混在同一个控制台里，也不再只告诉用户日志文件路径，而是为每个服务拉起独立日志终端窗口。

## 2. 本次改动范围

- `tools/latest/deploy-docker.ps1`
- `backend/tests/test_deploy_docker_script.py`

## 3. 行为说明

- `workstation` 本地进程继续把 `stdout/stderr` 写入 `release/runtime/logs/`
- `frontend` 本地进程继续把 `stdout/stderr` 写入 `release/runtime/logs/`
- `hybrid` / `full` / `workstation` / `frontend` 启动本地服务时，会分别为对应服务打开独立 PowerShell 日志窗口
- 每个日志窗口会持续跟随本服务的 `stdout` 与 `stderr` 文件，并在窗口内带前缀输出
- 同一服务重复部署前，会先尝试关闭上一次遗留的日志窗口，避免越开越多

## 4. 最小验证

执行：

```powershell
python -m pytest backend/tests/test_deploy_docker_script.py -q
```

覆盖点：

- 本地 `workstation` 仍按原路径启动
- `hybrid` 模式会分别为 `workstation` 与 `frontend` 记录独立日志窗口启动意图
- 远端 `WebBaseUrl` 场景下，当前 release 不会额外走 Docker
- 现有 `full` / `frontend` / `reuse` 等分支行为未回退

## 5. 测试钩子

- `ATS_LOG_VIEWER_RECORD_FILE`
  - 仅用于测试
  - 设定后不真正弹出 PowerShell 窗口，而是把将要启动的日志窗口参数写入指定文件

## 6. 结果结论

- 本地服务日志查看从“只能手动去文件里翻”调整为“按服务独立终端实时查看”
- `frontend` 与 `workstation` 不再争用同一终端输出
- 日志文件落盘能力仍保留，便于后续排障和打包收集
