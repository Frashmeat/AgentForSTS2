## 一、严重（High）

### 1.1 后端

**1. 事件循环阻塞风险**
- `backend/app/modules/platform/application/services/execution_orchestrator_service.py:888` 在被异步上下文调用的同步路径里使用 `asyncio.run()`，可能死锁
- 对照：`backend/routers/workflow.py:334-336` 的 `run_in_executor` 才是正确做法

**2. 过宽异常吞掉错误**
- `backend/routers/batch_workflow.py:447-450, 583-596` `except Exception as _e:` 仅记录后继续，调用方无法感知失败
- `backend/app/modules/platform/application/services/execution_orchestrator_service.py:873-886` 把所有异常归类为 `failed_system`，丢失系统错误与业务错误的区分（内存不足、系统中断都被误分类）

**3. 全局可变状态与线程竞态**
- `backend/app/modules/knowledge/infra/knowledge_runtime.py:694-700` `_RefreshTask._run()` 在 daemon 线程中无错误隔离
- `backend/project_utils.py:132-141` `_REFRESH_TASKS` / `_DETECT_TASKS` 字典无过期清理，存在内存泄漏

**4. 审批模块二重化**
- `backend/app/modules/approval/` 与 `backend/approval/` 两套并存
- `backend/app/modules/approval/application/services.py:20`、`backend/routers/approval_router.py:7`、`approval_facade_service.py` 均直接 import 旧包
- 后果：分层失效，维护者无法判断应改哪个实现

**5. N+1 查询**
- `backend/app/modules/platform/infra/persistence/repositories/job_query_repository_sqlalchemy.py:71-81` `list_jobs()` 主查询后再做 4 次 `_load_*_summaries()` 单独查询
- 对比：`backend/app/modules/platform/infra/persistence/repositories/job_repository_sqlalchemy.py:49` 在另一处用了 `selectinload(JobRecord.items)`，但主路径未改

**6. 事务边界不清**
- `backend/app/modules/platform/infra/persistence/repositories/job_repository_sqlalchemy.py:43,59` 用 `.flush()` 而非显式 `.commit()`，事务由调用者隐式管理
- `backend/routers/_auth_support.py:25-38` `auth_session_scope` 自动 commit；业务层缺乏显式事务控制

**7. 配置/密钥管理脆弱**
- `backend/app/shared/infra/config/settings.py:243` 直接 `subprocess.run(["setx", ...])` 写 Windows 环境变量；非 Windows 平台失效
- 密钥以明文 JSON 存储，未加密

**8. 资源清理不完整**
- `backend/routers/workflow.py:750-754` 与 `backend/routers/batch_workflow.py:393-399` `zipfile.ZipFile` 与文件操作 try 块缺少 `with` 语句保证关闭；如果 `shutil.rmtree()` 自身异常，资源未正确释放

### 1.2 前端

**9. 巨型组件**
- `frontend/src/features/batch-generation/view.tsx`（2,178 行，26+ 处 Hook 调用）规划/审批/执行多条流程耦合
- `frontend/src/components/SettingsPanel.tsx`（1,385 行）

**10. 类型安全漏洞（与 `tsconfig.json` 的 `strict: true` 矛盾）**
- `SettingsPanel.tsx:144,152,162,521,542` 多处 `any`：`buildConfigSaveBody(cfg: any, ...)`、`useState<any>`、动态 `path` 数组写入嵌套对象（`set()` 函数 line 518-536）

**11. 静默吞错**
- `SettingsPanel.tsx:252-254, 265-267` `getLatestDetectAppPathsTask()` / `getLatestRefreshKnowledgeTask()` 的 catch 块仅有注释；恢复失败既不告警也不打日志

**12. 跨实例全局可变状态**
- `SettingsPanel.tsx:156-157` 模块级 `retainedDetectionTaskId` / `retainedKnowledgeTaskId` 跨组件实例共享，与 `mountedRef` 检查非原子，多实例场景下竞态

**13. 异步竞态**
- `SettingsPanel.tsx:379-429` 多个 useEffect 用 `cancelled` / `mountedRef` 防泄漏，但 500ms 轮询 + 700ms debounce + 500ms debounce 三种节奏并存（line 318-330 与 365-372），快速切 Tab 时新请求可能在清理前发起

## 二、中等（Medium）

### 2.1 后端

**超长函数（职责过多）**
- `execution_orchestrator_service.py:96-278` `start_execution()` 184 行：日志/配置/权限/路由/计费/事务全混在一起
- `batch_workflow.py:245-737` `_handle_ws_batch()` **493 行**，5+ 层嵌套
- `routers/workflow.py:256-410` `_handle_ws_create()` 155 行；line 328-347 / 503-522 / 596-614 三段项目初始化代码 30+ 行重复
- `knowledge_runtime.py:702-757` `_run_refresh_impl()` 56 行混合配置/IO/反编译/下载

**重复代码**
- `backend/project_utils.py:320-595` `_find_sts2_via_registry()` 与 `_find_sts2_in_common_paths()` 包含重复遍历逻辑
- `_detect_sts2_with_progress()` 与 `detect_paths()` 70% 相同

**类型注解缺失**
- `backend/image/generator.py:52-117, 172-246, 251-301` `progress_callback` 缺类型；应为 `Callable[[str], Awaitable[None]]`
- `image/generator.py:179` `progress_callback=None` 无默认类型

**异步函数中的同步阻塞 IO**
- `batch_workflow.py:419` 在 `async def` 内直接 `PilImage.open(io.BytesIO(img_data)).convert("RGBA")`，未走 executor

**控制流复杂**
- `batch_workflow.py:633-679` `handle_control_message()` 嵌套 defer 与 retry；`approval_states`、`deferred_group_messages` 多个全局字典交互易不一致

**路由/应用服务职责重叠**
- `backend/routers/platform_jobs.py:26-80` 大量 `_build_*_service()` 工厂函数应下沉到容器，路由直接注入

**pytest.ini 路径硬编码**
- `backend/pytest.ini` 引用 `F:/WebCode/AgentForSTS2/.tmp/...`；与当前仓库根 `C:/...` 不符，CI/换机失效

**依赖未锁版本**
- `backend/requirements.txt`：`litellm>=1.57.0`、`rembg[gpu]>=2.0.62` 等使用 `>=`
- 前端 `package.json` 大量 `^x.y.z`

**测试基础设施薄弱**
- `backend/tests/conftest.py` 仅一个 tmp_path fixture
- 119 个测试文件多依赖 mock，缺统一的数据库 setup/teardown

**模块边界**
- `backend/app/modules/approval/` 仅有 application/domain，无 infra/persistence
- 与 `backend/approval/` 的分工未文档化

### 2.2 前端

**列表 key 反模式**
- `batch-generation/view.tsx:1324, 1342, 1642, 1664` 等多处使用 `key={index}` 或非稳定组合 key

**Props 爆炸**
- `frontend/src/features/single-asset/view.tsx:85-152` `SingleAssetFeatureViewProps` 80+ 字段；父组件每渲染都新建对象，子组件无 memo

**闭包陈旧**
- `batch-generation/view.tsx:432-448` `updateItem = useCallback(...)` 无依赖数组；WebSocket 回调（line 487+）会持有过期 ref

**deepClone 副作用**
- `SettingsPanel.tsx:524-535` `set()` 用 `structuredClone` 丢失 Date/Map 等非 JSON 类型

**自动保存竞态**
- `SettingsPanel.tsx:318-330` 700ms debounce 与 line 365-372 的 500ms debounce 并存；保存写入顺序错乱（line 625 vs 817）易破坏后端幂等

### 2.3 跨层

- **运行时配置查找复杂**：`backend/app/shared/infra/config/settings.py:24-36` 多分支路径解析，Docker/本地差异大
- **缺少集成测试**：测试套件以 mock 为主，无统一数据库 setup/teardown 基础设施

## 三、偏好/风格（Low）

### 3.1 后端

- **进度日志噪声**：`workflow.py` / `batch_workflow.py` 几乎每个阶段都 `await send("progress", ...)`，与项目"少注释、不冗余"风格冲突
- **冗余辅助函数**：`execution_orchestrator_service.py:40-44` `_short_text()` 顶层定义但仅一两处使用
- **`_safe_pack_id()` 字符集硬编码**：`project_utils.py:175-179` 不清楚为何排除 `!@#$` 等字符（仅凭名称难以理解）
- **缺少 lint/format/typecheck 配置**：无 `ruff`/`black`/`mypy` 配置
- **`requirements.txt` 未含开发工具依赖**：pytest 已含，但无 mypy/black/ruff

### 3.2 前端

- **错误处理不一致**：`shared/api/admin.ts` 用统一 `requestJson`，但 `batch-generation/view.tsx` 用 try-catch 且部分错误被吞
- **WebSocket 错误处理缺失**：`view.tsx:478` 设置错误后直接 return，未清理 socket
- **配置变更难以追踪**：`SettingsPanel` 用多个 `useState`（`llmKey`、`imgKey`、`imgSecret`、`cfg`）保存配置子集，签名计算（line 192-193）易出错——更适合 `useReducer` 或单 config 对象
- **`recovery.ts` 多个 normalizer 重复**：可抽工厂统一
- **TypeScript 严格模式覆盖不全**：`tsconfig.json:14` 启用 `"strict": true`，但大量 `any` 规避；应在 ESLint 规则中禁 `any` 或改用 `unknown` 配合类型守卫
- **缺少状态管理单元测试**：`state.ts` / `recovery.ts` 中 `batchWorkflowReducer`（line 352+）40+ action 分支无测试

### 3.3 跨层

- **`.gitignore` 复杂**：`docs/` 整目录忽略后再多层白名单（line 38-50），维护成本高
- **文档脱节**：`docs/02-现状/2026-03-31-后端现状总结.md:59-82` 仍记录已退役的 migration 开关
- **模块缺失 `__init__.py` 导出约定**：如 `backend/app/modules/approval/` 无显式公共 API 导出
- **LLM 模式规范化无行为文档**：`backend/app/shared/infra/config/settings.py:162-188` 逻辑复杂但缺说明
