## 3. 本阶段形成的实际能力

截至当前，仓库已经不再只是“讨论服务器模式以后怎么做”，而是已经具备以下可运行基础：

1. 平台可以向普通用户返回服务器模式可选 `CLI + 模型` 组合列表。
2. 平台可以读取和更新当前用户的默认服务器配置。
3. 平台任务已经可以记录“用户选了哪一个服务器配置组合”。
4. 单次执行已经可以记录“实际命中了哪一类凭据事实字段”。
5. 管理员查询链路已经具备最小权限边界，不再是完全裸露的只读入口。
6. 管理员已经可以查看服务器执行配置列表和服务器凭据列表。

## 4. 当前仍未完成的关键项

本专题还没有进入“可上线完成态”，主要缺口包括：

### 4.1 管理员写接口仍未落地

- 未实现新增服务器凭据
- 未实现编辑服务器凭据
- 未实现启用 / 禁用服务器凭据
- 未实现手动健康检测

### 4.2 凭据安全能力仍未落地

- 当前虽然已经有 `credential_ciphertext / secret_ciphertext` 字段，但尚未形成正式加密写入链路
- 在密文写入与读取规则未补齐前，不宜继续推进服务器凭据写接口

### 4.3 执行器与真实凭据命中尚未打通

- `ai_execution` 已有 `credential_ref / retry_attempt / switched_credential`
- 但真实执行链路尚未根据 `execution_profile + credentials pool` 去解析和写入这些字段

### 4.4 前端服务器模式体验尚未接入

- 运行时服务器配置选择弹窗尚未正式接后端新接口
- 设置页 `服务器模式` Tab 尚未实现
- 默认值失效后的前端兜底体验尚未形成

### 4.5 抗压与调度保护尚未开始

- 入队
- 单用户并发上限
- 创建 / 开始任务限流
- 凭据健康熔断
- 同组合自动切换一次

以上都还停留在实施计划层，没有进入实现阶段。

## 5. 发现的问题

| 严重程度 | 问题描述 | 根本原因 | 改进建议 |
| --- | --- | --- | --- |
| 必须改正 | 管理员写接口仍缺位，当前只能读不能管 | 安全前置能力此前不存在，必须先补管理员权限边界 | 下一阶段优先补密文写入与管理员写接口 |
| 必须改正 | 真实凭据池调度和执行器尚未打通 | 前期先优先冻结接口与数据模型，执行器改造被刻意后置 | 下一阶段先做 `execution_profile -> credential` 解析和执行写回 |
| 应当改正 | 当前接口文档虽然已回写，但前端尚未接新接口 | 前后端推进节奏不同步 | 等后端写接口稳定后，立刻启动前端服务器模式接入 |
| 应当改正 | 测试环境里的平台仓储 / 服务夹具最初没包含 auth 表，新增外键后暴露出问题 | 旧测试基建默认只建 platform 表，未考虑跨模块外键 | 后续新增跨域表时，优先检查测试夹具是否仍只建局部 schema |

## 6. 做得好的地方

- 先冻结接口、表结构和权限边界，再动代码，避免了服务器模式专题一开始就陷入返工。
- 管理员权限前置能力补得及时，使后续 `/api/admin/platform/*` 扩展有了可持续的安全边界。
- 文档回写基本同步，没有让“草案、计划、现状、代码”四套口径严重漂移。
- 任务被拆成了多个小提交，后续定位和回滚成本都更低。

## 7. 下一阶段建议

下一阶段建议严格按以下顺序推进：

1. 先补服务器凭据密文写入能力
   - 至少形成一套明确的写入 / 存储 / 读取口径

2. 再落管理员写接口
   - 新增服务器凭据
   - 启用 / 禁用服务器凭据
   - 手动健康检测

3. 再打通执行器与凭据池
   - `execution_profile -> server_credentials`
   - 写回 `credential_ref / retry_attempt / switched_credential`

4. 最后再接前端体验
   - 服务器模式选择弹窗
   - 设置页 `服务器模式` Tab

## 8. 当前阶段结论

当前可以明确认为：

- 本专题已经**进入代码实现阶段**
- 但仍处于**后端第一批基础能力落地阶段**
- 还**不具备直接进入前端全面接入或上线准备**的条件

一句话总结：

**服务器模式专题已经从“只会讨论”推进到“有真实后端骨架”，下一阶段的主攻方向应转向“凭据写入安全能力 + 管理员写接口 + 执行器真实调度链路”。**
