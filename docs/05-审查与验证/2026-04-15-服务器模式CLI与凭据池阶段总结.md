# 2026-04-15 服务器模式 CLI 与凭据池阶段总结

## 1. 原定目标

本阶段的目标不是一次性做完服务器模式全链路，而是把该专题从“纯草案讨论”推进到“可安全进入实现，并已有第一批后端落地”的状态。

本阶段重点包括：

- 把服务器模式专题从草案收束成正式实施入口
- 冻结接口、表结构和管理员权限边界
- 补齐管理员最小权限前置能力
- 落第一批服务器模式数据闭环
- 让任务主链开始记录服务器模式选择与执行事实

## 3. 本阶段形成的实际能力

截至当前，仓库已经不再只是“讨论服务器模式以后怎么做”，而是已经具备以下可运行基础：

1. 平台可以向普通用户返回服务器模式可选 `CLI + 模型` 组合列表。
2. 平台可以读取和更新当前用户的默认服务器配置。
3. 平台任务已经可以记录“用户选了哪一个服务器配置组合”。
4. 单次执行已经可以记录“实际命中了哪一类凭据事实字段”。
5. 管理员查询链路已经具备最小权限边界，不再是完全裸露的只读入口。
6. 管理员已经可以查看服务器执行配置列表和服务器凭据列表。
7. 管理员已经可以新增、编辑、启用、禁用服务器凭据。
8. 管理员已经可以手动检测单个凭据，并把结果落回数据库状态字段与检测历史表。
9. 平台执行编排已经具备“从 `execution_profile` 自动解析真实命中凭据并写回执行快照”的后端真源。
10. 平台真实 caller 在创建任务草稿时，已经能自动继承当前用户“可用的默认服务器配置”并写入任务快照。
11. 平台真实 caller 在确认开始任务时，已经能创建 `ai_execution` 并写入真实命中的 `provider / model / credential_ref`，把主链正式接到调度真源的记录层。
12. 平台内部 runner contract 已经可以携带与任务路由一致的 `execution_binding`，为后续具体 handler 摆脱 runtime config 提供了注入点。
13. 平台 runner 已经出现第一个真实消费 `execution_binding` 的具体 handler，为后续最小文本类服务器任务接入提供了实现样板。
14. `web` runtime 已具备最小平台 runner 装配，不再只能停留在“有 contract、无执行器”的状态。
15. `log_analysis` 已成为第一个能在服务器模式下从 `dispatching/running` 继续推进到完成态的最小真实任务类型。
16. `batch_generate/custom_code` 已成为第二个最小服务器文本任务类型，平台开始能在不依赖 `project_root` 的前提下直接消费批量规划结果。
17. `single_generate/custom_code` 已具备前后端最小服务器文本闭环，平台 `/api/platform/jobs` 与 `/api/me/jobs` 当前也可直接提交该纯文本子项，单资产页也已开放对应入口。
18. `single_generate/relic` 已具备后端最小服务器文本闭环，平台 `/api/platform/jobs` 与 `/api/me/jobs` 当前可直接提交该纯文本遗物描述并推进到完成态。
19. `single_generate/card` 已具备后端最小服务器文本闭环，平台 `/api/platform/jobs` 与 `/api/me/jobs` 当前可直接提交该纯文本卡牌描述并推进到完成态。
20. 其他暂未接通的服务器任务，当前也不再只是“停在 running 无下文”，而是会显式暴露 deferred 原因，便于继续收敛 payload 边界。
21. 用户中心历史列表已能直接区分“仍在执行中”和“已命中 deferred、等待继续接入”的 `running` 任务，平台侧阻塞状态开始具备列表级可观测性。
22. deferred 摘要已从前端临时拼装提升为正式 summary contract，后续其他列表页或兼容 `/api/platform/jobs` 调用方也可直接复用。
23. deferred 任务当前不会再形成“未真实执行却净消耗 1 次”的错误计费，平台开始具备最小的延期返还保护。
24. deferred 分支的 `ai_execution` 当前已能以 `completed_with_refund` 结束，管理员执行视图不再只能看到误导性的 `dispatching`。
25. 用户侧与平台侧任务主状态当前已能显式区分 `deferred` 与真实 `running`，服务器模式状态机开始摆脱“伪运行中”口径。
26. 前端运行时弹窗当前已能读取服务器执行配置、选择具体 `CLI + 模型` 组合，并把本次选择显式写入平台任务快照。
27. 设置页当前也已具备服务器模式默认配置管理入口，运行时弹窗与设置页开始共享同一套服务器配置真源。
28. 当前前端在默认服务器配置失效时，已经能够显式提示用户，并自动回退到推荐 / 首个可用配置。
29. 用户中心当前也已能直接展示任务绑定的执行配置，服务器任务的“选了哪套 CLI / 模型”开始具备列表级与详情级可见性。

## 4. 当前仍未完成的关键项

本专题还没有进入“可上线完成态”，主要缺口包括：

### 4.1 凭据安全与检测能力仍有边界

- 已补第一版密文写入 / 读取链路，并已打通平台执行编排内的调度读取真源
- 当前真实联网检测只覆盖 `openai + api_key`、`anthropic + api_key`
- `ak_sk` 当前只会返回 `degraded / unsupported_auth_type`，还不是完整的可用性检测

### 4.2 真实运行时 caller 已接入最小内部执行器，但还没有覆盖更广泛任务类型

- `ExecutionOrchestratorService` 已能根据 `execution_profile + credentials pool` 自动解析并写回命中快照
- 平台真实 caller 已经在“创建任务草稿”阶段补上默认服务器配置回填
- 平台真实 caller 现在已经在 `start_job` 阶段复用了 `ExecutionOrchestratorService.start_execution()`，并能把任务推进到 `ai_execution.status = dispatching`
- `ExecutionOrchestratorService.run_registered_steps()` 已能为内部 runner 构造按次执行绑定，并在需要时解密出本次执行所需凭据字段
- `ExecutionAdapter`、`text_generate_handler`、`log_analysis_handler` 与 `batch_custom_code_handler` 已证明具体 step handler 可以消费 `execution_binding`
- `web` runtime 已直接挂上最小 `workflow_runner / execution_adapter`
- 当前 `log_analysis`、`batch_generate/custom_code`、`single_generate/custom_code`、`single_generate/relic` 与 `single_generate/card` 已能真正进入内部执行器并回写完成态
- 其他未接通任务当前会通过 `ai_execution.deferred` 明确暴露阻塞原因，而不是继续完全静默
- 但该能力目前仍只覆盖最小文本类任务，更多任务类型、输入资产和服务器工作目录语义仍未接通

### 4.3 前端服务器模式体验仍未完整收口

- 运行时服务器配置选择弹窗已正式接后端新接口
- 运行时创建平台任务时已可显式携带所选执行配置，并可选择写回默认服务器配置
- 设置页 `服务器模式` Tab 已实现，当前也可直接保存 / 清空默认服务器配置
- 默认值失效时的基础前端兜底提示已补；若后续需要更细粒度引导与恢复建议，仍可继续增强

### 4.4 抗压与调度保护尚未开始

- 入队
- 单用户并发上限
- 创建 / 开始任务限流
- 凭据健康熔断
- 同组合自动切换一次

以上都还停留在实施计划层，没有进入实现阶段。

## 5. 发现的问题

| 严重程度 | 问题描述 | 根本原因 | 改进建议 |
| --- | --- | --- | --- |
| 必须改正 | 当前只有 `log_analysis`、`batch_generate/custom_code`、`single_generate/custom_code`、`single_generate/relic` 与 `single_generate/card` 被接成最小真实闭环，其他任务类型仍不能在服务器模式下真正跑完 | 目前仅有纯文本语义任务同时满足“消费执行绑定”和“payload 具备服务器可执行语义”两项前提 | 下一阶段继续扩展更多不依赖本地路径、上传图片的文本类任务，并逐步抽象通用 workflow 注册与完成态回写 |
| 必须改正 | 服务器模式平台任务 payload 目前还不具备真实服务器执行语义，直接接真实执行会出现“任务已提交但服务器无法可靠运行”的伪闭环 | 前端当前仍把用户本机 `project_root` 写入 payload；上传图片场景只传 `has_uploaded_image`，未把真实图片内容或可用引用带入平台任务 | 下一阶段先收敛“第一版允许进入服务器执行的任务边界”，优先支持不依赖本地路径、不依赖上传图片的最小闭环 |
| 必须改正 | 凭据健康检测目前仍有能力边界，`ak_sk` 与更多 provider 还未形成真实检测 | 第一版先把检测记录、状态回写和主流 `api_key` provider 跑通 | 下一阶段补更完整的 provider / auth_type 检测支持 |
| 可继续优化 | 当前 `job / job_item` 已显式返回 `deferred`，但更细粒度的“可重试 / 等待平台接入 / 等待资产持久化”仍复用同一状态值 | 第一版先收口的是主状态机真相，还没有把 deferred 子类型提升到正式状态枚举 | 后续可评估是否需要在查询 contract 中继续补更稳定的 deferred subtype 字段 |
| 可继续优化 | 当前 `job summary` 虽已并入最近 deferred 摘要，但仍只暴露最新一条原因码与消息 | 第一版目标是先解决列表级可观测性与额外查询成本，没有把更细粒度的 `job_item_id / ai_execution_id / occurred_at` 一并下沉 | 若后续需要更精确的列表级诊断，可考虑继续把最近 deferred 事件的时间与子项上下文并入 summary contract |
| 应当改正 | 测试环境里的平台仓储 / 服务夹具最初没包含 auth 表，新增外键后暴露出问题 | 旧测试基建默认只建 platform 表，未考虑跨模块外键 | 后续新增跨域表时，优先检查测试夹具是否仍只建局部 schema |

## 6. 做得好的地方

- 先冻结接口、表结构和权限边界，再动代码，避免了服务器模式专题一开始就陷入返工。
- 管理员权限前置能力补得及时，使后续 `/api/admin/platform/*` 扩展有了可持续的安全边界。
- 文档回写基本同步，没有让“草案、计划、现状、代码”四套口径严重漂移。
- 任务被拆成了多个小提交，后续定位和回滚成本都更低。

## 7. 下一阶段建议

下一阶段建议严格按以下顺序推进：

1. 继续扩展更多可服务器执行的文本类任务
   - `log_analysis`、`batch_generate/custom_code`、`single_generate/custom_code`、`single_generate/relic` 与 `single_generate/card` 已验证“按次执行绑定注入 + 最小 runner + 完成态回写”这条链可用
   - 下一步应继续把不依赖本地 `project_root`、不依赖上传图片的文本任务映射到该链路

2. 再收敛更广泛任务的服务器可执行语义
   - 当前 caller 已经复用 `execution_profile -> server_credentials` 真源并写入执行快照
   - 但 `payload` 中的本地路径、上传图片引用、服务器工作目录仍未统一
   - 当前虽已通过 `ai_execution.deferred` 显式暴露这些阻塞，但它们本身仍需要真正被消解

3. 再补更通用的运行时编排能力
   - 让更多 workflow 类型明确使用本次路由命中的 `provider / model / credential_ref`
   - 抽出更稳定的 workflow 注册和结果持久化样板

4. 再补凭据健康检测边界
   - `ak_sk` 检测支持
   - 更多 provider 的真实检测

5. 最后再继续扩展前端体验
   - 当前历史列表 deferred 摘要提示已补齐
   - 运行时服务器模式选择弹窗、设置页 `服务器模式` Tab 和默认值失效提示已接通，后续继续补更细粒度引导与恢复建议

## 8. 当前阶段结论

当前可以明确认为：

- 本专题已经**进入代码实现阶段**
- 但仍处于**后端第一批基础能力落地阶段**
- 还**不具备直接进入前端全面接入或上线准备**的条件
- 当前主链已经不再只停在 `dispatching`，`log_analysis` 已能在服务器模式下走到完成态
- 当前“按次执行参数注入”已经补到 runner contract，且已有文本类 handler 真正消费该绑定
- 下一阶段的主攻方向，应从“证明最小闭环可行”继续推进到“扩展更多服务器可执行任务并收敛 payload 语义”

一句话总结：

**服务器模式专题已经从“后端调度真源已打通”继续推进到“`log_analysis`、`batch_generate/custom_code`、`single_generate/custom_code`、`single_generate/relic` 与 `single_generate/card` 已形成五条最小服务器文本闭环，且 deferred 阻塞状态已在用户中心形成列表级可观测性”，下一阶段应继续扩展更多服务器可执行任务并补齐通用输入语义。**
