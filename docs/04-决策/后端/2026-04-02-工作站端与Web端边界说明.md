# 2026-04-02 工作站端与 Web 端边界说明

## 1. 目的

本文档用于固化 **当前代码实现** 中“工作站端后端”和“Web 端后端”的边界，避免后续再通过翻入口文件、路由装配逻辑和打包脚本重复确认。

这里描述的是 **当前实现边界**，不是不可变更的最终架构真理。后续如果入口、路由装配或迁移开关发生变化，需要同步更新本文。

## 2. 当前后端角色划分

当前后端应用工厂在 `backend/app_factory.py` 中定义了三个运行角色：

| 角色 | 入口文件 | 默认端口 | 包含前端静态文件 | 路由来源 |
| --- | --- | --- | --- | --- |
| `full` | `backend/main.py` | `7860` | 是 | 工作站路由 + 按迁移开关条件启用的平台 Web 路由 |
| `workstation` | `backend/main_workstation.py` | `7860` | 是 | 仅工作站路由 |
| `web` | `backend/main_web.py` | `7870` | 否 | 仅 Web 路由 |

可以把三者理解为：

- `workstation`
  面向本地创作工作站，负责前端界面承载、本地项目工作流、日志分析、Mod 分析、构建部署、审批与本地配置。
- `web`
  面向平台化服务端，负责 Job/Execution/Quota/Audit 这类数据库驱动的 HTTP API。
- `full`
  面向兼容态或同机联调态，不是一个独立职责域，而是“工作站端能力 + 部分 Web 端能力”的组合壳。

## 3. 代码中的正式边界

### 3.1 路由装配边界

`backend/app_factory.py` 中有两个核心路由集合：

#### 工作站端路由集合

- `routers.workflow`
- `routers.config_router`
- `routers.batch_workflow`
- `routers.log_analyzer`
- `routers.mod_analyzer`
- `routers.build_deploy`
- `routers.approval_router`

#### Web 端路由集合

- `routers.auth_router`
- `routers.me_router`
- `routers.platform_jobs`
- `routers.platform_admin`

应用工厂的实际装配规则如下：

- `role in {"full", "workstation"}` 时，加载全部工作站路由。
- `role == "web"` 时，加载全部 Web 路由。
- `role == "full"` 时，不直接等价于 `web`，而是只在迁移开关允许时按条件额外挂载：
  - `platform_jobs_api_enabled` -> `routers.platform_jobs`
  - `platform_service_split_enabled` -> `routers.platform_admin`

这意味着：

- `workstation` 是明确的“只含工作站路由”角色。
- `web` 是明确的“只含 Web 路由”角色。
- `full` 是兼容混合态，是否暴露 Web 平台接口取决于 migration flags。

### 3.2 静态前端挂载边界

只有 `full` 和 `workstation` 会执行前端静态资源挂载：

- 挂载目录：`frontend/dist`
- 挂载方式：`app.mount("/", StaticFiles(..., html=True), name="frontend")`

`web` 角色 **不挂载前端静态文件**，因此它应该被视为纯 API 服务，而不是前端承载服务。

这也是工作站端与 Web 端最明确的运行时边界之一：

- 工作站端 = UI 承载 + 本地操作接口
- Web 端 = API 服务，不负责前端页面托管

## 4. 工作站端职责边界

### 4.1 工作站端负责的能力

按当前路由实现，工作站端负责以下类型能力：

| 路由模块 | 典型路径 | 作用 | 边界判断 |
| --- | --- | --- | --- |
| `routers.workflow` | `/api/ws/create` `/api/project/create` `/api/project/build` `/api/project/package` | 单资产创建主流程、生图、代码生成、项目创建、构建与打包 | 明显依赖本地项目目录、图片生成、代码 Agent、本地文件系统 |
| `routers.batch_workflow` | `/api/plan` `/api/ws/batch` | 批量规划与批量资产创建工作流 | 面向本地 Mod 工程生成流程 |
| `routers.config_router` | `/api/config` `/api/config/detect_paths` `/api/config/test_imggen` | 工作站配置查看/修改、路径探测、图像生成测试 | 明显依赖本机环境和本地配置 |
| `routers.log_analyzer` | `/api/ws/analyze-log` | 读取本机游戏日志并交给 LLM 分析 | 直接读取本机 `godot.log`，是本地诊断能力 |
| `routers.mod_analyzer` | `/api/ws/analyze-mod` | 扫描本地 Mod 项目源码与 localization 内容并分析 | 直接读取项目目录，是本地项目分析能力 |
| `routers.build_deploy` | `/api/ws/build-deploy` | 调用 Code Agent 构建并把产物部署到本机 STS2 Mods 目录 | 直接依赖本地构建链路和游戏安装目录 |
| `routers.approval_router` | `/api/approvals/*` | 审批请求读取、批准、拒绝、执行 | 当前被归入工作站路由集合，主要为本地工作流执行链路服务 |

### 4.2 工作站端的判断标准

如果某个后端能力满足下列特征之一，当前实现上应优先归到工作站端：

- 直接依赖本地文件系统或项目目录。
- 直接依赖本机安装的 STS2、Godot、Mods 目录等环境。
- 与单人创作工作流强绑定，需要 WebSocket 持续推流到本地前端。
- 本质是“工作台操作能力”，而不是平台侧多用户任务管理能力。

### 4.3 关于审批接口的特殊说明

`approval_router` 目前在路由装配上属于工作站端，而不是 Web 端。

但它的内部实现已经有一层迁移兼容逻辑：

- 当 `platform_service_split_enabled` 打开时，会优先走 `platform.approval_facade_service`
- 否则走当前本地 `approval_store` / `approval_service`

因此审批能力目前的准确表述是：

- **路由归属上属于工作站端**
- **实现演进上已经为平台化服务拆分预留了 facade 接入点**

## 5. Web 端职责边界

### 5.1 Web 端负责的能力

按当前路由规划与当前实现边界，Web 端负责认证与平台化模块：

| 路由模块 | 典型路径 | 作用 | 边界判断 |
| --- | --- | --- | --- |
| `routers.auth_router` | `/api/auth/*` | 登录、注册、当前登录态、邮箱验证、找回密码等认证链路 | 明显属于平台身份体系与服务器登录态 |
| `routers.me_router` | `/api/me/*` | 当前用户 profile、quota、jobs 等当前用户视角接口 | 明显属于平台用户中心查询层 |
| `routers.platform_jobs` | `/api/platform/jobs` `/api/platform/jobs/{job_id}` `/api/platform/quota` | Job 创建、启动、取消、详情、事件、配额查询 | 明显是数据库驱动的平台任务接口 |
| `routers.platform_admin` | `/api/admin/jobs/{job_id}/executions` `/api/admin/executions/{execution_id}` `/api/admin/quota/refunds` `/api/admin/audit/events` | 执行记录、退款记录、审计事件等管理视图 | 明显属于平台后台管理能力 |

与前端边界一并约束：

- 新增认证页、用户中心页、任务详情页默认只调用 `/api/auth/*` 与 `/api/me/*`
- 旧 `/api/platform/*` 中显式 `user_id` 参数仅保留给兼容链路
- `user_id` 继续保留为后端内部正式主键，不因前端收口 `/api/me/*` 而删除

### 5.2 Web 端的判断标准

如果某个后端能力满足下列特征之一，当前实现上应优先归到 Web 端：

- 以 HTTP 查询/命令接口为主，而不是本地工作站 WebSocket 流。
- 强依赖数据库会话、仓储、审计、执行记录、配额等平台数据模型。
- 面向多用户、多任务、平台运行态，而不是面向本地创作工作台。
- 不需要承载前端静态资源。

### 5.3 Web 端的基础设施特征

`platform_jobs` 和 `platform_admin` 都通过 `request.app.state.container` 获取平台依赖，并显式使用：

- `platform.db_session_factory`
- 平台仓储工厂
- 平台应用服务 / 查询服务工厂

这说明 Web 端在当前实现中本质上是：

- **数据库与应用服务驱动的 API 服务**
- **不直接承担本地工作流执行界面**
- **当前 `/api/auth/*` 与 `/api/me/*` 已默认归到 `web-backend`**

补充当前前端落地：

- 工作站前端壳中的认证页位于 `/auth/*`
- 用户中心总览页位于 `/me`
- 用户中心任务详情页位于 `/me/jobs/:jobId`
- 这些页面默认调用 `web-backend` 的 `/api/auth/*` 与 `/api/me/*`

## 6. `full` 角色不是“正式 Web 端”

后续讨论部署和打包时，最容易混淆的是把 `full` 当成“完整生产形态”。当前代码里不应这样理解。

`full` 的准确语义是：

- 始终包含工作站端能力
- 始终挂载前端静态资源
- 仅在迁移开关开启时额外暴露部分 Web 平台接口

因此它更适合被理解为：

- 兼容阶段的聚合入口
- 本地联调入口
- 过渡期全量验证入口

而不是“Web 端正式部署入口”的同义词。

## 7. 打包与部署边界

当前 `tools/latest` 脚本已经按目标对这两个职责域进行了分离：

| 打包目标 | 实际含义 | 对应边界 |
| --- | --- | --- |
| `workstation` | 前端 + 工作站后端 | 给终端用户使用的本地工作站包 |
| `web` | 仅 Web 后端 | 给服务器部署的平台 API 包 |
| `frontend` | 仅前端静态站 | 纯静态托管场景 |
| `full` | 前端 + 工作站 + Web 后端 | 联调/兼容/一体化验证包 |

这与应用角色边界是一致的：

- 用户侧推荐理解为：`frontend + workstation`
- 服务器侧推荐理解为：`web`
- `full` 保留给过渡期兼容和联调，不建议作为长期职责边界表达

## 8. 后续判断规则

后续新增接口时，可以按下面的简化规则判断归属：

### 8.1 应归到工作站端的接口

- 要操作本地项目目录
- 要读取本机日志或本机配置
- 要驱动本地构建/部署
- 要给工作台页面做长连接推流

### 8.2 应归到 Web 端的接口

- 要管理 Job、Execution、Quota、Audit
- 要通过数据库进行持久化查询/命令
- 要面向服务器部署和平台侧调用
- 不依赖前端静态文件托管

### 8.3 需要特别警惕的混合接口

以下类型最容易越界，新增时应单独评审：

- “工作站触发，但平台落库”的混合工作流接口
- 审批相关接口
- Build/Deploy 的平台化改造接口
- 既想本地运行又想服务器运行的兼容接口

这类接口如果继续保留在工作站路由下，应该明确说明只是“兼容接入点”；如果要平台化，就应迁入 Web 端并补齐服务边界。

## 9. 当前建议

基于当前实现，后续默认采用以下口径：

- “工作站端后端” = `main_workstation.py` + 工作站路由集合 + 前端静态托管
- “Web 端后端” = `main_web.py` + `auth` / `me` / `platform_jobs` / `platform_admin` 四类平台 API
- 用户中心只展示平台模式任务；BYOK / 本机执行不会进入服务器历史
- “full” = 兼容聚合态，不作为长期职责命名基准

如果后面再讨论：

- 给用户打什么包
- 服务器部署什么包
- 某个新接口应该放在哪边
- 某个前端页面应该连哪个后端

都应优先以本文档中的边界定义为准。
