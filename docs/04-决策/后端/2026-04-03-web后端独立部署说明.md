# 2026-04-03 Web 后端独立部署说明

## 1. 目的

本文档用于明确当前阶段“独立部署 web 后端”到底指什么，以及在现有单前端入口方案下，`workstation-backend` 与 `web-backend` 的职责如何划分。

本轮目标是：

- 让 `web-backend` 可独立启动、独立维护、独立发布
- 不拆第二个前端应用
- 不更换 Python / FastAPI 技术栈

当前状态：

- 上述目标已完成并合并到主工作区
- 同一前端壳内已落地认证页、用户中心页与任务详情页
- 服务器模式已通过 `web-backend` 的 `/api/auth/*`、`/api/me/*` 与平台任务接口承接
- 运行角色已经收口为 `workstation` 与 `web`；`full` 不再是当前代码中的有效运行角色

## 2. 当前推荐部署口径

### 2.1 本地工作站

推荐运行：

- `tools\start_workstation.bat`

对应入口：

- `backend/main_workstation.py`

职责：

- 承接前端静态资源
- 承接本地工作流、本地配置、审批、构建部署等工作站能力

### 2.2 平台 Web 后端

推荐运行：

- `tools\start_web.bat`

对应入口：

- `backend/main_web.py`

职责：

- 承接认证接口
- 承接当前用户视角接口 `/api/auth/*`、`/api/me/*`
- 承接平台任务、配额、历史记录、返还聚合等平台 API

注意：

- `web-backend` 不负责前端静态资源托管
- `web-backend` 默认要求存在有效的 `database.url`
- Docker `web` 目标默认由 Web 容器托管一个同机 Workstation 子进程。部署脚本必须同时生成 `runtime\web.config.json` 与 `runtime\workstation.config.json`，并通过 `ATS_WORKSTATION_CONTROL_TOKEN` 共享内部控制 token。
- `tools\latest\package-release.ps1 web` 生成的 `services\web\backend` 必须包含 `main_web.py`、`main_workstation.py` 和 Workstation 内部路由；否则 Web 容器内无法 import `main_workstation:app`，部署脚本会在启动前拒绝旧 release。
- Web 派发平台任务前会先确认托管 Workstation 可用；配置缺失、启动失败或 capabilities 不可达时，任务失败摘要会带目标 URL 和明确原因，不再等到底层 `Connection refused`。
- 托管 Workstation stdout/stderr 写入 `runtime\logs\web-workstation.stdout.log` 与 `runtime\logs\web-workstation.stderr.log`，管理端 `/admin/runtime` 可查看这些日志路径。

### 2.3 旧兼容入口现状

当前 `backend/main.py` 和 `tools\start.bat` 只应理解为工作站兼容入口：

- `tools\start.bat`
- `backend/main.py`

说明：

- `backend/main.py` 当前调用 `create_app("workstation")`
- 当前代码不再支持 `create_app("full")`
- 需要 Web 平台 API 时，应启动 `backend/main_web.py` / `tools\start_web.bat`

### 2.4 独立前端双后端态

推荐入口：

- 用户侧正式打包：`tools\latest\package-release.ps1 hybrid`
- 用户侧 Docker 部署：`tools\latest\deploy-docker.ps1 hybrid -WebBaseUrl https://your-web-api.example.com`
- 用户侧本地验证：`tools\start_split_local.bat`
- 平台侧：`tools\start_web.bat`

对应构成：

- 独立静态前端站点
- 本地 `workstation-backend`
- 独立 `web-backend`

职责：

- 独立前端只负责页面与运行时配置加载
- `workstation-backend` 承接本地工作流、审批、配置、构建与工作台 WebSocket
- `web-backend` 承接认证、用户中心、平台任务、配额与管理接口

当前约束：

- 第一版只支持本机或 LAN 可达的 `workstation-backend`
- 不支持公网静态前端直接访问任意用户本机 workstation
- `hybrid` 用户侧 bundle 推荐发放“独立前端 + workstation-backend + launcher”，`web-backend` 单独部署

补充口径：

- `hybrid` 是正式推荐部署目标
- `full` 已退出当前实现基线，不再作为部署目标或职责边界
- `split-local` 只是 `hybrid` 形态的本地启动与验证入口，不等于发布目标
- `hybrid` Docker 部署必须显式提供 `-WebBaseUrl`，避免把平台接口误写到浏览器所在机器的 `127.0.0.1`

## 3. 当前单前端入口下的分流规则

本轮不拆第二个前端应用，因此前端仍是一个入口。但后端职责应明确分流：

### 3.1 workstation-backend 承接

- 本地项目工作流
- 本地 AI 配置与配置检测
- 本地日志分析
- 本地 Mod 分析
- 本地构建与部署
- 工作站审批链路

### 3.2 web-backend 承接

- 登录 / 注册 / 邮箱验证 / 找回密码
- 当前用户信息
- 当前用户平台任务列表与详情
- 当前用户视角的平台任务创建与确认开始
- 配额、返还、净消耗
- 平台管理与审计接口

### 3.3 独立前端下的前端分流规则

当前前端仍然只有一个应用，但在独立前端模式下需要通过运行时配置显式分流：

- `window.__AGENT_THE_SPIRE_API_BASES__.workstation`
  用于工作台 HTTP 接口，如 `/api/config`、`/api/plan`、`/api/approvals/*`
- `window.__AGENT_THE_SPIRE_API_BASES__.web`
  用于平台 HTTP 接口，如 `/api/auth/*`、`/api/me/*`、`/api/admin/*`
- `window.__AGENT_THE_SPIRE_WS_BASES__.workstation`
  用于工作台 WebSocket，如 `/api/ws/create`、`/api/ws/build-deploy`

这意味着：

- 工作台流量不再依赖前端站点同源
- 平台接口也不再要求和前端同源托管
- 更换部署地址时，优先替换 `runtime-config.js`，不重新构建前端

## 4. 最小配置要求

### 4.1 web-backend

至少应具备：

- `database.url`
- `runtime.web.cors_origins`
- `auth.session_cookie_name`
- `platform_execution.workstation_url`
- `platform_execution.workstation_config_path`
- `platform_execution.control_token_env`

其中：

- `database.url` 用于平台任务、用户、配额、历史等数据库读写
- `runtime.web.cors_origins` 用于独立部署时控制前端访问来源
- `auth.session_cookie_name` 用于后续认证链路的统一会话配置
- `platform_execution.*` 用于 Web 托管 Workstation 的启动、探测和内部派发。Docker `web` 目标中 `workstation_url` 必须指向 Web 容器内可达地址，默认是 `http://127.0.0.1:<workstation_port>`，不是宿主机或管理员电脑上的 Workstation。

### 4.2 workstation-backend

可继续复用当前本地配置体系，不强依赖数据库。

独立前端模式下至少应具备：

- `runtime.workstation.cors_origins`
- `auth.session_cookie_name`
- 如需跨站点 Cookie，还需根据部署方式设置 `auth.session_cookie_samesite`、`auth.session_cookie_secure`、`auth.session_cookie_domain`

## 5. 本轮边界

本轮明确：

- 先做 `web-backend` 独立部署基础
- 再继续 `/api/auth/*`、`/api/me/*`、用户中心与平台接入主链
- 在同一前端壳内补齐独立前端运行时分流与双后端访问能力

本轮不做：

- 第二个前端应用
- Java / Spring 重写
- Rust 重写
- 微服务拆分

## 6. 结论

当前“web 独立部署”指的是：

- 独立部署 `backend/main_web.py`
- 以独立 API 服务的方式承接平台能力
- 与工作站后端解耦维护和发布
- 并允许同一个前端壳在独立静态站点模式下同时访问 `workstation-backend` 与 `web-backend`

而不是：

- 立刻拆第二个前端应用
- 立刻切换语言或重写整套后端
