# 2026-04-03 Web 后端独立部署说明

## 1. 目的

本文档用于明确当前阶段“独立部署 web 后端”到底指什么，以及在现有单前端入口方案下，`workstation-backend` 与 `web-backend` 的职责如何划分。

本轮目标是：

- 让 `web-backend` 可独立启动、独立维护、独立发布
- 不拆第二个前端应用
- 不更换 Python / FastAPI 技术栈

当前状态：

- 上述目标已完成并合并到主工作区
- 同一前端壳内已落地认证页、用户中心页与任务详情页
- 服务器模式已通过 `web-backend` 的 `/api/auth/*`、`/api/me/*` 与平台任务接口承接

## 2. 当前推荐部署口径

### 2.1 本地工作站

推荐运行：

- `tools\start_workstation.bat`

对应入口：

- `backend/main_workstation.py`

职责：

- 承接前端静态资源
- 承接本地工作流、本地配置、审批、构建部署等工作站能力

### 2.2 平台 Web 后端

推荐运行：

- `tools\start_web.bat`

对应入口：

- `backend/main_web.py`

职责：

- 承接认证接口
- 承接当前用户视角接口 `/api/auth/*`、`/api/me/*`
- 承接平台任务、配额、历史记录、返还聚合等平台 API

注意：

- `web-backend` 不负责前端静态资源托管
- `web-backend` 默认要求存在有效的 `database.url`

### 2.3 兼容聚合态

兼容入口：

- `tools\start.bat`
- `backend/main.py`

说明：

- `full` 仅作为兼容态与联调用入口保留
- 不建议继续把 `full` 当作长期职责边界

### 2.4 独立前端双后端态

推荐入口：

- 用户侧：`tools\start_split_local.bat`
- 平台侧：`tools\start_web.bat`

对应构成：

- 独立静态前端站点
- 本地 `workstation-backend`
- 独立 `web-backend`

职责：

- 独立前端只负责页面与运行时配置加载
- `workstation-backend` 承接本地工作流、审批、配置、构建与工作台 WebSocket
- `web-backend` 承接认证、用户中心、平台任务、配额与管理接口

当前约束：

- 第一版只支持本机或 LAN 可达的 `workstation-backend`
- 不支持公网静态前端直接访问任意用户本机 workstation
- 若面向最终用户打包，推荐发放“独立前端 + workstation-backend + launcher”，`web-backend` 单独部署

## 3. 当前单前端入口下的分流规则

本轮不拆第二个前端应用，因此前端仍是一个入口。但后端职责应明确分流：

### 3.1 workstation-backend 承接

- 本地项目工作流
- 本地 AI 配置与配置检测
- 本地日志分析
- 本地 Mod 分析
- 本地构建与部署
- 工作站审批链路

### 3.2 web-backend 承接

- 登录 / 注册 / 邮箱验证 / 找回密码
- 当前用户信息
- 当前用户平台任务列表与详情
- 当前用户视角的平台任务创建与确认开始
- 配额、返还、净消耗
- 平台管理与审计接口

### 3.3 独立前端下的前端分流规则

当前前端仍然只有一个应用，但在独立前端模式下需要通过运行时配置显式分流：

- `window.__AGENT_THE_SPIRE_API_BASES__.workstation`
  用于工作台 HTTP 接口，如 `/api/config`、`/api/plan`、`/api/approvals/*`
- `window.__AGENT_THE_SPIRE_API_BASES__.web`
  用于平台 HTTP 接口，如 `/api/auth/*`、`/api/me/*`、`/api/admin/*`
- `window.__AGENT_THE_SPIRE_WS_BASES__.workstation`
  用于工作台 WebSocket，如 `/api/ws/create`、`/api/ws/build-deploy`

这意味着：

- 工作台流量不再依赖前端站点同源
- 平台接口也不再要求和前端同源托管
- 更换部署地址时，优先替换 `runtime-config.js`，不重新构建前端

## 4. 最小配置要求

### 4.1 web-backend

至少应具备：

- `database.url`
- `runtime.web.cors_origins`
- `auth.session_cookie_name`

其中：

- `database.url` 用于平台任务、用户、配额、历史等数据库读写
- `runtime.web.cors_origins` 用于独立部署时控制前端访问来源
- `auth.session_cookie_name` 用于后续认证链路的统一会话配置

### 4.2 workstation-backend

可继续复用当前本地配置体系，不强依赖数据库。

独立前端模式下至少应具备：

- `runtime.workstation.cors_origins`
- `auth.session_cookie_name`
- 如需跨站点 Cookie，还需根据部署方式设置 `auth.session_cookie_samesite`、`auth.session_cookie_secure`、`auth.session_cookie_domain`

## 5. 本轮边界

本轮明确：

- 先做 `web-backend` 独立部署基础
- 再继续 `/api/auth/*`、`/api/me/*`、用户中心与平台接入主链
- 在同一前端壳内补齐独立前端运行时分流与双后端访问能力

本轮不做：

- 第二个前端应用
- Java / Spring 重写
- Rust 重写
- 微服务拆分

## 6. 结论

当前“web 独立部署”指的是：

- 独立部署 `backend/main_web.py`
- 以独立 API 服务的方式承接平台能力
- 与工作站后端解耦维护和发布
- 并允许同一个前端壳在独立静态站点模式下同时访问 `workstation-backend` 与 `web-backend`

而不是：

- 立刻拆第二个前端应用
- 立刻切换语言或重写整套后端
