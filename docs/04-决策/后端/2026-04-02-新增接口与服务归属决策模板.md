# 2026-04-02 新增接口与服务归属决策模板

## 1. 目的

本文档用于在 **当前代码实现边界** 下，为后续新增的接口、应用服务、facade、数据落库需求提供统一的归属判断模板。

目标不是替代详细设计，而是先在立项或实现前快速回答四个问题：

- 这个能力应该归 `workstation` 还是 `web`？
- 它是正式能力，还是兼容迁移层 `facade`？
- 它是否允许直接落平台数据库？
- 它最终应该跟随哪个打包/部署目标发布？

本文档与 [2026-04-02-工作站端与Web端边界说明.md](I:/WebCode/AgentTheSpire/docs/04-决策/后端/2026-04-02-工作站端与Web端边界说明.md) 配套使用：

- 前一篇解决“当前已经怎么分”
- 本文解决“以后新增内容应该怎么判”

## 2. 归属分类

后续新增能力统一先落到以下四类之一：

| 分类 | 含义 | 何时使用 |
| --- | --- | --- |
| `workstation` | 工作站端正式能力 | 面向本地工作台、本地文件、本地构建、本地长流程 |
| `web` | Web 端正式能力 | 面向服务器部署、平台 API、数据库驱动查询/命令 |
| `shared` | 两端共享的基础能力 | 纯领域模型、协议对象、无部署归属的通用逻辑 |
| `hybrid` | 跨边界能力，必须评审 | 同时触达本地工作流与平台持久化，或兼具两端语义 |

补充说明：

- `shared` 不是“放不下就塞这里”的兜底目录。
- `hybrid` 不是正式归属，而是一个预警标签，表示必须单独评审拆分方案。
- `full` 不是归属分类，只是聚合运行角色，不能被用来回答“这个模块归谁”。

## 3. 六层边界模型

后续判断统一按六层看，不再只看“这个 API 路径像不像平台接口”。

### 3.1 入口层

看谁先发起这次调用。

| 问题 | 偏向 `workstation` | 偏向 `web` |
| --- | --- | --- |
| 谁是直接调用方？ | 本地工作台前端、本地用户操作 | 服务器侧调用、平台前端、平台后台 |
| 运行入口是什么？ | `main_workstation.py` / `main.py` 中的工作站能力 | `main_web.py` 或独立 Web 服务 |
| 是否要求本地 UI 挂载？ | 是 | 否 |

判定原则：

- 入口天然绑定本地工作台，默认归 `workstation`
- 入口天然面向服务器部署 API，默认归 `web`

### 3.2 路由层

看接口协议和交互方式。

| 特征 | 偏向 `workstation` | 偏向 `web` |
| --- | --- | --- |
| 协议形态 | WebSocket 长流程、阶段推流、人工交互确认 | HTTP 查询/命令、短事务、列表/详情 |
| 返回模式 | `stage_update`、流式 chunk、审批等待 | JSON 视图、分页、详情、状态变更 |
| 路由意图 | 驱动工作流 | 暴露平台资源 |

判定原则：

- 以“驱动一个长生命周期本地流程”为核心的路由，优先归 `workstation`
- 以“管理平台资源实体”为核心的路由，优先归 `web`

### 3.3 服务层

看业务能力本质属于哪一类应用服务。

| 问题 | 偏向 `workstation` | 偏向 `web` |
| --- | --- | --- |
| 服务在解决什么问题？ | 创作、分析、构建、部署、审批执行、本地流程控制 | Job 编排、执行记录、配额、审计、后台管理 |
| 服务依赖什么上下文？ | 项目目录、图片、日志、Mod 文件、本机工具链 | 用户、任务、执行、退款、审计事件、DB session |
| 是否必须依赖运行时本机环境？ | 经常需要 | 不应需要 |

判定原则：

- 与本机工程环境强绑定的服务，归 `workstation`
- 与平台业务对象强绑定的服务，归 `web`

### 3.4 Facade 层

看它是不是兼容层，而不是正式能力归属。

| 类型 | 说明 | 归属处理 |
| --- | --- | --- |
| compatibility facade | 旧工作站路由转调新平台服务 | 记为 `workstation` 路由下的兼容接入点 |
| service facade | 对平台服务做聚合包装，但只服务于 Web 端 | 归 `web` |
| adapter / translator | 协议转换、对象转换、兼容旧返回结构 | 优先归调用方一侧，不单独升级为“核心域” |

判定原则：

- `facade` 先回答“服务谁”，再回答“放哪边”
- 仅为了迁移兼容存在的 facade，不应被误认成正式业务边界
- facade 可以在工作站路由下，但它所桥接的正式能力可能已经属于 `web`

### 3.5 数据层

看数据是否属于平台持久化语义。

| 问题 | 偏向 `workstation` | 偏向 `web` |
| --- | --- | --- |
| 数据是否必须持久化落库？ | 通常不需要，或只需本地配置/临时状态 | 需要，且属于平台长期记录 |
| 数据对象属于什么？ | 本地配置、临时会话、工作流上下文、项目文件 | Job、Execution、Quota、Refund、Audit |
| 能否脱离数据库独立运行？ | 应该可以 | 通常不可以 |

判定原则：

- 凡是平台账务、审计、执行记录、任务状态等数据，归 `web`
- 凡是本地工作流的临时上下文，不应为了“以后可能有用”直接落平台数据库

### 3.6 部署层

看它最终跟哪种包一起发布。

| 部署目标 | 适用归属 |
| --- | --- |
| `workstation` | 工作站正式能力 |
| `web` | Web 正式能力 |
| `frontend` | 仅前端静态资源，不承载后端归属 |
| `full` | 聚合验证态，不作为归属依据 |

判定原则：

- 一个能力如果只有放进 `full` 才“解释得通”，通常说明它的边界还没定义清楚

## 4. 决策流程

后续新增能力时，按以下顺序判断：

### 第一步：看运行环境

如果能力必须依赖以下任一条件，默认先归 `workstation`：

- 本地项目目录
- 本机日志文件
- 本机 Godot / STS2 / Mods 环境
- 本地图片生成或构建工具链

如果能力不依赖上述环境，继续下一步。

### 第二步：看业务对象

如果核心对象是以下任一类，默认先归 `web`：

- Job
- Execution
- Quota
- Refund
- Audit
- 平台用户任务状态

如果不是，继续下一步。

### 第三步：看交互协议

如果是以下形态，优先归 `workstation`：

- WebSocket 长流程
- 多阶段流式推送
- 需要人工选择图片、人工审批、人工继续操作

如果是以下形态，优先归 `web`：

- HTTP 短命令
- HTTP 查询/列表/详情
- 平台后台管理 API

### 第四步：看是否需要持久化落库

如果需要新增数据库表或长期持久化平台状态，默认归 `web`。

如果只是本地会话上下文、工作流临时状态或本地配置，不要直接落平台库，默认归 `workstation` 或 `shared`。

### 第五步：看是否只是兼容桥接

如果新增内容只是：

- 在旧工作站路由里转调新平台服务
- 兼容旧接口返回结构
- 为迁移阶段保留调用入口

则标记为 `facade / compatibility`，不要把它误记为新的正式业务域。

### 第六步：无法单边解释时，标记为 `hybrid`

满足以下任一情况时，不要直接拍板，应标记为 `hybrid` 并单独评审：

- 本地工作流触发，但过程或结果要形成平台正式任务记录
- 既要操作本地项目，又要维护平台任务状态机
- 接口本身在工作站端，但数据所有权明显属于平台
- 业务希望“本地可跑、服务器也可跑”，但没有清晰拆层

## 5. 判定表

### 5.1 接口归属判定表

| 判定问题 | 是 | 否 |
| --- | --- | --- |
| 这个接口是否直接操作本地项目、本机日志、本机构建环境？ | `workstation` | 继续判断 |
| 这个接口是否以 Job / Execution / Quota / Audit 为核心资源？ | `web` | 继续判断 |
| 这个接口是否以 WebSocket 长流程和阶段推流为主？ | `workstation` | 继续判断 |
| 这个接口是否是纯 HTTP 查询/命令资源接口？ | `web` | 继续判断 |
| 这个接口是否只是迁移兼容入口？ | `facade / compatibility` | 继续判断 |
| 这个接口是否同时带本地执行语义和平台持久化语义？ | `hybrid`，必须评审 | 按前面结果落定 |

### 5.2 服务归属判定表

| 判定问题 | 结果 |
| --- | --- |
| 服务核心依赖是文件系统、项目目录、本机工具链 | `workstation service` |
| 服务核心依赖是 repository、db session、platform application service | `web service` |
| 服务只做协议聚合或旧新系统对接 | `facade` |
| 服务没有部署归属，只提供纯算法/纯模型/纯 DTO | `shared` |
| 服务既驱动本地执行又拥有平台状态所有权 | `hybrid`，先拆分再实现 |

### 5.3 数据落库判定表

| 数据类型 | 建议归属 |
| --- | --- |
| 本地配置、路径探测结果、本地临时工作流状态 | 不进平台库；归 `workstation` 或本地存储 |
| 平台任务状态、执行记录、配额、退款、审计日志 | `web`，可进平台库 |
| 既是本地执行上下文，又被拿来做平台状态源数据 | `hybrid`，要先定义主数据所有权 |

## 7. 设计约束

后续新增能力时，默认遵守以下约束：

1. 不要因为“平台以后也可能要用”就提前把工作站能力做成 `web`
2. 不要因为“先方便接前端”就把平台资源接口塞进工作站端
3. 不要把 facade 当成正式业务边界
4. 不要让 `shared` 承担运行时副作用
5. 不要让 `full` 成为逃避边界设计的借口

## 8. 需要单独评审的高风险场景

以下场景后续默认进入专项评审，不直接按模板快速落地：

1. 审批系统平台化
2. Build/Deploy 平台化
3. 工作流执行平台落库
4. 本地工作流与平台任务双写
5. 既要离线本地可用，又要在线平台统一追踪的能力

这些场景往往不是“接口放哪边”的问题，而是“主数据所有权和执行所有权怎么拆”的问题。

如果一轮讨论后，`推荐归属` 仍然写不稳，就不要进入实现，先补边界设计。

## 10. 当前建议

基于当前仓库状态，后续新增内容默认按下面的口径执行：

- 本地创作、分析、构建、部署、交互式工作流，默认归 `workstation`
- 平台资源管理、任务状态、执行记录、审计与配额，默认归 `web`
- 纯模型、DTO、无副作用工具能力，才允许归 `shared`
- 同时涉及本地执行与平台数据所有权的能力，默认标 `hybrid` 并单独评审

如果未来平台化改造继续推进，优先策略应是：

- 正式能力迁到 `web`
- 旧工作站入口保留为兼容 facade
- 最终逐步压缩 `hybrid` 和兼容桥的数量
