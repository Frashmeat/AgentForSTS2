# PROJECT_SPEC.md

> 文档定位：本文是本仓库面向人类开发者、Codex、Claude 与其他 AI 助手的唯一项目协作规范入口。
>
> 事实依据：基于当前仓库 `AGENTS.md`、`.trellis/`、`docs/README.md`、现有实施计划与用户确认的文档目录迁移规则整理。
>
> 权威入口：协作规则以本文为准；项目事实与专题资料以 `docs/README.md` 及各目录 `README.md` 为准。
>
> 最后更新：2026-04-26

## 一、默认阅读顺序

进入本仓库处理任务时，默认按以下顺序阅读：

1. `PROJECT_SPEC.md`
2. `docs/README.md`
3. `.trellis/workflow.md`
4. 当前任务相关的专题目录 `README.md`
5. 当前任务相关的具体文档或代码

如果项目级规则、Trellis 规则与临时用户指令冲突，先指出冲突点，再按更具体、更贴近当前仓库的规则执行。

## 二、语言与沟通

- 默认使用简体中文。
- 结论先给，理由随后。
- 不把计划和实现混在一起汇报。
- 存在关键未知项时，先明确未知项，再给建议。
- git 提交、README、changelog、方案文档、任务说明，能用中文就优先使用中文；专有名词、命令名、API 名、文件名保持英文。

## 三、默认执行原则

- 默认先调查现状，再给计划，再进入实现。
- 中等及以上复杂任务，先说明当前理解、计划、影响范围、风险和验收标准。
- 在用户确认前，不直接进入复杂实现。
- 对已确认、无关键歧义、无重型验证、无高破坏动作的任务，可以连续推进到实现、文档同步、局部验证和收口说明。
- 当前项目尚未正式上线；除非用户明确要求，后续实现默认不为上线兼容、迁移兼容或旧口径并存额外保留包袱，应优先收口到当前主线实现。

## 四、Trellis 工作流

仓库根目录存在 `.trellis/`，默认将 Trellis 作为首选工作流之一。

- 新的中等及以上复杂任务，默认先执行 `$start` 对应的只读初始化步骤。
- 后端开发前按需执行 `$before-backend-dev`。
- 前端开发前按需执行 `$before-frontend-dev`。
- 多文件改动、功能开发、重构或正式修复，默认按 Trellis 的 Task Workflow 思路推进。
- 交付前如需规范检查，优先建议使用 `$check-backend`、`$check-frontend`、`$check-cross-layer`、`$finish-work`。
- Codex 不支持 Trellis 的 `/parallel` 多 Agent 流水线；即使使用 Trellis，也继续遵循多 Agent 默认关闭规则。

## 五、多 Agent 规则

多 Agent / 多子进程默认关闭。

如果任务适合拆分并行，必须先询问用户：

> 这个任务适合拆成多个子任务并行处理，是否开启多 Agent？默认不开启。

只有用户明确同意后，才能开启子 Agent。

## 六、验证边界

- 默认复用已有证据，不重复执行同一类重验证。
- 只需要局部验证时，不跑全量验证。
- `build`、全量测试、启动服务、部署、数据库重置等重型或高影响动作默认不自动执行，先说明建议命令和目的，再由用户决定是否运行。
- 静态阅读、定点搜索、语法检查、单文件或小范围测试属于轻量验证，可按任务需要执行。
- 若验证命令因环境缺依赖失败，必须说明失败原因，不得输出“测试通过”。

## 七、文档同步规则

每次执行完任务，必须同步更新相关文档。

若任务改变实现基线、接口、流程、目录结构、验证方式或约束条件，必须更新对应文档，而不是只改代码。

同步矩阵：

| 改动类型 | 必须回看或更新 |
| -------- | -------------- |
| 后端接口 | API 接口文档、前端调用说明、变更记录 |
| 前端页面行为 | 系统流程文档、页面操作说明、需求口径 |
| 数据库表结构 | 建库脚本、测试数据、数据库设计文档 |
| 配置或脚本 | README、部署文档、脚本说明 |
| 权限或认证 | 权限说明、接口文档、流程文档 |
| 文档目录调整 | `docs/README.md` 和对应目录 `README.md` |
| 破坏性变更 | 相关规范、迁移说明、旧口径删除说明 |

如果不确定该更新哪份文档，先读 `docs/README.md` 和相关目录 `README.md`。

## 八、文档目录与命名

本仓库保留当前文档主结构：

```text
docs/
  01-总览/
  02-现状/
  03-方案/
  04-决策/
  05-审查与验证/
  90-归档/
  superpowers/
```

新增正式文档优先使用：

```text
主题（用途）.md
```

用途词优先限制为：

- `规范`
- `索引`
- `说明`
- `计划`
- `方案`
- `设计`
- `审查`

新增计划、方案、审查等存在推进状态的文档时，使用 `未开始/`、`进行中/`、`已经完成/` 子文件夹表达状态，不再使用 `未开始-`、`进行中-`、`已经完成-` 文件名前缀。

状态变化时，移动到对应状态子文件夹，并同步更新上级目录 `README.md` 或相关索引链接。

## 九、文首格式

当前有效文档顶部建议写清：

```md
# 文档标题

> 文档定位：说明本文负责什么，不负责什么。
>
> 事实依据：说明是否基于当前代码、当前系统、历史讨论或人工确认。
>
> 权威入口：说明相关上级入口或专题目录。
>
> 最后更新：YYYY-MM-DD
```

计划、方案、审查类文档还应补充当前状态、适用范围、不纳入范围、风险与约束、完成标准。

## 十、后端运行时 AI 提示词

仓库根目录不再保留 `AGENTS_CODEX.md`，后端也不再自动注入该文件。

后端运行时全局提示词统一通过 `llm.custom_prompt` 配置注入。`PROJECT_SPEC.md` 是项目协作规范入口，不作为后端每次生成任务的自动 prompt 输入。

## 十一、Git 与安全

- 不执行会改写历史的高风险 git 操作，例如 `git reset --hard`、强制覆盖、强推。
- 不回滚用户已有改动，除非用户明确要求。
- 发现工作区已有无关改动时，忽略它们；如果影响当前任务，先说明再处理。
- 不输出密钥、令牌、证书等敏感信息。
